第 6 章 值类别：左值、右值与 std::move

这一章想让你带走什么

你在 Java / Python 里从没听过"左值""右值"——因为那些语言不需要它。但在 C++ 里，每一个表达式都有一个"值类别"，它决定了这个表达式能不能被赋值、能不能取地址、以及——最关键的——它的资源能不能被偷走。

这一章是下一章"移动语义"的地基。读完你会明白：std::move 其实什么都不搬，它只是一个类型转换；真正决定"拷贝还是移动"的，是表达式的值类别。

为什么需要这个概念

先看一个你早就会、但从没深究过的现象。这两行赋值，一行合法，一行编译不过：

C++
int x = 10;
x = 20;      // ✅ 合法：x 可以出现在 = 左边
10 = 20;     // ❌ 编译错误：10 不能出现在 = 左边

为什么 x 能放在等号左边，10 不能？因为 x 是一个有身份、有地址的对象——它在内存里占着一块地方，你可以往里写东西。而 10 只是一个临时的、马上就消失的值，它没有稳定的地址，写进去也没意义。

C++ 给这两类表达式起了名字，这就是"左值"和"右值"的由来：

最朴素的定义（先建立直觉，后面再精确）
  • 左值（lvalue）：有名字、有稳定地址的表达式。它是一个持久的对象。能放在 = 左边，能对它取地址 &。例：变量 x、arr[0]、*ptr。
  • 右值（rvalue）：临时的、即将消失的值，没有稳定地址。例：字面量 10、表达式结果 a + b、函数返回的临时对象。
名字的来历就是字面意思：left（能在赋值左边）和 right（只能在右边）。

一个判别小技巧：能取地址吗？

区分左值和右值，有个立竿见影的办法：试试能不能对它取地址 &。能取地址的是左值，不能的是右值。

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>

int f() { return 42; }

int main() {
    int x = 10;

    int* p = &x;        // ✅ x 是左值，能取地址
    std::cout << *p << '\n';

    // int* q = &10;    // ❌ 10 是右值，不能取地址
    // int* r = &f();   // ❌ f() 返回的临时值是右值，不能取地址
    // int* s = &(x+1); // ❌ x+1 的结果是右值

    std::cout << "&x = " << p << '\n';   // 左值有稳定地址
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

把注释掉的三行任意一行放开，点 🔗 在 Compiler Explorer 打开 编译——你会看到编译器明确报错 "lvalue required as unary '&' operand"。编译器用的正是"值类别"这套判断。

为什么 Java / Python 不需要这个？ 因为在那些语言里，一切对象都在堆上、由引用访问，你从不直接操心"这个表达式有没有地址、是不是临时的"。而 C++ 有栈上对象、有临时对象、能取地址、能就地构造——它必须在语言层面区分"持久的东西"和"临时的东西"，才能安全地决定资源归属。值类别就是这套区分的规则。

右值引用：一种"只绑定临时值"的引用

到这里都还是铺垫。真正的新东西是 C++11 引入的 右值引用，写作 T&&（两个 &）。

  • 普通引用 T&（一个 &，从第 8 章起细讲）只能绑定到左值；
  • 右值引用 T&& 专门绑定到右值——那些临时的、即将消失的值。
C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <string>

int main() {
    std::string s = "hello";

    std::string& lref = s;          // ✅ 左值引用绑左值
    // std::string& bad = "temp";   // ❌ 左值引用不能绑临时值

    std::string&& rref = "temp";    // ✅ 右值引用绑右值（临时 string）
    // std::string&& bad2 = s;      // ❌ 右值引用不能绑左值 s

    std::cout << lref << ", " << rref << '\n';
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

为什么要专门发明一种"只认临时值"的引用？因为临时值马上就要销毁了，它的资源没人要了——正好可以放心地偷走，而不必辛辛苦苦拷贝一份。这就是下一章移动语义的核心思想：对右值，转移；对左值，拷贝。 右值引用就是编译器用来识别"这是个可以被掏空的临时值"的抓手。

图 6-1　值类别决定资源归属：左值该拷贝，右值可移动。T& 绑左值，T&& 绑右值

std::move 的真相：它什么都不搬

现在来拆穿全 C++ 最容易误解的一个名字。std::move 听起来像"把数据搬走"，但它的实现是这样的（简化版）：

C++
// std::move 的本质：一个类型转换，仅此而已
template <typename T>
constexpr std::remove_reference_t<T>&& move(T&& x) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(x);
}
std::move(x) 不移动任何东西 它只是把 x 强制转换成右值引用，相当于对编译器说："我不再需要 x 的当前内容了，你可以把它当临时值对待、可以偷走它的资源。"
真正的"搬"发生在之后——当这个右值被传给移动构造函数或移动赋值时（下一章的主题）。std::move 本身只是贴了一张"可以掏空我"的标签。

用汇编印证"它什么都不做"。下面 std::move 作用在一个 int 上（int 没有资源可移动，移动等于拷贝），你会发现它没有产生任何额外指令：

C++ 🔬 看汇编 / 编译器处理
#include <utility>

int consume(int&& v) { return v; }

int test(int x) {
    return consume(std::move(x));   // move 只是把 x 转成右值，无运行时开销
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

点开 🔬 看汇编 / 编译器处理：std::move 这一层完全消失了，test 直接把 x 交给 consume。这再次印证——它是一个纯编译期的类型转换，零运行时成本。

危险：move 之后别再用它

既然 std::move(x) 是在宣告"x 的内容我不要了，随便掏空"，那么 move 之后 x 处于什么状态？答案是**"有效但未指定"（valid but unspecified）**——对象还活着（析构不会出问题），但里面装什么不保证。

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <string>
#include <utility>

int main() {
    std::string a = "hello world";
    std::string b = std::move(a);   // 把 a 的内容移动给 b

    std::cout << "b = " << b << '\n';           // "hello world"
    std::cout << "a = \"" << a << "\"\n";        // 通常为空，但标准不保证内容
    std::cout << "a.size() = " << a.size() << '\n';

    a = "reused";                    // ✅ 重新赋值总是安全的
    std::cout << "a = " << a << '\n';
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out
move 之后的对象，只能做两件事 一个被 std::move 掏空的对象，之后只保证能安全地：①销毁（离开作用域），②重新赋值。
读它的旧内容、或假设它还是原来的值，是 bug 的常见来源。规则很简单：move 完就当它没了，除非你重新给它赋值。

精确一点：五种值类别（了解即可）

前面用"左值 / 右值"两分法建立了直觉，够用来理解移动了。但如果你去查标准或看编译器报错，会遇到更细的术语。这里给你一张地图，不必背，遇到了能对上号即可：

C++
//                    表达式（expression）
//                    /                \
//              glvalue              rvalue
//             /       \            /      \
//        lvalue      xvalue（消亡值）    prvalue（纯右值）
//
//  lvalue  : 有名字的持久对象     —— 变量 x、*p、arr[i]
//  prvalue : 纯粹的临时值/字面量  —— 10、a+b、make()
//  xvalue  : "将亡"的对象         —— std::move(x) 的结果、返回 T&& 的调用
//  glvalue = lvalue + xvalue     —— 有身份（有地址）的表达式
//  rvalue  = prvalue + xvalue    —— 可被移动的表达式

两条主线索足以应付日常：

  • 有没有身份（能不能取地址） → 有则 glvalue（含 lvalue、xvalue），无则 prvalue；
  • 能不能被移动 → 能则 rvalue（含 prvalue、xvalue），不能则 lvalue。

std::move(x) 的作用，用这张图说就是：把一个 lvalue（x）变成一个 xvalue（"将亡值"），于是它就落入了 rvalue 阵营，可以匹配 T&&、触发移动。

不用记这五个词 真正要牢记的还是那句话：左值该拷贝，右值可移动；std::move 只是把左值贴上"可移动"的标签。五分类是精确化的补充，等你读第 16 章 lambda 捕获、或写完美转发模板（第 11 章）时会再见到 xvalue，到时候回头看这张图即可。

动手：观察值类别如何选择重载

下面这段程序用函数重载"探测"实参的值类别：传左值走一个版本，传右值走另一个。这正是移动语义在底层挑选"拷贝还是移动"的机制。运行前先预测每行走哪个分支：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <utility>

void inspect(const int& x) {   // 绑定左值
    std::cout << "左值版本: " << x << '\n';
}
void inspect(int&& x) {        // 绑定右值
    std::cout << "右值版本: " << x << '\n';
}

int main() {
    int a = 1;

    inspect(a);              // a 是左值 → ?
    inspect(42);             // 42 是右值 → ?
    inspect(a + 1);          // a+1 的结果是右值 → ?
    inspect(std::move(a));   // move(a) 是右值(xvalue) → ?
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

预期输出：

左值版本: 1
右值版本: 42
右值版本: 2
右值版本: 1

看懂这个重载决议，你就掌握了移动语义的钥匙——std::vector、std::string 内部正是用同一套 T& / T&& 重载，来决定"拷贝底层数组"还是"直接接管指针"。下一章，我们就把这套机制变成能省下真实开销的移动构造与移动赋值。

这一章的要点

  • C++ 中每个表达式都有值类别。朴素地分两类：左值（有名字、有地址、持久）和右值（临时、无地址、将消失）。
  • 判别技巧：能取地址 & 的是左值，不能的是右值。
  • 右值引用 T&& 专门绑定右值，是编译器识别"可被掏空的临时值"的抓手。
  • std::move 什么都不搬：它是一个零成本的类型转换，只是把左值贴上"可移动"标签；真正的移动发生在之后的移动构造/赋值里。
  • move 之后的对象处于**"有效但未指定"**状态，只应销毁或重新赋值。
  • 精确的五分类（lvalue / prvalue / xvalue / glvalue / rvalue）了解即可；核心口诀是左值该拷贝，右值可移动。
