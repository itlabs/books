第 4 章 栈、堆与对象生命周期

这一章想让你带走什么

在 Java / Python 里，"对象在哪、什么时候消失"是你几乎从不操心的问题——GC 全包了。在 C++ 里，这恰恰是你必须建立的直觉：一个对象是在栈上还是堆上？它什么时候诞生、什么时候销毁？

读完这一章，你会对"对象的生命周期"有一幅清晰的画面。这幅画面是下一章 RAII 的前提——只有先看清"对象何时销毁"，才能理解 C++ 如何靠它自动管理资源。

你其实一直活在"全在堆上"的世界

先看清你的起点。在 Java 里，除了 int、double 这些基本类型，所有对象都在堆上：

java
Point p = new Point(1, 2);   // 对象在堆上，p 是栈上的一个引用

Python 更彻底——一切皆对象，全在堆上，连整数都是。你写 x = 5，x 只是一个指向堆中整数对象的名字。

所以你从未真正区分过"栈对象"和"堆对象"，因为在你的世界里对象只住一个地方。C++ 把这两个地方都交给你，而且默认用栈。

两种存储：栈与堆

图 4-1　栈：随函数调用自动伸缩、后进先出、极快；堆：按需申请、位置零散、需要管理归还

栈是随函数调用自动管理的一块内存。每调用一个函数，就在栈上"压入"一帧，存放这个函数的局部变量；函数返回时，这一帧整个"弹出"，里面的局部变量随之销毁。你不需要做任何事，编译器全安排好了。栈分配快到几乎免费——就是移动一个指针。

堆是一大块可以按需申请的内存。你（或标准库容器）在需要时向它要一块，用完必须归还。堆灵活（大小可动态决定、生命周期可超出当前函数），但分配更慢，且谁申请、谁负责归还。

默认在栈上：这是 C++ 的常态

在 C++ 里，一个普通的局部变量定义，对象就直接建在栈上——没有 new，没有引用间接层：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>

struct Point { int x, y; };

int main() {
    Point p{1, 2};          // p 就是栈上的一个 Point 对象，不是引用
    std::cout << "p.x = " << p.x << '\n';
    std::cout << "sizeof(p) = " << sizeof(p) << " 字节\n";
    std::cout << "&p = " << &p << "  (栈上的地址)\n";
}                            // main 结束，p 自动销毁
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

对比一下你熟悉的写法差异——同样是"创建一个对象",默认落点完全不同：

Java：对象在堆
java
Point p = new Point(1, 2);
// 必须 new
// p 是引用，对象在堆上
// GC 之后某刻回收
C++：默认在栈
C++
Point p{1, 2};
// 不需要 new
// p 就是对象本身
// 离开作用域立即销毁

生命周期：作用域说了算

栈对象的生命周期规则简单得惊人:从定义处诞生，到所属的 {} 作用域结束时销毁，顺序与诞生相反。这是 C++ 里最可靠的一条规律，请把它刻进直觉。

下面这段程序用一个会"喊出生死"的类，把生命周期直接打印出来：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>

struct Noisy {
    int id;
    Noisy(int i) : id(i) { std::cout << "  构造 #" << id << '\n'; }
    ~Noisy()            { std::cout << "  析构 #" << id << '\n'; }
};

int main() {
    std::cout << "进入 main\n";
    Noisy a(1);
    {
        std::cout << "进入内层作用域\n";
        Noisy b(2);
        Noisy c(3);
        std::cout << "离开内层作用域\n";
    }                       // b、c 在此销毁（逆序：先 c 后 b）
    std::cout << "回到 main\n";
}                           // a 在此销毁
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

运行输出：

进入 main
  构造 #1
进入内层作用域
  构造 #2
  构造 #3
离开内层作用域
  析构 #3
  析构 #2
回到 main
  析构 #1

两个关键点，请盯住：

  1. 销毁是自动的——没有一句 delete，b、c 在内层 } 处就销毁了，a 在 main 的 } 处销毁。
  2. 销毁是逆序的——先构造的后销毁（后进先出），因为它们是栈上的一叠。
这就是 C++ 一切自动资源管理的物理基础 "对象离开作用域时，它的析构函数确定地、立即地被调用"——这个保证是 Java / Python 给不了的（GC 只保证"未来某时"回收，且不保证及时）。下一章的 RAII 正是把这个铁律利用到极致：把资源的释放写进析构函数，资源就会随对象自动、准时地释放。

什么时候才需要堆？

既然栈这么好，为什么还要堆？三种情况栈搞不定：

  1. 对象要活得比当前函数久——函数都返回了，它还得存在。
  2. 大小在运行时才知道——比如读入 n 个元素，n 是运行时才确定的。
  3. 对象太大——栈空间有限（通常几 MB），放不下巨型数据。

这时才需要堆。但请记住第 1 章的告诫：现代 C++ 几乎不直接写 new / delete。堆分配被封装进了两样东西：

  • 标准库容器（std::vector、std::string 等）：它们在内部用堆存元素，但对你表现为栈上的值对象；
  • 智能指针（std::unique_ptr、std::shared_ptr，第 8 章细讲）：它们持有堆对象，并在自己销毁时自动释放。

看栈上的 vector 如何悄悄使用堆：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <vector>

int main() {
    std::vector<int> v;          // v 本身是栈对象
    v.reserve(3);
    for (int i = 0; i < 3; ++i) v.push_back(i * 10);

    std::cout << "&v      = " << &v      << "  (v 这个对象在栈上)\n";
    std::cout << "v.data()= " << v.data() << "  (元素数组在堆上)\n";
    for (int x : v) std::cout << x << ' ';
    std::cout << '\n';
}                                 // v 析构：它自动 free 掉堆上的元素数组
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

打印出来的两个地址通常相差很远——&v 在栈区，v.data() 在堆区。而你从头到尾没写一句内存管理代码：vector 的析构函数替你把堆内存还了。这就是"栈对象管理堆资源"的 RAII 雏形。

图 4-2　vector 是"栈上的小管家"：自己住栈上，管着堆上的元素数组，析构时替你归还

悬垂：把栈对象的地址带出作用域

理解了"栈对象离开作用域即销毁"，就能看懂 C++ 里一类经典 bug——返回局部对象的地址或引用。这在 Java / Python 里根本不可能发生（那里的对象只要还有引用就不会被回收），但 C++ 会让你实实在在踩到：

C++
int& dangling() {
    int local = 42;
    return local;      // ❌ 返回局部变量的引用
}                      // local 在这里就销毁了

int main() {
    int& r = dangling();
    std::cout << r << '\n';   // 未定义行为：r 指向已销毁的对象
}

local 在 dangling 返回的瞬间就销毁了，r 绑定到一块已经不属于任何对象的内存——这叫悬垂引用（dangling reference）。

C++ 不会拦着你，但编译器会警告 这段代码是"未定义行为"：可能碰巧打印出 42，也可能崩溃，换个编译器/优化级别结果就变。好消息是现代编译器能识别这种模式——把上面的代码放进 Compiler Explorer，加上 -Wall，g++ / clang 会给出 warning: reference to local variable 'local' returned。永远不要忽略这类警告。正确做法是返回值（靠第 3 章讲的值语义 + RVO，没有性能损失），而不是返回局部对象的引用或地址。

动手：把生命周期看个通透

下面这段程序把本章所有要点串在一起：栈对象、作用域、逆序销毁、以及容器如何在析构时清理堆。运行前，先在脑子里预测输出顺序：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <vector>
#include <string>

struct Tracked {
    std::string name;
    Tracked(std::string n) : name(std::move(n)) {
        std::cout << "构造 " << name << '\n';
    }
    ~Tracked() { std::cout << "析构 " << name << '\n'; }
};

int main() {
    Tracked outer("外层");
    {
        Tracked inner("内层");
        std::vector<Tracked> box;
        box.reserve(2);            // 预留容量，避免扩容时的额外搬移
        box.emplace_back("盒中A");
        box.emplace_back("盒中B");
        std::cout << "--- 内层作用域即将结束 ---\n";
    }   // 先析构 box（连同它装的元素），再析构 inner
    std::cout << "--- 回到外层 ---\n";
}       // 最后析构 outer
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

运行输出：

构造 外层
构造 内层
构造 盒中A
构造 盒中B
--- 内层作用域即将结束 ---
析构 盒中A
析构 盒中B
析构 内层
--- 回到外层 ---
析构 外层

看两层规律：栈上的具名对象（outer、inner、box）严格按"后构造先析构"逆序销毁——所以内层的 box 先于 inner 销毁，outer 最后销毁。而 box 销毁时，会把它装着的元素一并析构（容器元素按下标顺序销毁，所以是先 A 后 B）。全过程自动，无需一句 delete。

那行 box.reserve(2) 是干嘛的？ 如果去掉它，你会在输出里看到多余的构造/析构：vector 装第二个元素时容量不够，会另开一块更大的堆内存，把已有元素搬过去、销毁旧的。这正是第 3 章"无意的拷贝"和第 7 章移动语义要解决的场景。reserve 提前备好容量，就没有这些搬移了——这里用它是为了让输出干净，聚焦生命周期本身。

搞清楚了"对象何时诞生、何时销毁"，我们就可以进入下一章：C++ 如何把这个铁律变成一套强大的资源管理武器——RAII。

这一章的要点

  • Java / Python 的对象几乎都在堆上；C++ 的对象默认在栈上，且变量就是对象本身（不是引用）。
  • 栈：随函数调用自动伸缩、后进先出、极快、自动销毁；堆：按需申请、灵活、需要管理归还。
  • 栈对象的生命周期由作用域决定：定义处诞生，} 处销毁，顺序与构造相反。
  • "离开作用域即确定地、立即地调用析构函数"是 C++ 的铁律，也是一切自动资源管理的基础。
  • 需要堆时（对象要活得更久 / 大小运行时定 / 太大），优先用容器和智能指针，而非裸 new/delete。
  • 返回局部对象的引用或地址会产生悬垂引用（未定义行为）；用返回值代替，靠值语义 + RVO 没有额外开销。
