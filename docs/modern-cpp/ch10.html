第 10 章 类、构造与特殊成员函数

这一章想让你带走什么

你早就会设计类——字段、方法、构造函数，这些概念 Java / Python 都有。这一章聚焦 C++ 类里你还不熟的那部分：初始化列表、以及决定对象如何被拷贝、移动、销毁的五个特殊成员函数。

这一章把第 5、7、8 章的线索收拢成一条清晰的类设计准则：Rule of Zero / Three / Five。掌握它，你就知道什么时候该手写这些函数、什么时候一个都不用写（而且后者才是常态）。

构造函数与成员初始化列表

先看一个和 Java 不同的细节。C++ 构造函数初始化成员，应该用成员初始化列表（构造函数参数列表后、函数体前那段 : 成员(值)），而不是在函数体里赋值：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <string>

class Point {
    int x_;
    int y_;
public:
    // 冒号后面就是成员初始化列表
    Point(int x, int y) : x_(x), y_(y) {
        std::cout << "构造 (" << x_ << ", " << y_ << ")\n";
    }
    int x() const { return x_; }
    int y() const { return y_; }
};

int main() {
    Point p(3, 4);
    std::cout << p.x() << ", " << p.y() << '\n';
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out
为什么用初始化列表，而不是在函数体里赋值？ 成员在进入构造函数函数体之前就已经被初始化了。如果你在函数体里写 x_ = x;，成员其实经历了"先默认初始化、再赋值"两步；而初始化列表是直接用给定值构造，一步到位。
对 int 无所谓，但对有构造成本的成员（如 std::string、其他类对象）就有真实差别。而且有些成员必须用初始化列表：const 成员、引用成员、以及没有默认构造函数的类型——它们没有"先默认再赋值"的机会。习惯用初始化列表，总是对的。
成员初始化顺序的坑 成员的初始化顺序由它们在类中声明的顺序决定，不是由初始化列表里的书写顺序决定。如果 y_ 的初始化依赖 x_，就要保证 x_ 声明在前。把初始化列表的顺序写得和声明顺序一致，能避免这类隐蔽 bug（开 -Wall 时编译器也会警告顺序不一致）。

五个特殊成员函数

有些成员函数即使你不写，编译器也会在需要时自动生成。它们统称"特殊成员函数",一共五个,决定了一个对象如何被创建、复制、转移和销毁：

图 10-1　五个特殊成员函数：一个析构 + 两个拷贝（构造/赋值）+ 两个移动（构造/赋值）
  • 析构函数 ~T()：对象销毁时调用（第 4、5 章）。
  • 拷贝构造 T(const T&) 和拷贝赋值 T& operator=(const T&)：复制一个对象（第 7 章）。
  • 移动构造 T(T&&) 和移动赋值 T& operator=(T&&)：转移资源（第 7 章）。

第 7 章你已经见过拷贝构造和移动构造的手写实现。这里的新知识是:它们作为一个整体互相关联,而这份关联催生了一条重要准则。

Rule of Zero：最好一个都别写

先说结论，也是现代 C++ 最推荐的做法：

Rule of Zero（零之法则） 如果你的类的每个成员都是"行为良好"的类型（标准库容器、智能指针、以及同样遵守此法则的类），那就一个特殊成员函数都不要写。编译器自动生成的五个版本会正确地逐成员拷贝、移动、析构——因为每个成员都知道如何管理自己。
你的类因此变得简短、正确、且自动获得高效的移动。这应该是你设计类的默认状态。

看一个遵守 Rule of Zero 的完整类——它管理着字符串和一个动态数组，却没有一句拷贝/移动/析构代码：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <string>
#include <vector>

// 成员都是"行为良好"的类型 → 无需写任何特殊成员函数
class Playlist {
    std::string name_;
    std::vector<std::string> songs_;   // 自己管理堆内存
public:
    Playlist(std::string name) : name_(std::move(name)) {}
    void add(std::string song) { songs_.push_back(std::move(song)); }
    const std::string& name() const { return name_; }
    std::size_t size() const { return songs_.size(); }
};

int main() {
    Playlist a("最爱");
    a.add("歌1");
    a.add("歌2");

    Playlist b = a;              // 自动拷贝（深拷贝，vector/string 负责）
    Playlist c = std::move(a);   // 自动移动（廉价，vector/string 负责）

    std::cout << b.name() << ": " << b.size() << " 首\n";   // 最爱: 2 首
    std::cout << c.name() << ": " << c.size() << " 首\n";   // 最爱: 2 首
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

拷贝、移动、析构全部由 std::string 和 std::vector 成员自己搞定，Playlist 自动继承了正确且高效的行为。这就是第 7 章"把裸资源交给 RAII 类型持有"建议的落地。

Rule of Five：要写就五个一起写

那什么时候必须手写?只有当你的类直接管理裸资源（裸指针、文件句柄等）时——但即便那样，也有一条铁律：

Rule of Five（五之法则） 如果你手写了五个特殊成员函数中的任何一个（通常是因为你有裸资源，需要自定义析构），那么另外四个你都要认真考虑、通常也要一起提供。
原因：需要自定义析构，几乎总意味着"逐成员浅拷贝"是错的（会双重释放，见第 7 章）。这时如果你只写了析构、却放任编译器生成默认拷贝，就等于埋了一颗雷。要么五个都管好，要么（更好）用 Rule of Zero 让你根本不需要写。

它的历史前身是 Rule of Three（C++11 之前只有拷贝构造、拷贝赋值、析构三个）；C++11 加入移动语义后，扩展为五个。

= default 与 = delete：精确控制生成

C++11 给了你两个关键字，用来精确指挥编译器如何处理这些函数：

  • = default：明确要求编译器"给我生成默认版本"——比隐式生成更清晰地表达意图。
  • = delete：明确"禁止"某个函数,谁调用它就编译报错。
C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <memory>

// 一个"独占、不可拷贝、可移动"的句柄类型
class UniqueHandle {
    std::unique_ptr<int> data_;
public:
    UniqueHandle(int v) : data_(std::make_unique<int>(v)) {}

    UniqueHandle(const UniqueHandle&) = delete;             // 禁止拷贝
    UniqueHandle& operator=(const UniqueHandle&) = delete;  // 禁止拷贝赋值
    UniqueHandle(UniqueHandle&&) = default;                 // 允许移动（默认实现）
    UniqueHandle& operator=(UniqueHandle&&) = default;      // 允许移动赋值

    int value() const { return *data_; }
};

int main() {
    UniqueHandle a(42);
    // UniqueHandle b = a;            // ❌ 编译错误：拷贝被 delete
    UniqueHandle b = std::move(a);    // ✅ 移动被允许
    std::cout << "b.value = " << b.value() << '\n';   // 42
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

这正是 std::unique_ptr 自己的设计方式——拷贝被 delete、移动被 default,从而在类型层面强制了"独占所有权"。你也可以用同样的手法,让自己的类型表达"这个东西不可复制"。

一个实用直觉：拷贝生成的自动抑制 现代 C++ 有个规则：一旦你声明了析构函数或任何移动操作，编译器就不再自动生成移动操作（拷贝也可能被抑制）。这背后的哲学正是 Rule of Five——编译器在提醒你"既然你开始手动管理了，就请把该管的都管上"。这也是又一个"优先 Rule of Zero"的理由：让编译器全权处理，就不用记这些微妙规则。

explicit：挡住意料之外的隐式转换

最后一个类设计的常用工具。一个能用单个参数调用的构造函数,默认会成为隐式转换的通道,有时会带来意外:

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>

struct Meters {
    double value;
    explicit Meters(double v) : value(v) {}   // explicit：禁止隐式转换
};

void printDistance(Meters m) {
    std::cout << m.value << " 米\n";
}

int main() {
    printDistance(Meters(100));   // ✅ 显式构造，清晰
    // printDistance(100);        // ❌ 若无 explicit，这会被悄悄当成 100 米
    Meters d(42);
    std::cout << d.value << '\n';
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

如果没有 explicit，printDistance(100) 会编译通过——编译器悄悄把 100 转成 Meters(100)。这种隐式转换有时方便，但更多时候是 bug 温床。

准则：单参数构造函数默认加 explicit 除非你确实想要隐式转换（少数情况，如自定义字符串类型想从 const char* 隐式构造），否则给单参数构造函数都加上 explicit。这能挡住一整类"参数被悄悄转换成了别的类型"的隐蔽错误。

动手：观察 Rule of Zero 类的完整生命周期

下面这个类完全遵守 Rule of Zero，我们通过一个会"报告生死"的成员，观察拷贝、移动、析构如何被自动、正确地处理。运行前先预测输出：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <string>
#include <vector>
#include <utility>

// 会报告自己被拷贝/移动/析构的探针
struct Probe {
    std::string tag;
    Probe(std::string t) : tag(std::move(t)) { std::cout << "  构造 " << tag << '\n'; }
    Probe(const Probe& o) : tag(o.tag)       { std::cout << "  拷贝 " << tag << '\n'; }
    Probe(Probe&& o) noexcept : tag(std::move(o.tag)) { std::cout << "  移动 " << tag << '\n'; }
    ~Probe() { std::cout << "  析构 " << (tag.empty() ? "(已移空)" : tag) << '\n'; }
};

// 遵守 Rule of Zero：不写任何特殊成员函数
class Container {
    Probe probe_;
public:
    Container(std::string t) : probe_(std::move(t)) {}
    // 拷贝/移动/析构全部由编译器按成员自动生成
};

int main() {
    std::cout << "创建 c1:\n";
    Container c1("A");

    std::cout << "拷贝 c1 -> c2:\n";
    Container c2 = c1;                 // 自动拷贝 → 成员 Probe 被拷贝

    std::cout << "移动 c1 -> c3:\n";
    Container c3 = std::move(c1);      // 自动移动 → 成员 Probe 被移动

    std::cout << "作用域结束（逆序析构 c3, c2, c1）:\n";
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

预期输出：

创建 c1:
  构造 A
拷贝 c1 -> c2:
  拷贝 A
移动 c1 -> c3:
  移动 A
作用域结束（逆序析构 c3, c2, c1）:
  析构 A
  析构 A
  析构 (已移空)

Container 一行拷贝/移动/析构代码都没写，但它的成员 Probe 被正确地拷贝、移动、析构了——最后 c1 因为被移动过，其成员的 tag 已被移空，析构时打印 (已移空)（呼应第 6 章"move 后有效但未指定"）。这就是 Rule of Zero 的威力：正确性免费、且自动获得移动优化。

第二部分到此完整了：从值语义、生命周期、RAII、值类别、移动、所有权、引用与智能指针、const，一直到类的设计。你已经建立起 C++ 最核心的心智模型。接下来第三部分转向另一个维度——泛型与编译期，从模板开始。

这一章的要点

  • 用成员初始化列表（: 成员(值)）初始化成员，而非函数体内赋值；const/引用/无默认构造的成员必须这样初始化。成员按声明顺序初始化。
  • 五个特殊成员函数：析构、拷贝构造、拷贝赋值、移动构造、移动赋值——决定对象如何销毁、复制、转移。
  • Rule of Zero：成员都行为良好时，一个都别写，让编译器自动生成——类设计的默认状态。
  • Rule of Five：一旦手写了其中一个（通常因直接管理裸资源），另外四个都要一起管好——否则埋雷。
  • = default 显式要求生成默认版本，= delete 禁用某函数（如 unique_ptr 禁拷贝）。
  • 单参数构造函数默认加 explicit，挡住意料之外的隐式转换。
