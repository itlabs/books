第 3 章 值语义：变量就是对象本身

这一章想让你带走什么

第 1 章抛出了本书最重要的一句话：C++ 变量默认是对象本身，不是引用。这一章把它彻底讲透——你会真正理解 b = a 为什么是拷贝、函数传参默认发生了什么、以及为什么这套"值语义"是 C++ 一切内存管理的起点。

这是全书的认知拐点。这一章立住了，后面的栈/堆、RAII、移动、所有权都会顺理成章。

两种世界观：引用语义 vs 值语义

你在 Java / Python 里习惯的，叫引用语义（reference semantics）：变量是一个指向对象的"把手"，对象本身住在别处（堆上）。

图 3-1　引用语义（Java / Python）：a 和 b 是两个把手，指向堆上同一个对象。改一个，另一个"也变了"

C++ 默认是值语义（value semantics）：变量直接就是那块存着数据的内存。两个变量是两块独立的内存。

图 3-2　值语义（C++）：a 和 b 各自是一个完整的对象，互不相干

同一段代码，在两种世界观下结果完全不同：

Java：引用语义
java
Point a = new Point(1, 2);
Point b = a;
b.x = 99;
// a.x 也变成 99
// 因为 a、b 是同一个对象
C++：值语义
C++
Point a{1, 2};
Point b = a;
b.x = 99;
// a.x 仍是 1
// 因为 b 是 a 的一份独立拷贝

自己跑一遍确认：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
struct Point { int x, y; };

int main() {
    Point a{1, 2};
    Point b = a;     // 拷贝：b 是新对象
    b.x = 99;        // 只改 b

    std::cout << "a.x = " << a.x << '\n';   // 1
    std::cout << "b.x = " << b.x << '\n';   // 99
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

= 到底做了什么

在 C++ 里，一个变量定义式 Point b = a; 里的 =，做的是用 a 初始化 b——它会把 a 里每个成员的值拷贝一份到 b。对 Point 这种简单结构，就是把 8 个字节（两个 int）原样复制过去。

这引出一个关键区分，也是初学者最容易混淆的地方：

初始化 vs 赋值——两个 = 不是一回事
  • Point b = a;  这是初始化：b 刚诞生，用 a 的内容把它建起来（调用拷贝构造函数）。
  • b = a;  （b 已存在）这是赋值：把已经存在的 b 的内容替换成 a 的（调用拷贝赋值运算符）。
第 9 章会讲这两个函数怎么自定义，现在只需记住：两种情况都在拷贝，都不会让 b 变成 a 的别名。

函数传参：默认也是拷贝

值语义最容易"咬"到人的地方，是函数传参。在 Java / Python 里，你把一个对象传进函数，函数内部拿到的是同一个对象的引用，改它会影响外面。在 C++ 里，默认按值传递——函数拿到的是一份拷贝。

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
struct Point { int x, y; };

void moveRight(Point p) {   // 按值：p 是实参的一份拷贝
    p.x += 100;             // 改的是拷贝，外面看不到
}

int main() {
    Point a{1, 2};
    moveRight(a);
    std::cout << a.x << '\n';   // 仍是 1
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

想让函数真的改到外面的对象，你得显式要求用引用（第 7 章细讲），加一个 &：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
struct Point { int x, y; };

void moveRight(Point& p) {  // 按引用：p 就是外面那个对象
    p.x += 100;
}

int main() {
    Point a{1, 2};
    moveRight(a);
    std::cout << a.x << '\n';   // 变成 101
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out
方向反过来了 在 Java / Python 里，"共享"是默认，"拷贝"要你特意做（clone()、copy.deepcopy()）。在 C++ 里，"拷贝"是默认，"共享"才要你特意用 & 或指针去要求。这个方向的翻转，是你切换心智模型时最需要盯住的一点。

值语义不是"浅拷贝"——它拷贝整个对象

这里要特别澄清一个 Java / Python 程序员极易踩的坑。在那些语言里，"拷贝一个对象"往往是浅的：拷贝了引用，内部的数组、列表还是共享的。

C++ 的值语义是深的（对于设计良好的类型而言）：拷贝一个 std::vector，会把它内部的所有元素也拷贝一份到新的内存里。两个 vector 从此完全独立。

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <vector>

int main() {
    std::vector<int> a{1, 2, 3};
    std::vector<int> b = a;   // 拷贝整个数组内容，不是共享底层

    b[0] = 99;                // 只改 b

    std::cout << "a[0] = " << a[0] << '\n';   // 1
    std::cout << "b[0] = " << b[0] << '\n';   // 99
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out
图 3-3　拷贝一个 vector 会连底层元素一起复制，得到两个完全独立的数组

这正是值语义强大又危险的地方：它给你独立、无别名、可预测的对象，代价是拷贝可能很贵。拷贝一个装了一百万个元素的 vector，就是实打实地复制一百万个元素。

性能陷阱：无意的拷贝 因为拷贝是默认行为，一不小心就会拷贝一个大对象。最常见的是这两种：
  • 按值传参：void process(std::vector<int> data) —— 每次调用都拷贝整个 vector。想只读就用 const std::vector<int>&。
  • 循环里 for (auto x : container) —— 每轮拷贝一个元素。想只读就用 for (const auto& x : container)。
C++ 后来引入的移动语义（第 6 章）正是为了在"该转移而非拷贝"时避免这种开销。

编译器视角：拷贝真的发生了吗？

值语义听起来到处在拷贝，性能岂不很差？这里有个漂亮的地方：编译器会消除掉没有意义的拷贝。

看这个返回一个大对象的函数。按值语义的字面意思，make() 里的 p 要拷贝一份返回，再拷贝一份给 result——两次拷贝。但现代编译器会做返回值优化（RVO），直接在 result 的内存里构造对象，一次拷贝都不发生：

C++ 🔬 看汇编 / 编译器处理
struct Point { int x, y; };

Point make() {
    Point p{1, 2};
    return p;        // 看似要拷贝返回
}

int main() {
    Point result = make();   // 看似又拷贝一次
    return result.x + result.y;
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

点开 🔬 看汇编 / 编译器处理：你会发现根本没有拷贝的痕迹，编译器直接算出了 3。这就是第 1 章说的"零成本抽象"——值语义在源码层面给你清晰的语义，编译器在底层把不必要的开销抹掉。第 6 章和第 18 章会深入这套机制。

为什么 C++ 选择值语义作为默认？ 因为值语义带来局部推理的能力：看到一个 Point a，你就知道它是一个独立的对象，不会因为别处某个隐藏的别名而被偷偷改掉。没有别名，就没有一大类并发 bug 和"谁动了我的对象"的调试噩梦。C++ 的立场是：默认给你安全的独立性，需要共享时你再显式地要——而不是反过来。

动手：把三种语义并排看清

下面这段程序把"拷贝独立""引用共享""按值传参"三件事放在一起。点开运行前，先逐行预测每个输出：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>

struct Counter { int n = 0; };

void byValue(Counter c)  { c.n += 1; }   // 拷贝，不影响外面
void byRef(Counter& c)   { c.n += 1; }   // 引用，改到外面

int main() {
    Counter a;
    Counter b = a;      // 拷贝：b 独立
    b.n = 10;

    byValue(a);         // a 会变吗？
    std::cout << "after byValue: a.n = " << a.n << '\n';

    byRef(a);           // a 会变吗？
    std::cout << "after byRef:   a.n = " << a.n << '\n';

    std::cout << "b.n = " << b.n << '\n';
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

预期输出：

after byValue: a.n = 0
after byRef:   a.n = 1
b.n = 10

如果这三行你都预测对了，说明值语义的直觉已经建立起来了。

这一章的要点

  • C++ 默认值语义：变量就是对象本身，两个变量是两块独立内存。
  • = 是拷贝（初始化用拷贝构造，赋值用拷贝赋值运算符），不是让变量互为别名。
  • 函数传参默认按值（拷贝）；要改到外面的对象，需显式用引用 &。方向和 Java / Python 相反：拷贝是默认，共享要特意要求。
  • 值语义是深的：拷贝容器会连元素一起复制，得到完全独立的对象——强大但可能很贵。
  • 无意的拷贝是常见性能陷阱；移动语义（第 6 章）用来避免它。
  • 编译器会用 RVO 等手段消除不必要的拷贝，让值语义成为"零成本抽象"。
