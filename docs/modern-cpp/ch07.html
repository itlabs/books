第 7 章 拷贝、移动与所有权

这一章想让你带走什么

三条线索在这一章交汇：第 3 章的值语义（拷贝是默认）、第 5 章的 RAII（对象持有资源）、第 6 章的值类别（右值可移动）。把它们拼起来，你会得到 C++ 资源管理的完整图景。

核心问题只有一个：当一个持有资源的对象被拷贝或移动时，它的资源该怎么办？回答这个问题，就理解了拷贝构造、移动构造、以及"所有权"这个贯穿现代 C++ 的概念。

问题的由来：拷贝一个持有资源的对象

回忆第 5 章的 RAII 对象——它在构造时获取资源、析构时释放。现在假设它持有一块堆内存，我们拷贝它一份，会发生什么？

先看一个故意写错的版本，把问题暴露出来：

C++
class Buffer {
    int* data_;
    std::size_t size_;
public:
    Buffer(std::size_t n) : data_(new int[n]), size_(n) {}
    ~Buffer() { delete[] data_; }
    // 没有自定义拷贝：编译器生成的默认拷贝会逐成员复制
};

void bug() {
    Buffer a(10);
    Buffer b = a;   // 默认拷贝：把指针 data_ 原样复制！
}   // a 和 b 析构时，都对同一块内存 delete[] —— 双重释放，崩溃

问题出在：编译器默认生成的拷贝是逐成员复制，它把指针 data_ 的值（那个地址）复制给了 b。于是 a.data_ 和 b.data_ 指向同一块堆内存。当两者先后析构，同一块内存被 delete[] 两次——双重释放，未定义行为，通常直接崩溃。

图 7-1　浅拷贝的灾难：默认拷贝只复制指针，两个对象共享同一块内存，析构时双重释放

拷贝构造：做一份真正独立的副本

要符合第 3 章的值语义（拷贝应得到独立的对象），我们得自己定义拷贝行为——分配一块新内存，把内容深拷贝过去：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <algorithm>
#include <cstddef>

class Buffer {
    int* data_;
    std::size_t size_;
public:
    explicit Buffer(std::size_t n) : data_(new int[n]{}), size_(n) {
        std::cout << "  构造(" << n << ")\n";
    }
    ~Buffer() { delete[] data_; }

    // 拷贝构造：分配新内存，复制内容 —— 深拷贝
    Buffer(const Buffer& other) : data_(new int[other.size_]), size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
        std::cout << "  拷贝构造（复制了 " << size_ << " 个元素）\n";
    }

    void set(std::size_t i, int v) { data_[i] = v; }
    int  get(std::size_t i) const  { return data_[i]; }
    std::size_t size() const { return size_; }
};

int main() {
    Buffer a(3);
    a.set(0, 100);

    Buffer b = a;      // 调用拷贝构造：b 得到独立的内存
    b.set(0, 999);     // 改 b 不影响 a

    std::cout << "a[0] = " << a.get(0) << '\n';   // 100
    std::cout << "b[0] = " << b.get(0) << '\n';   // 999
}   // a、b 各自 delete[] 自己的内存，安全
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

现在 b = a 会分配新内存、逐元素复制，a 和 b 彻底独立——这正是第 3 章 std::vector 拷贝时发生的事。深拷贝安全，但有代价：复制了整整一份数据。如果 a 之后再也不用了，这份拷贝就纯属浪费。这正是移动语义登场的地方。

移动构造：偷走资源，而不是复制

回到第 6 章的洞察：右值是即将消失的临时值，它的资源没人要了，可以放心偷走。 移动构造函数就是来"偷"的——它不分配新内存、不复制内容，而是接管源对象的指针，再把源对象置空：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <utility>
#include <cstddef>

class Buffer {
    int* data_;
    std::size_t size_;
public:
    explicit Buffer(std::size_t n) : data_(new int[n]{}), size_(n) {}
    ~Buffer() { delete[] data_; }

    Buffer(const Buffer& other) : data_(new int[other.size_]), size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
        std::cout << "  拷贝构造（贵：复制 " << size_ << " 个元素）\n";
    }

    // 移动构造：接管 other 的资源，不分配、不复制
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {   // 偷走指针
        other.data_ = nullptr;                       // 源置空：它不再拥有这块内存
        other.size_ = 0;
        std::cout << "  移动构造（廉价：只挪了指针）\n";
    }

    std::size_t size() const { return size_; }
};

Buffer makeBuffer() {
    return Buffer(1'000'000);    // 返回一个临时对象（右值）
}

int main() {
    Buffer a(1'000'000);

    std::cout << "从左值拷贝：\n";
    Buffer b = a;                // a 是左值 → 拷贝构造（复制一百万个元素）

    std::cout << "从右值移动：\n";
    Buffer c = std::move(a);     // move(a) 是右值 → 移动构造（只挪指针）

    std::cout << "从临时对象移动：\n";
    Buffer d = makeBuffer();     // 临时对象是右值 → 移动（且常被 RVO 直接消除）
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

运行输出：

从左值拷贝：
  拷贝构造（贵：复制 1000000 个元素）
从右值移动：
  移动构造（廉价：只挪了指针）
从临时对象移动：

三处的差别，正是第 6 章值类别在起作用：

  • Buffer b = a;——a 是左值，编译器选拷贝构造，老老实实复制一百万个元素；
  • Buffer c = std::move(a);——std::move(a) 是右值，编译器选移动构造，只挪了几个指针；
  • Buffer d = makeBuffer();——临时对象是右值，本该移动，但连移动都没打印！因为编译器用 RVO（第 3 章）直接在 d 里构造，连移动都省了。
图 7-2　拷贝复制整块资源（两份），移动只转移指针的归属（还是一份，换了主人）

所有权：谁负责释放？

上面反复出现一个词——"接管""归属""主人"。这就是现代 C++ 最重要的概念之一：所有权（ownership）。

所有权 = 释放资源的责任 一块资源（内存、文件、锁……）在任一时刻应当有明确的、唯一的拥有者，由它负责在析构时释放。
· 拷贝创造了一个新的、独立的拥有者，各自管理各自的资源。
· 移动把所有权从一个对象转移给另一个：源对象交出资源（置空，不再负责释放），目标对象接手（今后由它释放）。
移动后源对象为什么必须置空？正是为了保证"唯一拥有者"——否则源和目标都以为自己拥有那块内存，又会双重释放。

这解释了第 6 章那条规则的由来：move 之后源对象处于"有效但未指定"状态——因为它的资源已经被交出去了，它成了一个空壳，只能安全析构（delete[] nullptr 是安全的）或重新赋值。

好消息：多数时候你不用手写这些

上面为了讲清原理，手写了 Buffer 的拷贝和移动。但现实中——

你几乎永远不必自己写这些函数 标准库的 std::vector、std::string、std::unique_ptr 等已经正确实现了拷贝和移动。只要你的类的成员都是这些"行为良好"的类型，编译器自动生成的拷贝/移动就是正确的——你什么都不用写。
真正需要手写拷贝/移动/析构的，只有当你的类直接管理裸资源（裸指针、文件句柄）时。而现代 C++ 的建议恰恰是别这么做：把裸资源交给一个专门的 RAII 类型（如 unique_ptr）去持有，你的类就退化成"成员都行为良好"的情形，回到零手写。这条经验叫 Rule of Zero（第 10 章展开）。

用行为良好的成员改写 Buffer——整个类不再需要任何手写的拷贝、移动、析构：

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <vector>
#include <utility>

class Buffer {
    std::vector<int> data_;    // 用 vector 持有资源，它自己会拷贝/移动/释放
public:
    explicit Buffer(std::size_t n) : data_(n) {}
    // 不写拷贝构造、移动构造、析构 —— 编译器按成员自动正确生成
    std::size_t size() const { return data_.size(); }
};

int main() {
    Buffer a(1'000'000);
    Buffer b = a;              // 自动深拷贝（vector 负责）
    Buffer c = std::move(a);   // 自动移动（vector 负责，廉价）
    std::cout << "b.size = " << b.size() << ", c.size = " << c.size() << '\n';
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

这段代码既安全（无双重释放、无泄漏）又高效（移动是廉价的），而你一行资源管理代码都没写。这才是现代 C++ 日常的样子：理解拷贝与移动的原理，是为了在读代码、调性能、以及偶尔必须管理裸资源时心里有数——而不是为了每天手写它们。

动手：数一数拷贝和移动各发生了几次

下面用一个会报告自己每次拷贝/移动的类型，观察把它放进 vector 时的行为。运行前先想想：push_back 一个具名变量 vs 一个临时对象，分别触发拷贝还是移动？

C++ 🔗 在 Compiler Explorer 打开
#include <iostream>
#include <vector>
#include <utility>

struct Widget {
    Widget()                        { std::cout << "  构造\n"; }
    Widget(const Widget&)           { std::cout << "  拷贝\n"; }
    Widget(Widget&&) noexcept       { std::cout << "  移动\n"; }
};

int main() {
    std::vector<Widget> v;
    v.reserve(4);                       // 预留，避免扩容干扰观察

    Widget w;
    std::cout << "push_back(w) —— w 是左值：\n";
    v.push_back(w);                     // 拷贝

    std::cout << "push_back(std::move(w)) —— 右值：\n";
    v.push_back(std::move(w));          // 移动

    std::cout << "push_back(Widget{}) —— 临时对象是右值：\n";
    v.push_back(Widget{});              // 构造 + 移动
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

预期输出：

  构造
push_back(w) —— w 是左值：
  拷贝
push_back(std::move(w)) —— 右值：
  移动
push_back(Widget{}) —— 临时对象是右值：
  构造
  移动

看懂这个计数，你就真正掌握了值语义、值类别、拷贝与移动这条主线。std::vector 内部正是靠这套 const T& / T&& 重载，在"复制元素"和"接管元素"之间自动做出最优选择——这也是为什么现代 C++ 能兼顾"值语义的清晰"和"移动带来的高效"。

到这里，本书的核心地基——对象、内存、生命周期、所有权——就打好了。接下来几章会用它们去理解引用与智能指针（第 8 章）、const（第 9 章）和类的设计（第 10 章）。

这一章的要点

  • 默认拷贝是逐成员浅复制；对持有裸资源（如裸指针）的类，这会导致双重释放。
  • 拷贝构造做深拷贝：分配新资源、复制内容，得到独立对象——安全但可能贵。
  • 移动构造转移所有权：接管源对象的资源指针，并把源置空——廉价，且只对右值触发。
  • 所有权是"释放资源的责任"；任一时刻资源应有唯一拥有者。拷贝创造新拥有者，移动转移拥有者。移动后源置空正是为了维持"唯一拥有者"。
  • 标准库类型已正确实现拷贝/移动；成员都行为良好时，编译器自动生成的版本就是对的——Rule of Zero，你无需手写。
  • 只有直接管理裸资源时才需手写这些函数，而现代 C++ 建议把裸资源交给 RAII 类型，从而回避这种需要。
