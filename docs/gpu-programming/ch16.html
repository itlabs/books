第 16 章 性能模型：occupancy、带宽与 Roofline

这一章想让你带走什么

前面你学会了写 kernel，也知道 GPU"多数时候卡在带宽"。这一章给你两把量尺，把"卡在哪儿"从直觉变成可计算的数字：一把是 occupancy（占用率）——你实际挂在 SM 上的 warp 占硬件上限的比例，它决定了 GPU 有没有足够的"备胎 warp"去掩盖延迟；另一把是 Roofline 模型——把 kernel 画在一张"性能 vs 算术强度"的图上，一眼看出它是被带宽还是被算力顶住了，以及离天花板还有多远。

读完你能用两个浏览器里能跑的小程序，算出任意 kernel 的 Roofline 天花板和 occupancy，并知道下一步该优化访存还是优化计算——这正是决定"值不值得动手"的判断力。

occupancy：你到底挂了多少个 warp

回忆第 1 章和第 3 章那句反复出现的话：GPU 靠并行掩盖延迟。当一个 warp 卡在等显存（几百个周期）时，warp 调度器立刻切到另一个就绪的 warp 接着算，访存的空档就被填满了。这个魔法能不能生效，取决于一件事——SM 上到底还有没有别的 warp 可切。

occupancy（占用率） 就是量这件事的指标，定义极简单：

occupancy = 一个 SM 上实际驻留的活跃 warp 数 ÷ 该 SM 支持的最大 warp 数

比如一个 SM 最多挂 64 个 warp，你实际只挂了 16 个，occupancy 就是 25%。warp 越少，调度器手里的"备胎"越少，一旦大家同时卡在访存上，算术单元就只能空转——延迟没被掩盖，吞吐掉下来。

那什么东西限制了你能挂多少 warp？答案是每个 SM 的资源是固定的，被同时驻留的所有 block 瓜分。三种资源最常成为瓶颈：

  • 每线程寄存器数：SM 的寄存器堆总量固定（如 65536 个）。你的 kernel 每线程用得越多，能同时驻留的线程（warp）就越少。这是最常见的 occupancy 杀手。
  • 每 block 的 shared memory：SM 的 shared memory 总量固定（如 64–228 KB）。每 block 申请得越多，能并存的 block 越少。
  • block 大小与硬上限：SM 还有"最多 N 个 block""最多 M 个 warp"的硬性天花板。block 太小（如 32 线程）会先撞到 block 数上限，挂不满 warp。
图 16-1　每线程寄存器用量减半，同一块寄存器堆就能多塞一倍的 block/warp——这就是"降寄存器换 occupancy"的直接来源

但 100% occupancy 不是目标

这里有个新手最容易走进的误区：把 occupancy 当成越高越好的分数去刷。不是的。 occupancy 的唯一作用是提供足够的备胎 warp 来掩盖延迟——一旦掩盖住了，再往上加没有任何收益。经验上，很多 kernel 在 50% 左右就已经把延迟藏得很好，再往上收益趋近于零。

更微妙的是，追高 occupancy 常常要付代价：为了少用寄存器，你可能被迫把本可以放在寄存器里的数据反复从显存里读——结果 occupancy 上去了，访存反而变多，净亏。真正会调优的人做的是权衡：让每个线程多用点寄存器（把复用数据留在片上）、接受略低的 occupancy，往往比硬刷满 occupancy 更快。这也是 NVIDIA 的 Volta 架构师那句名言的意思——"低占用率也能跑满带宽"。

occupancy 的正确用法 occupancy 是手段不是目的。它要够高到能掩盖延迟（通常几十个 warp 就够），但没必要顶满。当你的 kernel 慢，先看它是不是 occupancy 太低（比如寄存器爆了，只剩 12% ）导致延迟没被掩盖；如果 occupancy 已经过半还是慢，问题多半在别处（访存模式、算法），继续刷 occupancy 是白费力气。

Roofline：一张图看清 kernel 撞的是哪面墙

occupancy 告诉你"延迟藏没藏住"，但没告诉你这个 kernel 的理论上限在哪。Roofline 模型就是干这个的——它把整台机器的能力和你 kernel 的性质画在同一张图上。

图的两根轴（都取对数）：

  • 横轴：算术强度（arithmetic intensity），单位 FLOP/byte——第 3 章那个"每从显存搬 1 字节能做几次浮点运算"。这是 kernel 自身的性质。
  • 纵轴：可达到的性能，单位 FLOP/s。

"屋顶"由两条线拼成，任何 kernel 能达到的性能不可能超过它：

$$\text{可达性能} = \min(\text{峰值算力},\ \text{峰值带宽} \times \text{算术强度})$$

  • 右边一段是平屋顶：等于峰值算力（如 60 TFLOP/s），一条水平线——再高的算术强度也突破不了硅片的算力上限。
  • 左边一段是斜屋顶：等于峰值带宽 × 算术强度，在对数坐标里是一条斜率为 1 的直线——强度越低，带宽这堵墙就把你压得越低。

两条线相交处叫脊点（ridge point），它的横坐标正好是第 3 章的平衡点 = 峰值算力 ÷ 峰值带宽（我们那块 GPU 是 60 TFLOP/s ÷ 3 TB/s = 20 FLOP/byte）。脊点把世界切成两半：

  • 脊点左边（强度 < 20）：受斜屋顶限制，带宽受限（memory-bound）。想更快，就得优化访存（减少读写、提高强度）。
  • 脊点右边（强度 > 20）：受平屋顶限制，算力受限（compute-bound）。想更快，就得优化计算（用 Tensor Core、减少指令）。
图 16-2　Roofline：斜屋顶（带宽）与平屋顶（算力）在脊点相交。向量加法落在最左端被带宽压制，矩阵乘落在右侧能逼近峰值算力。kernel 的横坐标是它自己的性质，纵坐标上限由屋顶决定

它的用法非常实操：测出你 kernel 的算术强度，找到它在图上的横坐标，看它头顶的屋顶有多高，再对比它实测到的性能。 如果实测点贴着屋顶，说明这台机器对这个 kernel 已经压榨到极限了，别折腾了；如果离屋顶还很远，说明有优化空间——而往哪个方向优化，由它在脊点的左边还是右边决定。

CPU 的性能直觉
C++
// 优化 CPU 单线程/多线程：
// - 提高 ILP（指令级并行）：
//   让乱序引擎有活干
// - 吃好缓存：分块、预取、
//   减少 cache miss
// - 分支预测友好、SIMD 向量化
// 核心：让"少数强核"尽快跑完
GPU 的性能直觉
CUDA C++
// 优化 GPU kernel：
// - occupancy：挂够 warp 掩盖延迟
// - 带宽：合并访存、复用 shared
//   memory、算子融合减少读写
// - 提高算术强度，往脊点右边挪
// 核心：喂饱"海量弱核"的吞吐

动手（一）：算出每个 kernel 的 Roofline 天花板

下面这个程序能在浏览器里直接跑——点 ▶ 运行。它沿用第 3 章那块 GPU 的参数（60 TFLOP/s、3 TB/s），先算出脊点，再对一列 kernel 逐个算出它头顶的屋顶高度、能摸到峰值算力的百分之几，以及它到底带宽受限还是算力受限。

Python 
# Roofline 天花板计算器：给定峰值算力与带宽，判断每个 kernel 的上限
peak_flops = 60e12      # 峰值算力 60 TFLOP/s（沿用第 3 章）
peak_bw    = 3.0e12     # 显存带宽 3 TB/s（沿用第 3 章）

ridge = peak_flops / peak_bw   # 脊点 = 平衡点
print(f"脊点（ridge point）= {ridge:.0f} FLOP/byte")
print(f"  强度 < {ridge:.0f} → 带宽受限（在斜屋顶下）")
print(f"  强度 > {ridge:.0f} → 算力受限（在平屋顶下）\n")

# (名字, 算术强度 FLOP/byte)
kernels = [
    ("向量加法 c=a+b",   1/12),   # 每元素 1 次运算、搬 12 字节
    ("SAXPY y=a*x+y",    2/12),   # 2 次运算、搬 12 字节
    ("stencil 5 点",     10/8),   # 约 10 次运算、搬 8 字节
    ("矩阵乘 (大 N)",     50.0),   # tiling 后强度很高
]

for name, ai in kernels:
    ceiling = min(peak_flops, peak_bw * ai)   # 屋顶：两条线取低者
    bound = "带宽受限" if ai < ridge else "算力受限"
    pct = ceiling / peak_flops * 100          # 能摸到峰值算力的百分之几
    print(f"{name}：强度 {ai:.3f} FLOP/byte")
    print(f"  屋顶 = {ceiling/1e12:.3f} TFLOP/s（峰值的 {pct:.2f}%）→ {bound}")

运行结果：

脊点（ridge point）= 20 FLOP/byte
  强度 < 20 → 带宽受限（在斜屋顶下）
  强度 > 20 → 算力受限（在平屋顶下）

向量加法 c=a+b：强度 0.083 FLOP/byte
  屋顶 = 0.250 TFLOP/s（峰值的 0.42%）→ 带宽受限
SAXPY y=a*x+y：强度 0.167 FLOP/byte
  屋顶 = 0.500 TFLOP/s（峰值的 0.83%）→ 带宽受限
stencil 5 点：强度 1.250 FLOP/byte
  屋顶 = 3.750 TFLOP/s（峰值的 6.25%）→ 带宽受限
矩阵乘 (大 N)：强度 50.000 FLOP/byte
  屋顶 = 60.000 TFLOP/s（峰值的 100.00%）→ 算力受限

看清了吗？向量加法头顶的屋顶只有 0.25 TFLOP/s——连这块卡峰值算力的 0.42% 都摸不到，因为它被带宽死死压在斜屋顶的最左端。哪怕你把它优化到完美，它也永远碰不到 60 TFLOP/s。而矩阵乘落在脊点右边，屋顶就是满血的 60 TFLOP/s。请动手改：把 矩阵乘 的强度从 50 慢慢往下调，看它调到多少（提示：正好是脊点 20）才从"算力受限"翻转成"带宽受限"、屋顶开始往下掉——那一刻你就亲手穿过了脊点。

动手（二）：occupancy 计算器，看降寄存器如何拉满 warp

第二个程序同样能在浏览器里跑。它给一个 SM 的资源上限，输入你 kernel 的每线程寄存器、每 block 的 shared memory、每 block 线程数，算出到底哪种资源先耗尽、能驻留几个 block、最终 occupancy 是多少。我们跑两次：同一个 kernel，只把每线程寄存器从 64 降到 32，看 occupancy 怎么从 50% 跳到 100%。

Python 
# occupancy 计算器：给定每线程/每 block 的资源用量与 SM 上限，算能驻留多少 warp
WARP_SIZE = 32

# 一个 SM 的硬件上限（取某数据中心级 GPU 的大致公开量级）
SM_MAX_WARPS     = 64        # 每 SM 最多驻留的 warp 数
SM_MAX_BLOCKS    = 32        # 每 SM 最多驻留的 block 数
SM_REGISTERS     = 65536     # 每 SM 的寄存器总数
SM_SHARED_BYTES  = 65536     # 每 SM 的 shared memory 字节数（这里取 64 KB）

def occupancy(regs_per_thread, shared_per_block, threads_per_block):
    warps_per_block = (threads_per_block + WARP_SIZE - 1) // WARP_SIZE

    # 1) 寄存器限制：一个 block 用掉的寄存器 = 每线程寄存器 × block 线程数
    regs_per_block = regs_per_thread * threads_per_block
    lim_reg = SM_REGISTERS // regs_per_block if regs_per_block else SM_MAX_BLOCKS

    # 2) shared memory 限制
    lim_shmem = SM_SHARED_BYTES // shared_per_block if shared_per_block else SM_MAX_BLOCKS

    # 3) warp 数 / block 数硬上限
    lim_warps = SM_MAX_WARPS // warps_per_block

    # 真正能驻留的 block 数 = 各限制的最小值
    blocks = min(lim_reg, lim_shmem, lim_warps, SM_MAX_BLOCKS)
    active_warps = blocks * warps_per_block
    occ = active_warps / SM_MAX_WARPS

    print(f"每线程 {regs_per_thread} 寄存器、每 block {shared_per_block} B shared、"
          f"{threads_per_block} 线程/block（= {warps_per_block} warp）")
    print(f"  限制来源：寄存器允许 {lim_reg} block，shared 允许 {lim_shmem} block，"
          f"warp 上限允许 {lim_warps} block")
    print(f"  → 实际驻留 {blocks} 个 block = {active_warps} 个 warp，"
          f"occupancy = {occ*100:.0f}%\n")
    return occ

print("A：寄存器吃紧的 kernel（每线程 64 寄存器）")
occupancy(regs_per_thread=64, shared_per_block=0, threads_per_block=256)

print("B：把寄存器降到 32（比如少用中间变量 / 加 __launch_bounds__）")
occupancy(regs_per_thread=32, shared_per_block=0, threads_per_block=256)

运行结果：

A：寄存器吃紧的 kernel（每线程 64 寄存器）
每线程 64 寄存器、每 block 0 B shared、256 线程/block（= 8 warp）
  限制来源：寄存器允许 4 block，shared 允许 32 block，warp 上限允许 8 block
  → 实际驻留 4 个 block = 32 个 warp，occupancy = 50%

B：把寄存器降到 32（比如少用中间变量 / 加 __launch_bounds__）
每线程 32 寄存器、每 block 0 B shared、256 线程/block（= 8 warp）
  限制来源：寄存器允许 8 block，shared 允许 32 block，warp 上限允许 8 block
  → 实际驻留 8 个 block = 64 个 warp，occupancy = 100%

A 里最"抠门"的资源是寄存器：64 × 256 = 16384 个寄存器一个 block，65536 / 16384 = 4，只挂得下 4 个 block、32 个 warp。B 把每线程寄存器减半，寄存器允许的 block 翻倍到 8，占用率就顶满了。请动手改：给 B 加上 shared_per_block=16384（每 block 要 16 KB shared memory），再运行——你会看到限制来源从寄存器变成了 shared memory（65536/16384 = 4），occupancy 又掉回 50%。这就是"三种资源，谁先耗尽谁说了算"的木桶效应。

怎么知道 kernel 用了多少寄存器？

上面的计算器要你填"每线程寄存器数"——这个数不是你猜的，是 nvcc 编译时确定并能打印出来的。加上 -Xptxas -v（把详细信息传给 PTX 汇编器）就能看到：

终端
nvcc -Xptxas -v -o app app.cu
# 输出里会有类似：
#   ptxas info : Used 32 registers, 16384 bytes smem, ...

如果某个 kernel 寄存器用超了、压低了 occupancy，你可以用 __launch_bounds__ 给编译器一个承诺——"我这个 kernel 每 block 最多开这么多线程、我希望每个 SM 至少驻留这么多 block"，编译器会据此限制寄存器用量（宁可把多余的变量溢出到本地内存，也要保证 occupancy）：

CUDA C++
// __launch_bounds__(每 block 最大线程数, 期望每 SM 最少驻留 block 数)
// 告诉编译器：请把寄存器控制在能让 SM 挂下 4 个 256 线程 block 的水平
__global__ void __launch_bounds__(256, 4)
myKernel(const float* in, float* out, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) out[i] = in[i] * 2.0f + 1.0f;
    // 编译器会尽量把寄存器压到 65536/(256*4) = 64 个以内
}
别为了 occupancy 盲目压寄存器 __launch_bounds__ 强制降寄存器，代价是编译器可能把变量溢出（spill）到本地内存——本地内存其实在显存里，又慢又走带宽。结果 occupancy 上去了，却多了一堆访存，净亏。正确姿势是：先用 -Xptxas -v 看寄存器和 occupancy，再用第 16 章这两个模型判断瓶颈到底在不在 occupancy；确认在，再动 __launch_bounds__，并对比前后实测速度。凭感觉刷 occupancy 是新手最常见的浪费。

把两把尺子合起来用

occupancy 和 Roofline 回答的是两个不同的问题，实战里要配合着用：

  1. 先用 Roofline 定方向：测 kernel 的算术强度，看它在脊点左边还是右边。左边（多数 kernel）→ 主攻访存；右边 → 主攻计算。再看实测性能离屋顶多远，判断还有没有空间。
  2. 再用 occupancy 排查"延迟没藏住"：如果 kernel 明明带宽受限，实测带宽却远没跑满，很可能是 occupancy 太低（寄存器爆了/block 太小），备胎 warp 不够，访存延迟没被掩盖。这时提 occupancy 有效。
  3. occupancy 够用就停手：过半还慢，回到 Roofline 找真瓶颈，别再刷占用率。

这套"测强度 → 定方向 → 查 occupancy → 对比屋顶"的流程，就是专业 GPU 优化的骨架。附录 C 会教你用真实的 profiler（Nsight Compute）自动测出这些数字，它甚至直接给你画好 Roofline 图。

小结与下一步

你现在手里有两把定量的尺子，不用再靠"感觉这里慢"来优化了。Roofline 告诉你 kernel 撞的是带宽墙还是算力墙、离天花板多远；occupancy 告诉你有没有挂够 warp 去掩盖延迟。而 Roofline 反复指向同一个结论：绝大多数 kernel 落在脊点左边，被带宽压制——所以下一步最值钱的杠杆，就是把访存优化到极致。

下一章我们就把它彻底讲透：一个 warp 的 32 次访存怎么合并成一次大传输、shared memory 的 bank conflict 怎么让访存悄悄慢 32 倍、以及怎么用向量化加载一次搬更多字节。这是把带宽受限 kernel 真正提速的第一实战杠杆。

这一章的要点

  • occupancy = 实际驻留的活跃 warp ÷ SM 支持的最大 warp。它的作用是提供足够的备胎 warp 掩盖访存延迟。
  • 限制 occupancy 的三种资源：每线程寄存器、每 block 的 shared memory、block 大小/硬上限，谁先耗尽谁说了算（木桶效应）。降寄存器常能显著拉高 occupancy。
  • 100% 不是目标：够高到能掩盖延迟即可（常常 50% 就够）；盲目刷高常以多访存/寄存器溢出为代价，得不偿失。
  • Roofline 模型：可达性能 = min(峰值算力, 峰值带宽 × 算术强度)。斜屋顶（带宽）与平屋顶（算力）在脊点相交，脊点横坐标 = 峰值算力 ÷ 峰值带宽（本书那块卡是 20 FLOP/byte）。
  • 脊点左边 = 带宽受限（优化访存），右边 = 算力受限（优化计算）。多数真实 kernel 在左边。
  • 用 nvcc -Xptxas -v 看寄存器用量，用 __launch_bounds__ 约束它——但要先确认瓶颈真在 occupancy，再动手。
