第 11 章 CUDA 进阶：用 shared memory 做 tiling 矩阵乘

这一章想让你带走什么

上一章把 CUDA 的编程模型系统化了，但那些 kernel 都是"读一次、算一下、写一次"的访存受限类型。这一章讲 GPU 优化里最经典、最值得吃透的一个案例：矩阵乘法。你会先看到最朴素的写法为什么慢——它把同一份数据从慢速全局内存里反复读了几十上百遍；再学 tiling（分块） 这招：让一个 block 的线程协作把小块数据搬进 __shared__ 高速内存，反复复用，把一个访存受限的问题硬生生推向算力受限。

读完你能亲手推导 __syncthreads() 为什么一轮要放两个、各放在哪；用一个浏览器里能跑的模拟器数出 tiling 把全局内存读取次数砍掉了几倍；并读懂一个完整、正确、能和 CPU 结果对拍的 tiled 矩阵乘程序。

先看朴素矩阵乘：为什么它是访存灾难

矩阵乘 C = A × B，三者都是 N×N。定义你早就会：C[i][j] 等于 A 的第 i 行和 B 的第 j 列逐元素相乘再求和。搬到 GPU 上，最自然的写法就是第 1 章那套思路——一个线程负责算一个 C[i][j]：

CUDA C++
// 朴素矩阵乘：每个线程算一个 C[row][col]
__global__ void matmul_naive(int N, const float* A, const float* B, float* C) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (row < N && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < N; ++k)
            sum += A[row * N + k] * B[k * N + col];   // 每次都从全局内存读
        C[row * N + col] = sum;
    }
}

代码没错，但看那个内层循环：算一个 C[row][col] 要从全局内存读一整行 A（N 个元素）加一整列 B（N 个元素）。全 N×N 个输出元素都这么干，总读取次数是 2N³。

真正致命的是重复。想想 C[0][0] 和 C[0][1]：它俩都要读 A 的第 0 行——同一份数据被不同线程从慢速全局内存里各读了一遍。整个第 0 行 A，会被第 0 行所有 N 个输出线程各读一次，即读了 N 遍。B 的每一列同理。回忆第 3 章的算术强度：这个 kernel 每读一个 float（4 字节）只做一次乘加，强度低得可怜，妥妥的访存受限——算力单元大把时间在等数据从显存爬过来。

问题的本质：复用没有被利用 矩阵乘里数据的复用度极高（每个 A 元素要参与 N 个输出的计算），但朴素写法把每次复用都变成一次全局内存访问。优化的全部思路就一句话：把要反复用的数据，先搬到快的地方（shared memory），在那儿反复用完再说。这正是第 3 章那块"片上内存比显存快两个数量级"派上用场的地方。

tiling：让一个 block 协作搬砖、就地复用

tiling（分块 / blocking） 的想法：不要让每个线程各自跑去全局内存读整行整列，而是让一个 block 里的线程组成一个小方阵（比如 16×16），大家协作把 A 的一小块和 B 的一小块搬进 __shared__ 内存，然后所有线程都从这块共享的快内存里取数做部分和。搬一次，被整个 block 复用 TILE 次。

具体地，一个 block 负责算 C 的一个 TILE×TILE 小方块。因为算这个小方块需要 A 的一整条行带和 B 的一整条列带（都太大放不进 shared memory），就把它们沿 k 方向也切成一个个 TILE×TILE 的小块，一块一块地滑过去：

  1. block 里每个线程从全局内存各读一个 A 元素和一个 B 元素，填进 As[ty][tx] 和 Bs[ty][tx]。TILE×TILE 个线程一起，正好把两个小块填满。
  2. 同步（__syncthreads()）：确保整块都装载完，谁都别提前开算。
  3. 每个线程从 shared memory 里取 TILE 个数做部分和累加——这一步的访存全在快内存里，且每个搬进来的元素被复用了 TILE 次。
  4. 再同步：确保所有线程都算完这块、用完这块 shared memory，才能覆盖写下一块。
  5. 滑到下一个 k 方向的 tile，回到第 1 步，直到覆盖完整条行带/列带。
图 11-1　tiling 方案：算 C 的一个小块，需要 A 的一条行带与 B 的一条列带；把它们切成 TILE×TILE 的小块逐个搬进 shared memory，block 内线程协作装载、就地复用、累加部分和，滑完整条带即得结果

两个 __syncthreads()，一个都不能省

__shared__ 内存是一个 block 内所有线程共享的，所以谁读谁写必须协调。每轮 tile 迭代放两个 barrier，位置和理由如下：

第一个 barrier：装载后、计算前
CUDA C++
As[ty][tx] = A[...];   // 我只填一个格子
Bs[ty][tx] = B[...];
__syncthreads();       // ← 等全 block 填满
// 若省掉：我可能用到别的线程
// 还没写进去的格子 → 读到垃圾
for (k) sum += As[ty][k]*Bs[k][tx];
第二个 barrier：计算后、覆盖前
CUDA C++
for (k) sum += As[ty][k]*Bs[k][tx];
__syncthreads();       // ← 等全 block 算完
// 若省掉：跑得快的线程会用
// 下一块数据覆盖 As/Bs，而慢线程
// 还在读这一块 → 读到被冲掉的值
// 下一轮再 As[ty][tx] = A[...];

一句话记：第一个 barrier 保证"数据都到齐了才开始算"，第二个保证"都算完了才敢覆盖"。两个都是防止 shared memory 上的数据竞争，缺一个结果就会随机出错——而且往往在小矩阵上"碰巧"对，一放大就崩，是极难查的 bug。（barrier 的机制见第 8 章。）

动手：数一数 tiling 到底省了多少次全局读

下面这段程序能在浏览器里直接跑——点 ▶ 运行。它不做真的乘法，只数全局内存读取次数：朴素版每算一个输出读 2N 次（一行 A + 一列 B），tiling 版则是每个 block 沿 k 方向滑动、每个线程每 tile 只从全局内存读 2 个元素。对比两个数字，你会看到 tiling 恰好把全局读取砍掉约 TILE 倍。

Python 
# 数全局内存读取次数：朴素矩阵乘 vs tiling 矩阵乘
def count_naive(N):
    # 每个 C[i][j] 读一整行 A（N 个）+ 一整列 B（N 个）= 2N 次，共 N*N 个输出
    return N * N * (2 * N)

def count_tiled(N, TILE):
    nb = (N + TILE - 1) // TILE            # 每个维度切成几个 tile
    # 一个 block 负责 C 的一个 TILE×TILE 小块，沿 k 方向滑过 nb 个 tile；
    # 每个 tile 里 TILE×TILE 个线程各从全局内存读 A、B 各 1 个 = 2 次
    reads_per_block = nb * (TILE * TILE * 2)
    return (nb * nb) * reads_per_block     # 共 nb×nb 个 block

for N, TILE in [(64, 16), (1024, 32)]:
    na = count_naive(N)
    ti = count_tiled(N, TILE)
    print(f"N={N}, TILE={TILE}")
    print(f"  朴素版全局读取：{na:,} 次")
    print(f"  tiling 全局读取：{ti:,} 次")
    print(f"  省了 {na / ti:.0f} 倍（≈ TILE={TILE}）\n")

运行结果：

N=64, TILE=16
  朴素版全局读取：524,288 次
  tiling 全局读取：32,768 次
  省了 16 倍（≈ TILE=16）

N=1024, TILE=32
  朴素版全局读取：2,147,483,648 次
  tiling 全局读取：67,108,864 次
  省了 32 倍（≈ TILE=32）

结论很干脆：tiling 把全局内存读取砍掉约 TILE 倍，砍掉的那部分变成了对 shared memory 的访问——而后者快两个数量级。用第 3 章的语言说，tiling 把算术强度提高了 TILE 倍，让 kernel 从访存受限往算力受限方向挪了一大步。请动手改：把 TILE 从 16 调到 32，再看 N=64 那组——省的倍数也跟着变成 32。这就是为什么 tile 越大理论上越省……但别急，看下面的警告。

tile 不是越大越好 shared memory 每个 SM 就那么几十到上百 KB，两个 TILE×TILE 的 float 数组要占 2 × TILE² × 4 字节。TILE=32 就要 8 KB，再大很快会撑爆 shared memory，或让每个 SM 能同时驻留的 block 数（occupancy，第 16 章）掉下来，反而更慢。实践中 16 或 32 是常见甜点值，具体要测。

动手：一个完整、正确、能对拍的 tiled 矩阵乘

下面是完整可编译的程序：tiled 矩阵乘 kernel + host 端填充小矩阵、运行、和 CPU 参考实现逐元素对拍，最后打印 OK。这段代码的索引是这一章的重点，请对着图 11-1 慢慢读。点 🔗 在 Compiler Explorer 打开 看它编成的 PTX，或用下面的命令本地编译运行。

CUDA C++ 🔗 在 Compiler Explorer 打开（看 PTX）
#include <cstdio>
#include <cstdlib>
#include <cmath>
#include <cuda_runtime.h>

#define TILE 16

// tiled 矩阵乘：C = A × B，三者都是 N×N，行主序
__global__ void matmul_tiled(int N, const float* A, const float* B, float* C) {
    __shared__ float As[TILE][TILE];       // 本 block 共享的 A 小块
    __shared__ float Bs[TILE][TILE];       // 本 block 共享的 B 小块

    int tx = threadIdx.x, ty = threadIdx.y;
    int row = blockIdx.y * TILE + ty;      // 本线程负责的 C 行
    int col = blockIdx.x * TILE + tx;      // 本线程负责的 C 列

    float sum = 0.0f;
    int numTiles = (N + TILE - 1) / TILE;  // 沿 k 方向要滑过几个 tile
    for (int t = 0; t < numTiles; ++t) {
        // 1. 协作装载：每个线程各填 As、Bs 的一个格子（带边界检查，越界填 0）
        int aCol = t * TILE + tx;          // A 这一 tile 的列下标
        int bRow = t * TILE + ty;          // B 这一 tile 的行下标
        As[ty][tx] = (row < N && aCol < N) ? A[row * N + aCol] : 0.0f;
        Bs[ty][tx] = (bRow < N && col < N) ? B[bRow * N + col] : 0.0f;

        __syncthreads();                   // 2. 等全 block 装载完，才能开算

        // 3. 从 shared memory 取数做部分和（快内存，且每个元素被复用 TILE 次）
        for (int k = 0; k < TILE; ++k)
            sum += As[ty][k] * Bs[k][tx];

        __syncthreads();                   // 4. 等全 block 算完，才能覆盖 As/Bs
    }
    if (row < N && col < N)
        C[row * N + col] = sum;            // 写回结果
}

int main() {
    const int N = 100;                     // 故意取非 TILE 整数倍，测边界处理
    size_t bytes = N * N * sizeof(float);

    float *hA = (float*)malloc(bytes);
    float *hB = (float*)malloc(bytes);
    float *hC = (float*)malloc(bytes);
    for (int i = 0; i < N * N; ++i) {      // 填一点非平凡的值
        hA[i] = (float)((i % 7) - 3);
        hB[i] = (float)((i % 5) - 2);
    }

    float *dA, *dB, *dC;
    cudaMalloc(&dA, bytes); cudaMalloc(&dB, bytes); cudaMalloc(&dC, bytes);
    cudaMemcpy(dA, hA, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(dB, hB, bytes, cudaMemcpyHostToDevice);

    // block 是 TILE×TILE 的线程方阵；grid 铺满整个 C
    dim3 block(TILE, TILE);
    dim3 grid((N + TILE - 1) / TILE, (N + TILE - 1) / TILE);
    matmul_tiled<<<grid, block>>>(N, dA, dB, dC);
    cudaDeviceSynchronize();

    cudaMemcpy(hC, dC, bytes, cudaMemcpyDeviceToHost);

    // CPU 参考实现，逐元素对拍
    bool ok = true;
    for (int i = 0; i < N && ok; ++i)
        for (int j = 0; j < N && ok; ++j) {
            float ref = 0.0f;
            for (int k = 0; k < N; ++k) ref += hA[i * N + k] * hB[k * N + j];
            if (fabsf(ref - hC[i * N + j]) > 1e-3f) ok = false;
        }
    printf("C[0][0] = %.1f, 校验 %s\n", hC[0], ok ? "OK" : "FAILED");

    cudaFree(dA); cudaFree(dB); cudaFree(dC);
    free(hA); free(hB); free(hC);
    return 0;
}
本地运行：nvcc -std=c++20 -O3 -arch=sm_90 main.cu -o app && ./app

终端
nvcc -arch=sm_80 matmul_tiled.cu -o matmul_tiled && ./matmul_tiled
# 输出：C[0][0] = ..., 校验 OK
三处最容易写错的索引 （1）装载 A 用 A[row * N + aCol]，行固定为本线程的 row、列随 tile 滑动（aCol = t*TILE + tx）；装载 B 用 B[bRow * N + col]，列固定为本线程的 col、行随 tile 滑动。方向别搞反。（2）内层累加是 As[ty][k] * Bs[k][tx]——共享的中间维 k 在 As 里做列下标、在 Bs 里做行下标。（3）边界越界时填 0.0f，这样部分和不受影响，N 不是 TILE 整数倍也正确。上面特意取 N=100 就是逼这条边界逻辑上场。

这就是 CUDA 的"超能力"

回头看这一章做了什么：同一个矩阵乘，我们没换算法（还是 O(N³) 的乘加），只是换了数据放在哪、被谁复用——把反复要读的小块提到 __shared__ 里就地复用，全局内存读取砍掉 TILE 倍。这种对片上内存的显式、精细控制，正是上一章说的"CUDA 最贴近硬件、给你最多控制"的含金量所在。cuBLAS 里真正的高性能矩阵乘还会在此之上叠寄存器分块、双缓冲、Tensor Core（第 19 章），但内核思想就是你刚学会的这个 tiling。

小结与下一步

你现在掌握了 GPU 优化的头号范式：用 shared memory 做 tiling，把访存受限的问题推向算力受限。这套"协作搬砖 + 就地复用 + 两次同步"的模式，会在归约、扫描、卷积、attention 等几乎所有重要 kernel 里反复出现。

有意思的是，这个思想完全不是 CUDA 独有的——它是 GPU 硬件层级决定的普适规律。下一章我们切换到 SYCL：你会看到同样的 tiling 矩阵乘，用一套单一源码、跨厂商的现代 C++ 写出来——__shared__ 变成 local_accessor、__syncthreads() 变成 item.barrier()，但那颗"搬进快内存、协作复用"的心，一模一样。

这一章的要点

  • 朴素矩阵乘每个线程算一个 C[i][j]，从全局内存读一整行 A + 一整列 B，总读取 2N³，同一份数据被反复读几十上百遍——访存受限。
  • tiling（分块）：让一个 block 的线程协作把 A、B 的 TILE×TILE 小块搬进 __shared__，沿 k 方向滑动、就地复用，把全局读取砍掉约 TILE 倍，算术强度提高 TILE 倍（呼应第 3 章）。
  • 每轮 tile 迭代要放两个 __syncthreads()：装载后（保证数据到齐再算）、计算后（保证算完再覆盖），防 shared memory 数据竞争，一个都不能省。
  • 关键索引：装载 A 行固定列随 tile 滑、装载 B 列固定行随 tile 滑；累加 As[ty][k]*Bs[k][tx]；越界填 0 以正确处理 N 非 TILE 整数倍。
  • 这种对片上内存的显式精细控制是 CUDA"贴近硬件"的招牌；同样的 tiling 思想在 SYCL/Triton/CuTe 里换个外衣照样成立（下一章起）。
