第 18 章 并行原语：归约与扫描的经典模式

这一章想让你带走什么

前面你学会了把"每个元素一个线程"的活儿映射到 GPU，那都是各干各的、互不来往的计算。这一章处理一类新问题：结果依赖于所有元素——求一个数组的和、最大值，或每个位置的前缀和。这类问题在 CPU 上一个 for 循环就搞定，可它天生带着"上一步喂给下一步"的串行依赖，怎么让几万个线程一起做？

答案是两个被反复使用的并行原语（parallel primitive）：归约（reduction）和扫描（scan / 前缀和）。读完你会明白它们怎么把看似串行的问题拆成 log N 步的并行，并亲手跑两个浏览器里的模拟器，看着数组在每一步塌缩或铺开；最后写出一个完整的、用 shared memory + 树形归约 + warp shuffle + atomicAdd 求和的 CUDA 程序。

问题：这次每个线程不能只顾自己

回忆你在 CPU 上求一个数组的和，代码简单到不用想：

C++
// CPU：一个累加器，从头扫到尾
float sum = 0.0f;
for (int i = 0; i < n; ++i)
    sum += a[i];        // 每一步都依赖上一步的 sum

这段代码有一个你平时不会注意、但对 GPU 是致命的性质：每一次 sum += a[i] 都要读上一次的 sum。第 5 次加法必须等第 4 次算完，第 4 次又要等第 3 次……n 个元素排成一条长度为 n 的依赖链，只能一个接一个走。CPU 上无所谓，反正就几个线程，单核跑得飞快。可 GPU 有几万个线程，你让它们排成一条链依次加，等于让一支千人军队走独木桥——绝大多数线程在干等。

并行原语要解决的核心矛盾 归约和扫描的输出依赖全部输入，朴素写法是一条长度 n 的串行依赖链。并行化的关键洞察是：加法（以及 max、min、乘法……）满足结合律，((a+b)+c)+d 可以重排成 (a+b)+(c+d)。一旦允许重新结合，那条长链就能折叠成一棵深度只有 log₂n 的树——这就是把串行变并行的全部秘密。

归约：把一条链折成一棵树

先看最简单的归约——数组求和。串行是 n-1 次加法排成一条链；树形归约则让相邻的元素两两相加，一轮下来元素个数减半，log₂n 轮后只剩一个。

拿 8 个数举例。第 1 轮：让 4 个线程各做一次加法，8 个数变 4 个；第 2 轮：2 个线程，变 2 个；第 3 轮：1 个线程，变 1 个。log₂8 = 3 轮就出结果，而串行要 7 步。数据一大，差距是指数级的：一百万个元素，串行一百万步，树形归约只要 20 步。

CPU：一条 n 步的依赖链
C++
float sum = 0;
for (int i = 0; i < n; ++i)
    sum += a[i];
// 第 i 步依赖第 i-1 步
// 深度 = n，无法并行
// n=1e6 → 一百万步
GPU：一棵 log₂n 层的树
CUDA C++
// 每一步：活跃线程数减半
for (int s = n/2; s >= 1; s >>= 1) {
    if (tid < s)
        a[tid] += a[tid + s];
    __syncthreads();   // 等这一层都算完
}
// 深度 = log₂n
// n=1e6 → 约 20 步

具体怎么两两相加，有讲究。一种自然的想法是"隔一个加一个"（stride 从 1 开始翻倍），但那会让活跃线程分散、还容易在 shared memory 上撞 bank conflict（第 17 章）。更好的写法叫顺序寻址（sequential addressing）：让 stride 从 n/2 开始减半，每一步都是前半段加后半段——活跃的永远是编号最小的一批线程 tid < stride，它们连续排布，既避免了 warp 内的分支分歧（第 2 章），访问 shared memory 也是连续的、不撞 bank。下面这张图就是顺序寻址的样子：

图 18-1　顺序寻址的树形归约：stride 从 4 减半到 1，每一步"前半段 += 后半段"，活跃线程数 4→2→1；log₂8=3 步就把 8 个数求成一个和

在一个 block 里做归约

一个 block 里的线程能共享 shared memory、能用 __syncthreads() 互相等待（第 8 章）——这正好是树形归约需要的两样东西。标准套路分三步：

  1. 搬进 shared memory：每个线程从全局内存读一个元素，放进 block 的 shared memory 数组，__syncthreads() 确保全搬完。
  2. 树形折叠：按上面的顺序寻址循环，每一层折叠后 __syncthreads()，等这一层所有线程都写完，下一层才能读。
  3. 收尾：折到只剩一个 warp（32 个线程）时，warp 内的线程本就锁步执行，不再需要 __syncthreads()——改用更快的 warp shuffle（第 8 章的 __shfl_down_sync）在寄存器里直接交换、相加，省掉最后几轮的 shared memory 往返和同步。
为什么最后一个 warp 要特殊处理 __syncthreads() 是整个 block 的 barrier，不便宜。当活跃线程降到 32 个以内时，它们全在同一个 warp 里、天然锁步，继续调用 __syncthreads() 纯属浪费。warp shuffle 让线程直接读另一个线程寄存器里的值，无需经过 shared memory，也无需同步——这是归约 kernel 里最经典的一处优化。但要记住：shuffle 只在同一个 warp 内有效，跨 warp 仍得靠 shared memory + __syncthreads()。

合并 block 之间的结果

一个 block 只能归约它自己那部分数据，得到一个部分和（partial sum）。一百万个元素、每 block 处理 256 个，就有约 4096 个部分和要再合成一个总和。跨 block 无法用 __syncthreads()（第 4 章讲过，block 之间不能安全互等），有两条常见路子：

  • 第二个 kernel：第一个 kernel 把每个 block 的部分和写进一个长度为 blocks 的数组，再启动第二个 kernel 归约这个短数组。两次启动之间是天然的全局同步点。
  • atomicAdd 汇总：每个 block 算完后，用一次原子加（第 8 章）把自己的部分和累加到一个全局累加器上。省了第二个 kernel，代价是几千次原子加的争用——但每个 block 只加一次（不是每个线程都加），争用可以接受。本章的完整程序用的就是这条路。

扫描：比它看起来难得多的原语

第二个原语是扫描（scan），也叫前缀和（prefix sum）。给定数组 [a0, a1, a2, ...]，扫描输出每个位置之前所有元素的累积。它分两种：

  • 包含式（inclusive）：out[i] = a0 + a1 + ... + ai（含自己）。
  • 排他式（exclusive）：out[i] = a0 + ... + a(i-1)（不含自己，第 0 位是单位元 0）。
图 18-2　包含式扫描：out[i] 是输入前 i+1 项之和。图中 out[4]=15=3+1+7+0+4；最右端 25 就是整个数组的总和

扫描比归约棘手，因为它的每个输出都不同——归约只要一个总和，扫描要 n 个各不相同的累积值。朴素串行版还是一个循环：

C++
out[0] = a[0];
for (int i = 1; i < n; ++i)
    out[i] = out[i-1] + a[i];   // 又是一条 n 步的依赖链

看起来彻头彻尾是串行的——out[i] 明明白白依赖 out[i-1]。但和归约一样，靠结合律，它照样能拆成 log N 步的并行。

两种并行扫描

Hillis-Steele 扫描（step-efficient）最直观：第 d 步（d = 1, 2, 4, ...）让每个位置加上距它 d 格的左邻 x[i] += x[i-d]。log₂n 步后每个位置就累积了它左边所有元素。本章的模拟器演示的就是它。它步数少（log n 步），但每步几乎所有线程都在加，总加法次数是 O(n log n)——比串行的 O(n) 还多，属于不做无用功但做了重复功。

**Blelloch 扫描（work-efficient）**更省，总加法次数降到 O(n)，代价是两趟遍历树：

  • 上扫（up-sweep / reduce）：自底向上做一次树形归约，顺便在树的内部节点上留下部分和。走完根节点拿到总和。
  • 下扫（down-sweep）：把根置为单位元（0），自顶向下走，每个节点把自己的值传给右孩子、把"左孩子旧值 + 自己"传给……最终每个叶子恰好得到它左边所有元素之和（排他式扫描）。

上扫下扫各 log n 步、总功 O(n)，是 GPU 库（如 CUB、Thrust）里大规模扫描的底子。概念上你只需记住一句：扫描 = 一趟自底向上的归约 + 一趟自顶向下的分发。

为什么值得为扫描费这么大劲 扫描是一大堆算法的地基原语。直方图用它把各桶计数变成桶的起始偏移；基数排序用它算出每个元素该落到的位置；流压缩（stream compaction）——"把数组里满足条件的元素紧凑地挑出来"——本质就是对一个 0/1 标记数组做排他式扫描，扫描结果直接告诉每个被保留元素该写到输出的第几格。你会发现，很多"看起来必须串行"的数据整理，底层都是一次扫描。归约和扫描就是并行算法的加减法，值得吃透。

动手：树形归约模拟器

下面这段程序能在浏览器里直接跑——点 ▶ 运行。它用纯 Python 忠实复刻图 18-1 的顺序寻址归约：每一步把前半段加上后半段，打印数组状态，并报告本步有几个线程在干活。你会看到活跃线程数 4 → 2 → 1 一路减半，log₂8 = 3 步得到总和。

Python 
# 树形归约模拟器：把 8 个数求和，看每一步活跃线程数如何减半
data = [3, 1, 7, 0, 4, 1, 6, 3]
n = len(data)
a = data[:]                        # 复制一份，就地归约
print(f"初始数组（{n} 个）：{a}   （CPU 串行求和 = {sum(data)}）")

step = 0
stride = n // 2
while stride >= 1:
    step += 1
    active = stride                # 本步活跃线程数 = stride
    for tid in range(stride):      # 只有 tid < stride 的线程活跃
        a[tid] += a[tid + stride]  # 前半段 += 后半段
    print(f"第 {step} 步：stride={stride}，{active} 个线程活跃 -> {a}")
    stride //= 2

print(f"\n归约完成，a[0] = {a[0]}  （与 CPU 求和一致？{a[0] == sum(data)}）")
print(f"{n} 个元素只用了 {step} 步 = log2({n})，而串行需要 {n-1} 次加法")

运行结果：

初始数组（8 个）：[3, 1, 7, 0, 4, 1, 6, 3]   （CPU 串行求和 = 25）
第 1 步：stride=4，4 个线程活跃 -> [7, 2, 13, 3, 4, 1, 6, 3]
第 2 步：stride=2，2 个线程活跃 -> [20, 5, 13, 3, 4, 1, 6, 3]
第 3 步：stride=1，1 个线程活跃 -> [25, 5, 13, 3, 4, 1, 6, 3]

归约完成，a[0] = 25  （与 CPU 求和一致？True）
8 个元素只用了 3 步 = log2(8)，而串行需要 7 次加法

注意归约后数组尾部是"垃圾"——只有 a[0] 是有效结果，a[1..] 留着中间值没人再管。这在 GPU 上完全 OK：block 只关心最后写出 sdata[0]。请动手改：把 data 换成 16 个数（比如 list(range(16))），再运行——你会看到步数变成 log₂16 = 4，而串行要 15 步。数据翻倍，串行步数翻倍，树形归约只多一步，这就是对数的威力。

动手：Hillis-Steele 扫描模拟器

再看扫描。下面这段程序同样能在浏览器里跑，实现图 18-2 的包含式扫描。它每一步让每个位置加上距它 d 格的左邻（d = 1, 2, 4），打印每步后的数组，最后和 CPU 串行前缀和核对。

Python 
# Hillis-Steele 包含式扫描：每个输出 = 到它为止所有输入之和
data = [3, 1, 7, 0, 4, 1, 6, 3]
n = len(data)
x = data[:]
print(f"输入：          {x}")

step = 0
d = 1
while d < n:
    step += 1
    old = x[:]                          # 先读旧值（模拟 double buffering，避免读到本步刚写的值）
    for i in range(n):
        if i >= d:
            x[i] = old[i] + old[i - d]  # 加上距离 d 的左邻
    print(f"第 {step} 步 d={d}：      {x}")
    d *= 2

# 用 CPU 串行前缀和核对
ref, acc = [], 0
for v in data:
    acc += v
    ref.append(acc)
print(f"\n最终（包含式扫描）：{x}")
print(f"CPU 串行前缀和：    {ref}")
print(f"一致？{x == ref}，共 {step} 步 = log2({n})")

运行结果：

输入：          [3, 1, 7, 0, 4, 1, 6, 3]
第 1 步 d=1：      [3, 4, 8, 7, 4, 5, 7, 9]
第 2 步 d=2：      [3, 4, 11, 11, 12, 12, 11, 14]
第 3 步 d=4：      [3, 4, 11, 11, 15, 16, 22, 25]

最终（包含式扫描）：[3, 4, 11, 11, 15, 16, 22, 25]
CPU 串行前缀和：    [3, 4, 11, 11, 15, 16, 22, 25]
一致？True，共 3 步 = log2(8)

注意那个 old = x[:]：扫描里每个位置要读的是上一步的值，如果就地边读边写，会读到本步刚被别的位置改过的数——GPU 上这对应"双缓冲（double buffering）"，用两块 shared memory 轮流当读/写缓冲，或在读写之间插 __syncthreads()。请动手改：把 old = x[:] 删掉、循环里直接用 x 代替 old，再运行——你会看到结果错了，因为破坏了"读旧值"的约束。这正是并行扫描最容易踩的坑。

动手：一个完整的 CUDA 求和归约程序

把上面的思路落地成一个完整、可编译的 .cu 程序：每个 block 用 shared memory + 树形归约把自己那段求成部分和，最后一个 warp 用 shuffle 收尾，再用 atomicAdd 把各 block 的部分和汇总到全局累加器。程序对全 1 数组求和（理论和 = n），与 CPU 结果核对后打印 OK 和总和。点代码块的 🔗 在 Compiler Explorer 打开 可看 PTX；本地有 GPU 就用下面的命令编译运行。

CUDA C++ 🔗 在 Compiler Explorer 打开（看 PTX）
#include <cstdio>
#include <cmath>
#include <cuda_runtime.h>

// 块级求和归约：shared memory + 树形折叠 + 最后一个 warp 用 shuffle 收尾
__global__ void reduce_sum(const float* in, float* out, int n) {
    extern __shared__ float sdata[];
    int tid = threadIdx.x;
    int i   = blockIdx.x * blockDim.x + threadIdx.x;

    // 1) 每个线程搬一个元素进 shared memory，越界的填加法单位元 0
    sdata[tid] = (i < n) ? in[i] : 0.0f;
    __syncthreads();

    // 2) 顺序寻址树形归约：活跃线程 tid<stride，前半段 += 后半段，直到剩一个 warp
    for (int stride = blockDim.x / 2; stride > 32; stride >>= 1) {
        if (tid < stride) sdata[tid] += sdata[tid + stride];
        __syncthreads();          // 等这一层写完，下一层才能读
    }

    // 3) 最后一个 warp（32 线程）天然锁步：用 shuffle 收尾，无需 __syncthreads
    if (tid < 32) {
        float v = sdata[tid] + sdata[tid + 32];        // 先把 64 个并成 32 个
        for (int offset = 16; offset > 0; offset >>= 1) // 32→16→8→4→2→1
            v += __shfl_down_sync(0xffffffff, v, offset);
        // 4) 每个 block 只由 0 号线程做一次 atomicAdd，把部分和汇总到全局
        if (tid == 0) atomicAdd(out, v);
    }
}

int main() {
    const int n = 1 << 20;                 // 1,048,576 个元素
    const int bytes = n * sizeof(float);

    float* h_in = (float*)malloc(bytes);
    double cpu_sum = 0.0;
    for (int i = 0; i < n; ++i) {
        h_in[i] = 1.0f;                    // 全 1，理论总和 = n
        cpu_sum += h_in[i];
    }

    float *d_in, *d_out;
    cudaMalloc(&d_in, bytes);
    cudaMalloc(&d_out, sizeof(float));
    cudaMemcpy(d_in, h_in, bytes, cudaMemcpyHostToDevice);
    cudaMemset(d_out, 0, sizeof(float));   // 全局累加器清零

    int threads = 256;                     // 32 的倍数、且 >= 64（收尾要用 tid+32）
    int blocks  = (n + threads - 1) / threads;
    size_t shmem = threads * sizeof(float);
    reduce_sum<<<blocks, threads, shmem>>>(d_in, d_out, n);

    float gpu_sum = 0.0f;
    cudaMemcpy(&gpu_sum, d_out, sizeof(float), cudaMemcpyDeviceToHost);

    printf("CPU 求和 = %.0f\n", cpu_sum);
    printf("GPU 求和 = %.0f\n", gpu_sum);
    printf("%s\n", (fabs(gpu_sum - (float)cpu_sum) < 1e-3) ? "OK" : "FAILED");

    cudaFree(d_in); cudaFree(d_out); free(h_in);
    return 0;
}

小结与下一步

归约和扫描是本书从"每个线程各干各的"迈向"线程协作解决全局问题"的分水岭。它们的共同灵魂是同一句话：利用结合律，把一条长度 n 的串行依赖链，折叠成一棵深度 log n 的并行树。

你现在能用 shared memory + __syncthreads() + warp shuffle 写出一个高效的块级归约，也理解了扫描为什么"看着串行、实则可并行"，以及它是直方图、排序、流压缩这些算法的地基。

到这里，本书的性能与算法主线（occupancy、Roofline、访存优化、并行原语）就铺完了。下一章我们抬头看看现代 GPU 硬件为 AI 时代加了什么新武器：Tensor Core（一条指令做一整块矩阵乘）、异步拷贝与 TMA（搬数据不挡住计算）、以及软件流水线（一边算一边预取下一块）。它们是 FlashAttention、CUTLASS 这些"快得离谱"的库背后的真正引擎——而这也解释了 Triton、CuTe DSL 这些高级 DSL 为何存在：把这些硬件能力用起来，而不必手写 PTX。

这一章的要点

  • 归约（求和/max）和扫描（前缀和）的输出依赖全部输入，朴素写法是一条长度 n 的串行依赖链；靠结合律能折成深度 log n 的并行树。
  • 树形归约每一步活跃线程减半，log₂n 步出结果；顺序寻址（stride 从 n/2 减半、前半段+=后半段）能避免 warp 分歧和 bank conflict（第 2、17 章）。
  • 块级归约三步走：搬进 shared memory → 树形折叠（每层 __syncthreads()）→ 最后一个 warp 用 shuffle 收尾（第 8 章）；跨 block 用第二个 kernel 或 atomicAdd 汇总。
  • 扫描分包含式/排他式，看着串行实则可并行：Hillis-Steele（log n 步、O(n log n) 功）直观；Blelloch（上扫+下扫、O(n) 功）更省——记住"扫描 = 自底向上归约 + 自顶向下分发"。
  • 扫描是直方图、基数排序、流压缩等一大类算法的地基原语；归约换个单位元和运算就能算平方和、最大值（第 20 章 softmax 会用到）。
