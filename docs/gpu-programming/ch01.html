第 1 章 你会并行编程了——那 GPU 到底不一样在哪

这一章想让你带走什么

你会写多线程代码，懂锁、懂缓存、也许还调过 SIMD。这一章不教你任何 GPU 语法，而是先讲清一件事：当你从 CPU 走进 GPU，真正改变的不是 API，而是"一个线程值多少钱"这个最底层的假设。

读完这一章，你会明白后面每一章为什么这么安排，也会用一个能在浏览器里直接跑的小实验，亲眼看到"几个线程"和"成千上万个线程"的世界观差别。

先说结论：线程变便宜了，但也变笨了

在 CPU 上，你对"线程"有一套根深蒂固的直觉。你知道线程很贵：创建一个要向操作系统申请，调度有开销，所以你会用线程池复用它们；你的机器上能真正并行的线程数，大约等于核心数，8 核、16 核、顶天 128 核。你写并行代码时，脑子里想的是"几个强壮的工人，各自干一大块活"。

C++ 🔗 在 Compiler Explorer 打开
// CPU 的心智模型：几个强壮的线程，每个处理一大段数据
#include <thread>
#include <vector>

void add_range(float* c, const float* a, const float* b, int lo, int hi) {
    for (int i = lo; i < hi; ++i) c[i] = a[i] + b[i];  // 一个线程干一整段
}

int main() {
    const int N = 1'000'000;
    std::vector<float> a(N, 1.0f), b(N, 2.0f), c(N);
    const int T = 8;                       // 8 个线程，约等于核心数
    std::vector<std::thread> pool;
    for (int t = 0; t < T; ++t)
        pool.emplace_back(add_range, c.data(), a.data(), b.data(),
                          t * N / T, (t + 1) * N / T);   // 每人分 1/8
    for (auto& th : pool) th.join();
}
本地编译运行：g++ -std=c++23 -O2 -Wall main.cpp && ./a.out

八个线程，每个循环处理十二万五千个元素。这是 CPU 世界的标准姿势：线程少而强，每个线程内部还要靠循环、靠缓存、靠乱序执行榨干性能。

现在看 GPU 的做法。同样是向量加法，你不再写"每个线程处理一段"，而是写一个线程只处理一个元素，然后一次性启动一百万个线程：

CUDA C++ 🔬 看 PTX / 编译器处理
// GPU 的心智模型：一百万个线程，每个只处理一个元素
// __global__ 表示这是一个"kernel"——会在 GPU 上、被成千上万个线程各跑一遍的函数
__global__ void add(float* c, const float* a, const float* b, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;  // 我是第几个线程？（全局编号）
    if (i < n) c[i] = a[i] + b[i];                  // 就只算这一个元素
}
本地运行：nvcc -std=c++20 -O3 -arch=sm_90 --ptx main.cu -o main.ptx # 生成 PTX 中间码

而在 CPU（host）那边，你只要写一行就能把它一次性铺开成上百万个线程——具体怎么分配显存、怎么启动，留到第 9 章完整走一遍，这里先看形状：

C++
const int N = 1'000'000;
int threads = 256;                          // 每个 block 256 个线程
int blocks  = (N + threads - 1) / threads;  // 凑够约 100 万个线程（向上取整）
add<<<blocks, threads>>>(c, a, b, N);       // <<<...>>> 是 CUDA 的启动语法，一次全部启动

点开上面 CUDA 代码块的 🔬 看 PTX / 编译器处理，你能看到 nvcc 把这个 kernel 编译成的 PTX（GPU 的"汇编"）——现在不用读懂它，先记住这个入口，后面讲编译器处理时会反复用到。

差别不在"用不用并行"（两边都并行），而在并行的粒度和心态：

CPU：几个强壮的线程
C++
// 8 个线程
// 每个线程用循环处理 12.5 万个元素
// 线程贵，所以要复用、要分块
// 单线程很快，内部还有缓存和乱序执行
GPU：海量弱小的线程
CUDA C++
// 100 万个线程
// 每个线程只处理 1 个元素
// 线程极便宜，随手就开几百万个
// 单线程很慢，靠"人多"和内存带宽取胜
本书第一个、也是最重要的一个观念 在 CPU 上，线程又贵又强，你让少数几个线程各自处理一大块数据；在 GPU 上，线程又便宜又弱，你启动成千上万个，每个只处理一小片。写 GPU 代码的第一步，是把"我要怎么分块给几个线程"换成"如果每个数据元素都有一个自己的线程，它该做什么"。这叫数据并行（data parallelism）的思维。
图 1-1　同一个向量加法，两种世界观：CPU 用几个线程各扫一段；GPU 给每个元素配一个线程，一次全启动

这一个差别，会像多米诺骨牌一样推倒你对很多事情的默认预期：分支（if）在 GPU 上为什么可能变慢？为什么访问内存的顺序比访问的次数还重要？为什么 GPU 上几乎不用锁，却大量用一种叫 barrier 的东西？——在 GPU 上，答案常常和你的 CPU 直觉相反。

为什么 GPU 要这么设计？一句话：赌带宽，不赌延迟

CPU 是一台延迟优化的机器。它假设你的程序里下一步依赖上一步，所以拼命缩短单条指令链的时间：巨大的缓存、分支预测、乱序执行、深流水线——全都是为了让单个线程尽快跑完。

GPU 是一台吞吐优化的机器。它假设你有海量互不依赖的活儿要干，所以干脆不追求单个线程快，而是追求单位时间内完成的总量最大。它的秘密武器是"用并行掩盖延迟"：当一批线程卡在等内存时，硬件瞬间切换到另一批就绪的线程接着算，访存的等待就被算力填满了，几乎不浪费。

类比：餐厅的两种开法 CPU 像一家精致小馆：几个顶级大厨，每人一条龙做完一道复杂菜，追求每道菜出餐尽量快（低延迟）。GPU 像一座巨型食堂：几百个只会做一个动作的帮厨，站成流水线，追求每小时出餐总量最大（高吞吐）。你不会让食堂做分子料理，也不会让小馆一小时供应上万份——用错场景，两边都难看。这就是为什么不是所有代码都适合 GPU。

因为赌的是吞吐，GPU 必须同时挂着远超核心数的线程（NVIDIA 的旗舰卡能同时驻留几万个线程）。线程一多，硬件就没本钱给每个线程都配独立的取指、译码、调度——于是它做了一个关键的取舍：让一大群线程共享同一套控制逻辑，齐步走同一条指令。这就是下一章的主角 SIMT，也是"线程变笨了"的根源：单个 GPU 线程没有自己的大脑，它只是一大队士兵里喊着同一个口令迈步的一个。

你的 CPU 经验，哪些能用、哪些要改

好消息是，你不是从零开始。很多硬件常识在 GPU 上更重要，不是没用了：

你已经懂的 在 GPU 上
缓存局部性、访存比计算贵 更极端：GPU 算力过剩，绝大多数 kernel 卡在访存上，优化访存是主线
SIMD（一条指令处理多个数据） 演化成 SIMT（第 2 章）：写起来像标量，跑起来像向量
线程、同步、数据竞争 概念还在，但同步主要靠 barrier（第 8 章），锁几乎不用
内存层级（寄存器 / L1 / L2 / RAM） 层级更多也更显式：shared memory 要你亲手管（第 7 章）
分支预测让 if 几乎免费 反过来：一群线程走进不同分支会串行执行，即 warp 分歧（第 6 章）

需要换掉的直觉主要是三条，正好是本书前三部分要反复敲打的：

  1. 一个线程只干一点点活，而不是一大块（本章）。
  2. 一群线程齐步走同一条指令（SIMT，第 2 章），所以分支和访存模式的代价和 CPU 完全不同。
  3. 内存层级要你显式管理，数据放哪儿、怎么访问，往往比算法本身更影响速度（第 7 章起）。

动手：亲眼看看"少而强"对"多而弱"

光看不够。GPU 我们还没法在浏览器里跑，但"数据并行"的思维方式可以先用 Python 模拟出来。下面这段程序能在浏览器里直接运行——点右上角 ▶ 运行。它不真的用 GPU，而是把两种世界观并排写出来，让你看到：不管是"几个线程各扫一段"还是"每个元素一个线程"，算出来的结果完全一样，变的只是你怎么描述这件事。

Python 
# 用纯 Python 模拟两种并行世界观：结果相同，思维不同
N = 16
a = [i for i in range(N)]          # [0, 1, 2, ..., 15]
b = [i * 10 for i in range(N)]     # [0, 10, 20, ..., 150]

# ---------- CPU 世界观：4 个"线程"，每个处理一整段 ----------
def cpu_style(a, b, num_threads=4):
    c = [0] * len(a)
    chunk = len(a) // num_threads
    for t in range(num_threads):                 # 想象这 4 次是 4 个线程"同时"跑
        lo, hi = t * chunk, (t + 1) * chunk
        print(f"  线程 {t}：负责下标 {lo}..{hi-1} 这一整段")
        for i in range(lo, hi):                  # 每个线程内部还要循环
            c[i] = a[i] + b[i]
    return c

# ---------- GPU 世界观：每个元素配一个"线程"，各算一个 ----------
def gpu_style(a, b):
    c = [0] * len(a)
    def kernel(tid):                             # 这就是一个 GPU 线程做的全部事情
        c[tid] = a[tid] + b[tid]                 # 没有循环！只处理"我这一个"
    for tid in range(len(a)):                    # 想象这 16 次是 16 个线程"同时"跑
        kernel(tid)
    return c

print("CPU 世界观（4 个强壮线程）：")
r1 = cpu_style(a, b)
print("GPU 世界观（16 个弱小线程）：每个线程只做 c[tid] = a[tid] + b[tid]")
r2 = gpu_style(a, b)

print("\n结果一样吗？", r1 == r2)
print("结果：", r2)

运行结果：

CPU 世界观（4 个强壮线程）：
  线程 0：负责下标 0..3 这一整段
  线程 1：负责下标 4..7 这一整段
  线程 2：负责下标 8..11 这一整段
  线程 3：负责下标 12..15 这一整段
GPU 世界观（16 个弱小线程）：每个线程只做 c[tid] = a[tid] + b[tid]
结果一样吗？ True
结果： [0, 11, 22, 33, 44, 55, 66, 77, 88, 99, 110, 121, 132, 143, 154, 165]

注意两个 kernel 函数体的差别：CPU 版内部有个 for i in range(lo, hi) 的循环，GPU 版一行、没有循环——它只关心"我是 tid 号线程，我算 c[tid]"。这个"从循环体里抽出一次迭代，让它成为一个线程"的动作，就是把 CPU 代码搬上 GPU 的核心套路。请改一下代码：把 gpu_style 里的 kernel 改成计算 c[tid] = a[tid] * b[tid] + 1，再运行看看——你会发现，你已经在用 GPU 的思维写"kernel"了。

别急着把一切都塞进 GPU 数据并行的前提是大量互不依赖的活儿。如果任务只有几百个元素、或每一步都强依赖上一步的结果（比如递归下降的解析器），GPU 那几万个线程会大部分闲着，搬数据进出显存的开销还可能比省下的计算更多。GPU 快，是在"又多又独立"的活儿上快。判断一个任务适不适合 GPU，是比会写 kernel 更值钱的能力，第 16 章会给你一套定量的方法。

这本书怎么帮你完成切换

你不缺编程能力，缺的是并行的新心智模型。所以本书的策略始终是对照：先用一段你熟悉的 CPU 代码建立锚点，再看 GPU 怎么做、为什么这么做，并尽量把抽象机制画出来、跑出来。

全书的路线：

  1. 换心智模型（第 1–3 章）：从"几个线程"切到"成千上万个线程"，看清 SIMT 和 GPU 硬件长什么样。
  2. 核心编程模型（第 4–9 章，地基）：执行层级（Grid/Block/Thread 与 NDRange/Work-group/Work-item）、真正的调度单位 warp/sub-group、内存层级、同步，最后写出第一个完整的真 kernel。这一部分立住了，GPU 就不再神秘。
  3. 四大技术逐一展开（第 10–15 章）：CUDA、SYCL、Triton、CuTe DSL——你会发现它们是同一个模型的不同外衣。

搞清楚了"线程变便宜也变笨了"这一点，我们就可以进入下一章，去看那个让线程"变笨"的核心机制：一条指令怎么同时驱动一大群线程——SIMT。

这一章的要点

  • CPU 线程又贵又强，让少数线程各干一大块；GPU 线程又便宜又弱，启动成千上万个，每个只干一点点。这是数据并行思维。
  • 把 CPU 代码搬上 GPU 的核心套路：从循环体里抽出一次迭代，让它成为一个线程。
  • GPU 是吞吐优化的机器，赌的是带宽而非延迟，用海量并行掩盖访存等待——所以它必须同时挂着远超核心数的线程。
  • 你的硬件直觉大多更重要了（访存比计算贵），但有三条要改：一个线程只干一点、一群线程齐步走（SIMT）、内存层级要显式管理。
  • GPU 只在"又多又独立"的活儿上快；判断任务适不适合 GPU，比会写 kernel 更值钱。
  • 本书用对照 + 可视化 + 可运行的方式，帮你把 CPU 的心智模型平滑切换到 GPU。
