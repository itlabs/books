第 4 章 执行层级（一）：Grid、Block、Thread

这一章想让你带走什么

前三章你换好了心智模型：海量弱小的线程、齐步走的 SIMT、由 SM 拼成的硬件。这一章讲清你启动 kernel 时写的那行神秘符号 <<<grid, block>>> 到底是什么：GPU 把成千上万个线程组织成 Grid → Block → Thread 三层结构，而你在 kernel 里靠 threadIdx、blockIdx、blockDim、gridDim 四个内建变量算出"我是第几个线程"。

读完你能默写出那行几乎每个 CUDA 程序都有的咒语 int i = blockIdx.x * blockDim.x + threadIdx.x，说清它为什么长这样、为什么后面总跟一句 if (i < n)，还会跑一个浏览器里的模拟器，亲眼看到二维网格是怎么铺满一张图像的。

从"我要开几个线程"到"我要怎么组织线程"

在 CPU 上启动并行很直接：你想要几个线程，就 emplace_back 几次，数量大约等于核心数。线程之间是平的——8 个线程就是 8 个独立的个体，没有分组、没有层级。

GPU 不一样。你要一次启动几十万个线程，硬件不可能把它们当成一盘散沙来调度。所以 CUDA 强制你把线程组织成两层：

  • 先把线程打包成一个个 block（线程块），每个 block 里有固定数量的线程；
  • 再把这些 block 排成一个 grid（网格）。

于是启动一个 kernel，你不是说"给我 100 万个线程"，而是说"给我一个由 blocks 个 block 组成的 grid，每个 block 里有 threads 个线程"。这就是那行 <<<grid, block>>> 的含义——CUDA 里叫执行配置（execution configuration）：

CUDA C++
// 启动配置：<<<block 的数量, 每个 block 里的线程数>>>
add<<<blocks, threads>>>(...);
//     ^grid      ^block
// 总线程数 = blocks * threads
CPU：你挑线程数 ≈ 核心数
C++
// 线程是平的、少而贵
int T = std::thread::hardware_concurrency(); // 8
std::vector<std::thread> pool;
for (int t = 0; t < T; ++t)
    pool.emplace_back(worker, t);
// 你决定的是"几个"，通常刚好压满核心
GPU：你挑 grid/block 维度，海量超订
CUDA C++
// 线程分两层、多而贱，随手超订
int threads = 256;                 // 每 block 线程数
int blocks  = (n + 255) / 256;     // block 数量
add<<<blocks, threads>>>(...);
// 你决定的是"怎么切成 block×thread"
// 总数远超核心数，硬件排队轮流上
核心区别：CPU 你定"数量"，GPU 你定"形状" CPU 上线程数量约等于核心数，多了只会增加调度开销。GPU 上你几乎总是大幅超订（oversubscribe）：启动的线程数远超物理核心，硬件靠 warp 调度器让它们排队轮流上、用别人的计算掩盖等内存的空档（第 3 章）。所以在 GPU 上你要想的不是"开几个"，而是"把这堆线程切成什么形状的 block × grid"。

三层结构：grid 装 block，block 装 thread

把这两层画出来，就是本章最该刻进脑子里的一张图。一个 grid 是若干 block 的数组，一个 block 是若干 thread 的数组：

图 4-1　Grid 是 block 的数组，block 是 thread 的数组。每个线程用 blockIdx.x × blockDim.x + threadIdx.x 把"我在第几个 block 的第几个"换算成一条拉直的全局下标

四个内建变量：kernel 里"我是谁"

kernel 是同一份代码被几十万个线程各跑一遍。那每个线程怎么知道自己该处理哪个数据？靠 CUDA 在 kernel 内部自动提供的四个内建变量。它们不用你声明，进 kernel 就能读：

变量 含义 类比
threadIdx 我在自己 block 里排第几（从 0 起） 我在班里的学号
blockIdx 我这个 block 在 grid 里排第几 我们班在年级里的班号
blockDim 一个 block 有多少线程 每班多少人
gridDim grid 里有多少 block 年级有多少班

每个变量都有 .x、.y、.z 三个分量（下一节讲多维）。一维情况下只用 .x。要把"第 blockIdx.x 班的第 threadIdx.x 号"换算成"全年级第几号"，公式就是班号 × 每班人数 + 学号：

CUDA C++
// 几乎每个 CUDA kernel 开头都有这一行
int i = blockIdx.x * blockDim.x + threadIdx.x;
//      \_____第几个 block_____/   \__block 内序号__/

这条式子就是把二层结构"拉直"成一维全局下标的标准套路。block 0 的线程占 i = 0 .. blockDim-1，block 1 接着占 blockDim .. 2*blockDim-1，无缝拼接，正好铺满 gridDim * blockDim 个连续下标。

为什么非要分两层？block 是硬件的分界线

你可能会想：既然最后都要拉直成一维 i，那 CUDA 干脆让我写"我要 100 万个线程"不就行了，为什么逼我拆成 block × thread 两层？

答案在第 3 章那张硬件图里：一个 block 会被整体调度到一个 SM 上执行，绝不跨 SM。这个约束带来两个后果，正是分层的全部意义：

  • block 内的线程能廉价协作：它们同在一个 SM 上，可以共享那块片上 shared memory，还能用 __syncthreads() 在 block 内设 barrier 互相等待（第 7、8 章）。这是 GPU 上线程协作的主要方式。
  • 不同 block 之间几乎无法协作：它们可能被分到不同 SM、甚至不同时间执行，硬件不保证谁先谁后。block 之间没有 barrier、不能安全地互相等待。
为什么这个设计是必要的，而非碍事 正因为 block 之间互不依赖、可以任意顺序执行，GPU 才能自由伸缩：同一个 kernel，在有 20 个 SM 的入门卡上就少并行几个 block、在有 130 个 SM 的旗舰卡上就多并行几个，代码一字不改。block 是"一份能独立完成、可被任意 SM 领走的工作包"。你把协作限制在 block 内，换来的是跨硬件的可扩展性——这正是 GPU 编程模型最聪明的取舍之一。
陷阱：别指望用全局内存在 block 之间"手动同步" 新手常想：我在全局内存里放个标志位，block A 写、block B 轮询等待，不就同步了？不行——如果 block B 先被调度、把 SM 占满，而 block A 还在排队根本没开始，B 就会永远等下去，直接死锁。跨 block 的同步只有一个可靠办法：把 kernel 拆成两次启动，两次启动之间天然是全局同步点。

一维不够用：dim3 与二维网格

到目前为止我们只用 .x。但很多数据天生是二维的——图像、矩阵。硬要用一维下标去访问 image[y * width + x] 也能写，但 CUDA 允许你直接用二维的 grid 和 block，让线程坐标和像素坐标自然对应。

执行配置里的两个参数其实是 dim3 类型——一个含 .x/.y/.z 的三元组。写 <<<blocks, threads>>> 时如果传的是整数，等价于只设了 .x（.y=.z=1）。要开二维，就显式构造 dim3：

CUDA C++
// 处理一张 width × height 的图像：让线程排成二维
dim3 block(16, 16);                    // 每个 block 是 16×16 = 256 个线程
dim3 grid((width  + 15) / 16,          // x 方向需要多少个 block（向上取整）
          (height + 15) / 16);         // y 方向需要多少个 block
process<<<grid, block>>>(img, width, height);

kernel 里就同时用 .x 和 .y 各算一维，得到像素的 (x, y)：

CUDA C++
__global__ void process(unsigned char* img, int width, int height) {
    int x = blockIdx.x * blockDim.x + threadIdx.x;   // 全局列
    int y = blockIdx.y * blockDim.y + threadIdx.y;   // 全局行
    if (x < width && y < height) {                   // 二维也要边界检查
        int idx = y * width + x;                     // 二维坐标转一维偏移
        img[idx] = 255 - img[idx];                   // 例：反色
    }
}
二维只是"方便"，不是"更快" 二维 grid/block 不会凭空提速，它只是让代码更贴合数据的形状：线程的 (x, y) 直接就是像素坐标，索引算术更清晰、越界判断更自然。底层硬件依然把 block 内的线程按一维顺序切成 warp（先按 x、再 y、再 z），所以第 6、7 章讲访存时，你仍要关心"哪些线程落在同一个 warp"。三维（.z）同理，常用于体数据/三维网格。

边界检查：if (i < n) 不是可选项

注意上面每个 kernel 都有一句 if (i < n)（或二维的 if (x < width && y < height)）。这不是防御性编程的洁癖，而是几乎必然发生的情况：

block 里的线程数通常固定（比如 256），而你的数据量 n 很少正好是 256 的整数倍。于是 blocks = (n + 255) / 256 向上取整后，启动的线程总数 blocks * 256 会略多于 n。多出来的那些线程算出的 i 会 ≥ n，如果不拦住，它们就会去写 data[i] 这个越界地址——轻则结果错乱，重则非法访存崩溃。

CUDA C++
__global__ void add(float* c, const float* a, const float* b, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n)                 // 挡住超出数据范围的"多余线程"
        c[i] = a[i] + b[i];
}

这就引出下一个问题：block 大小该选多少？

block 大小为什么常取 32 的倍数

你会发现示例里 block 大小总是 128、256、512 这类数——都是 32 的倍数。原因埋在第 2 章：硬件把 block 内的线程按 32 个一组切成 warp（线程束），warp 才是真正的调度和执行单位。

如果你把 block 设成比如 100 个线程，硬件仍会切成 4 个 warp（ceil(100/32)=4，共 128 个 warp 槽位），最后一个 warp 里只有 4 个线程干活、28 个"空转",白白浪费了近一个 warp 的执行资源。取 32 的倍数就没有这种零头浪费。

先记结论，第 6 章讲透 block 大小取 32 的倍数（常用 128 / 256），因为硬件以 32 线程的 warp 为单位调度，非整数倍会在最后一个 warp 里浪费槽位。至于该在 128、256、512 里选哪个，取决于寄存器用量、shared memory 用量和 occupancy——那是第 16 章性能模型的内容。现在你只需养成"block 大小填 32 的倍数"这个肌肉记忆。

动手：亲手算全局索引，看二维网格铺满图像

下面这个模拟器能在浏览器里直接跑——点 ▶ 运行。它不碰 GPU，而是用纯 Python 忠实复刻 CUDA 的索引算术：先展示一维下 blockIdx / threadIdx 怎么拼成全局 i，再展示二维下 2×2 个 block 怎么各自认领一张 6×4 图像的一块，最后演示"多余线程被 if (i < n) 拦下"。

Python 
# 模拟 CUDA 的全局索引：i = blockIdx.x * blockDim.x + threadIdx.x
def global_index(block_idx, block_dim, thread_idx):
    return block_idx * block_dim + thread_idx

blockDim = 4   # 每个 block 有 4 个线程
gridDim  = 2   # grid 里有 2 个 block

print("=== 一维：grid 有 2 个 block，每个 block 4 个线程 ===")
for b in range(gridDim):
    for t in range(blockDim):
        i = global_index(b, blockDim, t)
        print(f"  blockIdx.x={b} threadIdx.x={t} -> i = {b}*{blockDim}+{t} = {i}")

print("\n=== 二维：用 2x2 个 block 覆盖一张 6x4 的图像 ===")
W, H = 6, 4
bdx, bdy = 3, 2    # 每个 block 是 3x2 个线程（blockDim.x=3, blockDim.y=2）
gdx, gdy = 2, 2    # grid 是 2x2 个 block（gridDim.x=2, gridDim.y=2）
owner = [[None] * W for _ in range(H)]
for by in range(gdy):
    for bx in range(gdx):
        for ty in range(bdy):
            for tx in range(bdx):
                x = bx * bdx + tx          # 全局列 = blockIdx.x*blockDim.x + threadIdx.x
                y = by * bdy + ty          # 全局行 = blockIdx.y*blockDim.y + threadIdx.y
                owner[y][x] = f"b({bx},{by})"
print("每个像素由哪个 block 负责（block 坐标 bx,by）：")
for y in range(H):
    print("  " + "  ".join(owner[y][x] for x in range(W)))

print("\n=== 边界检查：n=10，但 3 个 block x 4 线程 = 12 个线程 ===")
n = 10
bd = 4
gd = (n + bd - 1) // bd          # = 3
for b in range(gd):
    for t in range(bd):
        i = b * bd + t
        if i < n:
            print(f"  i={i:2d} -> 处理 data[{i}]")
        else:
            print(f"  i={i:2d} -> if(i<n) 拦下，什么都不做")

看二维那张表：每个 block 认领一个 3×2 的矩形区域，四个 block 拼起来正好盖满 6×4 的图像——这就是 GPU 处理图像的标准分工。请动手改：把 blockDim 从 4 改成 3，再看一维那段输出，你会发现 i 依然是 0,1,2,…7 连续无缝——改 block 大小不改变全局下标的连续性，这正是那条索引公式的威力。再把 n 从 10 改成 12，看边界检查那段，多余线程就消失了。

动手：一个完整可编译的 CUDA 程序

下面是一个完整、可编译的 .cu 程序：kernel 把每个线程自己的全局索引 i 写进数组，host 端做完整的 cudaMalloc → 启动 → cudaMemcpy 拷回 → cudaFree，最后打印头尾几个元素验证。点代码块上的 🔗 在 Compiler Explorer 打开 可以看 nvcc 生成的 PTX；本地有 GPU 的话按下面命令编译运行。

CUDA C++ 🔗 在 Compiler Explorer 打开（看 PTX）
#include <cstdio>
#include <cuda_runtime.h>

// 每个线程把自己的全局索引写进 out[i]
__global__ void write_index(int* out, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;  // 我是全局第几个线程
    if (i < n)                                       // 挡住多余线程
        out[i] = i;
}

int main() {
    const int n = 1000;                 // 故意不是 256 的整数倍
    const int bytes = n * sizeof(int);

    int* h_out = (int*)malloc(bytes);   // host 端缓冲

    int* d_out = nullptr;               // device 端缓冲
    cudaMalloc(&d_out, bytes);

    int threads = 256;                              // block 大小取 32 的倍数
    int blocks  = (n + threads - 1) / threads;      // 向上取整 = 4
    printf("启动配置：%d 个 block × %d 线程 = %d 个线程，覆盖 n=%d\n",
           blocks, threads, blocks * threads, n);

    write_index<<<blocks, threads>>>(d_out, n);     // <<<grid, block>>>
    cudaDeviceSynchronize();                        // 等 kernel 跑完

    cudaMemcpy(h_out, d_out, bytes, cudaMemcpyDeviceToHost);  // 拷回

    printf("前 5 个： %d %d %d %d %d\n",
           h_out[0], h_out[1], h_out[2], h_out[3], h_out[4]);
    printf("后 3 个： %d %d %d\n",
           h_out[n - 3], h_out[n - 2], h_out[n - 1]);

    cudaFree(d_out);                    // 释放显存
    free(h_out);                        // 释放主存
    return 0;
}
本地运行：nvcc -std=c++20 -O3 -arch=sm_90 main.cu -o app && ./app
终端
nvcc ch04_write_index.cu -o ch04_write_index && ./ch04_write_index

跑起来 h_out[i] 应当正好等于 i（前 5 个是 0 1 2 3 4，后 3 个是 997 998 999）。这验证了一件事：那 4 × 256 = 1024 个线程里，前 1000 个各自写对了自己的格子，多出来的 24 个被 if (i < n) 干净地挡在了外面。试着把 write_index 里的 out[i] = i 改成 out[i] = blockIdx.x，重新编译运行——你会看到输出变成一段段相同的班号（前 256 个是 0，接着 256 个是 1……），直观地看出"谁属于哪个 block"。

小结与下一步

你现在掌握了 CUDA 的软件执行层级：怎么用 <<<grid, block>>> 组织海量线程，怎么用四个内建变量在 kernel 里定位自己，以及为什么要分成 block、为什么 block 大小取 32 的倍数、为什么每个 kernel 都得带边界检查。

这套 Grid/Block/Thread 词汇是 NVIDIA CUDA 的说法。下一章我们看同一套概念在 SYCL/OpenCL 里的另一套名字——NDRange、work-group、work-item——它们和本章的概念几乎一一对应，但在"你到底指定什么数字"这件事上有一个极易踩坑的差别。理解了本章，你会发现下一章大半是"换个词汇表"。

这一章的要点

  • 启动 kernel 用 <<<grid, block>>>（执行配置）：GPU 强制你把线程组织成 grid → block → thread 两层结构，总线程数 = block 数 × 每 block 线程数，且几乎总是大幅超订。
  • kernel 里靠四个内建变量定位自己：threadIdx（block 内序号）、blockIdx（block 号）、blockDim（每 block 线程数）、gridDim（block 数）。
  • 把二层拉直成一维全局下标的标准公式：int i = blockIdx.x * blockDim.x + threadIdx.x。
  • 分两层是因为一个 block 整体落在一个 SM 上：block 内线程能用 shared memory 和 __syncthreads() 廉价协作，block 之间几乎无法协作——换来的是跨硬件的自由伸缩。
  • 用 dim3 可开二维/三维 grid/block，让线程坐标直接对应像素/矩阵坐标；这是"方便"而非"更快"。
  • 边界检查 if (i < n) 不可省：向上取整导致的多余线程会越界写。
  • block 大小取 32 的倍数（硬件按 32 线程的 warp 调度，第 6 章）。
