第 9 章 你的第一个真 kernel：向量加法的完整生命周期

这一章想让你带走什么

第 1 章给你看过 GPU 版向量加法的 kernel，但把"分配显存、拷贝数据"那部分推给了第 9 章。第 3 章讲硬件时又说"一次 kernel 启动，host 先拷数据、再启动、再拷回"，同样留到第 9 章。现在，欠下的债一次还清。

这一章把一个真 CUDA 程序从头到尾走一遍：host 分配、cudaMalloc、拷贝上卡、选 grid/block、启动、同步与错误检查、拷回、验证、释放——八个步骤，一个不落。你会拿到一份正确、带错误检查的向量加法参考实现（本书后面都以它为基准），再看同一件事用 SYCL 的 USM 怎么写得更省事，最后用一个模拟器算清"这趟到底值不值得跑一趟 GPU"。

一次 kernel 的完整生命周期：八个步骤

CPU 上算 c = a + b，你 malloc 三个数组、写个循环、完事。GPU 上多了一层：数据住在 host 内存，计算发生在 device 显存，中间隔着一条 PCIe 总线。所以一个真 kernel 的生命周期是这样八步：

  1. host 端分配并初始化数据（malloc / std::vector），这是你要计算的输入。
  2. device 端分配显存（cudaMalloc），给输入输出各开一块。
  3. H2D 拷贝（cudaMemcpy ... HostToDevice）：把输入从 host 内存搬进显存。
  4. 选 grid / block 维度：block 取 32 的倍数，grid 用向上取整凑够线程。
  5. 启动 kernel（<<<blocks, threads>>>）：注意这是异步的，CPU 发完命令立刻返回。
  6. 同步 + 错误检查（cudaDeviceSynchronize + cudaGetLastError）：等 GPU 真的算完，并检查有没有出错。
  7. D2H 拷贝（cudaMemcpy ... DeviceToHost）：把结果从显存搬回 host。
  8. 验证 + 释放（cudaFree / free）：核对结果，归还所有内存。
图 9-1　八步生命周期的时间线：数据先 H2D 上卡、GPU 算、再 D2H 回来。启动异步、同步是必需的一步；总时间由三段拼成，传输两头往往比中间的计算还贵

第 4、5 步值得单独说。block 维度取 32 的倍数，是因为第 6 章讲过的：硬件按 warp（32 线程）调度，block 不是 32 的倍数会浪费最后一个 warp 的部分 lane。256 是常用的稳妥值。grid 用向上取整——因为 N 通常不能被 block 大小整除，你得多开一点线程盖住余数，再靠 kernel 里的 if (i < n) 边界检查让多出来的线程什么都不做：

CUDA C++
int threads = 256;
int blocks  = (N + threads - 1) / threads;   // 向上取整：ceil(N / threads)
// 例：N=1000, threads=256 → blocks=4 → 共启动 1024 个线程
// 多出的 24 个线程会被 kernel 里的 if (i < n) 挡在门外

错误检查：GPU 编程里最容易被省掉、也最坑人的一步

CPU 上函数出错通常立刻抛异常或返回错误码，你很难忽略。GPU 不一样，有两个陷阱：

  • kernel 启动是异步的：add<<<...>>>() 这行只是把任务丢进队列就返回了，此刻 kernel 可能还没开始跑。所以 kernel 内部的运行时错误（比如越界访问）不会在这一行报出来。
  • 错误是"粘"在后面的：CUDA 的 runtime API 返回错误码（cudaError_t），但很多人从不检查。一个错误可能要等到下一个 CUDA 调用才浮现，报错位置和真正的出错点对不上，极难排查。

所以专业的 CUDA 代码都会包一个 CHECK 宏，对每个 CUDA 调用检查返回值；kernel 启动后则用 cudaGetLastError()（抓启动配置错误，如 block 过大）加 cudaDeviceSynchronize()（等 kernel 跑完并抓运行期错误）。这不是洁癖，是能不能调试的分水岭。

不查错的 GPU 程序，出了问题几乎无法定位 最典型的翻车：kernel 里数组越界，程序不崩、也不报错，只是结果错得莫名其妙；或者 cudaMalloc 因显存不足失败返回了错误码，你没查，后面拿着无效指针继续跑，直到某个不相干的地方段错误。养成习惯：每个 CUDA 调用都用 CHECK 宏包起来，kernel 启动后紧跟一次 cudaGetLastError() + 同步。这一步在本书所有 CUDA 示例里都会出现。

动手：完整、带错误检查的向量加法（本书的基准实现）

下面是那份完整、正确、带错误检查的参考实现——本书后面每次说"回到第 9 章那个向量加法"，指的就是它。八个步骤在注释里一一标出。

CUDA C++ 🔗 在 Compiler Explorer 打开（看 PTX）
#include <cstdio>
#include <cstdlib>
#include <cmath>
#include <cuda_runtime.h>

// CHECK 宏：包住每个 CUDA runtime 调用，出错就打印位置并退出
#define CHECK(call)                                                        \
    do {                                                                   \
        cudaError_t err = (call);                                          \
        if (err != cudaSuccess) {                                          \
            fprintf(stderr, "CUDA 错误 %s:%d: %s\n", __FILE__, __LINE__,   \
                    cudaGetErrorString(err));                              \
            exit(EXIT_FAILURE);                                            \
        }                                                                  \
    } while (0)

__global__ void vec_add(const float* a, const float* b, float* c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;   // 全局线程下标
    if (i < n) c[i] = a[i] + b[i];                   // 边界检查：多出的线程不越界
}

int main() {
    const int N     = 1 << 20;                       // 1,048,576 个元素
    const size_t sz = N * sizeof(float);

    // 步骤 1：host 端分配并初始化
    float* h_a = (float*)malloc(sz);
    float* h_b = (float*)malloc(sz);
    float* h_c = (float*)malloc(sz);
    for (int i = 0; i < N; ++i) { h_a[i] = 1.0f; h_b[i] = 2.0f; }

    // 步骤 2：device 端分配显存
    float *d_a, *d_b, *d_c;
    CHECK(cudaMalloc(&d_a, sz));
    CHECK(cudaMalloc(&d_b, sz));
    CHECK(cudaMalloc(&d_c, sz));

    // 步骤 3：H2D 拷贝，把输入搬上显卡
    CHECK(cudaMemcpy(d_a, h_a, sz, cudaMemcpyHostToDevice));
    CHECK(cudaMemcpy(d_b, h_b, sz, cudaMemcpyHostToDevice));

    // 步骤 4：选 grid / block 维度（block 取 32 的倍数，grid 向上取整）
    int threads = 256;
    int blocks  = (N + threads - 1) / threads;

    // 步骤 5：启动 kernel（异步！这行发完命令就返回）
    vec_add<<<blocks, threads>>>(d_a, d_b, d_c, N);

    // 步骤 6：错误检查 + 同步
    CHECK(cudaGetLastError());        // 抓启动配置错误（如 block 过大）
    CHECK(cudaDeviceSynchronize());   // 等 kernel 真的跑完，并抓运行期错误

    // 步骤 7：D2H 拷贝，把结果搬回 host
    CHECK(cudaMemcpy(h_c, d_c, sz, cudaMemcpyDeviceToHost));

    // 步骤 8：验证 + 释放
    int errors = 0;
    for (int i = 0; i < N; ++i)
        if (fabsf(h_c[i] - 3.0f) > 1e-5f) ++errors;
    printf("N = %d，结果 %s（h_c[0] = %.1f，期望 3.0）\n",
           N, errors == 0 ? "全部正确" : "有错误", h_c[0]);

    CHECK(cudaFree(d_a)); CHECK(cudaFree(d_b)); CHECK(cudaFree(d_c));
    free(h_a); free(h_b); free(h_c);
    return 0;
}
本地运行：nvcc -std=c++20 -O3 -arch=sm_90 main.cu -o app && ./app

点开代码块的 🔗 在 Compiler Explorer 打开，你能看到 vec_add 编成的 PTX；本地有 NVIDIA 卡时，nvcc vecadd.cu -o vecadd && ./vecadd 会打印"全部正确"。你现在拥有了写任何简单 kernel 的完整骨架——把 vec_add 换成别的逐元素运算（c[i] = a[i]*b[i] + 1.0f），八步流程原封不动。

这份骨架就是 GPU 编程的"Hello World" 八个步骤 + CHECK 宏 + 边界检查 + 异步启动后的同步，构成了几乎所有 CUDA 程序的外壳。真实的复杂 kernel（矩阵乘、attention）在计算部分千变万化，但外面这层 host 端的"分配—拷贝—启动—同步—拷回—释放"骨架基本不变。把它记熟，你就再也不会对着一个新 kernel 不知从何写起。

同一件事，SYCL 用 USM 怎么写

CUDA 的显式 cudaMemcpy 很清晰，但也啰嗦：每块数据都要手动分配 device 版、手动拷来拷去。SYCL（第 5、12 章的主角）提供了一种更省事的选择——USM（统一共享内存，Unified Shared Memory）：用 malloc_shared 分配的指针，host 和 device 都能直接访问，数据的搬运交给运行时按需自动完成，你的代码里没有一句显式拷贝。

CUDA：显式 cudaMemcpy
CUDA C++
cudaMalloc(&d_a, sz);          // device 专用内存
cudaMalloc(&d_c, sz);
cudaMemcpy(d_a, h_a, sz,       // 手动 H2D
           cudaMemcpyHostToDevice);
vec_add<<<blocks,threads>>>(d_a,d_b,d_c,N);
cudaDeviceSynchronize();
cudaMemcpy(h_c, d_c, sz,       // 手动 D2H
           cudaMemcpyDeviceToHost);
SYCL：USM 隐式搬运
SYCL / DPC++
float* a = malloc_shared<float>(N, q);  // host/device 都能访问
float* c = malloc_shared<float>(N, q);
// 直接在 host 上写 a[i]，直接读 c[i]
q.parallel_for(N, [=](id<1> i){
    c[i] = a[i] + b[i];
}).wait();                     // 没有一句显式 memcpy

下面是完整、可编译的 SYCL 版本。注意它有多干净：一个 queue、malloc_shared 分配、一个 parallel_for、wait() 同步，就完事了。

SYCL / DPC++ 🔗 在 Compiler Explorer 打开
#include <sycl/sycl.hpp>
#include <cstdio>
#include <cmath>

int main() {
    namespace sy = sycl;
    const int N = 1 << 20;                       // 1,048,576 个元素

    sy::queue q;                                 // 默认选一个设备（GPU 优先）
    printf("运行设备：%s\n",
           q.get_device().get_info<sy::info::device::name>().c_str());

    // USM：host 和 device 都能直接访问的共享内存
    float* a = sy::malloc_shared<float>(N, q);
    float* b = sy::malloc_shared<float>(N, q);
    float* c = sy::malloc_shared<float>(N, q);

    for (int i = 0; i < N; ++i) { a[i] = 1.0f; b[i] = 2.0f; }  // 直接在 host 写

    // 启动并行计算；数据搬运由运行时按需完成，无需显式 memcpy
    q.parallel_for(sy::range<1>(N), [=](sy::id<1> idx) {
        int i = idx[0];
        c[i] = a[i] + b[i];
    }).wait();                                   // 等 kernel 跑完

    int errors = 0;
    for (int i = 0; i < N; ++i)
        if (std::fabs(c[i] - 3.0f) > 1e-5f) ++errors;
    printf("N = %d，结果 %s（c[0] = %.1f，期望 3.0）\n",
           N, errors == 0 ? "全部正确" : "有错误", c[0]);

    sy::free(a, q); sy::free(b, q); sy::free(c, q);
    return 0;
}
本地运行：icpx -fsycl -O2 -std=c++20 main.cpp -o app && ./app # 需要 Intel oneAPI

用 icpx -fsycl vecadd.cpp -o vecadd 编译（oneAPI 环境，见附录 A）。USM 让代码短了一半，但别被迷惑：数据该搬还得搬，只是搬运的时机和方式交给了运行时。省的是代码量和出错机会，省不掉的是 PCIe 上真实的传输时间——而那，正是决定"值不值得上 GPU"的关键。

USM 不是魔法，它只是把拷贝藏了起来 malloc_shared 的数据在 host 访问时可能在 host 侧，device 访问时被迁移到显存——迁移仍然走 PCIe，仍然要花时间。USM 有几种类型（malloc_device 只在显存、malloc_host 只在 host、malloc_shared 可迁移），第 12 章会细讲。这里的要点是：显式（CUDA）和隐式（USM）只是谁来写拷贝代码的区别，物理上的数据搬运和它的成本，两边完全一样。

动手：这一趟到底值不值得跑 GPU

第 3 章说过：只为做一次向量加法就把数据搬上 GPU 再搬回，"光搬运的时间可能就远超省下的计算"。现在你会写完整流程了，我们用一个模拟器把这句话算成数字。它用第 3 章的显存带宽（3 TB/s）、一条真实量级的 PCIe 带宽（16 GB/s），把总时间拆成 H2D + compute + D2H 三段，看看谁是大头。

Python 
# 建模向量加法 c=a+b 的端到端时间：H2D + compute + D2H
pcie_bw   = 16e9      # PCIe Gen4 x16 约 16 GB/s（CPU 内存 ↔ 显存）
device_bw = 3.0e12    # 显存带宽 3 TB/s（沿用第 3 章的量级）
copy_latency = 5e-6   # 每次 cudaMemcpy 的固定启动开销，约 5 微秒

def model(N):
    # H2D 搬 a、b 两个数组；D2H 搬回 c 一个数组；每个 float 4 字节
    h2d = copy_latency + (2 * N * 4) / pcie_bw
    d2h = copy_latency + (1 * N * 4) / pcie_bw
    # 设备上访存受限：读 a、读 b、写 c 共 3*N*4 字节，按显存带宽算
    compute = (3 * N * 4) / device_bw
    total = h2d + compute + d2h
    return h2d, compute, d2h, total

for N in [1_000, 1_000_000, 100_000_000]:
    h2d, compute, d2h, total = model(N)
    transfer = h2d + d2h
    print(f"N = {N} 个元素：")
    print(f"  H2D 拷贝 a,b：{h2d*1e6:8.3f} us")
    print(f"  设备上计算  ：{compute*1e6:8.3f} us")
    print(f"  D2H 拷回 c  ：{d2h*1e6:8.3f} us")
    print(f"  合计        ：{total*1e6:8.3f} us，其中传输占 {transfer/total*100:.2f}%")
    print()

print("结论：不管 N 大小，H2D+D2H 都占了约 99.5% 的时间——")
print("因为 PCIe(16 GB/s) 比显存(3 TB/s) 慢近 200 倍，而向量加法搬的字节和算的次数一样多。")
print("所以只为做一次向量加法就把数据搬上 GPU 再搬回，几乎永远不划算（呼应第 3 章）。")
print("GPU 划算的前提：数据搬上去后被很多个 kernel 反复使用，把传输成本摊薄。")

运行结果：

N = 1000 个元素：
  H2D 拷贝 a,b：   5.500 us
  设备上计算  ：   0.004 us
  D2H 拷回 c  ：   5.250 us
  合计        ：  10.754 us，其中传输占 99.96%

N = 1000000 个元素：
  H2D 拷贝 a,b： 505.000 us
  设备上计算  ：   4.000 us
  D2H 拷回 c  ： 255.000 us
  合计        ： 764.000 us，其中传输占 99.48%

N = 100000000 个元素：
  H2D 拷贝 a,b：50005.000 us
  设备上计算  ： 400.000 us
  D2H 拷回 c  ：25005.000 us
  合计        ：75410.000 us，其中传输占 99.47%

结论：不管 N 大小，H2D+D2H 都占了约 99.5% 的时间——
因为 PCIe(16 GB/s) 比显存(3 TB/s) 慢近 200 倍，而向量加法搬的字节和算的次数一样多。
所以只为做一次向量加法就把数据搬上 GPU 再搬回，几乎永远不划算（呼应第 3 章）。
GPU 划算的前提：数据搬上去后被很多个 kernel 反复使用，把传输成本摊薄。

这个结果很扎心，也很重要：向量加法这种算术强度极低的 kernel（第 3 章算过它是 0.083 次/字节），几乎 100% 的时间花在 PCIe 传输上，GPU 强大的算力完全用不上。改一行看看：把 compute 那行的 device_bw 换成 peak_flops = 60e12 并用 (1 * N) / peak_flops（按一次加法一个 FLOP 算），你会发现计算时间更小——因为向量加法连显存带宽都喂不饱，更别说算力了。这就是"是否值得上 GPU"的定量答案：只有当数据搬上去后被反复计算（把 H2D/D2H 成本摊到很多个 kernel 上），或者本身就是算力密集的任务，GPU 才划算。第 14 章的算子融合、第 16 章的 Roofline 模型，本质上都在对抗这个"传输/带宽成本"。

小结与下一步

到这里，本书的"地基"部分（第 4–9 章）完整落成。你从软件的执行层级出发，认识了真正的调度单位 warp、显式的内存层级、三种协作机制，现在把它们全部串成了一个能编译、能跑、带错误检查的真程序。你已经会写 GPU kernel 了——这不是夸张，逐元素类的 kernel 你现在完全能独立完成。

从下一章起进入第三部分：把这个模型用四种主流技术分别深入。第 10 章先啃最贴近硬件的 CUDA——同样的向量加法你已经写过，我们会在它之上展开 CUDA 的 stream、事件、更细的内存 API 和多 kernel 协作，看清这层"最原始的外衣"到底给了你多少控制力。

这一章的要点

  • 一个真 kernel 的生命周期是八步：host 分配 → cudaMalloc → H2D 拷贝 → 选 grid/block → 启动 → 同步+错误检查 → D2H 拷贝 → 验证+释放。
  • block 取 32 的倍数（对齐 warp），grid 用向上取整 (N+threads-1)/threads 凑够线程，再靠 kernel 里的 if (i < n) 挡住多出的线程。
  • kernel 启动是异步的，错误可能"粘"在后面的调用上——必须用 CHECK 宏包住每个 CUDA 调用，启动后紧跟 cudaGetLastError() + cudaDeviceSynchronize()，否则几乎无法调试。
  • SYCL 的 USM（malloc_shared）让 host/device 共享指针、免写显式拷贝，代码更短；但物理上的 PCIe 传输和成本一分不少，只是被运行时藏了起来。
  • 用时间模型算清"值不值得上 GPU"：向量加法约 99.5% 的时间耗在 PCIe 传输上，算力用不上。GPU 只有在"数据反复使用摊薄传输"或"算力密集"时才划算——这正是后续融合与 Roofline 章节的出发点。
