第 7 章 内存层级：global、shared、register 与合并访问

这一章想让你带走什么

第 3 章说过：多数 kernel 的瓶颈不是算得慢，而是数据喂不上来。这一章把 GPU 的内存层级摊开讲清楚：每个线程私有的寄存器（register）最快、每个 block 共享的shared memory 是你亲手管理的片上暂存区、片外的全局内存（global memory）又大又慢。

更重要的是那个能让性能差出几十倍的核心概念——合并访问（memory coalescing）：当一个 warp 的 32 条 lane 访问连续地址时，硬件把它们合并成一次宽传输；访问跨步/散乱时，则退化成几十次小传输。读完你会跑一个模拟器数清楚每种访存模式要几次传输，还会读到一个完整可编译的 CUDA 程序，第一次亲手用上 shared memory。

三级内存：越快越小，越大越慢

CPU 也有内存层级（寄存器 → L1 → L2 → L3 → 主存），但那套层级对你是透明的：你只管访问变量，硬件自动决定放哪级缓存。GPU 的层级不但更陡，还有一部分要你亲手管理。从快到慢，三个你必须区分的层级：

  • 寄存器（register）：每个线程私有，就在计算核心旁边，最快（0 延迟量级）。你 kernel 里的局部变量（int i、float sum）默认就住在寄存器里。每个线程能用的寄存器有限——用超了会"溢出"到慢的本地内存。
  • shared memory（共享内存）：每个 block 内共享，物理上就是第 3 章说的那块片上 L1 高速内存，比全局内存快一两个数量级。关键在于：它由你的程序显式管理——你决定往里放什么、什么时候放。它是这一章的主角。
  • 全局内存（global memory）：就是你买卡看的"24GB 显存"，片外 DRAM，又大又慢（一次访问几百个时钟周期）。所有线程都能读写，是 host 和 device 之间搬数据的落脚点。

此外还有两个你会偶尔碰到的：常量内存（constant memory）——小、只读、带专门缓存，适合放所有线程都读同一个值的参数；L2 缓存——全片共享，介于 SM 和全局内存之间，自动工作、不用你管。

图 7-1　GPU 内存金字塔。寄存器/shared 在 SM 片上、极快极小；全局内存在片外、又大又慢。速度和容量差着一两个数量级，作用域也各不相同

shared memory：你自己管的缓存

CPU 的缓存对你是自动、透明的：你访问一个数组，硬件默默把它周围的数据拉进 L1，下次命中就快。你从不写"把这块数据放进 L1"的代码——你没法这么写。

GPU 的 shared memory 恰恰相反：它是一块由你显式管理的片上高速内存。你在 kernel 里用 __shared__ 声明一块，然后亲手把全局内存里的数据搬进去、用完再说。硬件不会替你决定放什么。

CPU 缓存：自动、透明
C++
// 你只管访问，硬件自动缓存
for (int i = 0; i < n; ++i)
    sum += a[i];   // a[] 被自动拉进 L1
// 你无法、也不需要控制"放什么进缓存"
// 命中与否由硬件的替换策略决定
GPU shared memory：显式、手动
CUDA C++
// 你亲手声明、亲手搬、亲手同步
__shared__ float tile[256];
tile[threadIdx.x] = a[globalIdx]; // 你决定放什么
__syncthreads();                  // 等全组搬完
// 之后大家反复读 tile[]，不再碰慢的全局内存
一句话记住 shared memory 它是"你自己管的缓存"。CPU 的缓存是硬件替你猜哪些数据该留在近处；GPU 把这个决定权交给了你。它的黄金用法是数据复用：如果一块数据要被 block 里很多线程反复读，就先花一次慢的全局内存访问把它搬进 shared memory，之后所有复用都走快的片上访问。第 11 章的 tiling 矩阵乘就是这招的教科书级应用。

合并访问：这一章最值钱的概念

现在讲那个能让性能差出几十倍、却常被新手忽略的东西。回忆第 6 章：全局内存访问不是一个线程一个线程发出的，而是以 warp 为单位——一个 warp 的 32 条 lane 在同一条 load 指令上，同时各自要一个地址的数据。

硬件访问全局内存有个物理特性：它一次只能按固定大小的对齐块搬运（典型是 128 字节一段，即 32 个 float）。于是关键问题变成：这个 warp 的 32 个地址，落在几个 128 字节的段里？

  • 如果 32 条 lane 要的是连续的 32 个 float（lane i 读 a[base+i]），它们正好落在同一个 128 字节段里——硬件一次传输就全部搞定。这叫合并访问（memory coalescing）。
  • 如果 32 条 lane 的地址跨步或散乱（比如 lane i 读 a[i*32]），32 个地址落进 32 个不同的段——硬件只能发 32 次传输，其中每次搬来的 128 字节里只用了 4 字节，97% 的带宽被浪费。
图 7-2　合并访问 vs 跨步访问。warp 的 32 个地址落在同一个 128 字节段里就是 1 次传输；散落在 32 个段里就是 32 次传输，带宽利用率天差地别
合并的判据是"warp 内相邻 lane 访问相邻地址" 这解释了一个新手常犯的错：处理二维数据时，让相邻线程沿列方向走（a[tid * width + col]），相邻 lane 的地址就差了一整行，彻底不合并；改成沿行走（a[row * width + tid]），相邻 lane 访问相邻地址，立刻合并。同样的算法、同样的数据量，仅仅换了循环下标的角色，就可能快几十倍。第 17 章会把访存优化（合并、向量化、bank conflict）讲成一套系统方法。

bank conflict：shared memory 也有它的"访存模式"（先一瞥）

shared memory 快，但它内部被分成 32 个 bank（存储体），一个 warp 同时访问时，如果多条 lane 撞进同一个 bank 的不同地址，就会 bank conflict（存储体冲突）——硬件被迫串行处理，快内存也变慢。规律和合并类似：warp 内 lane 访问 shared 数组的下标错开成不同 bank 才最快。这里先埋个点，第 17 章详解；现在只需知道"shared memory 也讲究访问模式"。

动手：数一数每种访存模式要几次传输

下面这个模拟器能在浏览器里直接跑——点 ▶ 运行。它把上面那张图变成可计算的：给定一个 warp 的访问步长（stride），算出这 32 条 lane 的地址落进几个 128 字节段，也就是需要几次内存传输。你会看到 stride=1 只要 1 次，stride=32 要 32 次。

Python 
# 模拟：一个 warp（32 条 lane）按某步长访问 float，需要几次 128 字节传输
WARP = 32
BYTES_PER_FLOAT = 4
SEGMENT = 128                       # 硬件一次搬运的对齐块大小（字节）

def transactions(stride, base_elem=0):
    """每条 lane 读 a[base + lane*stride]，返回覆盖到的 128 字节段数。"""
    segments = set()
    for lane in range(WARP):
        byte_addr = (base_elem + lane * stride) * BYTES_PER_FLOAT
        segments.add(byte_addr // SEGMENT)     # 这个地址属于哪个段
    return len(segments)

print(f"一个 warp = {WARP} 条 lane，每次搬运一个对齐的 {SEGMENT} 字节段\n")
print("步长 stride | 需要的传输次数 | 带宽利用率")
print("-" * 46)
for stride in [1, 2, 4, 8, 16, 32]:
    n = transactions(stride)
    used = WARP * BYTES_PER_FLOAT          # 真正需要的字节
    moved = n * SEGMENT                    # 实际搬运的字节
    print(f"{stride:>9}   | {n:>10} 次   | {used/moved*100:5.1f}%")

print("\n结论：")
print("  stride=1（合并）：32 个 float 正好铺满 1 个段 → 1 次传输，100% 有效。")
print("  stride 越大越散：段数越多、每段浪费越狠。")
print("  stride=32：32 个地址落进 32 个段 → 32 次传输，只用了 1/32 的带宽。")

运行结果：

一个 warp = 32 条 lane，每次搬运一个对齐的 128 字节段

步长 stride | 需要的传输次数 | 带宽利用率
----------------------------------------------
        1   |          1 次   | 100.0%
        2   |          2 次   |  50.0%
        4   |          4 次   |  25.0%
        8   |          8 次   |  12.5%
       16   |         16 次   |   6.2%
       32   |         32 次   |   3.1%

一张表把"访存模式决定性能"讲透了：从合并（stride=1，一次传输、满带宽）到最散（stride=32，32 次传输、只用 3% 带宽），算的活一模一样，仅仅因为地址的排布，带宽利用率就差了 32 倍。请动手改：把 transactions(stride) 调用改成 transactions(1, base_elem=1)（连续访问，但整体偏移一个 float）——你可能以为还是 1 次，但因为地址不再对齐到 128 字节边界，它会变成 2 次。这就是为什么真实优化里除了"连续"还讲究"对齐"。

动手：第一个用上 shared memory 的完整 kernel

下面是一个完整可编译的 CUDA 程序：每个 block 把自己那段数据搬进 shared memory，__syncthreads() 等全组搬完，然后每个线程去读自己 block 内的邻居——做一个"在 block 内把数据首尾颠倒"的操作。它同时演示了 shared memory 的两大要素：显式搬运 和 同步等待。点 🔗 在 Compiler Explorer 打开 可以看它编译出的 PTX。

CUDA C++ 🔗 在 Compiler Explorer 打开（看 PTX）
// 用 shared memory 在每个 block 内部把数据首尾颠倒
#include <cstdio>
#include <cuda_runtime.h>

#define BLOCK 256

__global__ void reverseInBlock(float* out, const float* in, int n) {
    __shared__ float tile[BLOCK];                 // 每个 block 一块片上暂存区

    int gid = blockIdx.x * blockDim.x + threadIdx.x;
    int t   = threadIdx.x;

    // 1) 合并访问：相邻线程读相邻地址，把本 block 的数据搬进 shared memory
    if (gid < n) tile[t] = in[gid];
    __syncthreads();                              // 等本 block 所有线程都搬完

    // 2) 每个线程读"block 内的镜像邻居"——数据已在快内存里，随便挑
    int mirror = blockDim.x - 1 - t;
    if (gid < n) out[gid] = tile[mirror];
}

int main() {
    const int N = 1024;
    size_t bytes = N * sizeof(float);

    float *hIn = (float*)malloc(bytes), *hOut = (float*)malloc(bytes);
    for (int i = 0; i < N; ++i) hIn[i] = (float)i;    // 0,1,2,...

    float *dIn, *dOut;
    cudaMalloc(&dIn, bytes);
    cudaMalloc(&dOut, bytes);
    cudaMemcpy(dIn, hIn, bytes, cudaMemcpyHostToDevice);

    int blocks = (N + BLOCK - 1) / BLOCK;
    reverseInBlock<<<blocks, BLOCK>>>(dOut, dIn, N);
    cudaDeviceSynchronize();

    cudaMemcpy(hOut, dOut, bytes, cudaMemcpyDeviceToHost);

    // 第 0 个 block 内应从 255 反到 0；打印头尾验证
    printf("out[0]   = %.0f  (期望 255)\n", hOut[0]);
    printf("out[255] = %.0f  (期望 0)\n",   hOut[255]);
    printf("out[256] = %.0f  (期望 511，第二个 block 内部反转)\n", hOut[256]);

    cudaFree(dIn); cudaFree(dOut);
    free(hIn); free(hOut);
    return 0;
}
本地运行：nvcc -std=c++20 -O3 -arch=sm_90 main.cu -o app && ./app

编译运行（本地有 NVIDIA GPU 时）：

终端
nvcc reverse.cu -o reverse && ./reverse

运行结果：

out[0]   = 255  (期望 255)
out[255] = 0  (期望 0)
out[256] = 511  (期望 511，第二个 block 内部反转)

注意这个 kernel 的结构，它是后面几乎所有"用 shared memory 提速"的 kernel 的骨架：搬进来 → __syncthreads() → 反复读。这里为什么必须 __syncthreads()？因为线程 0 想读 tile[255]，而 tile[255] 是线程 255 搬进去的——如果不等线程 255 写完就读，线程 0 会读到垃圾。这就是 GPU 上代替锁的协作方式：barrier（栅栏同步）。请动手改：把 mirror 那行改成 int mirror = (t + 1) % blockDim.x;（读右边邻居，循环回卷），重新编译运行，看输出怎么变——你会看到每个 block 内的数据整体左移了一位。

小结与下一步

你现在有了 GPU 内存的完整地图：寄存器（每线程、最快）、shared memory（每 block、片上、你亲手管）、全局内存（片外、又大又慢），以及那条铁律——warp 内相邻 lane 访问相邻地址才能合并成一次传输，否则退化成几十次。访存的顺序，比访存的次数更决定速度。

上面那个 reverse kernel 里，你不得不写下一行 __syncthreads()——没有它，程序就是错的。这个"等所有线程到齐再往下走"的动作，正是 GPU 协作的核心。下一章我们就专门讲协作与同步：__syncthreads() 这样的 barrier 到底保证了什么、第 6 章埋下的 warp 级 shuffle 怎么用、以及当多个线程要往同一个地址累加时，如何用原子操作避免数据竞争。

这一章的要点

  • GPU 内存分三级：寄存器（每线程私有、最快）、shared memory（每 block 共享、片上、你显式管理）、全局内存（片外、又大又慢、所有线程可见）；另有常量内存和自动的 L2。
  • shared memory 是"你自己管的缓存"：与 CPU 自动透明的缓存不同，你亲手声明、亲手搬数据、亲手同步。黄金用法是数据复用（第 11 章 tiling）。
  • 合并访问（memory coalescing） 是本章最值钱的概念：warp 的 32 条 lane 访问连续对齐地址时合并成 1 次宽传输；跨步/散乱时退化成多达 32 次小传输，带宽利用率暴跌。
  • 判据是"warp 内相邻 lane 访问相邻地址"；二维数据里沿行还是沿列遍历，可能带来几十倍性能差。
  • shared memory 也讲访问模式：多条 lane 撞进同一 bank 会 bank conflict（第 17 章详解）。
  • 用 shared memory 的骨架是：搬进来 → __syncthreads() → 反复读；那句同步不是可选的，缺了就是数据竞争——下一章展开。
