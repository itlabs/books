第 17 章 访存优化：合并、bank conflict 与向量化

这一章想让你带走什么

上一章的 Roofline 反复告诉你一件事：绝大多数 kernel 卡在带宽上。这一章就是把"优化访存"这把最值钱的杠杆彻底讲透——三个层面：全局内存的合并访问（memory coalescing）决定一个 warp 的 32 次读能不能压成一两次大传输；shared memory 的 bank conflict决定片上访存是 1 个周期还是被串行化 32 倍；向量化加载（float4）决定每条访存指令搬 4 字节还是 16 字节。

读完你能用一个浏览器里能跑的 bank 模拟器亲手数出冲突、并看到 padding 怎么把它清零，还能读懂一份完整可编译的矩阵转置代码——从"访存全乱"的朴素版，到"合并 + 无冲突"的 tiling 版。这是把带宽受限 kernel 提速的核心手艺。

第一层：全局内存的合并访问

第 7 章简单说过合并访问，这一章我们把机制讲到底。关键事实是：GPU 从全局内存（显存）取数据，不是一个字节一个字节取，而是一次搬一整段对齐的内存块——通常是 32、64 或 128 字节的事务（transaction）。这就像你从仓库取货，叉车一次搬一整个托盘，你要一颗螺丝它也搬一托盘回来。

一个 warp 有 32 个线程，它们的 32 次访存会被硬件尽量合并成尽可能少的事务。理想情况：32 个线程访问连续的 32 个 float（128 字节），刚好落在一个对齐的 128 字节块里——一次事务全部搞定，带宽利用率 100%。这就是合并访问（memory coalescing）。

反过来，如果 32 个线程访问的地址东一个西一个（跨度很大），每个线程都可能触发一次独立的事务——32 个线程要 32 次事务，每次还只用了搬回来那一整块里的一小格，带宽被浪费到 1/32。访问的字节数一样多，但因为不连续，实际打到显存上的流量翻了几十倍。 这就是第 1 章说的"访问顺序比访问次数还重要"的确切含义。

AoS vs SoA：一个数据布局的经典抉择

这个原理直接决定了你该怎么摆数据。假设每个粒子有 x、y、z 三个坐标，两种存法：

  • AoS（Array of Structs，结构体数组）：struct Particle{float x,y,z;}; Particle p[N]; —— 内存里是 x0 y0 z0 x1 y1 z1 ...，一个粒子的三个分量挨在一起。
  • SoA（Struct of Arrays，数组的结构体）：float x[N], y[N], z[N]; —— 内存里是 x0 x1 x2 ... y0 y1 y2 ...，同一分量的所有值挨在一起。

现在一个 warp 的 32 个线程各处理一个粒子、都要读它的 x。AoS 里，线程 0 读地址 0、线程 1 读地址 12（跳过 y0 z0）、线程 2 读地址 24……跨度 12 字节、完全不连续，合并失败。SoA 里，32 个线程读的是 x[0..31]——紧挨着的 128 字节，一次事务搞定。

图 17-1　一个 warp 都读 x 分量：AoS 里 x 被 y、z 隔开，访问散成多次事务；SoA 把所有 x 排在一起，一次事务合并搞定
AoS：直觉自然，但访存散
CUDA C++
struct Particle { float x, y, z; };
Particle* p;              // x0 y0 z0 x1 y1 z1 ...
int i = blockIdx.x*blockDim.x
      + threadIdx.x;
float xi = p[i].x;        // 跨度 12 字节
// warp 内 32 次读散开 → 合并失败
SoA：对 GPU 友好，合并
CUDA C++
float *x, *y, *z;         // x0 x1 x2 ... 各自连续
int i = blockIdx.x*blockDim.x
      + threadIdx.x;
float xi = x[i];          // 跨度 4 字节
// warp 内读 x[0..31] 连续 → 一次事务
GPU 上的黄金布局规则 让相邻的线程访问相邻的内存（data[threadIdx.x] 而不是 data[threadIdx.x * stride]）。这条规则把 CPU 时代"结构体挨在一起对缓存友好"的直觉反转了：CPU 上一个线程顺序扫一个结构体，AoS 更好；GPU 上一个 warp 的 32 个线程同时各取一个结构体的同一个字段，SoA 才能合并。数据布局往往比 kernel 里的算法更影响速度。

最经典的"访存不连续"陷阱是矩阵转置：out[x][y] = in[y][x]。读 in 时相邻线程读相邻列（连续，合并好），写 out 时相邻线程却要写相邻行（跨度是整行的宽度，完全不连续）——读合并了，写却散了。这个问题我们放到本章动手部分用完整代码解决。

第二层：shared memory 的 bank conflict

数据一旦搬进 shared memory（第 7 章那块片上高速内存），就没有"合并"问题了——但有它自己的坑：bank conflict（bank 冲突）。第 7 章提过一句，这里讲透。

shared memory 在硬件上被切成 32 个 bank（存储体），像 32 条并排的收银台。连续的 4 字节字轮流分到这 32 个 bank 上：字 0 在 bank 0、字 1 在 bank 1、……字 31 在 bank 31、字 32 又回到 bank 0。规则就一条：

bank 编号 = 字地址 % 32

一个 warp 的 32 个线程同时访问 shared memory 时：

  • 如果它们落在32 个不同的 bank（每个 bank 最多一个线程）——全部并行，1 个周期搞定。这是理想情况。
  • 如果有 k 个线程挤到同一个 bank（且不是读同一个地址的广播），这个 bank 只能一个一个服务——k-way 冲突，串行 k 轮。最坏情况 32 个线程全挤到一个 bank，慢 32 倍。
图 17-2　shared memory 有 32 个 bank，bank = 地址 % 32。stride-1 时 32 线程铺满 32 bank（1 周期）；stride-2 时只用一半 bank，每个被 2 个线程命中（2-way 冲突，串行 2 轮）

padding：一个字宽度就化解转置的冲突

bank conflict 最出名的现身场景，还是转置。在 shared memory 里开一块 tile[32][32]，写进去时按行、读出来时按列（转置就是行列互换）。问题来了：tile[ty][tx] 的地址是 ty*32 + tx，同一列 tile[0][k], tile[1][k], ..., tile[31][k] 的地址是 k, 32+k, 64+k, ...——它们 % 32 全等于 k，全挤在同一个 bank！一个 warp 读一整列 = 32-way 冲突，慢 32 倍。

修复手法优雅到近乎魔法：把列宽从 32 改成 33，声明成 tile[32][32+1]。多出来的这一列从不使用，纯粹是"占位符"，但它让 tile[ty][k] 的地址变成 ty*33 + k。现在同一列的地址是 k, 33+k, 66+k, ...，因为 33 与 32 互质，它们 % 32 的结果各不相同——32 个线程刚好又铺满 32 个 bank，冲突归零。多花 32 个 float 的 shared memory，换来 32 倍的访存吞吐。 这就是你在无数 GPU 代码里见到的 [TILE][TILE+1] 的来历。

动手（一）：数出 bank 冲突，看 padding 清零它

下面这个程序能在浏览器里直接跑——点 ▶ 运行。它对一个 warp（32 线程）的访问模式，逐个算 bank = 地址 % 32，统计最忙的 bank 被几个线程命中（就是几路冲突）。四种模式：连续（无冲突）、跨 2（2-way）、按列取（32-way 最坏）、以及 +1 padding 后（冲突消失）。

Python 
# bank conflict 模拟器：shared memory 有 32 个 bank，bank = 地址 % 32
NUM_BANKS = 32

def analyze(name, addresses):
    """给一个 warp（32 个线程）各自访问的 shared memory 地址（以 4 字节字为单位），
    统计每个 bank 被几个线程命中，最大冲突数 = 这次访问要串行几轮。"""
    hits = [0] * NUM_BANKS
    for addr in addresses:
        hits[addr % NUM_BANKS] += 1
    max_conf = max(hits)          # 命中同一 bank 的最多线程数 = k-way 冲突
    way = "无冲突（1 周期）" if max_conf == 1 else f"{max_conf}-way 冲突（串行 {max_conf} 轮）"
    print(f"{name}：最忙的 bank 被 {max_conf} 个线程命中 → {way}")
    return max_conf

# 32 个线程，线程 t 访问的字地址
tids = list(range(32))

# 模式 1：stride-1，addr = t —— 32 个线程刚好铺满 32 个 bank
analyze("stride-1  addr=t",        [t for t in tids])

# 模式 2：stride-2，addr = 2t —— 只落在偶数 bank，每个被 2 个线程命中
analyze("stride-2  addr=2t",       [2*t for t in tids])

# 模式 3：列访问 addr = t*32 —— 全部落到 bank 0，32-way 冲突（最坏）
analyze("列访问   addr=t*32",      [t*32 for t in tids])

# 模式 4：加 padding，列宽从 32 改成 33，addr = t*33 —— 冲突消失
analyze("padding  addr=t*33",      [t*33 for t in tids])

运行结果：

stride-1  addr=t：最忙的 bank 被 1 个线程命中 → 无冲突（1 周期）
stride-2  addr=2t：最忙的 bank 被 2 个线程命中 → 2-way 冲突（串行 2 轮）
列访问   addr=t*32：最忙的 bank 被 32 个线程命中 → 32-way 冲突（串行 32 轮）
padding  addr=t*33：最忙的 bank 被 1 个线程命中 → 无冲突（1 周期）

看模式 3 和模式 4：仅仅把跨度从 32 改成 33，最坏的 32-way 冲突就变回了无冲突。请动手改：把模式 2 的 2*t 改成 3*t（stride-3）再运行——你会发现它居然无冲突！因为 3 和 32 互质，3t % 32 遍历所有 bank。再试 4*t（stride-4，4 与 32 不互质）——4-way 冲突。规律：跨度与 32 的最大公约数就是冲突路数。 这正是 padding 用"互质"消冲突的数学原理。

第三层：向量化加载

还有一招提高带宽利用率的手法：向量化加载（vectorized load）。GPU 支持一条指令读取 128 位（16 字节）的 float4、int4 等类型。用它，一个线程一次就能搬 4 个 float，好处有两个：每条访存指令搬的字节数翻 4 倍（更容易打满事务带宽），并且访存指令数量减到 1/4（减轻指令发射压力，对访存密集 kernel 尤其有效）。

CUDA C++
// 标量版：一个线程搬 1 个 float，要发 4 条 load
float a0 = in[i], a1 = in[i+1], a2 = in[i+2], a3 = in[i+3];

// 向量化版：一条指令搬 16 字节（4 个 float）
// 把指针 reinterpret 成 float4*，注意要求 16 字节对齐
float4 v = reinterpret_cast<const float4*>(in)[i];  // v.x v.y v.z v.w
out4[i] = make_float4(v.x*2, v.y*2, v.z*2, v.w*2);
向量化的前提：对齐与整除 float4 要求首地址按 16 字节对齐，且元素个数能被 4 整除，否则会读越界或触发未对齐访问。cudaMalloc 返回的指针天然满足 256 字节对齐，所以从头读一般安全；但如果你从数组中间某个奇数下标开始 reinterpret，对齐就破了。数据量不是 4 的倍数时，要单独处理尾巴。向量化是锦上添花，别在没测出访存指令是瓶颈前就上——先保证合并、再考虑向量化。

动手（二）：矩阵转置，从"访存全乱"到"合并 + 无冲突"

把三个层面串起来的最佳实战就是矩阵转置。下面是一个完整可编译的程序，包含两个 kernel：朴素版（读合并、写不合并），和 shared memory tiling 版（读写都合并，且用 +1 padding 消除 bank 冲突）。host 端做了完整的分配、拷贝、正确性校验，两个版本都会验证并打印 OK。

CUDA C++ 🔗 在 Compiler Explorer 打开（看 PTX）
// transpose.cu —— 矩阵转置：朴素版 vs shared memory tiling 版（+1 padding）
// 编译：nvcc -O2 -o transpose transpose.cu
#include <cstdio>
#include <cstdlib>
#include <cuda_runtime.h>

#define TILE 32

// ---------- 版本 1：朴素转置 ----------
// out 是 W 行 H 列（in 是 H 行 W 列）。读 in 时相邻线程读相邻列（合并好），
// 写 out 时相邻线程写相邻行（跨度 = H，不合并）——写访存被浪费。
__global__ void transposeNaive(const float* in, float* out, int W, int H) {
    int x = blockIdx.x * TILE + threadIdx.x;   // in 的列
    int y = blockIdx.y * TILE + threadIdx.y;   // in 的行
    if (x < W && y < H) {
        // out[x][y] = in[y][x]；out 的行宽是 H
        out[x * H + y] = in[y * W + x];
    }
}

// ---------- 版本 2：shared memory tiling + padding ----------
// 先把一块 tile 合并地读进 shared memory，再合并地写回 out。
// tile 列宽用 TILE+1，避免按列读写时的 32-way bank conflict。
__global__ void transposeTiled(const float* in, float* out, int W, int H) {
    __shared__ float tile[TILE][TILE + 1];     // +1 消除 bank conflict

    // 读阶段：相邻线程读 in 的相邻列 → 合并
    int x = blockIdx.x * TILE + threadIdx.x;
    int y = blockIdx.y * TILE + threadIdx.y;
    if (x < W && y < H)
        tile[threadIdx.y][threadIdx.x] = in[y * W + x];

    __syncthreads();

    // 写阶段：把本 block 的坐标转置后，相邻线程写 out 的相邻列 → 仍合并
    int xt = blockIdx.y * TILE + threadIdx.x;  // out 的列（对应 in 的行方向）
    int yt = blockIdx.x * TILE + threadIdx.y;  // out 的行（对应 in 的列方向）
    if (xt < H && yt < W)
        out[yt * H + xt] = tile[threadIdx.x][threadIdx.y];  // 从 shared 里按列取
}

int checkResult(const float* out, const float* in, int W, int H) {
    for (int r = 0; r < W; ++r)          // out 有 W 行、H 列
        for (int c = 0; c < H; ++c)
            if (out[r * H + c] != in[c * W + r]) return 0;
    return 1;
}

int main() {
    const int W = 1024, H = 768;         // in: H 行 W 列
    const size_t bytes = (size_t)W * H * sizeof(float);

    float* h_in  = (float*)malloc(bytes);
    float* h_out = (float*)malloc(bytes);
    for (int i = 0; i < W * H; ++i) h_in[i] = (float)i;

    float *d_in, *d_out;
    cudaMalloc(&d_in, bytes);
    cudaMalloc(&d_out, bytes);
    cudaMemcpy(d_in, h_in, bytes, cudaMemcpyHostToDevice);

    dim3 block(TILE, TILE);
    dim3 grid((W + TILE - 1) / TILE, (H + TILE - 1) / TILE);

    // --- 朴素版 ---
    cudaMemset(d_out, 0, bytes);
    transposeNaive<<<grid, block>>>(d_in, d_out, W, H);
    cudaMemcpy(h_out, d_out, bytes, cudaMemcpyDeviceToHost);
    printf("naive : %s\n", checkResult(h_out, h_in, W, H) ? "OK" : "FAIL");

    // --- tiling + padding 版 ---
    cudaMemset(d_out, 0, bytes);
    transposeTiled<<<grid, block>>>(d_in, d_out, W, H);
    cudaMemcpy(h_out, d_out, bytes, cudaMemcpyDeviceToHost);
    printf("tiled : %s\n", checkResult(h_out, h_in, W, H) ? "OK" : "FAIL");

    // 说明：两个版本结果都正确，但实测速度差异需要在真实 GPU 上用
    // profiler（如 Nsight Compute）测量——tiled 版写访存合并、无 bank
    // 冲突，通常明显更快。本机没有 GPU 时可只验证正确性。
    printf("（计时需在真实 GPU 上用 profiler 测量）\n");

    cudaFree(d_in); cudaFree(d_out);
    free(h_in); free(h_out);
    return 0;
}
本地运行：nvcc -std=c++20 -O3 -arch=sm_90 main.cu -o app && ./app

两个 kernel 算出的结果完全一样（都打印 OK），区别只在访存效率：朴素版写 out 时跨度是 H，一个 warp 的 32 次写散落在显存各处，合并失败；tiling 版先把数据合并地读进 shared memory，转置发生在片上（靠 +1 padding 避开 bank 冲突），再合并地写回 out——读、写两头都合并。在真实 GPU 上，tiling 版通常有数倍的带宽优势。你可以点代码块的 🔗 在 Compiler Explorer 打开 看两个 kernel 编译出的 PTX，对比它们的 load/store 指令。

为什么转置值得这么大费周章 转置本身一次计算都不做——它的算术强度几乎是 0，是纯带宽受限的极端例子（回到第 16 章的 Roofline，它贴在斜屋顶最左端）。正因为它"只搬不算"，它的全部性能就等于访存效率，成了检验合并/bank conflict 手艺的最佳练习。真实项目里，转置常常和别的计算融合在一个 kernel 里（第 14 章的算子融合），避免为了转置单独读写一趟显存。

小结与下一步

访存优化是把带宽受限 kernel 提速的头号杠杆，这一章给了你三层工具：全局内存要合并（相邻线程访问相邻地址，SoA 优于 AoS），shared memory 要避开 bank conflict（[TILE][TILE+1] 的 padding），还可以用向量化加载（float4）一次多搬几倍字节。转置这个例子把三者串在了一起。

有了合并访问和 shared memory 的手艺，你已经能写出高效的逐元素和局部访存了。但还有一类模式访存和计算深度交织、需要线程间协作——归约（reduction）和扫描（scan）：把一整个数组加成一个数、或算前缀和。它们看似简单，却把 warp 分歧、bank conflict、shuffle、原子操作全都用上了，是检验你到此为止所有知识的试金石。下一章我们就来把这两个并行原语写到极致。

这一章的要点

  • 全局内存按 32/64/128 字节事务成块传输。一个 warp 的 32 次访问若落在连续对齐块里 → 合并成一次事务；若散开 → 事务数暴增，带宽浪费到几十分之一。
  • 布局规则：相邻线程访问相邻地址。因此 GPU 上 SoA（float x[],y[],z[]）优于 AoS（struct{float x,y,z}）——与 CPU 缓存直觉相反。
  • shared memory 有 32 个 bank，bank = 地址 % 32。一个 warp 若 k 个线程挤同一 bank → k-way 冲突，串行 k 轮；跨度与 32 的最大公约数就是冲突路数。
  • 用 [TILE][TILE+1] padding（33 与 32 互质）把转置里的 32-way 列冲突清零，代价只是极少的 shared memory。
  • 向量化加载（float4/int4） 每条指令搬 16 字节、把访存指令数减到 1/4——但要求 16 字节对齐、元素数整除，是合并之后的锦上添花。
  • 矩阵转置是纯带宽受限的经典练习：朴素版写不合并，tiling+padding 版读写都合并且无 bank 冲突，结果相同而带宽差数倍。
