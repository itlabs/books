第 5 章 执行层级（二）：NDRange、Work-group、Work-item

这一章想让你带走什么

上一章你学会了 CUDA 的 Grid / Block / Thread。这一章告诉你一个好消息和一个坏消息。好消息：SYCL 和 OpenCL 用的是同一套执行模型，只是换了名字——NDRange / work-group / work-item，几乎一一对应 CUDA 的三层结构，你昨天学的今天全能复用。

坏消息：有一个地方极容易翻车——CUDA 的 <<<blocks, threads>>> 里你填的是"组数"和"每组线程数"，而 SYCL 的 nd_range 里你填的是"总 work-item 数"和"每组数"。同样是两个数字，含义完全不同。读完你会有一张"一图两词"的对照表刻在脑子里，还能亲眼看到把两套约定搞混会产生什么 bug。

一套模型，两套词汇

第 4 章讲的分层思想不是 CUDA 独有的。OpenCL 早年就定义了同样的模型，SYCL（现代 C++ 的跨厂商 GPU 方案，第 12 章细讲）继承了 OpenCL 的术语。三层结构完全对得上，只是每一层换了个名字：

CUDA SYCL / OpenCL 是什么
thread（线程） work-item（工作项） 最小执行单位，跑一份 kernel 代码
block（线程块） work-group（工作组） 能协作、共享片上内存、可 barrier 的一组 work-item
grid（网格） NDRange（N 维范围） 全部 work-item 的总集合
warp（线程束，32） sub-group（子组） 真正齐步走的硬件调度单位（第 6 章）

kernel 里定位"我是谁"的四个内建量也一一对应，只不过 CUDA 是变量（threadIdx.x），SYCL 是 nd_item 对象上的方法（.get_local_id()）：

CUDA SYCL（nd_item 上的方法） 含义
threadIdx.x item.get_local_id(0) 我在自己 work-group 内的局部序号
blockIdx.x item.get_group(0) 我这个 work-group 的编号
blockDim.x item.get_local_range(0) 每个 work-group 多少个 work-item
blockIdx.x*blockDim.x+threadIdx.x item.get_global_id(0) 全局 id（直接给你，不用自己乘加！）
SYCL 帮你算好了全局 id CUDA 里那条要你亲手写的 i = blockIdx.x * blockDim.x + threadIdx.x，在 SYCL 里被封装成一个方法 item.get_global_id(0) 直接返回。局部 id 用 get_local_id(0)，组号用 get_group(0)。少了手写乘加，也就少了一类下标算错的 bug——这是 SYCL 在人体工学上的一点小便利。括号里的 0 是维度号（x=0, y=1, z=2）。

同一张网格，两套标注

既然两套词汇描述的是同一个东西，那最有价值的一张图，就是把它们标在同一张网格上。下面这张图请记牢：以后你在 CUDA 文档和 SYCL 文档之间来回切换时，脑子里翻的就是它。

图 5-1　一张网格，两套词汇：thread=work-item、block=work-group、grid=NDRange。注意最下方——CUDA 的第一个参数是"组数"，SYCL 的第一个参数是"总数"，这是两者最容易混的地方

唯一的大坑：nd_range 填的是"总数"不是"组数"

现在讲这一章最重要、最容易让人栽跟头的一点。

启动一个 1024 元素、每组 256 的任务，两边都要给硬件两个数字，但这两个数字的含义不一样：

  • CUDA：add<<<4, 256>>> —— 第一个数是 block 的数量（4），第二个是每 block 线程数（256）。总线程数要你自己心算 4 × 256。
  • SYCL：nd_range<1>{range<1>{1024}, range<1>{256}} —— 第一个数是 全局 work-item 总数（1024），第二个是每组的 work-item 数（256）。组数要 SYCL 自己去算 1024 / 256。
CUDA：<<<组数, 每组线程数>>>
CUDA C++
int local  = 256;                 // 每 block 线程数
int groups = (N + 255) / 256;     // 你算出 block 数 = 4
// 第一个参数 = 组数
add<<<groups, local>>>(...);
// 总线程数 = groups * local，你自己心里有数
SYCL：nd_range{全局总数, 每组数}
SYCL / DPC++
size_t local  = 256;              // 每组 work-item 数
size_t global = N;                // 全局总数 = 1024，不是组数！
// 第一个参数 = 全局总 work-item 数
sycl::nd_range<1>{ sycl::range<1>{global},
                   sycl::range<1>{local} };
// 组数由 SYCL 算：global / local = 4
头号翻车现场：把 CUDA 的"组数"直接搬进 SYCL 当"全局数" 一个刚从 CUDA 转过来的人，习惯先算出 groups = 4，然后写 nd_range<1>{range<1>{4}, range<1>{256}}——他以为第一个参数是组数。后果有两种：要么运行时报错，因为 SYCL 要求 global 必须能被 local 整除，而 4 根本装不下一个 256 的组；要么侥幸不报错却只启动了 4 个 work-item，1024 个元素只算了前 4 个，其余 1020 个是垃圾值。记死这句：SYCL 的第一个数是"一共要多少个 work-item"，等于 CUDA 的 组数 × 每组数。

反过来，从 SYCL 转 CUDA 的人也会栽：他把 global=1024 直接填进 <<<1024, 256>>>，结果启动了 1024 × 256 = 262144 个线程，超订了 256 倍——好在有边界检查 if (i < n) 兜底不会崩，但绝大多数线程空转，还可能触发别的资源上限。方向搞反，两边都出事。

nd_item：SYCL 里的"我是谁"

在 SYCL 里，你把 kernel 写成一个 lambda，参数是一个 nd_item<1>（一维时）对象——它就相当于 CUDA kernel 里那组内建变量的集合体。你从它身上取各种 id：

SYCL / DPC++
q.parallel_for(
    sycl::nd_range<1>{sycl::range<1>{global}, sycl::range<1>{local}},
    [=](sycl::nd_item<1> item) {
        size_t g = item.get_global_id(0);   // 全局 id（≈ CUDA 的 i）
        size_t l = item.get_local_id(0);    // 组内局部 id（≈ threadIdx.x）
        size_t grp = item.get_group(0);     // 组号（≈ blockIdx.x）
        if (g < n)                          // 边界检查，和 CUDA 一样要写
            c[g] = a[g] + b[g];
    });
SYCL 还有个"简单版" parallel_for，但那不是本章的重点 如果你的 kernel 完全不需要组内协作（不用 shared memory、不用 barrier），SYCL 允许你只传一个 range（不是 nd_range），kernel 参数用 item 而非 nd_item，连"每组多少个"都不用操心，由运行时决定。但那样你就放弃了 work-group 这一层，也就用不了 sub-group、local memory 和 barrier。因为本书要打通到协作与优化，我们从头就用带 nd_range 的完整形式——它才和 CUDA 的 <<<grid, block>>> 一一对应。

动手：同一个问题，两套词汇，和一个经典 bug

下面这个模拟器能在浏览器里直接跑——点 ▶ 运行。它把同一个 1024 元素的问题分别用 CUDA 词汇和 SYCL 词汇描述一遍，让你看清"组数"和"全局数"这两个数字的换算关系；最后故意复现那个最常见的 bug：把 CUDA 的组数当成 SYCL 的 global 传进去，看它怎么漏掉 1020 个元素。

Python 
# 同一个 1024 元素的问题，用两套词汇分别描述
N = 1024

# ---------- CUDA 词汇：<<<blocks, threads>>> ----------
threads_per_block = 256                                  # block 大小
blocks = (N + threads_per_block - 1) // threads_per_block  # block 数量
print("CUDA: <<<blocks, threads>>>")
print(f"  你指定：blocks={blocks}, threads_per_block={threads_per_block}")
print(f"  总线程数 = blocks * threads_per_block = {blocks * threads_per_block}")

# ---------- SYCL 词汇：nd_range<1>{global, local} ----------
local   = 256          # work-group 大小（= CUDA 的 block 大小）
global_ = N            # 总 work-item 数（= 总线程数），不是组数！
groups  = global_ // local
print("\nSYCL: nd_range<1>{global, local}")
print(f"  你指定：global={global_}, local={local}")
print(f"  work-group 数 = global / local = {groups}")
print(f"  总 work-item 数 = global = {global_}")

# ---------- 最常见的 bug：把 CUDA 的 blocks 直接当成 SYCL 的 global ----------
print("\n--- bug：把 blocks 数误当成 global 传给 SYCL ---")
wrong_global = blocks                 # 4，本以为这是"组数"
print(f"  nd_range{{global={wrong_global}, local={local}}}")
print(f"  但 global({wrong_global}) < local({local})：只想启动 {wrong_global} 个 work-item，")
print(f"  却要求每组 {local} 个 —— 非法，且只覆盖 {wrong_global}/{N} 个元素，漏掉 {N - wrong_global} 个！")

运行结果：

CUDA: <<<blocks, threads>>>
  你指定：blocks=4, threads_per_block=256
  总线程数 = blocks * threads_per_block = 1024

SYCL: nd_range<1>{global, local}
  你指定：global=1024, local=256
  work-group 数 = global / local = 4
  总 work-item 数 = global = 1024

--- bug：把 blocks 数误当成 global 传给 SYCL ---
  nd_range{global=4, local=256}
  但 global(4) < local(256)：只想启动 4 个 work-item，
  却要求每组 256 个 —— 非法，且只覆盖 4/1024 个元素，漏掉 1020 个！

看清楚了吗？两套词汇描述的是同一个 1024/256 的分工，但你手填的第一个数一个是 4、一个是 1024。请动手改：把 N 改成 1000（不能被 256 整除），再看 SYCL 那段——你会发现 groups = global_ // local 算出 3，3 × 256 = 768 < 1000，漏了尾巴。这正是 SYCL 要求 global 能被 local 整除的原因：真实代码里，你得把 global 向上取整到 local 的整数倍（global = ((N + local - 1) // local) * local），再在 kernel 里用 if (g < N) 挡住多出来的 work-item——和 CUDA 的边界检查一模一样。

二维也一样：nd_range<2> 对应 CUDA 的二维 grid

上一章我们用二维 grid/block 处理图像。SYCL 用 nd_range<2> 做同样的事，规则完全平移：两个 range 参数仍是"全局总数"和"每组数"，只不过现在每个都是二维的。注意 SYCL 的维度顺序习惯是 {row, col}（先 y 后 x），和你写 dim3(x, y) 的直觉相反，这是另一个小坑：

SYCL / DPC++
// 处理 height × width 的图像：二维 NDRange
sycl::range<2> local{16, 16};                    // 每个 work-group 16×16
sycl::range<2> global{                           // 全局总数，向上取整到 local 倍数
    ((height + 15) / 16) * 16,                   // 维度 0 = 行（y）
    ((width  + 15) / 16) * 16};                  // 维度 1 = 列（x）
q.parallel_for(sycl::nd_range<2>{global, local},
    [=](sycl::nd_item<2> item) {
        size_t y = item.get_global_id(0);        // 维度 0 → 行
        size_t x = item.get_global_id(1);        // 维度 1 → 列
        if (y < height && x < width)
            img[y * width + x] = 255 - img[y * width + x];
    });

对照上一章 CUDA 的二维 kernel：get_global_id(1) 对应 x（列）、get_global_id(0) 对应 y（行）。只要记住 SYCL 的维度 0 是最外层（行）、CUDA 的 .x 是最内层（列），两边就不会串。

动手：一个完整可编译的 SYCL 向量加法

下面是完整、可编译的 SYCL 程序：用 sycl::queue 提交任务，用 USM（统一共享内存，malloc_shared） 分配 host 和 device 都能直接访问的内存（省去手动拷贝，让第一个例子尽量简单），parallel_for 配 nd_range 启动，最后打印结果。点代码块上的 🔗 在 Compiler Explorer 打开 看 icpx 的编译产物；本地装了 oneAPI 的话按下面命令编译。

SYCL / DPC++ 🔗 在 Compiler Explorer 打开
#include <sycl/sycl.hpp>
#include <cstdio>

int main() {
    sycl::queue q;                       // 默认选一个设备（GPU 优先）
    printf("运行设备：%s\n",
           q.get_device().get_info<sycl::info::device::name>().c_str());

    const size_t N = 1000;               // 故意不是 256 的整数倍
    const size_t local = 256;            // work-group 大小，取 sub-group(32) 的倍数
    // 把全局大小向上取整到 local 的整数倍：1024
    const size_t global = ((N + local - 1) / local) * local;

    // USM：host 与 device 都能直接访问，无需显式 memcpy
    float* a = sycl::malloc_shared<float>(N, q);
    float* b = sycl::malloc_shared<float>(N, q);
    float* c = sycl::malloc_shared<float>(N, q);
    for (size_t i = 0; i < N; ++i) { a[i] = float(i); b[i] = float(i) * 10.0f; }

    q.parallel_for(
        sycl::nd_range<1>{sycl::range<1>{global}, sycl::range<1>{local}},
        [=](sycl::nd_item<1> item) {
            size_t g = item.get_global_id(0);     // ≈ CUDA 的全局 i
            if (g < N)                            // 挡住向上取整多出的 work-item
                c[g] = a[g] + b[g];
        }).wait();                                // 等 kernel 完成

    printf("启动配置：global=%zu, local=%zu → %zu 个 work-group\n",
           global, local, global / local);
    printf("前 5 个： %.0f %.0f %.0f %.0f %.0f\n",
           c[0], c[1], c[2], c[3], c[4]);
    printf("后 3 个： %.0f %.0f %.0f\n", c[N - 3], c[N - 2], c[N - 1]);

    sycl::free(a, q);                    // USM 也要手动释放
    sycl::free(b, q);
    sycl::free(c, q);
    return 0;
}
本地运行：icpx -fsycl -O2 -std=c++20 main.cpp -o app && ./app # 需要 Intel oneAPI
终端
icpx -fsycl ch05_vector_add.cpp -o ch05_vector_add && ./ch05_vector_add

c[i] 应当等于 a[i] + b[i] = i + 10i = 11i，所以前 5 个是 0 11 22 33 44，后 3 个是 10967 10978 10989（即 11×997, 11×998, 11×999）。注意这段代码和你想象中的 CUDA 版几乎逐行对应：get_global_id(0) 对 blockIdx.x*blockDim.x+threadIdx.x，if (g < N) 对 if (i < n)，nd_range{global, local} 对 <<<blocks, threads>>>——只是那个第一个参数的含义变了。试着把 local 从 256 改成 128，重新编译：结果不变，但 work-group 数会从 4 变成 8——改 work-group 大小不改变结果，只改变分组方式，正如上一章改 blockDim 一样。

小结与下一步

你现在能在 CUDA 和 SYCL 两套词汇之间自由翻译了：work-item=thread、work-group=block、NDRange=grid，get_global_id 就是那条乘加公式的封装。最该记牢的是那个坑——SYCL 的 nd_range 第一个参数是全局总数，不是组数，它等于 CUDA 的 组数 × 每组数。

到这里，第 4、5 两章都在讲你显式写出来的两层（block/work-group 和 thread/work-item）。但第 2 章早就埋了伏笔：真正决定性能的，是一个你没直接写、却始终在幕后齐步走的单位——CUDA 的 warp（32 线程）和它在 SYCL 里的对应物 sub-group。下一章我们就掀开 block/work-group 的盖子，看这个"隐形的 32 人小队"到底怎么调度、怎么影响你的分支和访存——这是把前面所有概念和真实性能连起来的关键一章。

这一章的要点

  • SYCL/OpenCL 和 CUDA 是同一个执行模型的两套词汇：work-item=thread、work-group=block、NDRange=grid、sub-group=warp。
  • kernel 里用 nd_item 定位自己：get_global_id(0)（≈ CUDA 的 i，已帮你算好乘加）、get_local_id(0)（≈ threadIdx.x）、get_group(0)（≈ blockIdx.x）。
  • 最大的坑：nd_range<1>{global, local} 的第一个参数是全局 work-item 总数（= CUDA 组数 × 每组数），CUDA 的 <<<blocks, threads>>> 第一个参数是组数。搞反会漏算或超订。
  • SYCL 要求 global 能被 local 整除：真实代码把 global 向上取整到 local 的倍数，再用 if (g < N) 做边界检查——和 CUDA 完全一致。
  • 用 queue + parallel_for + nd_range 是与 CUDA 一一对应的完整写法；USM 的 malloc_shared 让第一个例子免去手动拷贝。
  • work-group 大小同样取 sub-group（32）的倍数，理由和 CUDA 的 block 一样，下一章讲透。
