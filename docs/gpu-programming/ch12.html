第 12 章 SYCL：单一源码的跨厂商 C++

这一章想让你带走什么

前两章你用 CUDA 把手伸进了 NVIDIA 的硬件——但 CUDA 只跑在 NVIDIA 卡上。这一章讲另一条路线：SYCL，一个由 Khronos 制定的开放标准。它的野心是"一份源码，到处编译"：用标准 C++ 编译器、在一个 .cpp 里同时写主机端和设备端代码，然后编译到 NVIDIA、AMD、Intel 的 GPU，乃至 CPU。

读完你会知道 SYCL 的 queue、两套内存模型（buffer/accessor 与 USM）、以及 nd_range + local_accessor（也就是 SYCL 版的 __shared__）怎么把第 11 章的 tiling 思路原样搬过来；你还会用一个浏览器里能跑的模拟器，看清"一份源码分发到多个后端、结果一致"到底是什么意思。最后你会掂量出这条路线的真实权衡：可移植性 vs 峰值性能。

为什么会有 SYCL：CUDA 好用，但它只认一家硬件

CUDA 是目前最成熟、生态最全的 GPU 编程模型——但它有一个前提：你的代码只能跑在 NVIDIA 的卡上。一旦你的部署环境里出现 AMD 的 MI 系列、Intel 的 Data Center GPU，或者你想让同一份代码在没有独显的机器上退化到 CPU 跑，CUDA 就不管用了。你要么维护多份代码，要么被一家硬件锁死（vendor lock-in）。

SYCL（读作"sickle"） 是 Khronos Group（就是定义 OpenGL、Vulkan、OpenCL 的那个组织）推出的开放标准，专门解决这个问题。它的设计目标可以浓缩成三句话：

  • 单一源码（single-source）：主机端逻辑和设备端 kernel 写在同一个 .cpp 文件里，不像 OpenCL 那样把 kernel 塞进一个字符串或单独的文件。
  • 就是现代 C++：没有新的 kernel 语言、没有 __global__ 这类扩展关键字。设备端代码是一个普通的 C++ lambda，用标准 C++17/20 编译器（配合 SYCL 实现）就能编。
  • 跨厂商：同一份源码，通过不同后端编译到 NVIDIA（走 PTX）、AMD、Intel（走 SPIR-V）的 GPU，以及 CPU。

主流的 SYCL 实现有两个，附录 A 会教你装：

  • Intel oneAPI DPC++：编译器命令是 icpx -fsycl。DPC++ 是 Data Parallel C++ 的缩写，本质是带 SYCL 扩展的 Clang。
  • AdaptiveCpp（原名 hipSYCL / Open SYCL）：一个开源实现，能把 SYCL 直接编到 CUDA、HIP（AMD）和 OpenMP（CPU）后端。
图 12-1　SYCL 的核心卖点：同一份 main.cpp 经 SYCL 编译器降级成后端无关的中间表示（Intel/AMD 走 SPIR-V，NVIDIA 走 PTX），再由各后端翻译成原生指令，跑到不同厂商的硬件甚至 CPU 上

和 CUDA 的第一眼对比：不再有独立的 kernel 语言

回忆第 10 章那个 CUDA kernel：它是一个带 __global__ 关键字的特殊函数，必须用 nvcc 这个专用编译器处理，且只出 NVIDIA 代码。SYCL 把这套东西全换成了标准 C++：设备端代码就是你提交给队列的一个 lambda，普通编译器就能编。

CUDA：专用关键字 + 专用编译器
CUDA C++
// 设备端是带 __global__ 的特殊函数
__global__ void add(float* c, const float* a,
                    const float* b, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) c[i] = a[i] + b[i];
}

// 用 <<<>>> 启动，必须 nvcc 编译，只出 NVIDIA
add<<<blocks, threads>>>(c, a, b, n);
// $ nvcc add.cu       ← 专用编译器
SYCL：普通 lambda + 标准编译器
SYCL / DPC++
// 设备端就是一个提交给 queue 的 lambda
q.parallel_for(sycl::range<1>(n), [=](sycl::id<1> i) {
    c[i] = a[i] + b[i];   // 就是普通 C++
});

// 同一份 .cpp，标准 C++ 编译器（+SYCL）即可
// $ icpx -fsycl add.cpp   ← 也能编到 NVIDIA/AMD/Intel

差别一目了然：CUDA 把设备代码标成"另一种东西"（特殊关键字、特殊语法、特殊编译器），SYCL 则坚持"设备代码也只是 C++"，把差异藏进库和编译器后端。这就是"单一源码"的字面含义。

核心概念：queue、device 与两套内存模型

SYCL 的运行时模型可以先记住三个词。

queue（队列）：你和一台设备之间的通道。你把工作（kernel）**提交（submit）**到 queue，运行时负责把它派到设备上跑。创建 queue 时你可以指定"选哪台设备"：

SYCL / DPC++
// 让运行时自动挑一台"最合适"的设备（通常是最强的 GPU）
sycl::queue q{sycl::default_selector_v};

// 或者明确要 GPU / CPU
sycl::queue q_gpu{sycl::gpu_selector_v};
sycl::queue q_cpu{sycl::cpu_selector_v};

// 问问它到底选中了谁
std::cout << q.get_device().get_info<sycl::info::device::name>() << "\n";

内存管理，SYCL 给了你两条路，理解它们的取舍是这一章的重点：

  1. buffer + accessor（缓冲区 + 访问器）：你把数据包进一个 sycl::buffer，kernel 里通过 accessor 访问。你声明每个 kernel 对每块 buffer 是读还是写，运行时据此自动构建依赖图、自动排好执行顺序、自动在主机和设备间搬数据。你不写一句 memcpy。
  2. USM（Unified Shared Memory，统一共享内存）：更像 CUDA。你用 malloc_device / malloc_shared / malloc_host 拿到裸指针，像操作普通指针一样在 kernel 里解引用；同步和数据搬运由你显式负责（q.memcpy、q.wait、事件依赖）。
怎么选？一句话 buffer/accessor 让运行时替你管依赖和数据搬运，代码更声明式、更不容易出错，适合刚上手或数据流复杂的场景。USM 是指针语义，控制更细、心智模型和 CUDA 几乎一一对应，从 CUDA 迁移过来最顺手，但同步要你自己盯。两者可以在一个程序里混用。本章的完整示例用 USM（贴近你刚学的 CUDA），而下面的模拟器帮你直观感受 buffer 的"自动依赖图"是怎么回事。

parallel_for：从 range 到 nd_range

提交 kernel 的主力是 parallel_for。它有两种粒度：

  • range（基础版）：你只说"一共多少个 work-item"，每个 work-item 拿到一个全局 id。没有 work-group 的概念，也就用不了 local memory。对应 CUDA 里"只关心全局线程号"的简单 kernel。
  • nd_range（进阶版）：你同时指定全局大小和work-group 大小（nd_range<1>{global, local}），kernel 收到的是 nd_item，能问出全局 id、组内 local id、以及第几组。只有 nd_range 才能用 local_accessor——那就是 SYCL 版的 shared memory，也才能用组内 barrier。

这里就要把第 11 章的词汇表接上了。SYCL 换了一套名字，但概念和 CUDA 一一对应：

CUDA（第 4–11 章） SYCL 说明
grid nd_range 的全局范围 所有线程/work-item 的总数
block work-group 一组能协作、共享局部内存的线程；调度到一个 SM
thread work-item 最小执行单位
warp（32 线程） sub-group 硬件真正齐步调度的一小队
blockIdx it.get_group(0) 我是第几组
threadIdx it.get_local_id(0) 我在组内的编号
全局线程号 it.get_global_id(0) 我的全局编号
__shared__ local_accessor 组内共享的片上高速内存
__syncthreads() it.barrier() 组内同步栅栏
换了外衣，没换身体 把第 11 章 tiling 矩阵乘的每一个 CUDA 概念，逐行替换成右列的 SYCL 名字，你就得到一个能跑在 Intel/AMD/NVIDIA 上的版本——算法、tiling 思路、barrier 的位置全都不变。你学的不是四种技术，是同一个执行模型的四种方言。附录 B 有更全的对照表。

错误处理：同步异常与异步异常

SYCL 用 C++ 异常报错，但有个坑要提前说清：设备上的错误是异步发生的（kernel 在后台跑，出错时你的主机代码早跑到别处了）。所以 SYCL 把错误分两类：

  • 同步异常：q.submit 之类的调用当场抛出，用普通 try/catch 就能接。
  • 异步异常：kernel 执行期间发生，被攒起来，直到你调用 q.wait_and_throw() 或队列析构时，交给你在建 queue 时注册的 async_handler 统一处理。

所以健壮的 SYCL 程序，建 queue 时通常都带一个 handler。下面的完整示例就演示了它。

动手（一）：模拟"一份源码，多个后端"

先在浏览器里跑一个模拟器，把 SYCL 最核心的那句"一份源码分发到多个后端、结果一致"变成你能看见的东西。它不真用 GPU：我们写一个"设备端 kernel"（一个函数，描述"一个 work-item 做什么"），把它降级成一份后端无关的中间表示，再交给三个"假装的后端"各自去"编译并执行"——你会看到三家硬件跑出完全一样的结果。

Python 
# 模拟 SYCL 的核心理念：一份源码 → 一份中间表示 → 多个后端 → 结果一致
# 你只写"一个 work-item 做什么"，后端负责翻译成各自的原生指令

def kernel(i, a, b):
    # 这就是设备端代码：一个 work-item 只算一个元素（和 CUDA 一个线程一个元素同理）
    return a[i] + b[i]

def backend(name, ir, kernel_fn, a, b, n):
    # 每个后端拿到同一份中间表示(IR)，翻译成自己的指令，再对每个 work-item 执行
    print(f"[{name}] 收到中间表示 {ir}，翻译成本后端原生指令并执行")
    out = [0] * n
    for i in range(n):          # 想象这 n 次是 n 个 work-item 并行
        out[i] = kernel_fn(i, a, b)
    return out

N = 8
a = [i for i in range(N)]         # [0,1,...,7]
b = [i * 10 for i in range(N)]    # [0,10,...,70]

# 编译器前端把同一份 kernel 降级成后端无关的中间表示
ir = "SPIR-V:add(v0,v1)"

r_intel  = backend("Intel  GPU", ir, kernel, a, b, N)
r_nvidia = backend("NVIDIA GPU", ir, kernel, a, b, N)
r_amd    = backend("AMD    GPU", ir, kernel, a, b, N)

print("\n三个后端结果一致吗？", r_intel == r_nvidia == r_amd)
print("结果：", r_intel)

运行结果：

[Intel  GPU] 收到中间表示 SPIR-V:add(v0,v1)，翻译成本后端原生指令并执行
[NVIDIA GPU] 收到中间表示 SPIR-V:add(v0,v1)，翻译成本后端原生指令并执行
[AMD    GPU] 收到中间表示 SPIR-V:add(v0,v1)，翻译成本后端原生指令并执行

三个后端结果一致吗？ True
结果： [0, 11, 22, 33, 44, 55, 66, 77]

这就是 SYCL 承诺的"一次编写、到处运行"的骨架：你只描述算法（那个 kernel 函数），后端无关的中间表示负责把它和具体硬件解耦，三家 GPU 各自翻译、各自执行，结果却一致。请动手改：把 kernel 改成 return a[i] * b[i] + 1，你会发现三个后端仍然自动保持一致——因为它们编译的是同一份源码，这正是可移植性的来源。

动手（二）：完整可编译的 SYCL 归约

下面是一个完整可编译的 SYCL 程序：对一个长度 1024 的数组求和。它用 nd_range 划分 work-group，用 local_accessor（= shared memory） 在组内做树形归约，再由每组的 0 号 work-item 写出本组的部分和——这正是第 11 章 tiling 那套"先搬进片上内存、组内协作、barrier 同步"的思路，只是换成了 SYCL 的名字。点代码块上的 🔗 在 Compiler Explorer 打开，或用本地 icpx -fsycl 命令编译。

SYCL / DPC++ 🔗 在 Compiler Explorer 打开
#include <sycl/sycl.hpp>
#include <cstdio>
#include <exception>

int main() {
    constexpr size_t N      = 1024;   // 元素总数
    constexpr size_t LOCAL  = 256;    // work-group 大小（= CUDA 的 blockDim）
    constexpr size_t GROUPS = N / LOCAL;  // 4 个 work-group

    // 建 queue 时注册 async_handler：统一接住 kernel 执行期的异步异常
    sycl::queue q{sycl::default_selector_v, [](sycl::exception_list el) {
        for (const std::exception_ptr& e : el) {
            try { std::rethrow_exception(e); }
            catch (const sycl::exception& ex) {
                std::fprintf(stderr, "异步异常: %s\n", ex.what());
            }
        }
    }};
    std::printf("运行设备: %s\n",
        q.get_device().get_info<sycl::info::device::name>().c_str());

    // 用 USM（指针语义，贴近 CUDA）：malloc_shared 主机和设备都能直接访问
    float* in      = sycl::malloc_shared<float>(N, q);
    float* partial = sycl::malloc_shared<float>(GROUPS, q);
    for (size_t i = 0; i < N; ++i) in[i] = 1.0f;   // 全 1，总和应为 N

    q.submit([&](sycl::handler& h) {
        // local_accessor 就是 SYCL 版的 __shared__：一块 work-group 内共享的片上内存
        sycl::local_accessor<float, 1> scratch(sycl::range<1>(LOCAL), h);

        h.parallel_for(
            sycl::nd_range<1>(sycl::range<1>(N), sycl::range<1>(LOCAL)),
            [=](sycl::nd_item<1> it) {
                size_t gid = it.get_global_id(0);   // 全局编号 = CUDA 全局线程号
                size_t lid = it.get_local_id(0);    // 组内编号 = threadIdx.x

                scratch[lid] = in[gid];             // 每个 work-item 搬一个到片上内存
                it.barrier(sycl::access::fence_space::local_space);  // = __syncthreads()

                // 组内树形归约：每一轮活跃的 work-item 减半
                for (size_t s = LOCAL / 2; s > 0; s >>= 1) {
                    if (lid < s) scratch[lid] += scratch[lid + s];
                    it.barrier(sycl::access::fence_space::local_space);
                }
                if (lid == 0)                        // 每组 0 号写出本组部分和
                    partial[it.get_group(0)] = scratch[0];
            });
    }).wait();   // 等 kernel 跑完（USM 需要显式同步）

    float total = 0.0f;                              // 主机端把各组部分和加起来
    for (size_t g = 0; g < GROUPS; ++g) total += partial[g];
    std::printf("求和结果 = %.1f（期望 %.1f）\n", total, (double)N);

    sycl::free(in, q);
    sycl::free(partial, q);
    return 0;
}
本地运行：icpx -fsycl -O2 -std=c++20 main.cpp -o app && ./app # 需要 Intel oneAPI

运行结果：

运行设备: <你的 GPU 或 CPU 名字>
求和结果 = 1024.0（期望 1024.0）

把它和第 11 章的 CUDA 版本并排看：local_accessor 对 __shared__、it.barrier(...) 对 __syncthreads()、get_local_id 对 threadIdx、get_group 对 blockIdx——归约的算法一模一样。请动手改一行：把 in[i] = 1.0f 改成 in[i] = (float)i，那么总和应变成 0+1+...+1023 = 523776，把 %.1f 的期望值也相应改掉，编译运行验证一下——你会发现片上归约的逻辑完全不用动。

可移植性不是免费的午餐 SYCL 的承诺是"到处能跑"，但不是"到处都最快"。同一份代码在 NVIDIA 上，往往比手写 CUDA 慢一点，因为多了一层抽象；某些厂商最新硬件的特性（比如 NVIDIA 最新的异步拷贝、特定的 Tensor Core 指令）可能在 SYCL 里暂时没有对等接口，或要靠后端扩展才能摸到。所以工程上的常见权衡是：用 SYCL 拿到"一份代码覆盖多家硬件"的可移植性，在真正的性能热点上，再针对具体后端做特化调优。可移植性和榨干峰值，本就是一对需要你按场景取舍的目标。

小结与下一步

你现在见过了第二种 GPU 编程方言。它和 CUDA 的执行模型同构，卖点是开放标准 + 单一源码 + 跨厂商：一个 .cpp、标准 C++ 编译器、queue 提交工作、nd_range + local_accessor 把 tiling 原样搬过来，代价是要接受抽象层带来的性能与新特性上的取舍。

CUDA 和 SYCL 都还在线程/work-item 级思考——你得亲手算全局下标、亲手管 local memory、亲手写 barrier。下一章我们换一个更高的抽象层：OpenAI 的 Triton。它让你用 Python 写 kernel，而且不再写"一个线程做什么"，而是写"一个 program 处理一整块（block）数据"——线程、合并访问、大部分 shared memory 管理，都交给编译器。抽象层往上跳一级，写 ML kernel 的效率会截然不同。

这一章的要点

  • SYCL 是 Khronos 的开放标准：单一源码（主机+设备同在一个 .cpp）、就是标准现代 C++（无专用 kernel 语言）、跨厂商（NVIDIA/AMD/Intel GPU 及 CPU）。实现有 Intel oneAPI DPC++（icpx -fsycl）和 AdaptiveCpp。
  • 运行模型核心是 queue：向它提交工作，并借 device selector 选设备。
  • 两套内存模型：buffer + accessor（声明读写，运行时自动建依赖图、自动搬数据）与 USM（malloc_device/shared，裸指针语义、显式同步，最贴近 CUDA）。
  • parallel_for 有 range（基础）和 nd_range（带 work-group，才能用 local_accessor = shared memory 和 it.barrier()）两种粒度。
  • 概念与 CUDA 一一对应：block=work-group、thread=work-item、warp=sub-group、__shared__=local_accessor、__syncthreads()=it.barrier()。
  • 错误处理分同步异常和异步异常（后者交给建 queue 时注册的 async_handler）。
  • 权衡：拿到可移植性，代价是相对手写 CUDA 可能损失峰值性能、够不到某些最新硬件特性——按场景取舍。
