第 13 章 Triton：用 Python 写块级 kernel

这一章想让你带走什么

前面三章（CUDA、SYCL）你都在线程级思考：算全局下标、管 shared memory、写 barrier，每一行都在操心"我这一个线程/work-item 做什么"。这一章是全书最大的一次心智跳跃：OpenAI 的 Triton 把抽象层从线程级抬到了块级（block-level）。

你用 Python 写 kernel，@triton.jit 把它编译到 GPU。你不再写带 threadIdx 的逐线程代码——每个"program"（tl.program_id）处理一整块元素，你操作的是长度为 BLOCK_SIZE 的向量，用"指针+偏移+掩码"来 tl.load/tl.store。块内的线程调度、合并访问、大部分 shared memory 管理，编译器替你做了。读完你会用一个浏览器里能跑的模拟器亲手体会"块级下标 + 掩码"，再看到教科书级的 Triton 向量加法长什么样。

一次抽象层的跃迁：从"一个线程"到"一个 program"

回忆第 1 章的 CUDA 向量加法：你写的是"一个线程处理一个元素"，然后启动上百万个线程。写 kernel 时你满脑子是 blockIdx.x * blockDim.x + threadIdx.x——线程级的下标算术。

Triton 让你换个粒度想问题。你写的是一个 program（程序实例），每个 program 负责处理一整块（BLOCK_SIZE 个）连续元素。program 之间由一个网格（grid）组织，grid = 需要多少个 program。在一个 program 内部，你不碰单个线程——你操作的是长度为 BLOCK_SIZE 的向量：一次 tl.load 读进一整块，做一次向量加法，一次 tl.store 写回一整块。至于这一块 512 个元素在硬件上被切成多少个线程、怎么合并访存、要不要走 shared memory——Triton 编译器全包了。

图 13-1　同一个向量加法：CUDA 让你面对 32 个线程、每个 1 个元素、下标全靠手算；Triton 让你面对 1 个 program、一个 BLOCK_SIZE 长的向量，线程和访存由编译器在底下自动摊开

grid、program_id 与掩码：块级模型的三件套

要读懂任何 Triton kernel，先掌握三个概念。

grid（网格）= program 的数量。 假设数组有 n_elements 个元素、每块 BLOCK_SIZE 个，那你需要 ceil(n_elements / BLOCK_SIZE) 个 program。Triton 提供 triton.cdiv(a, b)（向上取整除法）帮你算，启动语法是 kernel[grid](...)——方括号里放网格。

program_id = 我是第几个 program。 kernel 内用 pid = tl.program_id(axis=0) 取到自己的编号（一维网格用 axis=0）。这正对应 CUDA 的 blockIdx.x。有了 pid，我这一块负责的元素下标就是：

块起点 = pid * BLOCK_SIZE
这一块的所有下标 = 块起点 + [0, 1, 2, ..., BLOCK_SIZE-1]

其中那个 [0, 1, ..., BLOCK_SIZE-1] 用 tl.arange(0, BLOCK_SIZE) 生成——它是一个向量，这就是"块级"的精髓：offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE) 一次算出一整块的下标。

掩码（mask）= 处理"最后一块凑不齐"。 如果 n_elements 不能被 BLOCK_SIZE 整除，最后一个 program 会算出一些越界的下标。你不能让它们真的去读写内存。解决办法是造一个布尔向量 mask = offsets < n_elements，把它传给 tl.load(ptr + offsets, mask=mask) 和 tl.store(...)——为真的位置才真读写，为假的位置被安全地丢弃（load 时可给 other= 指定默认值）。这是 CUDA 里 if (i < n) 边界检查的向量化版本。

CUDA：线程级下标 + 手动合并
CUDA C++
__global__ void add(float* c, const float* a,
                    const float* b, int n) {
    // 每个线程一个标量下标
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n)                 // 逐线程边界检查
        c[i] = a[i] + b[i];    // 标量读、算、写
    // 合并访问是否达成？靠你安排下标
    // shared memory？要用就得自己声明、自己 sync
}
Triton：块级向量 + 掩码
Triton (Python)
@triton.jit
def add(a_ptr, b_ptr, c_ptr, n, BLOCK: tl.constexpr):
    pid = tl.program_id(axis=0)          # 我是第几块
    offs = pid * BLOCK + tl.arange(0, BLOCK)  # 一整块下标(向量)
    mask = offs < n                      # 向量化边界检查
    a = tl.load(a_ptr + offs, mask=mask) # 一次读一整块
    b = tl.load(b_ptr + offs, mask=mask)
    tl.store(c_ptr + offs, a + b, mask=mask)  # 一次写一整块
    # 合并访问、线程切分：编译器自动处理
}
一句话记住 Triton 的映射关系 一个 Triton program ≈ 一个 CUDA block；一个 program 处理的 BLOCK_SIZE 个元素 ≈ 一个 CUDA block 负责的那片数据。但关键区别在于：写 CUDA 你在 block 内部还要继续想"每个线程"；写 Triton 你停在块这一层，操作整块向量，块内的线程、合并访问、shared memory 交给编译器。你从"想线程"升级成了"想向量"。
为什么是 ≈，不是 =？ 因为它们只是角色对应，粒度并不相同。最容易踩的一点：BLOCK_SIZE 是一个 program 处理的元素个数，不是线程数——这和 CUDA 的 blockDim（那就是线程数）不一样。一个 BLOCK_SIZE=1024 的 program 在硬件上用多少线程，由编译器和 num_warps 决定，常常远少于 1024、每个线程循环处理多个元素。再加上：CUDA block 是"你要亲手管理其内部线程的容器"，Triton program 是"你只在块级写向量、内部交给编译器的抽象单元"。角色相同（网格里一个处理一片数据、彼此不能直接同步的独立单元），但粒度和归属不同——所以是 ≈ 而非 =。

为什么这对 ML kernel 更高效

块级抽象不是为了偷懒，是为了生产力和性能兼得，这在写机器学习算子时尤其明显：

  • 少写易错的下标算术：ML 里满是 softmax、layernorm、attention 这类"沿某个维度做归约再广播"的模式。用 CUDA 手写，光是把 tile 搬进 shared memory、算好每个线程的下标、插对 barrier，就能耗掉你大半天还容易错。Triton 里这些多数是编译器的活。
  • 编译器接管访存优化：合并访问、向量化、shared memory 的分配与同步——这些在 CUDA 里是你反复调优的重点，Triton 编译器会根据 BLOCK_SIZE 和硬件自动生成。
  • BLOCK_SIZE 是可调旋钮：它是一个 tl.constexpr（编译期常量）。改一个数字重编译，就能为不同 GPU 找到最优块大小——下一章的自动调优（autotuning） 就建立在这上面。
  • 融合天然：因为你在块级操作整块向量，把"读—算—算—写"多个步骤写进一个 kernel（算子融合）非常自然，能省掉中间结果反复读写显存的开销。这是 Triton 相对手写 CUDA 的真正杀手锏，下一章细讲。

动手（一）：用纯 Python 模拟块级下标 + 掩码

在碰真 Triton 之前，先在浏览器里跑一个只用标准库的模拟器，把"grid → program → offsets → mask"这条链亲手走一遍。我们故意选一个不能被块大小整除的长度（10 个元素，块大小 4），好让你看清最后一块的掩码是怎么救场的。

Python 
# 用纯 Python 模拟 Triton 的块级编程模型：program / offsets / mask
# 关键：数组长度 10 不能被 BLOCK 4 整除，看最后一块怎么靠掩码处理越界

N = 10          # 元素总数（故意不是 4 的倍数）
BLOCK = 4       # 每个 program 处理一块

a = [i for i in range(N)]        # [0,1,2,...,9]
b = [i * 10 for i in range(N)]   # [0,10,20,...,90]
c = [0] * N

def cdiv(x, y):                  # 向上取整除法，等价于 triton.cdiv
    return (x + y - 1) // y

num_programs = cdiv(N, BLOCK)    # grid：需要多少个 program
print(f"元素总数 N={N}, BLOCK={BLOCK} → 需要 {num_programs} 个 program\n")

def program(pid):
    # 一个 program 做的事：算一整块的下标向量，造掩码，读—加—写
    offs = [pid * BLOCK + k for k in range(BLOCK)]   # 这一块的下标(向量)
    mask = [o < N for o in offs]                     # 边界掩码：越界的为 False
    # 向量化 load：掩码为假的位置不真读，用默认值 0 顶替
    va = [a[o] if m else 0 for o, m in zip(offs, mask)]
    vb = [b[o] if m else 0 for o, m in zip(offs, mask)]
    vc = [x + y for x, y in zip(va, vb)]             # 一整块一起加
    print(f"program {pid}: offs={offs} mask={mask}")
    print(f"           load a={va} b={vb} → c={vc}")
    # 向量化 store：只写掩码为真的位置，越界的丢弃
    for o, m, val in zip(offs, mask, vc):
        if m:
            c[o] = val

for pid in range(num_programs):  # 想象这几个 program 在 GPU 上并行
    program(pid)

print("\n最终 c =", c)
print("验证：c[i] 应等于 a[i]+b[i] =", [a[i] + b[i] for i in range(N)])
print("一致吗？", c == [a[i] + b[i] for i in range(N)])

运行结果：

元素总数 N=10, BLOCK=4 → 需要 3 个 program

program 0: offs=[0, 1, 2, 3] mask=[True, True, True, True]
           load a=[0, 1, 2, 3] b=[0, 10, 20, 30] → c=[0, 11, 22, 33]
program 1: offs=[4, 5, 6, 7] mask=[True, True, True, True]
           load a=[4, 5, 6, 7] b=[40, 50, 60, 70] → c=[44, 55, 66, 77]
program 2: offs=[8, 9, 10, 11] mask=[True, True, False, False]
           load a=[8, 9, 0, 0] b=[80, 90, 0, 0] → c=[88, 99, 0, 0]

最终 c = [0, 11, 22, 33, 44, 55, 66, 77, 88, 99]
验证：c[i] 应等于 a[i]+b[i] = [0, 11, 22, 33, 44, 55, 66, 77, 88, 99]
一致吗？ True

看第 3 个 program（pid=2）：它算出的下标是 [8, 9, 10, 11]，但数组只到下标 9。掩码 [True, True, False, False] 把越界的 10、11 挡在门外——load 用默认值顶替，store 直接不写。这就是 Triton kernel 里那句 mask = offs < n_elements 的全部含义，也是块级模型处理"最后一块凑不齐"的标准姿势。请动手改：把 N 改成 12（正好 3 块整除），再运行——你会看到所有 mask 都变成 True，因为不再有越界。

为什么模拟器把 store 写成一个 for？ 真 Triton 里 tl.store(c_ptr + offs, vc, mask=mask) 是一次向量写操作，掩码直接作用在硬件层。我们用纯 Python 没有向量类型，只能用一个 for 循环逐个写、逐个判断掩码来模拟那次向量写的语义。请始终记住：在真 Triton 里，这一整块是一条指令，不是循环。
"program 内没有循环"是要点，不是巧合 注意模拟器里 program() 函数体内对一整块数据的处理，靠的是对向量的整体操作（列表推导只是 Python 没有向量类型的替代品）。真 Triton kernel 里，一个 program 处理一整块不写逐元素循环——你写的每一步（load、加法、store）都直接作用在长度 BLOCK_SIZE 的张量上。如果你在 Triton kernel 里写了个 for i in range(BLOCK_SIZE) 去逐个处理元素，多半就想错了粒度。

动手（二）：教科书级的 Triton 向量加法

下面是真正的 Triton 向量加法——Triton 官方教程的第一课，几乎是所有 Triton 代码的模板。它需要一块 NVIDIA GPU 才能跑（pip install triton torch），所以标为 triton 只高亮、不在浏览器运行。把它和上面的模拟器对照，你会发现结构一模一样：tl.program_id 就是 pid、tl.arange 就是那个 [0..BLOCK)、mask = offs < n 就是边界掩码。

Triton (Python) 
import torch
import triton
import triton.language as tl

@triton.jit
def add_kernel(
    a_ptr, b_ptr, c_ptr,   # 三个数组在显存里的首地址（指针）
    n_elements,            # 元素总数（运行时值，用于掩码）
    BLOCK_SIZE: tl.constexpr,  # 每块多少元素（编译期常量，可调旋钮）
):
    pid = tl.program_id(axis=0)              # 我是第几个 program（= blockIdx.x）
    block_start = pid * BLOCK_SIZE           # 我这一块的起始下标
    offsets = block_start + tl.arange(0, BLOCK_SIZE)  # 一整块的下标(向量)
    mask = offsets < n_elements              # 越界的位置为 False（向量化边界检查）

    a = tl.load(a_ptr + offsets, mask=mask)  # 一次读进一整块 a
    b = tl.load(b_ptr + offsets, mask=mask)  # 一次读进一整块 b
    c = a + b                                # 一整块向量相加
    tl.store(c_ptr + offsets, c, mask=mask)  # 一次写回一整块 c


def add(a: torch.Tensor, b: torch.Tensor) -> torch.Tensor:
    c = torch.empty_like(a)
    n_elements = a.numel()
    # grid：需要多少个 program。triton.cdiv 是向上取整除法
    grid = lambda meta: (triton.cdiv(n_elements, meta["BLOCK_SIZE"]),)
    # 用 kernel[grid](...) 启动，BLOCK_SIZE 作为编译期常量传入
    add_kernel[grid](a, b, c, n_elements, BLOCK_SIZE=1024)
    return c


if __name__ == "__main__":
    torch.manual_seed(0)
    size = 98432                              # 故意不是 1024 的倍数，考验掩码
    a = torch.rand(size, device="cuda")
    b = torch.rand(size, device="cuda")
    c_triton = add(a, b)
    c_torch = a + b                           # 用 PyTorch 的结果做基准
    max_diff = (c_triton - c_torch).abs().max().item()
    print(f"最大误差 = {max_diff}")           # 应为 0.0（同样的浮点加法）
    print("结果正确！" if max_diff == 0.0 else "结果不一致")
本地运行：pip install triton torch # 需要 NVIDIA GPU；然后：python main.py

在有 GPU 的机器上，它会打印：

最大误差 = 0.0
结果正确！

几个值得盯住的细节：

  • kernel 里全是向量操作，没有一句 for 遍历块内元素——tl.load/加法/tl.store 每一步都作用在整块上。这就是块级思维落到代码上的样子。
  • BLOCK_SIZE 通过关键字参数在启动时传入（BLOCK_SIZE=1024），因为它是 tl.constexpr，编译器会把它当常量特化生成代码。
  • grid 是一个 lambda，接收 meta（含 BLOCK_SIZE），返回一个元组 (num_programs,)。size=98432 不是 1024 的整数倍，最后一个 program 就靠 mask 兜底——和你在模拟器里看到的第 3 个 program 完全一个道理。
  • 启动语法 add_kernel[grid](...)：方括号里是网格，圆括号里是 kernel 参数。
Triton 也是可移植的，但路线不同于 SYCL Triton 最初瞄准 NVIDIA，如今也在扩展 AMD 等后端支持。但它的可移植性和 SYCL 不是一回事：SYCL 是"同一份 C++ 编到多家硬件"，Triton 是"用一个更高的 Python DSL 描述块级计算，由编译器针对目标硬件生成高效 kernel"。它牺牲了 CUDA 那种对硬件的极致细粒度控制，换来的是写 ML kernel 的开发速度——对绝大多数深度学习算子，这笔交易非常划算。

小结与下一步

Triton 让你在块级写 kernel：用 Python + @triton.jit，每个 program 处理一整块 BLOCK_SIZE 元素，你操作向量而非线程，用"指针 + tl.arange 偏移 + 掩码"完成 tl.load/tl.store，把线程调度、合并访问、shared memory 管理交给编译器。这比逐线程写 CUDA 在 ML 场景下高效得多。

但你现在看到的向量加法，还只是 Triton 的"Hello World"——它是个访存受限的算子，Triton 相对 CUDA 的优势还没真正体现。Triton 真正的回报在下一章：算子融合（fusion） 让你把一串操作压进一个 kernel、省掉中间结果反复进出显存；自动调优（autotuning） 让你把 BLOCK_SIZE 等参数交给框架自动搜出最优组合。等你写完一个融合的、自动调优的 softmax，才会真正明白为什么现代深度学习框架的底层，越来越多是 Triton 写的。

这一章的要点

  • Triton（来自 OpenAI）把抽象层从线程级抬到块级：你用 Python 写 kernel，@triton.jit 编译到 GPU，不写 threadIdx。
  • 核心模型：grid = program 数量；每个 program 用 pid = tl.program_id(axis=0) 拿到编号，处理一整块 BLOCK_SIZE 元素；offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE) 一次算出整块下标（向量）。
  • 掩码 mask = offsets < n_elements 是 CUDA 边界检查 if (i < n) 的向量化版本，专治"最后一块凑不齐"，配合 tl.load(ptr+offs, mask=mask) / tl.store(...) 使用。
  • 映射关系：一个 Triton program ≈ 一个 CUDA block；BLOCK_SIZE 元素 ≈ 一个 block 处理的数据；但你想向量、不想线程，块内线程/合并访问/shared memory 由编译器接管。
  • 启动用 kernel[grid](...)，网格大小用 triton.cdiv(n, BLOCK) 向上取整；BLOCK_SIZE 是 tl.constexpr 编译期常量，也是自动调优的旋钮。
  • 块级模型让写 ML 算子（softmax、attention 等）更高产，且天然适合融合——真正的回报（融合 + 自动调优）在第 14 章。
