第 6 章 真正的调度单位：Warp 与 Sub-group

这一章想让你带走什么

第 4、5 章讲了你在代码里写的执行层级——block / work-group。但硬件从来不会把一个 block 当成一个整体去调度。它会把 block 悄悄切成一片片 32 个线程的小队，NVIDIA 叫它 warp（线程束），这才是 GPU 真正的调度单位。SYCL/OpenCL 里的对应物叫 sub-group（子组）。

读完你会明白：为什么 block 大小几乎总该是 32 的倍数、"lane（通道）"是什么、第 2 章的 warp 分歧放到"一个 block 有好几个 warp"的尺度上是什么样，以及为什么一个 SM 上挂着越多 warp、GPU 反而越快。你还会跑一个模拟器，亲眼看到线程怎么打包进 warp、非 32 倍数的 block 浪费了多少条 lane，以及"对齐到 warp 边界的分歧其实不要钱"。

block 不是一整块：硬件把它切成 warp

回忆第 3 章那张硬件图：你启动的一个 block 会被整个分配到一个 SM 上。但 SM 内部并不是拿一个 block 当一个整体来跑的——它的调度器一次只认 32 个线程一组的小队。于是 SM 一拿到 block，做的第一件事就是把 block 里的线程按 threadIdx 顺序、每 32 个切成一个 warp。

一个 128 线程的 block，会被切成 4 个 warp：

  • warp 0：threadIdx.x = 0…31
  • warp 1：threadIdx.x = 32…63
  • warp 2：threadIdx.x = 64…95
  • warp 3：threadIdx.x = 96…127

这 32 个线程从此绑定在一起：它们同一时刻执行同一条指令（这正是第 2 章的 SIMT），一起被调度上算术单元，一起等内存。warp 才是 GPU 调度、执行、访存的原子单位——block 只是你写代码时的组织单位，warp 是硬件真正认的东西。

三层里多出来的一层 你写代码看到的是 grid → block → thread 三层（第 4 章）。但硬件在 block 和 thread 之间硬塞了一层 warp：grid → block →（硬件切分）warp → thread。这一层你在源码里几乎看不见，却决定了分歧、访存、占用率这三件最影响性能的事。理解 warp，是从"能写出正确的 kernel"迈向"能写出快的 kernel"的分水岭。

Lane：warp 内的门牌号

warp 里的 32 个线程，每个占一条 lane（通道），编号 0…31。一个线程的 lane 编号就是它在 warp 内部的位置：

CUDA C++
int lane   = threadIdx.x % 32;   // 我在自己 warp 里排第几（0..31）
int warpId = threadIdx.x / 32;   // 我属于 block 里的第几个 warp

lane 这个概念现在看着没用，但它是第 8 章 warp 级原语的基础：像"把 lane 3 的值直接递给 lane 0"这种线程间通信，靠的就是 lane 编号来点名。先记住这个门牌号。

图 6-1　硬件把 128 线程的 block 切成 4 个 warp。每个 warp 是 32 条 lane 的小队；分歧发生时，代价被局限在出现分歧的那个 warp 里

为什么 block 大小几乎总该是 32 的倍数

既然 SM 每 32 个线程切一个 warp，那当 block 大小不是 32 的倍数时会怎样？答案很直接：硬件会补齐到 32 的整数倍，多出来的 lane 全程空转、却照样占资源。

举个扎心的例子：一个 33 线程的 block。硬件切出来是 2 个 warp——第一个 warp 装满 32 个线程，第二个 warp 只装了 1 个线程，剩下 31 条 lane 全程"关灯"。你花了两个 warp 的调度和寄存器成本，却只干了 33 个线程的活，第二个 warp 有 97% 的算力在空转。

选 block 大小的第一条规矩 把 block 大小选成 32 的倍数（常见的是 128、256、512）。选 33、100、200 这种数，尾部一定有 lane 被浪费；更糟的是，warp 是资源分配的单位，那个"只装了 1 个线程的 warp"照样占满一个 warp 份额的寄存器和调度槽。这不是玄学，是硬件切分规则的直接后果——下面的模拟器会把浪费的 lane 精确数给你看。

Active mask：谁开着灯

第 2 章讲分歧时说"该走的干活、其余关灯"。硬件用什么记录"此刻这个 warp 里哪些 lane 开着灯"？——一个 32 位的整数，叫 active mask（活跃掩码）：第 i 位为 1 表示 lane i 此刻活跃（要执行当前指令），为 0 表示被屏蔽。

warp 遇到 if 分歧时，硬件先用"满足条件的 lane"这套掩码跑 if 分支，再用"取反的掩码"跑 else 分支——两趟，串行。掩码就是"关灯/开灯"的物理实现。第 8 章你会看到，很多 warp 级原语（如 __ballot_sync 返回的正是一个投票掩码）直接把这个 mask 交到你手里用。

换到 SYCL：sub-group

warp 是 NVIDIA 的叫法，也写死了 32 这个数。可移植的 SYCL/OpenCL 里，同一个概念叫 sub-group（子组），而且它的大小不保证是 32：在 NVIDIA 卡上通常是 32，在某些 AMD 卡上是 64，在 Intel GPU 上可能是 8/16/32。所以在 SYCL 里，你不能把 32 写死，必须在运行时查询。

CUDA：warp 与 lane（32 写死）
CUDA C++
// warp 大小是常量 32
int lane   = threadIdx.x % warpSize;  // warpSize==32
int warpId = threadIdx.x / warpSize;

// warp 级原语，隐含 32 条 lane
unsigned m = __activemask();
float v = __shfl_down_sync(m, x, 1);
SYCL：sub-group（大小要查询）
SYCL / DPC++
// 大小随硬件变，必须在运行时问
auto sg = it.get_sub_group();
uint32_t sz  = sg.get_local_range()[0]; // 8/16/32/64
uint32_t lid = sg.get_local_id()[0];    // 我在 sub-group 里排第几

// 对应的 shuffle：sycl::shift_group_left(sg, x, 1)
写可移植代码的纪律 CUDA 里 warpSize 恒为 32，你可以（虽然不推荐）当常量用；但一旦转到 SYCL，32 就成了 bug 的温床。规矩很简单：凡是和"一个调度小队里有几个线程"有关的逻辑，都去查询 sub_group_size，绝不硬编码。附录 B 有 CUDA ↔ SYCL 的完整术语对照表。

占用率预告：为什么"warp 越多越好"

第 3 章说 GPU 靠"用并行掩盖延迟"——某个 warp 卡在等显存（几百个周期）时，调度器立刻切到另一个就绪的 warp 接着算，等待就被填满了。这里的关键词是"另一个就绪的 warp"：只有当一个 SM 上同时驻留足够多的 warp，调度器才有牌可打。

一个 SM 能同时驻留的 warp 数是有上限的（由寄存器、shared memory 等资源决定）。"实际驻留的 warp 数 ÷ 硬件上限"就是 占用率（occupancy）。占用率低，意味着一旦手头的 warp 都在等内存，调度器无 warp 可切，算术单元就空转——延迟藏不住了。

把三章串起来 第 1 章："GPU 用海量并行掩盖延迟"；第 3 章："warp 调度器负责切换"；这一章补上最后一环：调度器能切换的前提，是 SM 上驻留着很多 warp。这就是为什么 GPU 要你开成千上万个线程——不是为了"同时算完"，而是为了给调度器足够多的 warp 来掩盖访存延迟。占用率怎么定量计算、多高才够，是第 16 章的主题。

warp 级原语：一瞥（细节在第 8 章）

既然一个 warp 的 32 个线程本就绑在一起、齐步执行，硬件干脆提供了让它们直接交换寄存器值、绕过内存的指令，叫 warp 级原语（warp-level primitives）。两个最常用的：

  • __shfl_sync / __shfl_down_sync：让一条 lane 直接读另一条 lane 的寄存器值（shuffle，洗牌）。
  • __ballot_sync：让 warp 里每条 lane 投一票（真/假），返回一个 32 位掩码。

它们的威力在于不碰 shared memory、不碰全局内存，纯寄存器间通信，极快。经典用法是"warp 内求和"——用 shuffle 做一棵折叠的归约树，5 步（log₂32）就把 32 个值加成 1 个：

CUDA C++
// warp 内 32 个 lane 的值求和，全程不碰内存（预告第 8、18 章）
__inline__ __device__ float warpReduceSum(float val) {
    // 每一步把"距离减半"的 lane 的值加过来：16→8→4→2→1
    for (int offset = 16; offset > 0; offset >>= 1)
        val += __shfl_down_sync(0xffffffff, val, offset);
    return val;   // lane 0 拿到的就是整个 warp 的总和
}

0xffffffff 是那个 32 位掩码，表示"32 条 lane 全部参与"。现在不用抠语义，只需记住：warp 不只是调度单位，还是一个能高速内部协作的小团队。归约、扫描这些经典并行模式（第 18 章）都建立在它之上。

动手：线程如何打包进 warp，分歧到底贵在哪

下面这个模拟器能在浏览器里直接跑——点 ▶ 运行。它做三件事：把一个 block 按 32 切成 warp、数出非 32 倍数时浪费了多少条 lane，然后把第 2 章的分歧模型升级到整个 block（好几个 warp）的尺度，对比两种分工：一种让分歧发生在 warp 内部，一种让分歧对齐到 warp 边界。你会看到后者几乎不花钱。

Python 
# 模拟：线程怎么打包进 warp + 分歧在 warp 尺度上的代价
WARP = 32

def pack_into_warps(block_size):
    num_warps = (block_size + WARP - 1) // WARP     # 不够 32 的向上补齐
    total_lanes = num_warps * WARP
    wasted = total_lanes - block_size                # 空转的 lane
    print(f"  block={block_size:>4} → {num_warps} 个 warp（{total_lanes} 条 lane），"
          f"干活 {block_size}，空转 {wasted}")
    return num_warps, wasted

print("一、线程如何打包进 warp（注意非 32 倍数的浪费）")
for bs in [32, 128, 256, 33, 100]:
    pack_into_warps(bs)

def branch_cost(block_size, predicate, if_cost=3, else_cost=3):
    """把 block 切成 warp，逐 warp 计分：
    warp 内全走同一分支 → 只付那一条分支的钱；
    warp 内走向不同（分歧）→ if 和 else 都要付，串行。"""
    num_warps = (block_size + WARP - 1) // WARP
    total = 0
    diverged_warps = 0
    for w in range(num_warps):
        preds = [predicate(w * WARP + lane)
                 for lane in range(WARP)
                 if w * WARP + lane < block_size]
        any_if   = any(preds)
        any_else = any(not p for p in preds)
        cost = (if_cost if any_if else 0) + (else_cost if any_else else 0)
        if any_if and any_else:
            diverged_warps += 1
        total += cost
    return total, diverged_warps, num_warps

BS = 128  # 4 个 warp
# 方案 A：按奇偶分工 → 每个 warp 都是一半一半 → 每个 warp 都分歧
a_total, a_div, nw = branch_cost(BS, lambda tid: tid % 2 == 0)
# 方案 B：按 warp 边界分工（warpId 的奇偶）→ 每个 warp 内部一致 → 无分歧
b_total, b_div, _  = branch_cost(BS, lambda tid: (tid // WARP) % 2 == 0)

print(f"\n二、同一个 128 线程 block（{nw} 个 warp），两种分工的代价")
print(f"  方案 A 按奇偶（tid%2）：{a_div}/{nw} 个 warp 分歧，共花 {a_total} 步")
print(f"  方案 B 按 warp 边界   ：{b_div}/{nw} 个 warp 分歧，共花 {b_total} 步")
print(f"  → 结果完全一样，但 A 比 B 慢了 {a_total / b_total:.0f} 倍")
print("  原因：A 的分歧落在每个 warp 内部（串行两条分支）；")
print("        B 的分歧对齐到 warp 边界，每个 warp 只走一条路 → 不花钱。")

运行结果：

一、线程如何打包进 warp（注意非 32 倍数的浪费）
  block=  32 → 1 个 warp（32 条 lane），干活 32，空转 0
  block= 128 → 4 个 warp（128 条 lane），干活 128，空转 0
  block= 256 → 8 个 warp（256 条 lane），干活 256，空转 0
  block=  33 → 2 个 warp（64 条 lane），干活 33，空转 31
  block= 100 → 4 个 warp（128 条 lane），干活 100，空转 28

二、同一个 128 线程 block（4 个 warp），两种分工的代价
  方案 A 按奇偶（tid%2）：4/4 个 warp 分歧，共花 24 步
  方案 B 按 warp 边界   ：0/4 个 warp 分歧，共花 12 步
  → 结果完全一样，但 A 比 B 慢了 2 倍
  原因：A 的分歧落在每个 warp 内部（串行两条分支）；
        B 的分歧对齐到 warp 边界，每个 warp 只走一条路 → 不花钱。

两个结论一次看清。第一段：block=33 白白空转 31 条 lane——这就是"block 大小要取 32 的倍数"的量化证据。第二段：同样是"一半线程走 if、一半走 else"，按奇偶分让每个 warp 都撕成两半、串行跑两条分支；按 warp 边界分让每个 warp 内部意见一致，各走各的路，总代价直接减半。请动手改：把方案 B 的判据从 (tid // WARP) % 2 == 0 改成 tid < 64（前 64 个线程走 if）——因为 64 正好是 2 个 warp 的边界，你会发现它依然 0 分歧。这就是第 2 章那句话的精确版：分歧的代价按 warp 结算，让走向不同的线程落到不同的 warp 里，分歧就是免费的。

小结与下一步

你现在知道了 GPU 真正的调度单位不是 block，而是 block 被切出来的一个个 32 线程的 warp（SYCL 里叫 sub-group，大小要查询）。这把第 2 章的分歧、第 3 章的延迟掩盖，都落到了"warp"这个具体的硬件粒度上。

warp 内部能高速协作，靠的是它们共享一个东西——但 warp 之间、block 里不同 warp 之间要交换数据，就得靠内存。而内存，正是决定 GPU 性能的另一半答案。下一章我们拆开 GPU 的内存层级：寄存器、shared memory、全局内存三级差着数量级的速度，以及那个能让性能差出几十倍的核心概念——合并访问（memory coalescing）：当一个 warp 的 32 条 lane 访问连续地址时，会发生什么好事。

这一章的要点

  • 硬件从不把 block 当整体调度，而是按 threadIdx 顺序每 32 个线程切一个 warp（NVIDIA）。warp 才是 GPU 真正的调度、执行、访存单位。
  • warp 内每个线程占一条 lane（0…31），lane = threadIdx.x % 32；lane 编号是 warp 级原语点名通信的基础。
  • block 大小几乎总该取 32 的倍数：33 线程会切成 2 个 warp、白白空转 31 条 lane，还照样占资源。
  • active mask 是 warp 里"哪些 lane 开着灯"的 32 位记录；分歧就是用两套互补的掩码把 if/else 串行跑两趟。
  • SYCL/OpenCL 的对应物是 sub-group，大小不保证是 32（可能 8/16/32/64），必须运行时查询，不能硬编码。
  • SM 上驻留越多 warp，调度器越有牌可切来掩盖延迟——这就是占用率（occupancy），第 16 章定量展开。
  • warp 级原语（__shfl_down_sync、__ballot_sync）让 32 条 lane 绕过内存直接协作，是归约/扫描的基础（第 8、18 章）。
  • 分歧的代价按 warp 结算：warp 内分歧才串行付费；把分歧对齐到 warp 边界，它就是免费的。
