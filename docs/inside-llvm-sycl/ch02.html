第 2 章 LLVM IR：Module / Function / BasicBlock / Instruction 与 SSA

这一章想让你带走什么

第 1 章你已经知道 IR 是 LLVM 的中心，也扫过一眼 define i32 @add 长什么样。这一章把镜头推到最近，正式解剖 IR 的数据结构：一个 Module 里装着若干 Function，一个 Function 里装着若干 BasicBlock，一个 BasicBlock 里装着一串 Instruction。这是「容器」的一面。

但 IR 真正的精髓在另一面：所有指令又通过 Value / User / Use 三个类织成一张 def-use 图——「谁产生了这个值、谁又用了它」。再叠加 SSA（每个值只被赋值一次）这条铁律，你就理解了为什么 LLVM 的优化能写得又快又准。读完你能对着 llvm/include/llvm/IR/ 下那几个头文件（Module.h、Function.h、BasicBlock.h、Instruction.h、Value.h）把 IR 的骨架讲清楚，也为第 4 章「写一个改 IR 的 Pass」打好地基。

两条正交的结构：容器树 vs def-use 图

初读 LLVM IR 最容易只看到「一层套一层」的容器关系，而漏掉更重要的那张图。其实 IR 上同时存在两套正交的结构：

  • 容器树（嵌套/包含关系）：Module ⊃ Function ⊃ BasicBlock ⊃ Instruction。回答的是「这条指令在物理上位于哪里、按什么顺序排列」。
  • def-use 图（数据依赖关系）：一条指令算出一个值，别的指令把这个值当操作数用。回答的是「这个值是谁算的、又被谁用了」。它跨越基本块、跨越容器层级。

同一条 add 指令，既是它所在基本块指令链表里的一个节点（容器树的一环），又是 def-use 图里的一个节点（它 def 出 %sum，同时 use 了 %a、%b）。把这两套结构分开看，是读懂 IR 的关键。

图 2-1　IR 上并存的两套结构：左边的容器树管「物理位置与顺序」，右边的 def-use 图管「数据依赖」。优化 Pass 大多在右边这张图上工作——顺着 use 找到所有用到某值的地方，是改 IR 最常见的动作

容器树：从 Module 到 Instruction

先看容器这一面。四层各对应 llvm/include/llvm/IR/ 下一个头文件里的一个类。

Module（Module.h） 是一次编译的顶层容器，对应一个翻译单元（大致一个 .ll / .bc 文件）。它装着这个模块里所有的函数（Function）、全局变量（GlobalVariable）、以及目标数据布局（DataLayout）、目标三元组（target triple）等元信息。你可以把它想成「一个 IR 文件的根」。

Function（Function.h） 是一个函数。它有返回类型、参数列表（Argument），函数体则是一串按顺序排列的基本块。空函数体（只有声明、没有定义）就是一个 declare。

BasicBlock（BasicBlock.h） 是一段顺序执行、中途不分叉也不汇入的指令序列。它有两条硬性规定：

  • 只能从第一条指令进入（不能跳到基本块中间）。
  • 必须以且仅以一条「终结指令（terminator）」结尾——ret、br（分支）、switch 等。控制流只在基本块的末尾发生转移。

正因为这两条，基本块成了控制流图（CFG）的天然节点：块与块之间靠终结指令连边。

Instruction（Instruction.h / Instructions.h） 是最小的操作单位：一次加法（add）、一次内存读（load）、一次调用（call）。绝大多数指令会「算出一个值」，这个值可以被后面的指令当操作数用——这就接到了下一节的 def-use 图。

用侵入式链表串起来，不是 std::vector Function 里的 BasicBlock、BasicBlock 里的 Instruction，都用 LLVM 自己的侵入式链表 ilist 串接（节点自带前后指针）。好处是：在基本块中间插入 / 删除一条指令是 O(1) 的，且插删时已有的指针、迭代器不失效。优化 Pass 天天在做「删掉这条、在那条前面插一条」，链表比数组合适得多。你在源码里会看到 BB.getInstList()、I.eraseFromParent() 这类 API，背后就是这个链表。

def-use 图：Value、User、Use 三个类

这是 LLVM IR 设计里最漂亮的一块，也是新手最容易忽略的。核心是三个类（都在 llvm/include/llvm/IR/ 下）：

  • Value（Value.h） —— 「一个可以被使用的值」。凡是能被别的指令当操作数引用的东西，都是 Value：指令的结果、函数参数、常量、甚至函数本身和基本块（br 要引用目标块）。几乎 IR 里的一切都是 Value 的子类。
  • User（User.h） —— 「一个会使用别的值的东西」。指令是典型的 User：add 这个 User 使用了 %a、%b 两个 Value。User 是 Value 的子类（因为一条指令既用别人、自己也是个能被用的值）。
  • Use（Use.h） —— 连接两者的一条边：它记录「哪个 User 的哪个操作数位，指向哪个 Value」。

关系是这样：每个 Value 内部挂着一条 use 链表，串起所有「使用了我」的 Use（这叫它的 users）；每个 User 则持有一组 Use，指向它「用到的」那些 Value（这叫它的 operands）。于是这张图能双向遍历：

  • 从一条指令出发问「我用了哪些值」→ 遍历它的 operands（I.operands() / I.getOperand(i)）。
  • 从一个值出发问「谁用了我」→ 遍历它的 users（V.users() / V.use_begin()）。

第二个方向是优化的命根子。比如做常量传播：算出某个 %x 恒等于常量 42，我就顺着 %x 的 users 找到所有用到它的指令，把操作数换成 42。做死代码消除：一个指令的结果没有任何 user（use 链表为空）且无副作用，它就是死的，可以删。这些都是 O(该值被用到的次数)，不用扫全函数——因为 def-use 图把「谁用了我」直接存好了。

def-use 图是「可导航的数据依赖」 别的编译器常常要「扫一遍代码才知道某个变量在哪被用到」。LLVM 把这层关系物化成 Use 边、挂在每个 Value 上，让「谁定义了它 / 谁使用了它」变成一次链表遍历。你写的几乎每个优化 Pass，本质都是在这张图上走：找到某个 Value，顺着它的 users 改写或删除。记住 V.users() 和 I.getOperand(i) 这两个方向，就抓住了操作 IR 的手柄。

SSA：每个值只被赋值一次，与 PHINode

def-use 图之所以如此干净好用，靠的是 IR 遵守 SSA（Static Single Assignment，静态单赋值） 形式：

每个值（Value）在程序文本里只被定义一次。

这就是为什么 IR 里到处是 %1、%2、%sum 这种编号只增不改的临时值，而不是像 C 里那样反复给同一个 x 赋值。一个值定义出来后就不再变，于是「这个值是什么」有唯一答案——def-use 图里每个 Value 只有一个「定义点」，分析可以直接顺着边推理，不必担心「它在别处又被改了」。

但真实程序有分支和循环，同一个变量在不同路径上会有不同的值。比如：

C++
// 源码里的 x 在两条路径上取不同值
int x;
if (cond) x = 10;
else      x = 20;
return x;          // 这里的 x 到底是哪个？

SSA 怎么表达「这里的 x 取决于走了哪条路径」？答案是 PHINode（φ 节点）——一种特殊指令（PHINode，定义在 llvm/include/llvm/IR/Instructions.h），放在多条控制流汇合处的基本块开头，形式是「若来自前驱块 A 则取值 v1，来自前驱块 B 则取值 v2」：

LLVM IR
; if/else 汇合处，用 phi 选出正确的 x
if.then:
  br label %if.end
if.else:
  br label %if.end
if.end:
  %x = phi i32 [ 10, %if.then ], [ 20, %if.else ]  ; 来自 then 取 10，来自 else 取 20
  ret i32 %x

phi 让「合流处的值」依然只被定义一次（就是这条 phi 指令），从而在有分支的情况下也维持了 SSA。你在读循环相关的 IR 时会大量看到 phi——循环归纳变量（i = i + 1）就是靠一个 phi 在「循环入口的初值」和「上一轮迭代的值」之间选择的。

SSA 不等于「变量只能赋值一次」 SSA 约束的是 IR 里的值，不是源码里的变量。源码里 x = x + 1 反复给 x 赋值完全合法——前端会把它拆成一串各自单次定义的 SSA 值（%x1、%x2…），或者先用内存（alloca + load/store）绕过，再由 mem2reg Pass 提升成寄存器 SSA 值并插入必要的 phi。所以别被 alloca 满天飞的 -O0 IR 迷惑——那是「还没转成漂亮 SSA」的形态，开了优化后 phi 才会现身。

亲手看一眼：带分支的 IR

把上面 if/else 那段编出来（clang -O1 -S -emit-llvm），核心 IR 大致是：

LLVM IR
define i32 @pick(i1 %cond) {
entry:
  br i1 %cond, label %if.then, label %if.else   ; 终结指令：按 cond 分支
if.then:
  br label %if.end                              ; 终结指令：无条件跳
if.else:
  br label %if.end
if.end:                                         ; 两条路径在此汇合
  %x = phi i32 [ 10, %if.then ], [ 20, %if.else ]
  ret i32 %x
}

对着它把这一章的东西全认一遍：@pick 是 Function；entry/if.then/if.else/if.end 是四个 BasicBlock，每个都以一条终结指令（br 或 ret）结尾；%cond、10、20、%x 都是 Value；那条 phi 是个 User，它的 operands 里既有值（10、20）也有来源块（%if.then、%if.else）；而 ret 这个 User 使用了 %x——顺着 %x 的 users 你只会找到这一条 ret。这一小段就是容器树、def-use 图、SSA、phi 四件事的合影。

动手：用 def-use 图做死代码消除

把 def-use 图和 SSA 浓缩成一个能在浏览器里跑的小模型：几条 SSA 指令，每条 def 出一个值、use 若干个值。我们建出「谁用了谁」的 use 链，然后反复删除「没有任何 user 且无副作用」的指令——这正是 DCE（第 4 章 DCEPass）的内核。点 ▶ 运行。

Python 
# 用 def-use 图做死代码消除（DCE）的极简模型

# 每条指令：名字（它 def 出的 SSA 值）、用到的操作数、是否有副作用
instrs = [
    {"def": "%a",   "uses": [],            "side_effect": False},  # %a = load ...
    {"def": "%b",   "uses": [],            "side_effect": False},
    {"def": "%sum", "uses": ["%a", "%b"],  "side_effect": False},  # %sum = add %a, %b
    {"def": "%dead","uses": ["%a"],        "side_effect": False},  # 没人用 → 死代码
    {"def": None,   "uses": ["%sum"],      "side_effect": True},   # store %sum（有副作用，是"根"）
]

def build_users(instrs):
    # 反向索引：值 -> 用到它的指令下标集合（这就是 def-use 图的 use 链）
    users = {i["def"]: set() for i in instrs if i["def"]}
    for idx, ins in enumerate(instrs):
        for v in ins["uses"]:
            if v in users:
                users[v].add(idx)
    return users

alive = [True] * len(instrs)
removed = True
round_no = 0
while removed:                     # 反复扫，直到没有可删的（删一条可能让别的也变死）
    removed = False
    round_no += 1
    users = build_users([ins for i, ins in enumerate(instrs)])
    # 重新统计每个值当前还有几个"活着的" user
    live_user_count = {i["def"]: 0 for i in instrs if i["def"]}
    for idx, ins in enumerate(instrs):
        if not alive[idx]:
            continue
        for v in ins["uses"]:
            if v in live_user_count:
                live_user_count[v] += 1
    for idx, ins in enumerate(instrs):
        if not alive[idx]:
            continue
        # 有副作用的指令是"根"，永远不删；没有 def 的（如 store）也当有副作用
        if ins["side_effect"] or ins["def"] is None:
            continue
        if live_user_count.get(ins["def"], 0) == 0:   # 没有任何活着的 user
            print(f"[第 {round_no} 轮] 删除死指令 {ins['def']}（无 user、无副作用）")
            alive[idx] = False
            removed = True

print("\n存活的指令：")
for idx, ins in enumerate(instrs):
    if alive[idx]:
        tag = ins["def"] if ins["def"] else "store（副作用根）"
        print(f"  {tag}")

运行结果：

[第 1 轮] 删除死指令 %dead（无 user、无副作用）

存活的指令：
  %a
  %b
  %sum
  store（副作用根）

看清这个过程：%dead 虽然算了个值，但没有任何 user 且无副作用，第一轮就被删。%a 虽然也被 %dead 用过，但它还被 %sum 用着，所以活下来。真实的 DCE 就是这样从「有副作用的根」（store、ret、call）出发，把不可达到根的值全部回收——而它能一眼看出「谁用了谁」，全靠 def-use 图这张预先织好的网。请动手改：把最后那条 store %sum 删掉（模拟没人用 %sum），再运行——你会看到 %sum、%a、%b 像多米诺骨牌一样被连锁删除，因为失去根之后它们全成了死代码。

小结与下一步

你现在真正「读得懂」一段 IR 了：

  • IR 上有两套正交结构：容器树（Module ⊃ Function ⊃ BasicBlock ⊃ Instruction，用侵入式链表串接，插删 O(1)）与 def-use 图（数据依赖）。
  • def-use 图由三个类织成：Value（能被用的值，IR 里几乎一切都是它）、User（会用别的值的东西，如指令）、Use（连接两者的一条边）。它能双向走：I.getOperand(i) 找操作数、V.users() 找使用者。
  • 后一个方向（顺着 users 改写/删除）是几乎所有优化 Pass 的核心动作。
  • IR 遵守 SSA：每个值只定义一次；分支汇合处用 PHINode 选值以维持 SSA。-O0 的 alloca 形态经 mem2reg 才提升成漂亮的寄存器 SSA。
  • BasicBlock 必以一条终结指令结尾，这让它成为 CFG 的天然节点。

下一章我们看 IR 的三种形态：这一章讲的都是内存里的 C++ 对象（Module/Function/…），但同一段 IR 还能落成人可读的文本 .ll、和紧凑的二进制 .bc。看清三者如何无损互转、各自用什么工具读写，你就理解了第 1 章说的「IR 能落盘」到底是怎么实现的。

这一章的要点

  • IR 有两套正交结构：容器树（Module/Function/BasicBlock/Instruction，头文件在 llvm/include/llvm/IR/，用 ilist 侵入式链表串接）与 def-use 图。
  • def-use 图 = Value（Value.h，能被引用的值）+ User（User.h，Value 的子类，持有 operands）+ Use（Use.h，一条边）；每个 Value 挂着 use 链表记录「谁用了我」。
  • 两个遍历方向：I.getOperand(i)/I.operands()（我用了谁）与 V.users()（谁用了我）；后者是优化 Pass 改写/删除的核心手柄。
  • IR 是 SSA：每个值只定义一次；控制流汇合处用 PHINode（Instructions.h）选值。-O0 的 alloca+load/store 经 mem2reg 提升为寄存器 SSA。
  • BasicBlock 只能从头进入、必以一条终结指令（ret/br/switch…）结尾，因而是 CFG 的天然节点。
