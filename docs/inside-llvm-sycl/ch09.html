第 9 章 异构 offload：bundler / wrapper / linker-wrapper 工具链

这一章想让你带走什么

第 8 章设备代码编成了 SPIR-V（或第 11 章的 PTX），但它还只是一段孤立的二进制。要让程序真正跑起来，它必须和主机端代码打包进同一个可执行文件，运行时才能在启动 kernel 时把它取出来交给驱动。这一章讲的就是完成这件事的 offload 工具链——一组 clang-offload-* / clang-linker-wrapper 工具，源码在 clang/tools/ 下。

你会看清「胖二进制（fat binary）」是怎么造出来的：设备镜像如何被嵌入主机目标文件的特殊 section、如何被包装成带注册代码的 bitcode、以及链接时那个包装器（linker-wrapper）如何扫描并抽取这些嵌入数据、驱动设备端链接、最终产出一个「主机 + 多个设备镜像」合体的可执行文件。这是连接「编译期」（前 8 章）和「运行期」（第 10 章 kernel 上设备）的关键一环。

问题：主机和设备代码，最终得在一个文件里

回顾第 1、8 章那条链：一份源码，设备部分编成了 SPIR-V/PTX，主机部分编成了普通 x86 目标文件。现在面对一个很实际的工程问题：这些东西最后怎么变成一个能双击运行的可执行文件？

约束是苛刻的：

  • 只能有一个可执行文件：用户期望 ./myapp 就能跑，不想管理一堆散落的 .spv/.ptx。
  • 可能有多个设备镜像：同一个 kernel 可能编了 Intel GPU 的 SPIR-V、NVIDIA 的 PTX、甚至 CPU fallback——都得塞进去，运行时按实际设备挑。
  • 主机端要能在运行时找到它们：程序启动时得知道「我身上带着哪些设备镜像、每个对应什么设备」，才能在提交 kernel 时取用。

解决方案就是 胖二进制（fat binary）：把设备镜像作为数据嵌进主机可执行文件里，再附上一段启动时自动运行的注册代码，把这些镜像登记到 offload 运行时。造这个胖二进制的活，就分给了下面几个工具。

图 9-1　offload 打包链路。设备镜像先被嵌进主机目标文件的 .llvm.offloading section；链接时 clang-linker-wrapper 扫描并抽取它们、驱动设备端链接、再用 wrapper 生成"启动时注册镜像"的代码，最终产出主机与设备合体的胖二进制

工具家族：clang/tools/ 下的 offload 工具

这套工具都在 clang/tools/ 下，各司一职。先认清楚谁干什么（本书代码库即 intel/llvm 的 sycl 分支，以下都以它为准）：

  • clang-offload-bundler（clang/tools/clang-offload-bundler/）：打包 / 拆包。把多个目标的设备对象捆成一个「bundled 文件」，或反过来拆开。它建立在一个 OffloadBundler API 之上。经典 offload 流程里用它把不同 target 的产物打成一捆。
  • clang-offload-wrapper（clang/tools/clang-offload-wrapper/）：包装成可注册的 bitcode。它的源码注释说得很直白——「接收 offload 目标二进制，生成一个 wrapper bitcode 文件，把这些二进制作为数据打包进去；wrapper bitcode 还包含初始化代码，在程序启动时把目标二进制注册到 offload 运行时」。这就是图里那段「启动时自动注册」代码的来源。
  • clang-linker-wrapper（clang/tools/clang-linker-wrapper/）：链接期的总指挥。注释：「作为链接任务的包装器……用来为 offload 创建链接好的设备镜像。它扫描链接器的输入里嵌在 .llvm.offloading section 的 offload 数据、抽取出来……再交给设备端链接任务生成最终设备镜像」。它是把前面几件事在链接时串起来的那个工具。
  • clang-sycl-linker（clang/tools/clang-sycl-linker/）：SYCL 专用的设备链接器。注释：「执行链接 SYCL 设备镜像所需的一系列步骤」——链接 LLVM bitcode、链接 bitcode 库（如 libclc/libdevice）、跑 SYCL 特有的 post-link 步骤、生成最终镜像。
  • 还有 clang-offload-extract、clang-offload-deps、clang-nvlink-wrapper、offload-arch/（枚举本机 GPU，含 NVPTXArch.cpp、LevelZeroArch.cpp 等）作为辅助。
此分支里没有 clang-offload-packager 上游主线 LLVM 有一个 clang-offload-packager 工具，但在本书用的 intel/llvm sycl 分支里它并不存在（只有一个 driver 测试 clang/test/Driver/offload-packager.c 顺带提到这个名字）。这类差异是"读真实源码"必然会遇到的——上游 LLVM 和厂商分支的工具集会有出入。本章按 sycl 分支实际存在的工具来讲（bundler / wrapper / linker-wrapper / sycl-linker）。你在自己的树里 ls clang/tools/ | grep offload 确认一下，是个好习惯。

关键机制一：把设备镜像「嵌」进主机对象

胖二进制的核心技巧，是把设备镜像当作普通数据塞进主机目标文件的一个特殊 section——在 LLVM offload 里这个 section 叫 .llvm.offloading。

为什么用 section？因为目标文件（ELF 等）本来就支持任意命名的数据 section，链接器会原样保留、合并它们。于是设备镜像搭上主机对象的"便车"：主机代码正常编译链接，设备镜像作为 .llvm.offloading section 里的字节一路跟着走，最后落在可执行文件里。这一步在编译设备端时由前端/驱动完成（把设备产物 embed 进去）。

这带来一个漂亮的性质：对系统链接器（ld）来说，带 offload 数据的目标文件和普通目标文件没区别——它只是多了个它不认识但会保留的 section。真正"读懂"这个 section 的是下一步的 linker-wrapper。

为什么 .bc（第 3 章）在这里很关键 被嵌进 .llvm.offloading 的设备侧数据，很多时候是 LLVM 位码（.bc）而非最终的 SPIR-V/PTX——因为设备端的最终链接（把多个翻译单元的设备 .bc 合并、链上 libclc/libdevice）要推迟到链接期才做。第 3 章说".bc 可嵌入是异构 offload 的物理前提"，这里就是兑现：正因为 IR 能落成可嵌入的 .bc，设备代码才能以数据形式搭主机对象的便车、延迟到链接期再统一处理。

关键机制二：linker-wrapper 在链接期收网

普通编译最后一步是调链接器 ld 把一堆 .o 链成可执行文件。offload 编译则用 clang-linker-wrapper 包住这个链接步骤——它在真正调 ld 前后插入设备端的处理。它干三件事：

  1. 扫描并抽取：遍历所有输入目标文件，找出 .llvm.offloading section，把里面嵌的设备镜像/.bc 抽出来，按目标（Intel SPIR-V / NVIDIA PTX / …）分组。
  2. 驱动设备端链接：对每一组，调用对应的设备链接工具（SYCL 走 clang-sycl-linker，NVIDIA 走 clang-nvlink-wrapper 等）把该目标的所有设备 .bc 合并、链上设备库、跑 post-link、生成该目标最终的设备镜像（SPIR-V/PTX/…）。
  3. 生成注册代码并合并：把这些最终设备镜像交给 wrapper 逻辑，生成"启动时把镜像注册到 offload 运行时"的 bitcode，再连同主机对象一起交给系统链接器，链成最终胖二进制。

一句话：linker-wrapper 是把「散落在各主机对象里的设备数据」在链接期收拢、完成设备端链接、再打包注册的总调度。它让整条 offload 编译对用户表现为"就是普通的 clang -fsycl a.cpp -o a.out"，背后的设备打包全被这一层藏起来了。

offload 工具链 = 让「一个可执行文件装下主机+多设备」成为可能 设备镜像以 .bc/二进制形式嵌进主机对象的 .llvm.offloading section（搭便车）；clang-linker-wrapper 在链接期扫描抽取、按目标驱动设备端链接（clang-sycl-linker 等）、再用 wrapper 逻辑生成"启动时注册镜像"的代码，合成胖二进制。这套机制把"编译期产出的多份设备镜像"和"运行期要用它们的主机程序"缝合到一个文件里——正是它让第 10 章的运行时能在程序启动后"就地"找到镜像并上设备。

亲手看一眼：胖二进制里的 offload section

你可以用标准的 ELF 工具直接观察胖二进制里那个 offload section。编一个最小 SYCL 程序（或任意 offload 程序）后：

终端
# 看目标文件/可执行文件里有没有 offload section
llvm-objdump --section-headers a.out | grep offloading
# 或用 readelf 看 section 列表
readelf -S a.out | grep llvm.offloading

你会看到类似 .llvm.offloading 的 section 条目——那就是设备镜像"搭便车"的地方。还能用 offload 专用工具把内容抽出来看：

终端
# 用 clang-offload-extract 把嵌入的设备镜像抽取出来（sycl 分支提供）
clang-offload-extract a.out
# 或者用 bundler 拆一个 bundled 文件
clang-offload-bundler --unbundle --type=o --targets=... -i bundled.o -o device.out

这就是"IR/镜像能落盘、能被独立工具处理"（第 1、3 章）在 offload 场景的又一次体现：整条链路的每个中间产物都能停下来、用工具打开检查——调试 offload 问题时，第一步往往就是 readelf -S 看镜像到底有没有被正确嵌进去。

动手：模拟胖二进制的打包与运行时挑镜像

把这一章的机制——「设备镜像嵌进主机对象 → linker-wrapper 收拢并加注册代码 → 运行时按设备挑镜像」——浓缩成一个浏览器里能跑的小模型。点 ▶ 运行。

Python 
# 模拟 offload 胖二进制：嵌入设备镜像 → 链接期收拢注册 → 运行时按设备挑镜像

# ---- 编译期：每个主机对象里"嵌"着若干设备镜像（.llvm.offloading section）----
host_objects = [
    {"name": "main.o",  "offloading": [("spir64", "<VecAdd 的 SPIR-V>"),
                                        ("nvptx64", "<VecAdd 的 PTX>")]},
    {"name": "util.o",  "offloading": [("spir64", "<Helper 的 SPIR-V>"),
                                        ("nvptx64", "<Helper 的 PTX>")]},
]

# ---- clang-linker-wrapper：链接期扫描所有对象，抽取并按目标分组 ----
def link_wrapper(objects):
    images_by_target = {}                     # target -> [镜像...]
    for obj in objects:
        for target, image in obj["offloading"]:   # 扫描 .llvm.offloading
            images_by_target.setdefault(target, []).append(image)
            print(f"  [linker-wrapper] 从 {obj['name']} 抽出 {target} 镜像")
    # 对每个目标"设备端链接"成一份最终镜像（这里简化为拼接）
    final = {}
    for target, imgs in images_by_target.items():
        final[target] = " + ".join(imgs)
        print(f"  [sycl-linker] 链接 {target} 的 {len(imgs)} 份镜像 → 最终设备镜像")
    return final

print("=== 链接期：造胖二进制 ===")
device_images = link_wrapper(host_objects)
# wrapper 生成的"注册表"随可执行文件一起，程序启动时注册进运行时
fat_binary = {"host_code": "<x86 主机机器码>", "registered_images": device_images}
print(f"\n胖二进制携带的设备镜像目标：{list(fat_binary['registered_images'])}\n")

# ---- 运行期（第 10 章）：按实际选中的设备，挑对应目标的镜像 ----
def run_on(device_target):
    imgs = fat_binary["registered_images"]
    if device_target in imgs:
        print(f"[运行时] 选中设备 {device_target} → 取用镜像：{imgs[device_target]}")
    else:
        print(f"[运行时] 选中设备 {device_target} → 胖二进制里没有该目标的镜像，回退失败")

print("=== 运行期：按设备挑镜像 ===")
run_on("spir64")     # 跑在 Intel GPU
run_on("nvptx64")    # 跑在 NVIDIA GPU
run_on("amdgcn")     # 没编这个目标 → 找不到

运行结果：

=== 链接期：造胖二进制 ===
  [linker-wrapper] 从 main.o 抽出 spir64 镜像
  [linker-wrapper] 从 main.o 抽出 nvptx64 镜像
  [linker-wrapper] 从 util.o 抽出 spir64 镜像
  [linker-wrapper] 从 util.o 抽出 nvptx64 镜像
  [sycl-linker] 链接 spir64 的 2 份镜像 → 最终设备镜像
  [sycl-linker] 链接 nvptx64 的 2 份镜像 → 最终设备镜像

胖二进制携带的设备镜像目标：['spir64', 'nvptx64']

=== 运行期：按设备挑镜像 ===
[运行时] 选中设备 spir64 → 取用镜像：<VecAdd 的 SPIR-V> + <Helper 的 SPIR-V>
[运行时] 选中设备 nvptx64 → 取用镜像：<VecAdd 的 PTX> + <Helper 的 PTX>
[运行时] 选中设备 amdgcn → 胖二进制里没有该目标的镜像，回退失败

这个模型把打包链路演清楚了：编译期每个主机对象各自"背着"若干目标的设备镜像；linker-wrapper 在链接期把分散在多个 .o 里、同一目标的镜像收拢、各自链成一份最终镜像；运行时再按实际选中的设备去这份"注册表"里挑。请动手改：给 -fsycl-targets 只编 Intel（把两个对象里的 nvptx64 项都删掉），再 run_on("nvptx64")——你会看到运行时报"没有该目标的镜像"，这正是"编译时没为某设备编镜像、运行时就没法在它上面跑"的真实后果。

小结与下一步

你现在理解了"设备代码怎么和主机代码合成一个可执行文件"：

  • 目标是胖二进制：一个可执行文件里既有主机机器码、又有多份设备镜像、还有启动时把镜像注册到 offload 运行时的代码。
  • 核心技巧是把设备镜像（常以 .bc 形态，呼应第 3 章）嵌进主机对象的 .llvm.offloading section，搭主机对象的便车通过系统链接器。
  • 工具家族（clang/tools/）：clang-offload-bundler（打包/拆包）、clang-offload-wrapper（生成带注册代码的 wrapper bitcode）、clang-linker-wrapper（链接期扫描抽取、驱动设备链接、合成）、clang-sycl-linker（SYCL 设备端链接）。⚠️ 本 sycl 分支没有 clang-offload-packager。
  • clang-linker-wrapper 是链接期总指挥：扫 .llvm.offloading → 抽取分组 → 驱动设备端链接 → wrapper 加注册代码 → 合成胖二进制。
  • 用 readelf -S / llvm-objdump / clang-offload-extract 可直接观察和抽取胖二进制里的设备镜像。

下一章我们跨过"编译期"进入"运行期"：胖二进制跑起来后，那段注册代码把镜像登记到了 offload 运行时——liboffload 和 plugins-nextgen（源码在顶层 offload/ 目录）如何拿着这些镜像，通过针对 CUDA / AMDGPU / Level Zero / Host 的插件，把 kernel 真正加载到设备上、分配内存、发起执行。第 10 章带你看 kernel 上设备的"最后一步"。

这一章的要点

  • offload 的产物是胖二进制：一个可执行文件 = 主机机器码 + 多份设备镜像 + 启动时注册镜像的代码。
  • 设备镜像（多为可嵌入的 .bc，见第 3 章）被嵌进主机对象的 .llvm.offloading section，对系统链接器透明。
  • 工具（clang/tools/）：clang-offload-bundler（打包/拆包）、clang-offload-wrapper（生成含注册代码的 wrapper bitcode）、clang-linker-wrapper（链接期总指挥）、clang-sycl-linker（SYCL 设备链接）；辅助有 clang-offload-extract/clang-offload-deps/clang-nvlink-wrapper/offload-arch。
  • 本 sycl 分支不存在 clang-offload-packager（上游主线才有）——读真实源码要以自己的树为准。
  • clang-linker-wrapper 流程：扫 .llvm.offloading → 抽取按目标分组 → 调设备链接器（clang-sycl-linker 等）生成最终镜像 → wrapper 加注册代码 → 合成。
  • 观察工具：readelf -S/llvm-objdump --section-headers 看 section，clang-offload-extract/clang-offload-bundler --unbundle 抽取镜像。
