第 21 章 实战：free function kernels 扩展

这一章想让你带走什么

第 20 章讲了「扩展是什么、怎么组织」。这一章拿一个真实、已进主线的扩展当解剖对象，把「实现一个 SYCL 扩展」从抽象变成具体：sycl_ext_oneapi_free_function_kernels——它让你把 kernel 写成一个普通的自由函数（free function），加个属性标注一下，而不必包成 lambda 提交给 queue。

读完你会看清一个扩展从上到下的完整切面：它解决什么痛点、feature-test 宏叫什么、用户 API 长什么样、那个属性宏在头文件里如何展开成一个编译器认识的 IR 属性、以及运行时怎么靠这个属性找到并启动 kernel。你会同时用上前三部分的知识——属性落到 IR（第 15 章）、kernel 命名与集成头文件（第 14 章）、kernel_bundle（第 16 章）。这一章的结构（痛点 → 宏 → API → 编译器 → 运行时）就是你日后读任何 SYCL 扩展的通用套路。

痛点：为什么想把 kernel 写成自由函数

SYCL 2020 里，kernel 几乎总是以 lambda 的形式出现，提交给 queue：

SYCL / DPC++
q.submit([&](sycl::handler& h) {
    sycl::accessor a(buf, h);
    h.parallel_for(sycl::range<1>(n), [=](sycl::id<1> i) {
        a[i] *= 2;                 // kernel 体，藏在一个匿名 lambda 里
    });
});

这套写法对「就地提交一个 kernel」很顺手，但有几个场景它很别扭：

  • kernel 想被单独声明、单独引用：比如你想把 kernel 声明在一个头文件里、在别处定义，或者想按「函数地址」去引用一个 kernel。lambda 是匿名的、就地的，做不到。
  • 运行时编译 / 互操作：把一段单独的 kernel 交给运行时编译（第 23 章的 kernel_compiler），或和别的语言/工具产生的 kernel 对接时，一个有名字的普通函数比一个埋在提交调用里的 lambda 好处理得多。
  • 就是想要「一个 kernel 就是一个函数」的朴素心智模型：像写 CUDA 的 __global__ void k(...) 那样。

于是有了 free function kernels 扩展：把 kernel 写成一个具名的、命名空间作用域（或静态成员）的普通函数，用一个编译期属性标注「这是个 kernel」。它的规范文档在：

sycl/doc/extensions/experimental/sycl_ext_oneapi_free_function_kernels.asciidoc

（注意它在 experimental/ 目录下——第 20 章讲过扩展按成熟度分 experimental / supported / proposed 等目录。）它的 feature-test 宏是 SYCL_EXT_ONEAPI_FREE_FUNCTION_KERNELS（实验阶段，值为 1）。

别和「free function queries」搞混 扩展目录里还有一个名字很像的 sycl_ext_oneapi_free_function_queries（成熟度更高，在 supported/；本章的 kernels 扩展在 experimental/）。两者名字像、目的完全不同：queries 是「在 kernel 内部不通过 nd_item 参数、而用自由函数查询自己的全局 id」；kernels（本章）是「把 kernel 本身定义成一个自由函数」。读扩展时先看清 doc 全名——顺带也印证了第 20 章"目录即成熟度"：名字相近的两个扩展可以处在不同的成熟度阶段。

用户 API：一个函数 + 一个属性

先看这个扩展给用户的样子。一个 free function kernel 就是这样一个函数：

SYCL / DPC++
namespace syclexp = sycl::ext::oneapi::experimental;

// 用属性标注：这是一个 3 维的 nd-range kernel
SYCL_EXT_ONEAPI_FUNCTION_PROPERTY((syclexp::nd_range_kernel<3>))
void scale(float* data, float factor) {
    // 函数体就是 kernel 体。用 free function queries 拿到自己的全局 id
    size_t i = syclexp::this_nd_item<3>().get_global_id(0);
    data[i] *= factor;
}

对比第一节那个 lambda 版本，差别一目了然：kernel 现在是一个有名字、能被单独声明、参数显式的普通函数 scale。规范对它有几条硬性要求（都写在 doc 里，编译器会检查）：

  • 用且仅用 nd_range_kernel<Dims> 或 single_task_kernel 之一标注。
  • 声明在命名空间作用域，或作为 static 成员函数。
  • 返回类型必须是 void；不能有可变参数；不能有默认参数值。
  • 参数必须是合法的 kernel 参数类型；不允许 reducer 和 kernel_handler。

启动它不再是「提交一个 lambda」，而是按函数引用它、再启动（扩展提供的 nd_launch / single_task 等辅助 + kernel_function 把函数地址作为模板参数捕获）：

SYCL / DPC++
namespace syclexp = sycl::ext::oneapi::experimental;
// 拿到这个 free function kernel 对应的 kernel_id / kernel_bundle
sycl::kernel_id id = syclexp::get_kernel_id<scale>();
auto bundle = syclexp::get_kernel_bundle<sycl::bundle_state::executable>(ctx);
// 或直接按函数启动
syclexp::nd_launch<scale>(q, launch_config, data, 2.0f);

这里 get_kernel_id<scale>()、get_kernel_bundle<...>()（都在 sycl::ext::oneapi::experimental 命名空间）就是把「一个函数」接回 SYCL 2020 那套 kernel_id / kernel_bundle（第 16 章）体系的桥。

关键机制：属性宏如何变成一个 IR 属性

这是整章的技术核心，也是「实现一个扩展」最典型的手法：一个面向用户的漂亮宏，在设备编译时展开成一个编译器前端认识的 IR 属性。

那个宏 SYCL_EXT_ONEAPI_FUNCTION_PROPERTY 定义在：

sycl/include/sycl/ext/oneapi/free_function_kernel_properties.hpp

它的定义是条件编译的——设备端和主机端展开成完全不同的东西：

C++
// sycl/include/sycl/ext/oneapi/free_function_kernel_properties.hpp（示意）
#ifdef __SYCL_DEVICE_ONLY__
// 设备编译趟：展开成一个 clang 属性，把属性名/值作为 IR 元数据附到函数上
#define SYCL_EXT_ONEAPI_FUNCTION_PROPERTY(PROP)                          \
  [[__sycl_detail__::add_ir_attributes_function(                         \
      /* name  */ FunctionPropertyMetaInfo<decltype(PROP)>::name,        \
      /* value */ FunctionPropertyMetaInfo<decltype(PROP)>::value)]]
#else
// 主机编译趟：什么都不展开（主机代码里这个属性无意义）
#define SYCL_EXT_ONEAPI_FUNCTION_PROPERTY(PROP)
#endif

看这里就把第 12 章「两趟编译」和第 15 章「属性落到 IR」全串起来了：

  • 主机趟：宏展开成空，scale 就是个普通函数。
  • 设备趟（__SYCL_DEVICE_ONLY__）：宏展开成 [[__sycl_detail__::add_ir_attributes_function(...)]]——这是 DPC++ 前端专用的一个属性，作用是把一对「名字 + 值」的字符串作为 IR 属性附着到这个函数上。

对两个属性，附上的名字/值分别是：

用户写的 附到 IR 上的属性名 值
nd_range_kernel<Dims> "sycl-nd-range-kernel" Dims（1/2/3）
single_task_kernel "sycl-single-task-kernel" 0

于是 scale 编成的设备 IR 上就带着一条 "sycl-nd-range-kernel" = 3 的属性。这条属性就是编译器后续识别「这个函数是个 kernel」的唯一依据。

图 21-1　free function kernels 的完整切面。用户写的属性宏在设备趟展开成 add_ir_attributes_function，把 "sycl-nd-range-kernel"=3 附到函数 IR 上；前端 SemaSYCL.cpp 扫到它就把这个函数当 kernel 处理（生成入口、写集成头文件）；运行时再靠它找到并启动。这条「头文件宏 → IR 属性 → 编译器识别 → 运行时」的链路，是几乎所有需要编译器配合的 SYCL 扩展的共同骨架

编译器一侧：SemaSYCL 如何认出它

属性附到 IR 上只是「留了个记号」。真正把带记号的函数当作 kernel 处理的，是 DPC++ 前端的 SYCL 语义分析，代码在 clang/lib/Sema/SemaSYCL.cpp。几个关键函数（第 22 章会细讲前端支持，这里先建立印象）：

  • SemaSYCL::isFreeFunction(const FunctionDecl *FD) —— 判断一个函数是不是 free function kernel：它扫描该函数上的 SYCLAddIRAttributesFunctionAttr 属性，找有没有 "sycl-nd-range-kernel" / "sycl-single-task-kernel" 这对名字/值。
  • getFreeFunctionRangeDim(...) —— 从属性里取出维数（1/2/3）。
  • constructFreeFunctionKernelName(...) —— 为它构造要写进集成头文件的 kernel 名字。

也就是说，那个属性宏和前端的 isFreeFunction 是约定好的一对暗号：头文件负责「贴记号」，前端负责「认记号并据此干活」。前端认出后，做的事情和第 12、14 章讲的普通 kernel 一样——生成 kernel 入口函数、把名字和参数信息写进集成头文件，好让主机端和运行时能对上号。

这条属性还会被前端校验：如果属性没写在函数的第一个声明上，会报 err_free_function_first_occurrence_missing_attr；参数类型不合法（比如用了 reducer）也会被 unsupportedFreeFunctionParamType() 挡下。这些诊断也在 SemaSYCL.cpp 附近。

「头文件贴记号，编译器认记号」是扩展的通用套路 free function kernels 不是特例。大量需要编译器配合的 SYCL 扩展都走这个模式：在头文件里用一个宏/属性把用户意图编码成一条 IR 属性（add_ir_attributes_function / ...global_variable 等），编译器前端和后续 Pass 再去识别这条属性并生成对应代码。学会读这一条链，你就能自己看懂 device_global、compile-time properties 等一大批扩展——它们共用 __sycl_detail__ 这套 IR 属性机制（设计文档 sycl/doc/design/CompileTimeProperties.md）。

运行时一侧：从函数到 kernel_bundle

设备镜像里现在有了一个带 "sycl-nd-range-kernel" 属性、名字确定的 kernel。运行时怎么把用户的 get_kernel_id<scale>() 接上它？

思路和第 12 章那个「按修饰名对号」的模型完全一致，只是键从「lambda 的 kernel-name 类型」换成了「函数 scale 本身」：

  • get_kernel_id<scale>() —— 编译期把函数 scale 映射到一个稳定的名字（同样走名字修饰），运行时拿它当键去程序管理器（sycl/source/detail/program_manager/）里查设备镜像中的对应 kernel，返回一个 kernel_id。
  • get_kernel_bundle<bundle_state::executable>(ctx) —— 拿到包含该 kernel、可执行状态的 kernel_bundle（第 16 章的 kernel_bundle 体系）。
  • bundle.ext_oneapi_get_kernel<scale>() —— 从 bundle 里取出真正的 sycl::kernel 对象，可用来查询信息或启动。

所以「实现这个扩展」的运行时工作量其实不大——因为它复用了 SYCL 2020 已有的 kernel_bundle / program_manager 基础设施，只是新增了「用函数（而非 lambda 类型）作为查找键」的入口。这也是好扩展的特征：尽量搭在既有机制上，而不是另起一套。

动手：模拟属性驱动的 kernel 识别

把这一章的核心机制——「设备趟给函数贴 IR 属性 → 前端扫描属性、认出 kernel → 运行时按名字启动」——浓缩成一个能在浏览器里跑的小模型。点 ▶ 运行。

Python 
# 模拟 free function kernels：属性宏贴记号 → 前端认记号 → 运行时启动

# ---- 头文件里的宏：设备趟给函数"贴一条 IR 属性" ----
def function_property(kind, dims=0):
    # 返回一个"IR 属性"（名字, 值），模拟 add_ir_attributes_function 的效果
    if kind == "nd_range":
        return ("sycl-nd-range-kernel", dims)
    elif kind == "single_task":
        return ("sycl-single-task-kernel", 0)

# ---- 用户源码：三个函数，只有前两个贴了 kernel 属性 ----
functions = {
    "scale":  {"attr": function_property("nd_range", 3), "params": ["float* data", "float factor"]},
    "init":   {"attr": function_property("single_task"), "params": ["int* buf"]},
    "helper": {"attr": None, "params": ["int x"]},   # 普通设备函数，不是 kernel
}

# ---- 编译器前端 SemaSYCL::isFreeFunction 的模拟 ----
KERNEL_ATTRS = {"sycl-nd-range-kernel", "sycl-single-task-kernel"}
def is_free_function_kernel(fn):
    return fn["attr"] is not None and fn["attr"][0] in KERNEL_ATTRS

device_image = {}         # 修饰名 -> kernel 信息（前端识别出的 kernel 才进来）
for name, fn in functions.items():
    if is_free_function_kernel(fn):
        attr_name, attr_val = fn["attr"]
        mangled = f"_ZTS{len(name)}{name}"
        kind = "nd-range" if attr_name.endswith("nd-range-kernel") else "single-task"
        device_image[mangled] = {"kind": kind, "dims": attr_val, "params": fn["params"]}
        print(f"[前端] {name}: 扫到属性 {fn['attr']} → 认定为 {kind} kernel，写入集成头文件（{mangled}）")
    else:
        print(f"[前端] {name}: 无 kernel 属性 → 当作普通设备函数，不生成 kernel 入口")

print()

# ---- 运行时：get_kernel_id<Func>() / nd_launch<Func>() ----
def launch(func_name):
    mangled = f"_ZTS{len(func_name)}{func_name}"
    if mangled not in device_image:
        print(f"[运行时] launch<{func_name}>：失败——它不是一个 free function kernel")
        return
    k = device_image[mangled]
    print(f"[运行时] launch<{func_name}>：找到 {k['kind']} kernel，参数 {k['params']}，启动")

launch("scale")
launch("init")
launch("helper")     # 会失败：helper 没被认定为 kernel

运行结果：

[前端] scale: 扫到属性 ('sycl-nd-range-kernel', 3) → 认定为 nd-range kernel，写入集成头文件（_ZTS5scale）
[前端] init: 扫到属性 ('sycl-single-task-kernel', 0) → 认定为 single-task kernel，写入集成头文件（_ZTS4init）
[前端] helper: 无 kernel 属性 → 当作普通设备函数，不生成 kernel 入口

[运行时] launch<scale>：找到 nd-range kernel，参数 ['float* data', 'float factor']，启动
[运行时] launch<init>：找到 single-task kernel，参数 ['int* buf']，启动
[运行时] launch<helper>：失败——它不是一个 free function kernel

这个模型把整条链演完了：属性是唯一的暗号。scale 和 init 因为贴了 sycl-*-kernel 属性被前端认定为 kernel、写进镜像；helper 没贴，就只是个普通设备函数、启动不了。运行时完全不看函数体，只看「前端有没有据属性给它建 kernel 入口」。请动手改：给 helper 也加上 function_property("nd_range", 1)，你会看到它立刻被认定为 kernel、launch("helper") 也成功了——这正是「加一个属性就把一个普通函数变成 kernel」的本质。

小结与下一步

你完整解剖了一个真实扩展：

  • 痛点：SYCL 2020 的 kernel 只能是匿名 lambda，不利于「单独声明/引用 kernel」「运行时编译」「一个 kernel 就是一个函数」的场景。
  • API：SYCL_EXT_ONEAPI_FUNCTION_PROPERTY((nd_range_kernel<Dims>)) / (single_task_kernel) 标注一个具名自由函数；用 get_kernel_id<Func>() / nd_launch<Func>() 引用和启动。feature-test 宏 SYCL_EXT_ONEAPI_FREE_FUNCTION_KERNELS。
  • 核心机制：属性宏在设备趟展开成 [[__sycl_detail__::add_ir_attributes_function("sycl-nd-range-kernel", Dims)]]，把一条属性附到函数 IR 上；主机趟展开成空。
  • 编译器：SemaSYCL.cpp 的 isFreeFunction() 扫描该属性、认出 kernel，走和普通 kernel 一样的入口生成 + 集成头文件流程。
  • 运行时：复用既有 kernel_bundle / program_manager，只新增「用函数作查找键」的入口。
  • 记住那条通用套路：头文件贴 IR 属性 → 编译器认属性并生成代码 → 运行时按名字对号。

下一章我们把镜头对准这条链里最难的一环——编译器前端支持：SemaSYCL 到底怎么改、一个新属性怎么在 clang/include/clang/Basic/Attr.td 里定义、诊断信息怎么加。free function kernels 需要的前端配合相对轻（复用了通用 IR 属性机制），但有些扩展要动更深的前端逻辑，第 22 章带你看清「什么时候必须改编译器、怎么改」。

这一章的要点

  • free function kernels（sycl_ext_oneapi_free_function_kernels，实验阶段，宏 SYCL_EXT_ONEAPI_FREE_FUNCTION_KERNELS）让 kernel 写成具名自由函数，而非匿名 lambda。
  • 用户 API：SYCL_EXT_ONEAPI_FUNCTION_PROPERTY((nd_range_kernel<Dims>)) 或 (single_task_kernel) 标注函数；get_kernel_id<Func>() / get_kernel_bundle<...>() / nd_launch<Func>() 引用启动。头文件 sycl/include/sycl/ext/oneapi/free_function_kernel_properties.hpp。
  • 核心机制：属性宏设备趟展开成 add_ir_attributes_function，把 "sycl-nd-range-kernel"=Dims（或 "sycl-single-task-kernel"=0）附到函数 IR；主机趟展开为空。
  • 编译器：clang/lib/Sema/SemaSYCL.cpp 的 isFreeFunction() / getFreeFunctionRangeDim() / constructFreeFunctionKernelName() 认出并处理；属性须在首个声明上，否则 err_free_function_first_occurrence_missing_attr。
  • 运行时复用 SYCL 2020 的 kernel_bundle / program_manager，只加「按函数查找」的入口——好扩展搭在既有机制上。
  • 通用套路：头文件贴 IR 属性 → 编译器认属性生成代码 → 运行时按名字对号，适用于一大批 __sycl_detail__ 系扩展（见 CompileTimeProperties.md）。
