第 3 章 IR 的三种形态：内存对象、文本 .ll 与位码 .bc

这一章想让你带走什么

第 1 章说 IR 是「一等公民、能落盘」，第 2 章解剖了 IR 在内存里那套 C++ 对象（Module/Function/…）。这一章补上另外两种形态：人可读的文本 .ll，和紧凑的二进制 .bc（bitcode，位码）。三者承载的信息完全等价，可以无损来回转换——这正是「IR 能落盘」这句话的技术兑现。

读完你会清楚：三种形态各自长什么样、用什么工具互转（llvm-as / llvm-dis / opt / clang）、各自在什么场景用（调试看 .ll、存储传输用 .bc、编译进行时用内存对象），以及一个对后面很关键的事实：SYCL / offload 把设备端代码打包进胖二进制，靠的就是 .bc 这种可嵌入的形态。这一章短，但它把「IR 是可保存、可传递的实体」这个心智模型钉死。

一份 IR，三副面孔

同一段 IR，在 LLVM 里有三种可以互相转换的存在形式。它们不是三种不同的东西，而是同一份数据的三种编码——就像同一个整数可以写成十进制、十六进制、或内存里的补码字节。

  • 内存对象（in-memory）：第 2 章那套 Module / Function / Instruction C++ 对象，还有织在其上的 def-use 图。这是编译进行时的形态——clang 前端生成它、opt 的 Pass 修改它、后端消费它。它只活在进程内存里，进程结束就没了。
  • 文本 .ll（assembly / textual IR）：人能读、能用编辑器改的汇编样式文本。你在第 1、2 章看到的 define i32 @add ... 就是它。给人看的。
  • 位码 .bc（bitcode / binary IR）：紧凑的二进制编码。机器读写快、体积小，但人看不了。给机器存储和传输用的。

三者信息量严格相等：任意一种都能无损转成另外两种。选哪种，只看当下的目的——要给人看就用 .ll，要存盘/嵌入/快速加载就用 .bc，正在编译就是内存对象。

图 3-1　IR 的三种等价形态与互转工具。内存对象是编译进行时的形态；.ll 给人看；.bc 给机器存传。任意两种之间都能无损转换——这就是第 1 章「IR 能落盘」的实现

互转工具：llvm-as / llvm-dis / clang / opt

三种形态之间的转换，各有对应的命令行工具或代码路径：

.ll ↔ .bc：一对互逆的工具，名字直白——

终端
llvm-as  add.ll  -o add.bc     # assemble：文本 .ll → 二进制 .bc
llvm-dis add.bc  -o add.ll     # disassemble：二进制 .bc → 文本 .ll

llvm-as（汇编器）和 llvm-dis（反汇编器）互为逆操作，来回转多少次内容都不变。

源码 → .ll / .bc：由前端 clang 产出。第 1 章那条命令就是：

终端
clang -S -emit-llvm add.c -o add.ll    # -S 出文本 .ll
clang -c -emit-llvm add.c -o add.bc    # -c 出二进制 .bc

内存对象 ↔ 文本/位码：这是代码里的转换，也是上面工具背后真正干活的 API（都在 llvm/lib/ 下）：

  • 文本解析：lib/AsmParser/（parseIR / parseAssembly）把 .ll 文本读成内存 Module。
  • 文本打印：Module::print / Value::print（lib/IR/AsmWriter.cpp）把内存对象打成 .ll。你在调试里常用的 M.dump()、I.dump() 就走这里。
  • 位码写出：lib/Bitcode/Writer/（WriteBitcodeToFile）把内存 Module 编码成 .bc。
  • 位码读入：lib/Bitcode/Reader/（parseBitcodeFile）把 .bc 解回内存 Module。

opt 这个工具则同时会读写：它读入 .ll 或 .bc → 在内存对象上跑 Pass → 再写出 .ll 或 .bc。默认输出 .bc，加 -S 输出 .ll：

终端
opt -passes=dce add.ll -S -o out.ll    # 读 .ll、跑 DCE、写回 .ll（-S 表示文本输出）
为什么调试时几乎总加 -S opt 和 clang -emit-llvm 默认吐的是二进制 .bc，直接 cat 出来是乱码。养成习惯：想看 IR 就加 -S（输出文本 .ll），想把 IR 喂给下一个工具或存起来就不加（输出 .bc）。若手上已经有个 .bc 想读，llvm-dis some.bc -o - 打到标准输出即可。

各形态的分工：谁在什么时候用

三种形态不是摆设，它们对应真实的工程权衡：

形态 优点 缺点 典型场景
内存对象 可随机访问、可改、带 def-use 图 占内存、进程内才有 编译进行时：前端生成、Pass 修改、后端消费
文本 .ll 人可读可改、diff 友好 体积大、解析慢 调试、写测试（LIT）、教学、手写 IR
位码 .bc 紧凑、读写快、可嵌入 人不可读 存储、分发、LTO、嵌进胖二进制

.ll 的「diff 友好」在 LLVM 的测试体系里被用到极致：llvm/test/ 下大量 LIT 测试就是「一段 .ll 输入 + 期望的输出模式」，靠文本比对验证某个 Pass 的行为。你在第 5 章写 Pass 时会亲手写这种测试。

而 .bc 的「可嵌入」直接关系到本书后面的核心链路。LTO（Link Time Optimization，链接时优化） 就是把各编译单元编成 .bc 而非机器码存进 .o，等链接时再把所有 .bc 合起来做全局优化。更重要的是——

.bc 是异构 offload 的载体 SYCL / OpenMP offload 的设备端代码，在打包进「胖二进制（fat binary）」之前，正是以 LLVM 位码的形态存在的：设备编译趟产出设备端 .bc，用 llvm-link 合并多个 .bc，再交给 llvm-spirv 翻成 SPIR-V（第 8 章）或经 NVPTX 出 PTX（第 11 章），最后由打包工具（第 9 章）嵌进可执行文件。「IR 能落盘成可嵌入的 .bc」不是一个理论上的漂亮性质，而是整条异构编译链路能拆成独立工具、能把设备代码塞进主机可执行文件的物理前提。

亲手看一眼：同一段 IR 的两副面孔

拿第 1 章那个 add 走一遍完整的转换环，你能亲眼确认「无损」：

终端
# 1. 源码 → 文本 .ll
clang -S -emit-llvm add.c -o add.ll
# 2. 文本 → 二进制
llvm-as add.ll -o add.bc
# 3. 二进制 → 文本（转回来）
llvm-dis add.bc -o add.roundtrip.ll
# 4. 对比：两个 .ll 内容一致（除了文件名注释）
diff add.ll add.roundtrip.ll

.ll 里那段你已经熟悉：

LLVM IR
define i32 @add(i32 %a, i32 %b) {
entry:
  %sum = add i32 %a, %b
  ret i32 %sum
}

而它的 .bc 用 xxd add.bc | head -1 看，开头是魔数 42 43 C0 DE——正是 ASCII 的 BC 加两个标志字节，这是 bitcode 文件的身份标识。内容你读不了，但它和上面那段文本一字不差地等价。这就是「三副面孔、一份数据」。

动手：模拟三形态互转的「无损」性质

用一个浏览器里能跑的小模型演示核心性质：三种形态是同一份数据的不同编码，来回转换后内容不变（无损）。我们把「内存对象」表示成一个结构化字典，把 .ll 表示成文本，把 .bc 表示成紧凑字节，实现它们之间的往返。点 ▶ 运行。

Python 
# 模拟 IR 三形态互转：内存对象 <-> 文本 .ll <-> 位码 .bc，验证无损

# ---- 内存对象：一份结构化的 IR（对应第 2 章的 Module/Function/Instruction） ----
ir_object = {
    "function": "add",
    "ret_type": "i32",
    "args": [("i32", "%a"), ("i32", "%b")],
    "body": [("%sum", "add", ["%a", "%b"]), (None, "ret", ["%sum"])],
}

# ---- 内存对象 → 文本 .ll（对应 Module::print / AsmWriter） ----
def to_ll(obj):
    args = ", ".join(f"{t} {n}" for t, n in obj["args"])
    lines = [f"define {obj['ret_type']} @{obj['function']}({args}) {{", "entry:"]
    for dst, op, uses in obj["body"]:
        if dst:
            lines.append(f"  {dst} = {op} i32 {', '.join(uses)}")
        else:
            lines.append(f"  {op} i32 {uses[0]}")
    lines.append("}")
    return "\n".join(lines)

# ---- 文本 .ll → 位码 .bc（对应 llvm-as / BitcodeWriter，这里用紧凑编码示意） ----
def to_bc(ll_text):
    # 真正的 .bc 是紧凑二进制、带魔数 'BC\xC0\xDE'；这里用简化的字节序列示意
    magic = b"BC\xc0\xde"
    return magic + ll_text.encode("utf-8")   # 示意：把文本紧凑打包

# ---- 位码 .bc → 文本 .ll（对应 llvm-dis / BitcodeReader） ----
def from_bc(bc_bytes):
    assert bc_bytes[:4] == b"BC\xc0\xde", "不是合法的 bitcode（魔数不符）"
    return bc_bytes[4:].decode("utf-8")

# ---- 走一圈：对象 → .ll → .bc → .ll，验证无损 ----
ll1 = to_ll(ir_object)
print("== 文本 .ll 形态 ==")
print(ll1)

bc = to_bc(ll1)
print(f"\n== 位码 .bc 形态（前 8 字节）==\n{bc[:8]!r}  ...  共 {len(bc)} 字节")

ll2 = from_bc(bc)          # 转回来
print("\n== 从 .bc 转回 .ll，与原文一致？ ==")
print("一致！" if ll1 == ll2 else "不一致（丢信息了）")

运行结果：

== 文本 .ll 形态 ==
define i32 @add(i32 %a, i32 %b) {
entry:
  %sum = add i32 %a, %b
  ret i32 %sum
}

== 位码 .bc 形态（前 8 字节）==
b'BC\xc0\xdedefi'  ...  共 90 字节
== 从 .bc 转回 .ll，与原文一致？ ==
一致！

模型虽简化，但把关键性质演清楚了：三种形态是同一份 IR 的不同编码，.bc 开头那个 BC\xc0\xde 魔数是真实 bitcode 的身份标识；来回转换（对象 → .ll → .bc → .ll）后内容一字不差——无损。真实的 llvm-as/llvm-dis 做的正是这件事，只不过 .bc 用的是精心设计的紧凑位流编码而非简单打包。请动手改：给 ir_object["body"] 再加一条指令（比如 ("%prod", "mul", ["%sum", "%a"])），重新运行——三种形态会同步反映这条新指令，因为它们本就是一份数据。

小结与下一步

你现在彻底理解了第 1 章那句「IR 能落盘」：

  • 一份 IR 有三种等价形态：内存对象（编译进行时，第 2 章那套 C++ 对象）、文本 .ll（人可读，调试/测试用）、位码 .bc（紧凑二进制，存储/传输/嵌入用）。三者信息严格相等，可无损互转。
  • 互转工具：llvm-as（.ll→.bc）/ llvm-dis（.bc→.ll）/ clang -emit-llvm（源码→IR）/ opt（读写皆可，-S 出文本）；背后是 lib/AsmParser/、lib/Bitcode/、AsmWriter.cpp。
  • 分工：编译时用内存对象，看/测用 .ll，存/传/嵌用 .bc。
  • .bc 的可嵌入性是异构 offload 的物理前提——设备代码先以 .bc 存在，经 llvm-link、llvm-spirv/NVPTX，最终嵌进胖二进制。

第一部分的 IR 基础到此打完（第 1–3 章）。下一章起进入Pass 框架：你已经在第 4 章读过新 PassManager 的设计（无基类契约、AnalysisManager、PreservedAnalyses）——第 5 章将把它落地，带你亲手写一个 Function Pass、编成插件、用 opt -load-pass-plugin 挂进管线跑起来，并配上一个 .ll 的 LIT 测试。这一章的 .ll/opt -S 正是那时你天天要用的工具。

这一章的要点

  • 一份 IR 三种等价形态：内存对象（Module/Function…，编译进行时）、文本 .ll（人可读）、位码 .bc（紧凑二进制）；信息量相等，无损互转。
  • 工具：llvm-as（汇编，.ll→.bc）、llvm-dis（反汇编，.bc→.ll）、clang -S/-c -emit-llvm（源码→IR）、opt（跑 Pass，-S 输出文本）。默认输出是二进制，调试记得加 -S。
  • 代码路径：文本读写在 lib/AsmParser/ 与 AsmWriter.cpp（Module::print/dump）；位码读写在 lib/Bitcode/Reader/、Writer/（parseBitcodeFile/WriteBitcodeToFile）。
  • .bc 文件魔数是 42 43 C0 DE（BC\xC0\xDE）。
  • .bc 可嵌入，是 LTO 与异构 offload（设备代码先成 .bc，再经 llvm-link→llvm-spirv/NVPTX→胖二进制）的物理基础——呼应第 8、9、11 章。
