第 20 章 扩展的解剖：命名约定、feature-test 宏与 doc/extensions

这一章想让你带走什么

前面十九章你反复撞见 sycl::ext::oneapi::experimental 这样的命名空间、SYCL_EXT_ONEAPI_* 这样的宏、doc/extensions/experimental/ 里的一个个 .asciidoc。第四部分正式转向实践——如何给 SYCL 写一个扩展。而这一章是这一部分的地基：先把"扩展"这件事的解剖结构讲清，不写一行实现，只回答"一个 SYCL 扩展由哪些部件组成、它们叫什么、放在哪、彼此怎么对应"。

读完你会掌握四件事：① 扩展文档的目录与生命周期（proposed / experimental / supported / deprecated / removed 五态）；② 雷打不动的命名约定——扩展名、feature-test 宏名、文件名三者一致的规则；③ feature-test 宏在哪定义、怎么取值、用户怎么检测（`feature_test.hpp.in` → 生成 `feature_test.hpp`）；④ vendor 前缀（oneapi/intel/codeplay）与 C++ 命名空间的映射。这套"解剖学"是你读任何 SYCL 扩展、以及第 21–24 章动手写扩展的通用坐标系。

扩展是什么：SYCL 标准之外的官方增量

SYCL 是一份 Khronos 标准（第 12 章讲过 SYCL 2020 规范）。但标准的演进慢、覆盖面有限，而硬件和用户需求跑得快——于是 SYCL 规范第 6 章专门留了扩展机制：厂商可以在标准之外定义新特性，只要遵守一套命名和版本规则，就能既增加能力、又不破坏可移植性（用户能在编译期检测某扩展在不在）。

DPC++（intel/llvm）就是这套机制的重度使用者。你用过的很多东西其实都是扩展而非核心 SYCL：USM 的一些高级形态、ext_oneapi_graph（第 17 章那个显式命令图）、bindless images（第 23 章）、free function kernels（第 21 章）——全是扩展。它们的规范文档统一住在：

sycl/doc/extensions/

这一章就从这个目录的结构开始。

五个状态：一个扩展的生命周期

sycl/doc/extensions/ 下按扩展的成熟度分成五个子目录，每个扩展的 .asciidoc 规范文档住在其中之一。README.md 把五态定义得很清楚：

目录 含义 大致文档数
proposed/ 已提案、尚未实现 ~22
experimental/ 已实现，但 API 可能变 ~56
supported/ 完全支持、稳定 ~32
deprecated/ 仍支持，但即将移除 3
removed/ 曾经支持、现已移除 ~17

（数字随分支演进会变，以你手上的树为准。）一个扩展的典型一生就是沿这条链走：先在 proposed/ 里成型（此时可随意改），实现它的那个 PR 把文档移到 experimental/ 或 supported/；若日后被核心 SYCL 吸收或废弃，就移到 deprecated/ 再到 removed/。第 24 章会带你走完这条流程；这一章你只要记住：看一个扩展在哪个目录，就知道它有多稳。

experimental/ 目录 ≠ C++ 的 ::experimental 命名空间——但常常同步 你会看到两个"experimental"：文档目录 doc/extensions/experimental/，和 C++ 命名空间 sycl::ext::oneapi::experimental。它们概念上是两回事（一个是文档成熟度，一个是 API 所在的命名空间），但实践中高度相关——实验阶段的扩展，其 API 通常就放在 ::experimental 命名空间里，等稳定了再考虑上提。别把"目录名"当成"命名空间名"的硬绑定，但看到其一基本能猜到另一。

命名约定：扩展名 = 宏名 = 文件名（大小写不同而已）

这是整章最该背下来的一条规则。README-process.md（"Lifetime of an Extension"）把它写死了：

  • 扩展名：小写、下划线分词，形如 sycl_ext_<vendor>_<feature>——例如 sycl_ext_oneapi_bindless_images。
  • feature-test 宏名：和扩展名完全一样，只是全大写——SYCL_EXT_ONEAPI_BINDLESS_IMAGES。
  • 规范文件名：和扩展名一致，加 .asciidoc 后缀——sycl_ext_oneapi_bindless_images.asciidoc。

三者一一对应，是整个扩展体系的"主键"。几个真实的对应例子：

规范文件（experimental/） feature-test 宏 当前值
sycl_ext_oneapi_bindless_images.asciidoc SYCL_EXT_ONEAPI_BINDLESS_IMAGES 6
sycl_ext_oneapi_graph.asciidoc SYCL_EXT_ONEAPI_GRAPH 2
sycl_ext_oneapi_bfloat16_math_functions.asciidoc SYCL_EXT_ONEAPI_BFLOAT16_MATH_FUNCTIONS 1
sycl_ext_intel_cache_config.asciidoc SYCL_EXT_INTEL_CACHE_CONFIG 1
vendor 字符串只有三个，且文件名的 vendor 未必等于宏的 vendor <vendor> 部分几乎总是三者之一：oneapi（oneAPI 通用，默认选它）、intel（很 Intel 硬件专属才用）、codeplay（Codeplay 贡献的）。README-process.md 明说："For DPC++, we use the string 'oneapi' unless the extension is very specific to Intel hardware."。有个容易踩的坑：文件名的 vendor 和宏里的 vendor 可以不一致——例如 codeplay 贡献的 sycl_ext_codeplay_enqueue_native_command.asciidoc，它的 feature-test 宏却是 SYCL_EXT_ONEAPI_ENQUEUE_NATIVE_COMMAND（因为功能通用）。以文档里 Feature test macro 一节写的宏名为准。

feature-test 宏：在哪定义、怎么检测

宏名有了，值从哪来？答案是一个生成出来的头文件。模板源在：

sycl/source/feature_test.hpp.in

构建时 CMake 用 configure_file(...)（sycl/CMakeLists.txt，约 235 行）把它生成成 <build>/include/sycl/feature_test.hpp，随 SYCL 头一起安装。文件顶部就写着"DO NOT EDIT——原始定义在 feature_test.hpp.in"。里面绝大多数就是一行行硬编码的 #define：

C++
// sycl/source/feature_test.hpp.in（节选）
namespace sycl { inline namespace _V1 {
#define SYCL_EXT_INTEL_DEVICE_INFO 9
#define SYCL_EXT_ONEAPI_DEVICE_ARCHITECTURE 1
#define SYCL_EXT_ONEAPI_BINDLESS_IMAGES 6
#define SYCL_EXT_ONEAPI_BFLOAT16_MATH_FUNCTIONS 1
} }

少数宏是构建期决定的——比如后端相关的用 #cmakedefine01（CUDA/HIP 后端有没有编进来）：

C++
#cmakedefine01 SYCL_EXT_ONEAPI_BACKEND_CUDA
#cmakedefine01 SYCL_EXT_ONEAPI_BACKEND_HIP

用户侧的检测姿势因此非常标准——#ifdef + 比较版本值：

C++
#include <sycl/sycl.hpp>            // 会带进 feature_test.hpp
#if defined(SYCL_EXT_ONEAPI_BINDLESS_IMAGES) && SYCL_EXT_ONEAPI_BINDLESS_IMAGES >= 6
  // 用 bindless images v6 的 API
#else
  #error "需要支持 bindless_images v6 的 DPC++"
#endif

宏的值就是扩展的版本号。experimental/ 里的不稳定扩展，值常年是 1（文档里明说"本实验扩展不做版本区分，宏恒为此值"）；而版本化的扩展每加一批 API 就 +1（bindless images 已经到 6）。这就是"可移植扩展"的核心——用户先检测、后使用，宏不存在或版本不够就在编译期就报错，而不是运行期崩溃。

扩展的三位一体：文件名 = 扩展名 = feature-test 宏名（大小写差异），宏值 = 版本号 规范 .asciidoc 住在 sycl/doc/extensions/{proposed,experimental,supported,deprecated,removed}/ 之一（成熟度五态）。命名规则（README-process.md）：扩展名 sycl_ext_<vendor>_<feature> 小写下划线，feature-test 宏是它的全大写，文件名与扩展名一致。宏在 sycl/source/feature_test.hpp.in 里 #define（构建期 configure_file 生成 feature_test.hpp），值即版本号。用户用 #if defined(宏) && 宏 >= N 在编译期检测——这就是 SYCL 扩展"增能力而不破坏可移植性"的机制。

vendor 前缀 ↔ C++ 命名空间

命名约定还延伸到代码里。sycl/include/sycl/ext/ 下正好有三个 vendor 目录，和三个 vendor 字符串对齐：

sycl/include/sycl/ext/
├── oneapi/      → namespace sycl::ext::oneapi（实验特性再进 ::experimental）
├── intel/       → namespace sycl::ext::intel
└── codeplay/    → namespace sycl::ext::codeplay

对应关系用真实头文件一看便知：ext/intel/math.hpp 里是 namespace ext::intel::math；ext/codeplay/experimental/max_registers_query.hpp 里是 namespace ext::codeplay::experimental；ext/oneapi/experimental/root_group.hpp 里是 namespace ext::oneapi::experimental。所以看到一个 API 的命名空间，你就能反推它属于哪个 vendor、成熟度大概如何，进而去 doc/extensions/ 对应目录找它的规范。命名空间是代码里的路标，doc 目录是文档里的路标，两者靠 vendor 字符串对齐。

图 20-1　扩展的"三位一体"。一个扩展名同时对应三个落点：doc/extensions/ 里的规范（目录=成熟度）、feature_test.hpp.in 里的宏（值=版本）、include/sycl/ext/<vendor>/ 里的命名空间与头文件。用户靠 #if defined(宏)&&宏>=N 先检测后使用

一份规范文档长什么样：template.asciidoc

doc/extensions/ 顶层有个 template.asciidoc——写新扩展就从复制它开始。它规定了一份规范的标准骨架（章节），照着填空即可：

  • Notice —— 版权/商标声明。
  • Contact —— 报 issue 的 URL。
  • Dependencies —— 依赖哪版 SYCL 规范、依赖哪些其他扩展。
  • Status —— proposed / experimental / supported / … 的样板文字（决定它该放哪个目录）。
  • Backend support status（可选）—— 哪些后端支持。
  • Overview —— 概述：这扩展解决什么问题。
  • Specification → Feature test macro —— 定义宏名和版本值表，然后是各 API 的规范。
  • Implementation notes / Issues（可选、非规范性）。

README-process.md 开门见山："Start by making a copy of the template extension specification document, and follow the instructions in that document."——并要求遵守 SYCL 2020 规范第 6 章"Guidelines for portable extensions"。所以写扩展的第一步永远是文档，而不是代码：先把"要提供什么、宏叫什么、放哪个目录"想清楚。第 24 章会把这条流程走完整。

老文档的章节和现在的模板未必一致 template.asciidoc 是当前的模板，但仓库里有些年头较久的 supported/ 文档（如 sycl_ext_oneapi_enqueue_barrier.asciidoc）用的是更早的章节约定（== Introduction / == Version / == Feature Test Macro / == Specification changes…）。读老扩展时别指望章节名和模板逐字对上——抓住"它一定有 Overview + Feature test macro + Specification"这个不变的内核即可。

亲手逛一遍扩展目录

不用编译，直接在源码树里把这一章的结构走一遍：

终端
# 五个成熟度目录，各有多少扩展
for d in proposed experimental supported deprecated removed; do
  echo "$d: $(ls sycl/doc/extensions/$d/*.asciidoc 2>/dev/null | wc -l)"
done

# 写扩展的三个锚文件
ls sycl/doc/extensions/{README.md,README-process.md,template.asciidoc}

# feature-test 宏在哪定义、某扩展的值是多少
grep -n 'SYCL_EXT_ONEAPI_BINDLESS_IMAGES\|SYCL_EXT_ONEAPI_GRAPH' sycl/source/feature_test.hpp.in

# 三个 vendor 目录
ls sycl/include/sycl/ext/

挑一个你感兴趣的扩展（比如 sycl_ext_oneapi_graph），把它的三个落点串起来看：doc/extensions/experimental/ 里的 .asciidoc（规范）、feature_test.hpp.in 里的宏值（版本）、include/sycl/ext/oneapi/experimental/ 里的头（API）。这个"顺着扩展名找三处"的动作，就是你日后读任何扩展的起手式。

动手：模拟 feature-test 宏的检测与版本协商

把这一章的核心——「扩展名=宏名=文件名，用户按宏值先检测后使用」——用一个能在浏览器里跑的小模型演出来。点 ▶ 运行。

Python 
# 模拟 SYCL 扩展的命名三位一体 + feature-test 宏的版本检测

# ---- 实现方（DPC++）在 feature_test.hpp 里 #define 的宏：名 -> 版本值 ----
FEATURE_TEST = {
    "SYCL_EXT_ONEAPI_BINDLESS_IMAGES": 6,
    "SYCL_EXT_ONEAPI_GRAPH": 2,
    "SYCL_EXT_ONEAPI_BFLOAT16_MATH_FUNCTIONS": 1,
    # 注意：没有 SYCL_EXT_ONEAPI_FANCY_NEW —— 该扩展本实现没提供
}

# ---- 由扩展名推出三位一体（文件名/宏名/命名空间）----
def anatomy(ext_name):                       # ext_name 形如 sycl_ext_oneapi_graph
    vendor = ext_name.split("_")[2]          # oneapi / intel / codeplay
    return {
        "doc_file": f"doc/extensions/experimental/{ext_name}.asciidoc",
        "macro":    ext_name.upper(),
        "namespace": f"sycl::ext::{vendor}::experimental",
    }

# ---- 用户代码：#if defined(宏) && 宏 >= 需要的版本 ----
def user_check(ext_name, need_version):
    a = anatomy(ext_name)
    macro = a["macro"]
    have = FEATURE_TEST.get(macro)           # None = 宏未定义（扩展不存在）
    print(f"用户想用 {ext_name}（需 v{need_version}）")
    print(f"   文件: {a['doc_file']}")
    print(f"   命名空间: {a['namespace']}")
    if have is None:
        print(f"   #ifdef {macro} → 未定义 ✗  本实现不提供该扩展，#error 编译期报错\n")
    elif have < need_version:
        print(f"   {macro}={have} < {need_version} ✗  版本不够，#error 编译期报错\n")
    else:
        print(f"   {macro}={have} >= {need_version} ✓  放心使用该扩展的 API\n")

user_check("sycl_ext_oneapi_bindless_images", 6)   # 版本正好，OK
user_check("sycl_ext_oneapi_graph", 3)             # 需要 v3，但只有 v2 → 拒绝
user_check("sycl_ext_oneapi_fancy_new", 1)         # 宏根本没定义 → 拒绝

运行结果：

用户想用 sycl_ext_oneapi_bindless_images（需 v6）
   文件: doc/extensions/experimental/sycl_ext_oneapi_bindless_images.asciidoc
   命名空间: sycl::ext::oneapi::experimental
   SYCL_EXT_ONEAPI_BINDLESS_IMAGES=6 >= 6 ✓  放心使用该扩展的 API

用户想用 sycl_ext_oneapi_graph（需 v3）
   文件: doc/extensions/experimental/sycl_ext_oneapi_graph.asciidoc
   命名空间: sycl::ext::oneapi::experimental
   SYCL_EXT_ONEAPI_GRAPH=2 < 3 ✗  版本不够，#error 编译期报错

用户想用 sycl_ext_oneapi_fancy_new（需 v1）
   文件: doc/extensions/experimental/sycl_ext_oneapi_fancy_new.asciidoc
   命名空间: sycl::ext::oneapi::experimental
   #ifdef SYCL_EXT_ONEAPI_FANCY_NEW → 未定义 ✗  本实现不提供该扩展，#error 编译期报错

看清两件事：① 一个扩展名机械地推出三个落点——anatomy() 只做字符串变换（大写得宏名、ext/<vendor>/ 得命名空间、加后缀得文件名），这正是命名约定的价值：知其一即知其三。② feature-test 宏是"能力协商"的唯一凭据——版本够就用、不够或没有就在编译期（不是运行期）报错。这就是 SYCL 扩展既能加能力、又不破坏可移植性的关键。请动手改：把 FEATURE_TEST 里的 SYCL_EXT_ONEAPI_GRAPH 改成 3，重新运行——第二个检测立刻从"拒绝"变"通过"。这正是实现方发布新版本（把宏值 +1）后，用户代码自动解锁新 API 的过程。

小结与下一步

这一章给了第四部分一套坐标系——不写实现，只认部件：

  • 扩展规范住在 sycl/doc/extensions/，按成熟度分五态目录：proposed（提案未实现）/ experimental（实现了但 API 可能变）/ supported（稳定）/ deprecated（将移除）/ removed（已移除）。看目录即知稳定度。
  • 命名三位一体（README-process.md）：扩展名 sycl_ext_<vendor>_<feature>（小写下划线）= feature-test 宏（全大写）= 文件名（+.asciidoc）。vendor 为 oneapi（默认）/ intel（Intel 硬件专属）/ codeplay。
  • feature-test 宏在 sycl/source/feature_test.hpp.in 里 #define（构建期 configure_file 生成 feature_test.hpp），值即版本号；少数用 #cmakedefine01（后端相关）。用户用 #if defined(宏) && 宏 >= N 先检测后使用。
  • vendor 前缀 ↔ 命名空间：ext/oneapi/→sycl::ext::oneapi（实验特性再进 ::experimental）、ext/intel/→sycl::ext::intel、ext/codeplay/→sycl::ext::codeplay。命名空间是代码路标，doc 目录是文档路标，靠 vendor 对齐。
  • 写扩展从复制 template.asciidoc 开始（Notice/Contact/Dependencies/Status/Overview/Specification+Feature test macro/…），第一步永远是文档。

下一章（你已经能读到）——第 21 章 free function kernels 实战——就用这套坐标系解剖一个真实扩展的完整切面：痛点 → 宏 → API → 编译器 → 运行时。而第 22 章深入这条链最难的一环：需要编译器前端配合的扩展怎么写（SemaSYCL 与 Attr.td）；第 23 章巡礼 bindless images 与运行时编译两个大案例；第 24 章走完"从提案到落地"的完整流程与测试。这一章的部件清单，是后面每一章的地基。

这一章的要点

  • 扩展规范在 sycl/doc/extensions/ 的五态目录：proposed / experimental / supported / deprecated / removed（成熟度递进），看目录即知稳定度；写新扩展从 template.asciidoc 复制起步，遵 README-process.md 与 SYCL 2020 第 6 章。
  • 命名三位一体：扩展名 sycl_ext_<vendor>_<feature>（小写下划线）= feature-test 宏（全大写）= 文件名（.asciidoc）。vendor = oneapi（默认）/ intel / codeplay；文件名 vendor 未必等于宏 vendor（以文档 Feature test macro 一节为准）。
  • feature-test 宏定义在 sycl/source/feature_test.hpp.in（构建 configure_file 生成 feature_test.hpp，sycl/CMakeLists.txt ~235），值即版本号；后端相关用 #cmakedefine01（如 SYCL_EXT_ONEAPI_BACKEND_CUDA）。
  • 用户检测：#if defined(宏) && 宏 >= N 编译期先检测后使用——扩展"增能力不破坏可移植性"的机制。experimental 扩展宏常年为 1，版本化的随 API 增加而 +1（bindless images 已到 6）。
  • vendor ↔ 命名空间：include/sycl/ext/{oneapi,intel,codeplay}/ 对应 sycl::ext::{oneapi,intel,codeplay}，实验特性再进 ::experimental。
