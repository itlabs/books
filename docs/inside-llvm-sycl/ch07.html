第 7 章 后端与代码生成：Target、指令选择到机器码

这一章想让你带走什么

第 1–6 章你走完了 LLVM 的前半程：源码 → IR → 优化后的 IR。这一章补上最后一段——后端（backend）：把与目标无关的 LLVM IR 翻译成某款具体芯片的机器码。这是「懂硬件」的那一段，也是 llvm/lib/Target/<芯片>/（X86、AArch64、NVPTX、SPIRV…）里那一大堆代码在干的事。

你会建立一张后端流水线的地图：指令选择（ISel）→ 寄存器分配 → 指令调度 → 汇编输出，看清哪些是与芯片无关的通用框架（llvm/lib/CodeGen/）、哪些是每款芯片各写一份的目标描述（lib/Target/<芯片>/，大量用 .td TableGen 文件描述）。理解这一层，你才明白第 1 章「加一款芯片 = 加一个后端」具体要加什么，也才能读懂第 11 章 NVPTX（NVIDIA）后端、以及 SPIR-V 到底是"后端"还是"翻译器"这个区分。

后端要解决的落差：从 IR 到真实硬件

优化后的 LLVM IR 仍然是理想化、与硬件无关的：它有无限多个 SSA 虚拟寄存器（%1、%sum…想用多少有多少）、有抽象的类型（i32、i64）、指令是 add/load/br 这种通用操作。但真实芯片是残酷而具体的：

  • 寄存器数量有限（x86-64 通用寄存器就 16 个），IR 里无限的 SSA 值必须挤进去，挤不下的要"溢出（spill）"到栈上。
  • 指令是特定 ISA 的真实指令（x86 的 addl、ARM 的 add、还带各种寻址模式和操作数约束），不是抽象的 add。
  • 指令有执行时序、流水线约束，排列顺序影响性能。
  • 有调用约定（calling convention）：参数放哪个寄存器、返回值怎么传、谁负责保存哪些寄存器。

后端的工作，就是把这些落差一层层填平。它被组织成一条代码生成流水线，每一步解决其中一个落差。

图 7-1　后端代码生成流水线。指令选择 → 调度 → 寄存器分配 → 汇编输出，每步的算法骨架在 llvm/lib/CodeGen/（所有芯片共用），而每步需要的"这款芯片长什么样"由 lib/Target/<芯片>/ 的 TableGen 描述提供

通用框架 vs 目标描述：后端的两半

第 1 章说后端代码分两处，这里说清它们的分工——这是理解整个后端的钥匙：

  • llvm/lib/CodeGen/ —— 与具体芯片无关的代码生成框架。它实现了指令选择的骨架、寄存器分配算法（如贪心分配器 RegAllocGreedy）、指令调度器、栈帧布局逻辑等。这些算法对所有芯片都一样。
  • llvm/lib/Target/<芯片>/ —— 每款芯片一份的目标描述。它回答通用框架的提问：这款芯片有哪些寄存器？分几类（通用/浮点/向量）？有哪些指令、各自的操作数约束和汇编写法？调用约定是什么？IR 的 add i32 该选成哪条真实指令？

打个比方：lib/CodeGen/ 是一台通用的装配机器人，lib/Target/X86/ 是X86 这款产品的图纸。同一个机器人，换一套图纸就能装配另一款芯片的代码——这就是为什么加一款新芯片主要是"写一份目标描述"，而不用重写代码生成算法。

这些目标描述大量用 TableGen（.td 文件） 来写。TableGen 是 LLVM 自己的一种声明式描述语言：你用它声明"这款芯片有寄存器 R0–R15""有一条指令叫 ADD、汇编写法是 add $dst, $src1, $src2""IR 的 add 模式匹配到这条 ADD 指令"，构建时由 llvm-tblgen 工具把这些 .td 生成成 C++ 代码，编进后端。你在 lib/Target/X86/ 下会看到 X86RegisterInfo.td、X86InstrInfo.td 这类文件——它们不是手写 C++，而是声明式的芯片描述。

为什么用 TableGen 而不是手写 C++ 一款现代芯片有成百上千条指令、几十类寄存器、无数操作数约束。若全手写 C++，代码量爆炸且极易出错。TableGen 让你声明式地描述这些事实，再由工具批量生成寄存器信息、指令信息、指令选择匹配表、汇编器/反汇编器等一大堆本该重复的 C++ 代码。它是 LLVM 后端能支持这么多芯片的工程基础——写新后端，很大一部分工作就是写 .td。

流水线四步，各解决一个落差

① 指令选择（Instruction Selection, ISel）：把与目标无关的 IR 操作，翻译成目标芯片的真实指令。IR 的一条 add i32 %a, %b 在 x86 上要选成 ADD32rr，还可能把 load + add 一起匹配成一条带内存操作数的指令。LLVM 有几种 ISel 实现：经典的 SelectionDAG ISel（把基本块建成 DAG 再做模式匹配，最成熟）、新一些的 GlobalISel（可跨基本块，更模块化）、以及简单快速的 FastISel（O0 用，图快）。匹配规则大多来自 .td 里描述的指令模式（pattern）。这一步之后，指令变成了目标指令，但操作数仍用无限的虚拟寄存器。

② 指令调度（Scheduling）：重排指令顺序以更好地利用流水线（隐藏内存延迟、避免执行单元冲突），同时不改变语义。它需要芯片的时序模型（.td 里的 scheduling model 描述每条指令占几个周期、用哪个执行端口）。

③ 寄存器分配（Register Allocation）：把无限的虚拟寄存器映射到芯片有限的物理寄存器上。当同时活跃的值超过物理寄存器数，就得把一些值**溢出（spill）**到栈上、用时再加载回来。这是 NP 难问题，LLVM 用启发式（默认是 RegAllocGreedy 贪心分配器）。分配器需要知道芯片有哪些物理寄存器、分几类——又来自 .td。

④ 汇编 / 机器码输出（AsmPrinter / MC 层）：把最终的目标指令打印成汇编文本（.s），或直接由 MC 层编码成目标文件里的机器码字节（.o）。每条指令怎么写成汇编、怎么编码成二进制，同样在 .td 里描述。

llc 这个工具（第 1 章见过）就是把这整条后端流水线暴露成命令行：

终端
llc -march=x86-64 add.ll -o add.s      # IR → x86-64 汇编
llc -march=aarch64 add.ll -o add.s     # 同一份 IR → ARM64 汇编（换个后端）
llc -filetype=obj add.ll -o add.o      # 直接出目标文件（走 MC 层）

同一份 add.ll 喂给不同 -march，走不同的 lib/Target/ 后端，出不同芯片的代码——这就是第 1 章那张"N+M"图在后端侧的兑现。

SPIR-V、NVPTX：不太一样的「后端」

这一层对理解本书第二部分至关重要。LLVM 的 lib/Target/ 里除了 CPU（X86/AArch64/RISCV），还有几个面向 GPU / 虚拟 ISA 的特殊后端，它们的产物不是 CPU 机器码：

  • NVPTX（lib/Target/NVPTX/）：产出的是 PTX——NVIDIA 的一种虚拟汇编，本身还要再由 NVIDIA 驱动/工具编成真正的 GPU 机器码（SASS）。所以 NVPTX 后端的"机器码"其实是文本的 PTX。这是第 11 章的主角。
  • SPIRV（lib/Target/SPIRV/）：产出 SPIR-V。但注意第 8 章的提醒——SYCL 当前默认走的是 llvm/llvm-spirv/ 那个翻译器（SPIRV-LLVM-Translator），而不是这个后端；两者目标相似、历史不同。
GPU「后端」和 CPU 后端不是一码事 CPU 后端（X86 等）产出的是能直接在硬件上跑的机器码；而 NVPTX / SPIRV 后端产出的是还需再经一层厂商编译的虚拟 ISA（PTX / SPIR-V）。原因正是第 8 章讲的：GPU 的"最后一英里"（虚拟 ISA → 原生指令）由厂商驱动闭源完成，LLVM 只负责到虚拟 ISA 为止。所以看到 lib/Target/NVPTX/ 时别以为它像 X86 后端那样直出可执行机器码——它止步于 PTX。

亲手看一眼：同一段 IR，两款芯片的汇编

还是那个 add。用 llc 分别出 x86-64 和 AArch64 汇编，你能直观看到"同一份 IR、不同后端选出不同真实指令、遵循不同调用约定"：

LLVM IR
; add.ll（第 1 章那段）
define i32 @add(i32 %a, i32 %b) {
  %sum = add i32 %a, %b
  ret i32 %sum
}

llc -march=x86-64 add.ll -o - 大致得到（x86-64 调用约定：前两个整型参数在 edi、esi，返回值在 eax）：

add:
    leal    (%rdi,%rsi), %eax   # eax = edi + esi（用 lea 做加法是 x86 常见技巧）
    retq

llc -march=aarch64 add.ll -o - 则得到（AArch64 约定：参数在 w0、w1，返回值在 w0）：

add:
    add     w0, w1, w0         # w0 = w1 + w0
    ret

同一条 IR 的 add，在 x86 上被指令选择成了 leal、参数走 edi/esi，在 ARM 上是 add、参数走 w0/w1——指令选择、寄存器、调用约定全都由各自的目标描述决定。这就是后端在做的事。

动手：模拟寄存器分配与 spill

后端里最能体现"理想 IR 撞上残酷硬件"的一步是寄存器分配：无限虚拟寄存器 → 有限物理寄存器，挤不下就 spill。用一个浏览器里能跑的小模型演示：给定每个虚拟寄存器的"活跃区间"和有限的物理寄存器数，做一个简单的线性扫描分配，看谁拿到寄存器、谁被溢出到栈。点 ▶ 运行。

Python 
# 模拟寄存器分配（线性扫描简化版）：虚拟寄存器挤进有限物理寄存器，挤不下就 spill

NUM_PHYS_REGS = 3    # 这款"芯片"只有 3 个物理寄存器

# 每个虚拟寄存器的活跃区间 [start, end)：从哪条指令定义、到哪条指令最后一次使用
vregs = [
    ("%a", 0, 4),
    ("%b", 1, 3),
    ("%c", 2, 6),
    ("%d", 3, 5),   # 定义时 a,b,c 可能都还活着 → 可能挤不下
    ("%e", 5, 7),
]

free_regs = [f"R{i}" for i in range(NUM_PHYS_REGS)]  # 可用物理寄存器池
active = []          # 当前活跃且已分配寄存器的：(end, vreg, phys)
assignment = {}      # vreg -> 物理寄存器 或 "spill(栈)"

# 按活跃区间起点排序，线性扫描
for name, start, end in sorted(vregs, key=lambda v: v[1]):
    # 先回收：所有在 start 之前就结束的活跃区间，把寄存器还回池子
    still_active = []
    for a_end, a_name, a_phys in active:
        if a_end <= start:
            free_regs.append(a_phys)
            print(f"  指令{start}：{a_name} 活跃期结束，释放 {a_phys}")
        else:
            still_active.append((a_end, a_name, a_phys))
    active = still_active

    # 分配
    if free_regs:
        phys = free_regs.pop(0)
        assignment[name] = phys
        active.append((end, name, phys))
        print(f"指令{start}：{name} 分到物理寄存器 {phys}")
    else:
        assignment[name] = "spill(栈)"
        print(f"指令{start}：{name} 无空闲寄存器 → SPILL 到栈")

print("\n最终分配：")
for name, _, _ in vregs:
    print(f"  {name} -> {assignment[name]}")

运行结果：

指令0：%a 分到物理寄存器 R0
指令1：%b 分到物理寄存器 R1
指令2：%c 分到物理寄存器 R2
  指令3：%b 活跃期结束，释放 R1
指令3：%d 分到物理寄存器 R1
  指令5：%a 活跃期结束，释放 R0
  指令5：%d 活跃期结束，释放 R1
指令5：%e 分到物理寄存器 R0
最终分配：
  %a -> R0
  %b -> R1
  %c -> R2
  %d -> R1
  %e -> R0

这个模型演清了寄存器分配的核心张力：物理寄存器是稀缺资源，分配器按活跃区间"占用—释放"循环利用（%b 在指令 3 结束后，它的 R1 立刻被 %d 复用）。这一版恰好没触发 spill——因为同时活跃的虚拟寄存器数没超过 3。请动手改：把 NUM_PHYS_REGS 改成 2，重新运行——你会看到当三个值同时活跃、只有 2 个寄存器时，有一个被 SPILL 到栈。这正是"IR 里无限的 SSA 值撞上真实芯片有限寄存器"时后端必须做的取舍；真实的 RegAllocGreedy 用更聪明的启发式决定"溢出谁代价最小"，但基本矛盾一模一样。

小结与下一步

第一部分的最后一块地基就位——你现在有了从源码到机器码的完整 LLVM 心智模型：

  • 后端填平 IR 与真实硬件的落差：有限寄存器、真实指令、时序约束、调用约定。
  • 它分两半：llvm/lib/CodeGen/（与芯片无关的算法框架：ISel 骨架、寄存器分配、调度）+ llvm/lib/Target/<芯片>/（每款芯片一份的目标描述，大量用 TableGen .td 声明寄存器/指令/约定，由 llvm-tblgen 生成 C++）。
  • 流水线四步：指令选择（ISel）→ 指令调度 → 寄存器分配（挤不下就 spill）→ 汇编/机器码输出（AsmPrinter/MC）。命令行工具是 llc（-march= 选后端，-filetype=obj 出目标文件）。
  • NVPTX / SPIRV 是特殊后端：产出的是虚拟 ISA（PTX / SPIR-V），还需厂商再编一层——这直接连到第二部分。

第一部分（LLVM 基础）到此完成。 你已经掌握：三段式与 IR 中心（ch1）、IR 数据结构与 SSA（ch2）、IR 三形态（ch3）、Pass 框架设计（ch4）、写 Pass 与插件（ch5）、优化管线（ch6）、后端代码生成（ch7）。

这一章的要点

  • 后端把与目标无关的 IR 落到真实硬件，解决四个落差：有限物理寄存器、特定 ISA 指令、执行时序、调用约定。
  • 分两半：llvm/lib/CodeGen/（芯片无关算法：ISel 骨架、RegAllocGreedy、调度器）+ llvm/lib/Target/<芯片>/（每芯片一份目标描述）。
  • 目标描述大量用 TableGen（.td） 声明寄存器、指令、调用约定、指令选择模式，由 llvm-tblgen 生成 C++——写新后端主要就是写 .td。
  • 流水线四步：指令选择（SelectionDAG / GlobalISel / FastISel）→ 调度 → 寄存器分配（溢出 spill 到栈）→ AsmPrinter/MC 输出。
  • 工具 llc：-march= 选目标后端，同一份 IR 出不同芯片汇编；-filetype=obj 经 MC 层直出 .o。
  • NVPTX（→ PTX，第 11 章）、SPIRV（→ SPIR-V，第 8 章）是产出虚拟 ISA 的特殊后端，需厂商再编一层，别与 X86 那种直出机器码的 CPU 后端混为一谈。
