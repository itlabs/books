第 23 章 案例巡礼：bindless images 与 kernel_compiler（运行时编译）

这一章想让你带走什么

前三章把"扩展解剖学"讲全了：命名三位一体（第 20 章）、一个轻量扩展的完整切面（第 21 章）、需要编译器配合的两条路线（第 22 章）。这一章拿两个分量很重的真实扩展做样本，检验你能不能独立读懂一个大扩展的实现重心落在哪——而它们恰好落在截然不同的地方。

bindless images（sycl_ext_oneapi_bindless_images）：让你用不透明句柄（handle）而非 accessor 访问图像，突破 SYCL 2020 图像"编译期数量固定、靠 accessor 追踪依赖"的限制。它几乎不动编译器——重心在运行时 + UR（第 18 章）：一批新句柄类型、handler 上的新成员函数、`urBindlessImages*Exp` 那层 API。kernel_compiler（sycl_ext_oneapi_kernel_compiler）：让你在运行时把一段源码字符串（OpenCL C / SPIR-V 字节 / 甚至 SYCL）编译成可执行 kernel_bundle。它的重心更是别处——顶层 sycl-jit/ 子项目在进程内调用 clang 库把 SYCL 源码 RTC 成设备镜像。读完你会看到：同样是"扩展"，一个靠运行时 + UR、一个靠独立 JIT 子项目，实现重心可以差得非常远——这正是这一部分要教会你的判断力。

案例一：bindless images——为什么要"无绑定"

先看 SYCL 2020 的图像（image）有什么约束。经典 image 通过 accessor 访问（和 buffer 一样，第 16 章）：

SYCL / DPC++
sycl::image<2> img(host_ptr, format, range);
q.submit([&](sycl::handler& h) {
    sycl::accessor acc(img, h);            // 每个 image 一个 accessor
    h.parallel_for(range, [=](sycl::id<2> id) { /* 通过 acc 采样 */ });
});

这套设计的优点是自动依赖追踪——accessor 让运行时知道谁读谁写，自动排出执行顺序（第 17 章的图调度）。但它的约束也正来自这里：

  • 图像数量在编译期就固定：每个 image 要一个 accessor，你没法在运行时决定"用 1000 张纹理里的哪几张"。
  • 难做纹理数组 / 纹理图集（texture atlas）：GPU 图形里常见的"一大堆纹理按索引选"，accessor 模型很别扭。
  • 难和图形 API 互操作：Vulkan/DX 那边的纹理是句柄式的，对不上 accessor。

bindless images 的思路直接借鉴图形界的"bindless"：用不透明句柄访问图像，句柄可以像普通数据一样放进 kernel、存进数组、运行时索引。规范文档（experimental）：

sycl/doc/extensions/experimental/sycl_ext_oneapi_bindless_images.asciidoc

feature-test 宏 SYCL_EXT_ONEAPI_BINDLESS_IMAGES，当前值 6（feature_test.hpp.in:43）——第 20 章说过，值即版本号，能到 6 说明它已经迭代很多轮。目前主要支持 CUDA / HIP / Level Zero 上的 GPU。

它是"积木"，不是 image 的替代品 规范里明说 bindless images 提供的是更底层的构建块（building blocks），而非要取代 SYCL 2020 的 image。你付出的代价是失去自动依赖追踪（句柄不是 accessor，运行时不知道谁读谁写，要你自己用 event 排依赖），换来的是灵活性（运行时数量、索引、互操作）。这是一个典型的"低层能力换控制权"的扩展——和 USM（相对 buffer/accessor，第 16 章）放弃自动依赖追踪、换来指针式自由的取舍是一个味道。

实现重心：句柄类型 + handler 成员 + UR，几乎不碰编译器

bindless images 的实现几乎全在头文件 API + 运行时 + UR这三层，编译器前端基本不参与（对比第 22 章：它不需要新属性、不需要 SemaSYCL 认新形态）。拆开看：

① 头文件里的一批新类型（sycl/include/sycl/ext/oneapi/bindless_images*.hpp，主文件加 _descriptor/_memory/_mem_handle/_sampler/_interop 几个）：

  • unsampled_image_handle（bindless_images.hpp:36）——一个 struct，核心就是一个 uint64_t raw_handle。这就是那个"不透明句柄"——本质是个 64 位整数，能拷贝、能进 kernel、能放数组。
  • sampled_image_handle（:48）——带采样器的版本。
  • image_descriptor（bindless_images_descriptor.hpp:54）——描述图像的宽高、通道、格式。
  • image_mem（class）——设备图像内存的 RAII 封装。

② 分离"分配"和"创建句柄"两步（对比经典 image 一步到位）：

SYCL / DPC++
namespace syclexp = sycl::ext::oneapi::experimental;
// 1. 描述 + 分配设备图像内存
syclexp::image_descriptor desc(range, channels, format);
syclexp::image_mem mem(desc, q);
// 2. 从内存创建一个不透明句柄
syclexp::unsampled_image_handle h = syclexp::create_image(mem, desc, q);
// 3. 句柄像普通数据一样传进 kernel，运行时决定用哪个
q.parallel_for(range, [=](sycl::id<2> id) {
    auto px = syclexp::fetch_image<float>(h, sycl::int2(id[0], id[1]));
});

对应的运行时入口都在 handler 和自由函数里：alloc_image_mem（bindless_images.hpp:67/77）、free_image_mem、create_image（:356–479，好几个重载）、以及 handler 的成员 ext_oneapi_copy（handler.hpp:2275+，负责 host↔device 图像拷贝）。

③ 运行时 + UR 才是真正干活的地方：

  • 运行时实现在 sycl/source/detail/bindless_images.cpp——把上面的 API 翻译成 UR 调用。
  • UR 层（第 18 章）新增了一整套 urBindlessImages*Exp 入口（ur_api.h 里搜 urBindlessImages），如分配 image 内存、创建句柄、采样器。Exp 后缀标记它们是实验性 UR 入口——扩展的能力最终要落到 UR 这层适配器，再由各 backend adapter（Level Zero/CUDA/HIP）实现。
bindless images：重心在运行时 + UR，编译器几乎不参与 它用不透明句柄（unsampled_image_handle / sampled_image_handle，本质 uint64_t）替代 accessor 访问图像，突破"图像数量编译期固定 + accessor 依赖追踪"的约束，换来运行时数量/索引/图形互操作——代价是失去自动依赖追踪。实现分三层：头文件新类型（bindless_images*.hpp）、运行时（detail/bindless_images.cpp）、UR 的 urBindlessImages*Exp 入口。不需要新编译器属性、不动 SemaSYCL——这是"纯运行时/UR 扩展"的典型。

案例二：kernel_compiler——运行时把源码编成 kernel

第二个案例的重心又飘到了一个全新的地方。kernel_compiler 要解决的是：能不能在程序运行时，拿一段源码字符串（甚至是别的语言写的），当场编译成一个能在设备上跑的 kernel？这就是 RTC（runtime compilation，运行时编译）。

用处很实际：根据运行时参数生成/特化 kernel 源码、加载用户在配置里写的 kernel、和以 OpenCL C 或 SPIR-V 分发 kernel 的生态对接。它的规范其实是一组文档（第 20 章说的"一个扩展一个 doc"在这里破例）：

  • sycl_ext_oneapi_kernel_compiler（umbrella，伞文档）——总纲，SYCL 语言的 RTC 也写在这里。
  • sycl_ext_oneapi_kernel_compiler_opencl——OpenCL C 源码。
  • sycl_ext_oneapi_kernel_compiler_spirv——SPIR-V 字节。

对应三个 feature-test 宏：SYCL_EXT_ONEAPI_KERNEL_COMPILER=1（feature_test.hpp.in:47）、..._OPENCL=1（:48）、..._SPIRV=1（:49）。注意没有单独的 _sycl 文件和宏——SYCL 源码语言就用伞文档 + 主宏表达（这是"以文档 Feature test macro 一节为准"的又一例，第 20 章）。

用户 API：从源码字符串到 kernel_bundle

API 全在 sycl/include/sycl/kernel_bundle.hpp，核心是给 bundle_state 加了一个新状态 ext_oneapi_source（"源码态"），和一个 source_language 枚举（kernel_bundle_enums.hpp:23：opencl=0 / spirv=1 / sycl=2）：

SYCL / DPC++
namespace syclexp = sycl::ext::oneapi::experimental;
// 1. 用一段 OpenCL C 源码字符串建一个"源码态"bundle
std::string src = R"(
  __kernel void add(__global int* a, __global int* b) { a[get_global_id(0)] += b[...]; }
)";
auto src_bundle = syclexp::create_kernel_bundle_from_source(
    ctx, syclexp::source_language::opencl, src);        // kernel_bundle.hpp:1213
// 2. 运行时编译成可执行态
auto exe_bundle = syclexp::build(src_bundle);            // :1284
// 3. 按名字取出 kernel 启动
sycl::kernel k = exe_bundle.ext_oneapi_get_kernel("add");// :503
q.submit([&](sycl::handler& h){ h.set_args(...); h.parallel_for(range, k); });

关键 API 一览（都在 kernel_bundle.hpp）：create_kernel_bundle_from_source（:1213 收字符串 / :1229 收字节，SPIR-V 走字节那个）、compile（:1250/1270）、build（:1284/1307）、ext_oneapi_get_kernel（:503）。整套设计接回了第 16 章的 kernel_bundle 状态机——只是在 input → object → executable 之前多插了一个 ext_oneapi_source 源码态。

实现重心：三种语言，三条差得很远的路

真正精彩的是运行时实现——三种源码语言走三条完全不同的编译路径，都在 sycl/source/detail/kernel_compiler/ 下，由 device_image_impl.hpp 里一个按 MRTCBinInfo->MLanguage 分派的 switch 决定（device_image_impl.hpp 约 :1195（SYCL）/ :1377（opencl）/ :1393（spirv））：

  • SPIR-V 字节（最省事）：字节流本就是设备可消费的中间表示，直接拿去用（:1393 一带而过），无需真正"编译"。
  • OpenCL C：dlopen 打开 libocloc.so（Intel 的 offline compiler 库 ocloc），把 OpenCL C 交给它编成设备镜像。外挂一个已有编译器。
  • SYCL 源码（最重）：这才是重头戏——它把源码交给顶层 sycl-jit/ 子项目，在进程内调用 clang 库做真正的 SYCL 编译。

sycl-jit：进程内的 clang，把 SYCL 源码 RTC 成镜像

SYCL 源码的 RTC 路径值得单独看，因为它动用了一个独立的顶层子项目 sycl-jit/（和 sycl/、libdevice/ 平级，第 19 章见过这种顶层布局）。链路是：

  1. 运行时 dlopen libsycl-jit.so，调它导出的 compileSYCL（sycl/source/detail/jit_compiler.cpp 里加载）。
  2. libsycl-jit.so 的实现在 sycl-jit/jit-compiler/lib/rtc/DeviceCompilation.cpp——它在进程内实例化 clang 的 CompilerInstance / ToolInvocation，等于把一个 clang 编译器嵌进了你的运行时进程，就地把 SYCL 源码编成 LLVM IR。
  3. 再经 SPIRVLLVMTranslation.cpp 把 IR 翻成 SPIR-V（就是第 9 章那个 llvm-spirv 的能力，这里以库形式调用）。
  4. SPIR-V 交给正常的设备镜像加载路径（第 8、18 章）上设备。

换句话说，SYCL 的 RTC 不是"外挂一个编译器进程"，而是把编译器当库链进来、在进程内跑一遍完整的设备编译前端 + SPIR-V 翻译。设计文档在 sycl/doc/design/SYCL-RTC.md。

图 23-1　kernel_compiler 的三条运行时编译路径。SPIR-V 字节直接用；OpenCL C 外挂 libocloc.so；SYCL 源码 最重——dlopen libsycl-jit.so，在进程内用 clang 库（DeviceCompilation.cpp）把源码编成 IR、再翻成 SPIR-V。三条路殊途同归到正常的设备镜像加载路径

两个案例的对照：实现重心可以差得很远

把这一章的两个样本并排看，正是这一部分想教的判断力——"扩展"这个词底下，实现重心的分布千差万别：

维度 bindless images kernel_compiler
解决的问题 用句柄突破图像数量/依赖约束 运行时把源码编成 kernel
编译器前端 几乎不动 基本不动（但 SYCL 路径把整个编译器当库调）
头文件 一批新句柄/描述类型 给 bundle_state/source_language 加值
运行时 detail/bindless_images.cpp detail/kernel_compiler/ + dispatch
独立子项目 无 sycl-jit/（进程内 clang）
UR 层 重（urBindlessImages*Exp 一整套） 走既有镜像加载路径
复用了什么 UR 适配器机制 kernel_bundle 状态机（加源码态）

读一个陌生的大扩展时，第一件事就是定位它的重心：是纯库/运行时（bindless images）、还是要动编译器前端（第 22 章的 free function kernels）、还是引入独立子系统（kernel_compiler 的 sycl-jit）。定位准了，你才知道该去哪个目录读它的核心实现。

"一个扩展一个 doc/一个宏"有例外，以文档为准 第 20 章立的规矩（扩展名=宏名=文件名）在这两个案例里都遇到例外：kernel_compiler 是一伞三文档（umbrella + opencl + spirv），且 SYCL 语言没有独立的 _sycl 文件/宏。这再次印证第 20 章那条铁律：看某扩展到底有哪些宏、值多少，永远以文档的 Feature test macro 一节和 feature_test.hpp.in 为准，别靠命名规律硬推。

动手：模拟 kernel_compiler 的语言分派

把这一章 kernel_compiler 的核心——「同一个 API，按 source_language 分派到三条差得很远的编译路径」——用一个浏览器里能跑的小模型演出来。点 ▶ 运行。

Python 
# 模拟 kernel_compiler：create_kernel_bundle_from_source → build → 按语言分派

# ---- source_language 枚举（kernel_bundle_enums.hpp）----
OPENCL, SPIRV, SYCL = "opencl", "spirv", "sycl"

def create_kernel_bundle_from_source(language, source):
    # 得到一个"源码态"bundle（bundle_state::ext_oneapi_source）
    return {"state": "ext_oneapi_source", "lang": language, "src": source}

def build(src_bundle):
    """运行时编译：device_image_impl.hpp 里按 MLanguage 分派的 switch。"""
    lang = src_bundle["lang"]
    print(f"build(): 源码语言 = {lang}")
    if lang == SPIRV:
        # 最省事：字节流本就是设备可消费 IR，直接用
        print("  → [spirv] 字节流即设备 IR，无需编译，直接加载")
        image = "spirv-bytes(直接用)"
    elif lang == OPENCL:
        # 外挂已有编译器：dlopen libocloc.so
        print("  → [opencl] dlopen libocloc.so，交 ocloc 编成设备镜像")
        image = "ocloc 产出的设备镜像"
    elif lang == SYCL:
        # 最重：进程内跑 clang 库
        print("  → [sycl] dlopen libsycl-jit.so → compileSYCL")
        print("      DeviceCompilation.cpp: 进程内 clang CompilerInstance，源码 → LLVM IR")
        print("      SPIRVLLVMTranslation.cpp: IR → SPIR-V")
        image = "sycl-jit 进程内编译产出的 SPIR-V"
    else:
        raise ValueError(f"未知语言 {lang}")
    # 三条路殊途同归：都变成可执行 bundle
    return {"state": "executable", "image": image}

def ext_oneapi_get_kernel(exe_bundle, name):
    print(f"  ext_oneapi_get_kernel('{name}') ← {exe_bundle['image']}\n")

for lang, src in [
    (SPIRV,  "0x07230203..."),               # SPIR-V 魔数字节
    (OPENCL, "__kernel void add(...){...}"),  # OpenCL C 源码
    (SYCL,   "SYCL_EXTERNAL void k(...){...}"),
]:
    sb = create_kernel_bundle_from_source(lang, src)
    eb = build(sb)
    ext_oneapi_get_kernel(eb, "add")

运行结果：

build(): 源码语言 = spirv
  → [spirv] 字节流即设备 IR，无需编译，直接加载
  ext_oneapi_get_kernel('add') ← spirv-bytes(直接用)

build(): 源码语言 = opencl
  → [opencl] dlopen libocloc.so，交 ocloc 编成设备镜像
  ext_oneapi_get_kernel('add') ← ocloc 产出的设备镜像

build(): 源码语言 = sycl
  → [sycl] dlopen libsycl-jit.so → compileSYCL
      DeviceCompilation.cpp: 进程内 clang CompilerInstance，源码 → LLVM IR
      SPIRVLLVMTranslation.cpp: IR → SPIR-V
  ext_oneapi_get_kernel('add') ← sycl-jit 进程内编译产出的 SPIR-V

看清 kernel_compiler 的精髓：一个统一的 API（create_kernel_bundle_from_source → build → ext_oneapi_get_kernel）背后藏着三条重量完全不同的路——SPIR-V 几乎零成本、OpenCL 外挂 ocloc、SYCL 则在进程内跑一整套 clang 前端。这也解释了为什么 SYCL 的 RTC 要单独立一个 sycl-jit/ 顶层子项目：它做的事情和离线编译一样重。请动手改：给循环再加一种假想语言（比如 ("cuda", "__global__ ...")），你会看到 build() 抛 未知语言——这正对应实现里 switch 的 default 分支：扩展只支持文档里列出的那几种源码语言，其余在运行时就被拒。

小结与下一步

这一章用两个大扩展检验了"定位实现重心"的判断力：

  • bindless images（SYCL_EXT_ONEAPI_BINDLESS_IMAGES=6）：用不透明句柄（unsampled/sampled_image_handle，本质 uint64_t）替代 accessor 访问图像，突破"数量编译期固定 + accessor 依赖追踪"的约束，换来运行时数量/索引/图形互操作（代价：自己排依赖）。重心在头文件新类型 + 运行时 detail/bindless_images.cpp + UR 的 urBindlessImages*Exp，编译器几乎不参与。是"积木"不是替代品。
  • kernel_compiler（伞文档 + _opencl + _spirv，宏 ..._KERNEL_COMPILER/_OPENCL/_SPIRV 均=1，SYCL 语言无独立文件/宏）：运行时把源码字符串编成 kernel_bundle。API 给 bundle_state 加了 ext_oneapi_source 源码态，source_language 枚举 opencl/spirv/sycl。重心在 detail/kernel_compiler/ 的三路分派：SPIR-V 直接用、OpenCL dlopen libocloc.so、SYCL 最重——dlopen libsycl-jit.so，在进程内用 clang 库（sycl-jit/ 顶层子项目的 DeviceCompilation.cpp）RTC 成 IR 再翻 SPIR-V。设计文档 SYCL-RTC.md。
  • 对照：同样是扩展，实现重心可落在纯运行时+UR（bindless images）、编译器前端（free function kernels）、独立子项目（kernel_compiler 的 sycl-jit）——读大扩展第一步永远是定位重心去哪个目录。

  • bindless images（sycl_ext_oneapi_bindless_images，宏值 6，experimental，CUDA/HIP/L0）：不透明句柄 unsampled_image_handle/sampled_image_handle（本质 uint64_t）替代 accessor，突破图像数量编译期固定 + 自动依赖追踪的约束（代价：手动排依赖）。是底层积木非替代品。

  • API（kernel_bundle.hpp）：bundle_state::ext_oneapi_source 源码态 + source_language（opencl=0/spirv=1/sycl=2）；create_kernel_bundle_from_source（:1213 字符串/:1229 字节）→ build（:1284）→ ext_oneapi_get_kernel（:503）。复用第 16 章 kernel_bundle 状态机。
