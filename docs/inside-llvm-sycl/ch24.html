第 24 章 从提案到落地：写扩展的完整流程与测试

这一章想让你带走什么

第 20–23 章教你读扩展——命名三位一体、一个轻量扩展的切面、需要编译器配合的两条路线、两个大案例。这一章把它们倒过来：假设你要给 DPC++ 写一个新扩展，从零到进主线，完整流程是什么。这是全书正文的收尾，也是把前面所有知识串成"能动手"的一章。

读完你会掌握一条端到端的路：① README-process.md 定义的扩展生命周期（提案 proposed → 实现并移到 experimental/supported → 版本化演进 → 弃用 deprecated → 移除 removed）；② 先写文档、再写代码——复制 template.asciidoc 起草规范、定 feature-test 宏、选 vendor 与目录；③ 实现落到哪几层（头文件 / SemaSYCL / 运行时 / UR，取决于第 22–23 章教的"重心在哪"）；④ 最关键的 两级测试——sycl/test/（LIT，不上真机、查编译产物）与 sycl/test-e2e/（E2E，上真设备跑结果），扩展没有这两级测试兜底就进不了主线。这一章不引入新机制，而是把前面每一章的落点，编织成一张"怎么写一个扩展"的作战地图。

扩展的一生：README-process.md 定义的生命周期

第 20 章讲过五态目录（proposed / experimental / supported / deprecated / removed），当时是从"读者视角"看成熟度。现在从"作者视角"看：这五个目录其实是一个扩展要依次走过的驿站，规则写死在 sycl/doc/extensions/README-process.md（标题就叫 "Lifetime of an Extension"）：

  1. 提案（proposed/）：扩展先以纯文档形式落地 proposed/，尚无实现。此阶段规范可以随便改——因为还没人依赖它。目的是把设计摊开讨论。
  2. 实现并移动（→ experimental/ 或 supported/）：实现它的那个 PR 连同把文档从 proposed/ 移到 experimental/（API 还可能变）或 supported/（承诺稳定）。文档移动和代码实现在同一个 PR 里——这是硬规矩，保证"文档说支持"和"代码真支持"永远一致。
  3. 版本化演进：扩展加新 API 就把 feature-test 宏 +1（第 20 章：值即版本号，bindless images 已到 6）。用户靠 #if 宏 >= N 精确要求版本。
  4. 弃用（→ deprecated/）：功能被核心 SYCL 吸收、或有更好的替代时，移到 deprecated/——仍能用，但编译时给 deprecation 警告，告诉用户尽快迁移。
  5. 移除（→ removed/）：过了弃用期，实现删除，文档移到 removed/ 存档——留个"曾经有过、为何移除"的记录。
写扩展是"文档驱动"的：先提案文档，实现时文档随代码同 PR 移动目录 README-process.md（"Lifetime of an Extension"）规定：新扩展先以文档形式进 proposed/（无实现、可随意改）；实现它的 PR 同时把文档移到 experimental/（API 可变）或 supported/（稳定）；之后版本化演进（宏 +1）；被取代则 deprecated/ → removed/。核心纪律是文档与实现永远同步——目录位置就是"这个扩展现在有多可信"的权威声明。所以写扩展第一步永远是写规范，而不是写代码。

第一步：起草规范（复制 template.asciidoc）

README-process.md 开门见山："Start by making a copy of the template extension specification document"——从复制 sycl/doc/extensions/template.asciidoc 开始，按里面的说明填空，并遵守 SYCL 2020 规范第 6 章 "Guidelines for portable extensions"。起草时要定死这几件事（都是第 20 章的"三位一体"落到笔头）：

  • 扩展名 / 文件名 / 宏名：sycl_ext_<vendor>_<feature>，三者一致（大小写差异）。vendor 默认 oneapi，很 Intel 硬件专属才用 intel，Codeplay 贡献用 codeplay。
  • 放哪个目录：新提案进 proposed/（Status 一节写 proposed 样板文字）。
  • feature-test 宏与版本表：在 Specification → Feature test macro 一节声明宏名和初值（实验扩展一般从 1 起）。
  • 依赖：Dependencies 一节写依赖哪版 SYCL、哪些其他扩展。

template.asciidoc 规定的标准骨架（第 20 章列过）：Notice / Contact / Dependencies / Status / Backend support（可选）/ Overview / Specification（含 Feature test macro）/ Implementation notes / Issues。先把"提供什么能力、宏叫什么、放哪个目录、依赖什么"想清楚，再写一行实现——这是 DPC++ 扩展流程和普通功能开发最不一样的地方。

第二步：实现——重心在哪，前四章已经教过

规范定稿后才动代码。实现落到哪几层，完全取决于第 22–23 章教的"重心在哪"——这一章不重复机制，只把它当成一张选路表：

扩展类型 主要改动 本书对应
纯库扩展 只加头文件里的类/函数 + 运行时 （最常见，不用动编译器）
要把意图带到 IR（通用路线） 头文件里用 add_ir_attributes_* 编码属性 第 22 章通用路线、第 21 章
要语义检查/新 metadata（专用路线） Attr.td→AttrDocs.td→SemaSYCLDeclAttr.cpp→CodeGen 第 22 章专用路线
要认新 kernel 形态 SemaSYCL.cpp（isFreeFunction 那类）+ 诊断 第 21、22 章
纯运行时 + 后端能力 运行时 detail/ + UR 的 ur*Exp 入口 第 23 章 bindless images
引入独立子系统 顶层子项目（如 sycl-jit/） 第 23 章 kernel_compiler

无论走哪条，都要记住第 20 章的收尾：头文件里 feature-test 宏的定义（sycl/source/feature_test.hpp.in）要和文档同步添加——这是"扩展真的被实现了"在代码里的第一个可见证据，也是用户 #ifdef 能检测到它的前提。

第三步（也是能不能进主线的关卡）：两级测试

DPC++ 对扩展的测试要求很硬：没有测试的扩展进不了主线。测试分两级，分工清晰，都在 sycl/ 下：

LIT 测试（sycl/test/）——不上真机，查"编译产物对不对"

LIT 测试不需要真实 GPU，用 clang/opt 等把源码编到某个中间阶段，然后用 FileCheck 断言产物里该有/不该有什么。它验证的是编译器行为，跑得快、能在任何 CI 机器上跑。对扩展来说，典型的 LIT 断言：

  • 属性有没有正确落到 IR（第 21/22 章）：-fsycl-device-only -S -emit-llvm 后 FileCheck 那条 "sycl-nd-range-kernel" 字符串属性在不在。
  • 诊断对不对（第 22 章的专用路线）：// expected-error{{...}} 断言非法用法确实报了预期的 err_*。
  • feature-test 宏定义对不对：#ifdef 检测宏在预期条件下有定义、值正确。
终端
# LIT 测试长这样（示意）：一个 .cpp 里嵌 RUN 行 + CHECK/expected 断言
# RUN: %clangxx -fsycl -fsycl-device-only -S -emit-llvm %s -o - | FileCheck %s
# CHECK: define {{.*}} @{{.*}}scale{{.*}} #[[ATTR:[0-9]+]]
# CHECK: attributes #[[ATTR]] = { {{.*}}"sycl-nd-range-kernel"="3"{{.*}} }

E2E 测试（sycl/test-e2e/）——上真设备，查"跑出来对不对"

E2E（end-to-end）测试要真实设备（Intel GPU / CUDA / …），它把扩展的完整用法编译、上设备真跑，断言计算结果正确。sycl/test-e2e/ 下按功能分目录（上百个），比如 bindless images、kernel_compiler 各有自己的目录。它验证的是端到端语义——不光编译对，跑出来也对。E2E 测试因为要真机，通常在带 GPU 的专用 CI（llvm/buildbot/ 那套构建/测试基础设施）上跑。

一个负责任的扩展 PR 两级都要覆盖：LIT 保证"编译器把属性/诊断/宏处理对了"，E2E 保证"用户照文档写、在真设备上结果对"。两者缺一——只有 LIT 没 E2E，你不知道它真机上跑不跑得起来；只有 E2E 没 LIT，编译器的细节回归（比如属性名拼错）可能被忽略。

图 24-1　写扩展的作战地图。① 复制 template.asciidoc 进 proposed/ 起草规范 → ② 实现（重心按第 22–23 章定位）并让文档随代码同 PR 移到 experimental/、同步在 feature_test.hpp.in 加宏 → ③ 补齐 LIT + E2E 两级测试（关卡）→ ④ 进主线供用户 #ifdef 检测。之后沿五态目录演进

一个好扩展的软性准则

流程之外，README-process.md 和 SYCL 2020 第 6 章还隐含一些"好扩展"的判断，值得记住：

  • 可移植优先：能用通用 oneapi vendor 就别绑 intel；能靠 feature-test 宏协商能力就别加新驱动开关（第 22 章）。目标是"增能力而不破坏可移植性"。
  • 搭在既有机制上：第 21 章 free function kernels 复用 kernel_bundle、第 23 章 kernel_compiler 复用 bundle 状态机、bindless images 复用 UR 适配器——好扩展尽量不另起炉灶。这既省实现量，也让用户的既有心智模型能迁移过来。
  • 先 experimental 后 supported：不确定 API 会不会变，就先进 experimental/（宏恒为 1、命名空间进 ::experimental），等设计沉淀了再考虑上提到 supported/ 甚至推给 Khronos 进核心标准。别一上来就承诺稳定。
文档、宏、实现、测试——四者必须同时到位 最容易犯的错是"四件套"缺角：改了实现忘了移动/更新文档、加了宏忘了写 E2E、写了文档没同步 feature_test.hpp.in。DPC++ 的 code review 会卡这些。记住第 20 章的三位一体 + 这一章的两级测试，凑齐规范文档 + feature-test 宏 + 分层实现 + LIT/E2E 测试四件套，才是一个能进主线的完整扩展 PR。

动手：模拟"扩展 PR 完整性"检查

把这一章的核心——「一个扩展 PR 必须四件套齐全：规范文档 + feature-test 宏 + 分层实现 + 两级测试」——做成一个能在浏览器里跑的小检查器。点 ▶ 运行。

Python 
# 模拟 DPC++ 扩展 PR 的完整性门禁：四件套 + 生命周期目录检查

REQUIRED = ["spec_doc", "feature_macro", "implementation", "lit_test", "e2e_test"]
VALID_DIRS = {"proposed", "experimental", "supported", "deprecated", "removed"}

def review_extension_pr(name, artifacts, target_dir):
    print(f"=== 审查扩展 PR: {name}  (目标目录: {target_dir}/) ===")
    ok = True
    # 1. 目录合法性 + proposed 特例（proposed 不要求实现/测试）
    if target_dir not in VALID_DIRS:
        print(f"  ✗ 非法目录 {target_dir}（须是五态之一）"); ok = False
    proposed_only = (target_dir == "proposed")
    # 2. 四件套检查
    for item in REQUIRED:
        have = artifacts.get(item, False)
        # proposed 阶段：只需文档 + 宏声明，实现/测试可缺
        optional_now = proposed_only and item in ("implementation", "lit_test", "e2e_test")
        if have:
            print(f"  ✓ {item}")
        elif optional_now:
            print(f"  – {item}（proposed 阶段可暂缺）")
        else:
            print(f"  ✗ 缺少 {item}"); ok = False
    # 3. 命名三位一体（第20章）
    if not name.startswith("sycl_ext_"):
        print(f"  ✗ 扩展名不符 sycl_ext_<vendor>_<feature>"); ok = False
    print(f"  → {'可以合入 ✅' if ok else '打回，补齐后再提 ❌'}\n")
    return ok

# PR A：实验扩展，四件套齐全 → 通过
review_extension_pr("sycl_ext_oneapi_free_function_kernels",
    {"spec_doc": True, "feature_macro": True, "implementation": True,
     "lit_test": True, "e2e_test": True}, "experimental")

# PR B：想进 experimental 但没写 E2E → 打回
review_extension_pr("sycl_ext_oneapi_fancy",
    {"spec_doc": True, "feature_macro": True, "implementation": True,
     "lit_test": True, "e2e_test": False}, "experimental")

# PR C：纯提案进 proposed → 只需文档+宏，通过
review_extension_pr("sycl_ext_oneapi_future_idea",
    {"spec_doc": True, "feature_macro": True}, "proposed")

运行结果：

  ✓ feature_macro
  ✓ implementation
  ✓ lit_test
  ✓ e2e_test
  → 可以合入 ✅

=== 审查扩展 PR: sycl_ext_oneapi_fancy  (目标目录: experimental/) ===

  ✓ feature_macro
  ✓ implementation
  ✓ lit_test
  ✗ 缺少 e2e_test
  → 打回，补齐后再提 ❌

=== 审查扩展 PR: sycl_ext_oneapi_future_idea  (目标目录: proposed/) ===
  ✓ spec_doc
  ✓ feature_macro
  – implementation（proposed 阶段可暂缺）
  – lit_test（proposed 阶段可暂缺）
  – e2e_test（proposed 阶段可暂缺）
  → 可以合入 ✅

这个检查器把这一章编成了一道门禁：PR A 四件套齐全、进 experimental/，放行；PR B 少了 E2E 测试——真机上跑不跑得起来没人知道，打回；PR C 只是纯提案进 proposed/，此阶段允许暂无实现和测试（对应生命周期第 1 步"无实现、可随意改"），放行。这正是 DPC++ 对扩展的真实要求：目录决定门槛，experimental/supported 必须四件套齐全，proposed 只需文档先行。请动手改：把 PR C 的 target_dir 从 "proposed" 改成 "experimental"，重新运行——它立刻因为缺实现和两级测试被打回。这就是"想承诺支持，就得拿出实现和测试"的落地含义。

小结与下一步

这一章把前四章拆开看的扩展，倒过来编成了"写一个扩展"的完整流程：

  • 生命周期（README-process.md "Lifetime of an Extension"）：proposed（文档先行、无实现）→ 实现 PR 同时把文档移到 experimental（API 可变）/supported（稳定）→ 版本化演进（宏 +1）→ deprecated → removed。目录位置 = 权威的成熟度声明，文档与实现永远同 PR 同步。
  • 第一步永远是文档：复制 template.asciidoc，定死扩展名/文件名/宏名（三位一体）、vendor（默认 oneapi）、目标目录、feature-test 宏与版本、依赖；遵 SYCL 2020 第 6 章。
  • 实现分层看重心（第 22–23 章）：纯库 / 通用 IR 属性路线 / 专用属性路线（Attr.td→Sema→CodeGen）/ 认新 kernel 形态 / 运行时+UR / 独立子项目——重心不同，改的目录完全不同；同时在 feature_test.hpp.in 加宏。
  • 两级测试是进主线的关卡：sycl/test/（LIT，不上真机，FileCheck 查属性落 IR/诊断/宏）+ sycl/test-e2e/（E2E，上真设备跑结果、断言正确）。两级缺一不可。
  • 四件套：规范文档 + feature-test 宏 + 分层实现 + LIT/E2E 测试，凑齐才是能进主线的扩展 PR；好扩展可移植优先、搭在既有机制上、先 experimental 后 supported。

第四部分（扩展篇）到此完成，全书正文也就此收尾。 你从第一部分的 LLVM 基础、第二部分的 LLVM 核心机制、第三部分 SYCL 的真实实现（头文件→设备编译→运行时→UR→设备库），走到第四部分"如何给 SYCL 写扩展"——一条从"读懂编译器"到"改造编译器"的完整路径。接下来的三个附录是你日后动手时的常备手册：附录 A 手把手搭建与构建 intel/llvm（config、compile、常用命令）；附录 B LLVM/SYCL/SPIR-V/UR 术语与缩写对照；附录 C 源码导览地图——想看某个功能该去哪个目录。正文教你"为什么"，附录帮你"上手做"。

  • 第一步是文档：复制 template.asciidoc（Notice/Contact/Dependencies/Status/Overview/Specification+Feature test macro/…），定三位一体命名、vendor、目录、宏与版本；遵 SYCL 2020 第 6 章可移植扩展指南。
  • 实现重心按第 22–23 章定位（纯库 / 通用或专用属性路线 / 新 kernel 形态 / 运行时+UR / 独立子项目），并在 sycl/source/feature_test.hpp.in 同步加宏。
  • 两级测试（进主线关卡）：sycl/test/=LIT（不上真机，clang/opt+FileCheck 查属性落 IR、expected-error 查诊断、宏定义）；sycl/test-e2e/=E2E（真设备跑、断言结果，按功能分目录）。两级缺一不可。
  • 四件套：规范文档 + feature-test 宏 + 分层实现 + LIT/E2E 测试。好扩展：可移植优先（默认 oneapi、少加驱动开关）、搭在既有机制上、先 experimental 后 supported。
