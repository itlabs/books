第 5 章 Pass 框架（二）：写一个 Pass，并用插件挂进 opt

这一章想让你带走什么

第 4 章你把新 PassManager 的设计读通了：无基类的 run 契约、AnalysisManager 的按需缓存、PreservedAnalyses 收据。这一章把它全部落地——亲手写一个 Function Pass，编成一个动态库插件，用 opt -load-pass-plugin 挂进管线跑起来，最后配一个 .ll 的 LIT 测试验证它。

你会走完一个真实 LLVM 开发者天天走的闭环：写 PreservedAnalyses run(...) → 写插件入口 llvmGetPassPluginInfo 用 registerPipelineParsingCallback 把 "my-pass" 这个名字接进 PassBuilder → opt -load-pass-plugin=./MyPass.so -passes=my-pass in.ll -S。读完你不仅"知道" Pass 框架，而且"会用"它——这是第 6 章看懂 PassBuilder 如何拼出 O2 管线、乃至以后给 SYCL 加自定义 Pass 的动手基础。

一个能跑的最小 Pass

先把第 4 章的抽象契约变成一段能编译的代码。目标：写一个 Function Pass，遍历函数里的每条指令，数出有多少条，打印出来（不改 IR，纯观察——最简单的起步）。

C++
// CountInst.cpp —— 一个数指令条数的 Function Pass
#include "llvm/IR/PassManager.h"
#include "llvm/Passes/PassBuilder.h"
#include "llvm/Plugins/PassPlugin.h"
#include "llvm/Support/raw_ostream.h"
using namespace llvm;

namespace {
// 第 4 章的契约：一个有 run(Function&, FunctionAnalysisManager&) 的类就是 Function Pass
// 用 PassInfoMixin CRTP 混入拿到 name() 等样板，不是基类
struct CountInstPass : PassInfoMixin<CountInstPass> {
  PreservedAnalyses run(Function &F, FunctionAnalysisManager &AM) {
    unsigned Count = 0;
    for (BasicBlock &BB : F)          // 遍历容器树：函数 → 基本块 → 指令
      for (Instruction &I : BB)
        ++Count;
    errs() << "函数 " << F.getName() << " 有 " << Count << " 条指令\n";
    return PreservedAnalyses::all();  // 没改 IR → 所有分析都保留（第 4 章的收据）
  }
};
} // namespace

对着第 4 章逐条印证：CountInstPass 没有继承任何 Pass 基类，只是有一个签名固定的 run；PassInfoMixin<CountInstPass> 用 CRTP 提供 name()；因为它只读不改，run 诚实地返回 PreservedAnalyses::all()。这就是一个合法的 Function Pass 的全部——剩下的问题是：怎么让 opt 认识它？

插件入口：把 Pass 名字接进 PassBuilder

opt 不会凭空知道你的 .so 里有个叫什么名字的 Pass。你得提供一个约定好的导出函数 llvmGetPassPluginInfo，opt 加载动态库时会调它，你在里面把「文本名字 → 构造 Pass」这条规则注册进 PassBuilder。

C++
// 接上文，同一个 CountInst.cpp

// 注册回调：告诉 PassBuilder，当 -passes 里出现 "count-inst" 时，构造一个 CountInstPass
static void registerCB(PassBuilder &PB) {
  PB.registerPipelineParsingCallback(
      [](StringRef Name, FunctionPassManager &FPM,
         ArrayRef<PassBuilder::PipelineElement>) {
        if (Name == "count-inst") {          // 用户在 -passes= 里写的名字
          FPM.addPass(CountInstPass());       // 命中就把我们的 Pass 加进 Function 管线
          return true;                        // 返回 true 表示"这个名字我认领了"
        }
        return false;                         // 不是我的名字，交给别的回调
      });
}

// opt 加载 .so 时调用的约定入口。名字、签名都是固定的
extern "C" LLVM_ATTRIBUTE_WEAK ::llvm::PassPluginLibraryInfo
llvmGetPassPluginInfo() {
  return {LLVM_PLUGIN_API_VERSION, "CountInst", LLVM_VERSION_STRING, registerCB};
}

三个要点：

  • llvmGetPassPluginInfo 是硬约定：名字、extern "C"、返回 PassPluginLibraryInfo 都不能改——opt 就是按这个符号名去 .so 里找的。返回值里 LLVM_PLUGIN_API_VERSION 让 opt 校验 ABI 版本是否匹配（第 4 章研究里提到当前是 2）。
  • registerPipelineParsingCallback 是把「文本名字」翻译成「往管线里加哪个 Pass」的地方。-passes=count-inst 里的 count-inst 就是在这里被匹配的。返回 true = 认领这个名字。
  • 注册进的是 FunctionPassManager，因为我们写的是 Function Pass。如果写的是 Module Pass，就该用另一个重载往 ModulePassManager 里加。
两种「挂 Pass」的方式，这里用插件 把 Pass 接进 opt 有两条路：① 编进 LLVM 树内——在 llvm/lib/Passes/PassRegistry.def 里登记、重编 opt，LLVM 自带的 Pass 都这么做；② 做成外部插件——就是本章这样，编成独立 .so，用 -load-pass-plugin 动态加载，不用重编 LLVM。开发和实验用②快得多；真要进上游才用①。opt 里处理 -load-pass-plugin 的代码在 llvm/tools/opt/（optdriver.cpp / NewPMDriver.cpp）。

编译、加载、运行

把上面两段合成一个 CountInst.cpp，编成动态库（需要装了 LLVM 的开发头文件，用 llvm-config 拿编译参数）：

终端
# 编成插件动态库（Linux 下 .so；macOS 下 .dylib）
clang++ -shared -fPIC CountInst.cpp -o CountInst.so \
    $(llvm-config --cxxflags)

然后用 opt 加载并跑在一段 IR 上。还用第 2 章那个带分支的 pick 函数当输入 in.ll：

终端
opt -load-pass-plugin=./CountInst.so \
    -passes=count-inst \
    in.ll -disable-output

-passes=count-inst 里的名字，正是你在 registerPipelineParsingCallback 里匹配的那个字符串。-disable-output 表示我们只关心 Pass 打印的副作用、不需要输出 IR。运行会打印：

函数 pick 有 5 条指令

管线还能组合——-passes 接受一串用逗号分隔的 Pass，从左到右依次跑。比如先做一遍死代码消除再数指令：

终端
opt -load-pass-plugin=./CountInst.so \
    -passes='dce,count-inst' in.ll -S

这时你会看到：dce 先删掉死指令、返回它的 PreservedAnalyses 收据，count-inst 再在已经变短的函数上计数。第 4 章那个「一串 Pass 顺序跑、逐个 invalidate」的循环，你现在是在命令行上亲手驱动它。

让它真正改一次 IR

只读的 Pass 讲清了机制，但真正的变换 Pass 要动 IR。把 CountInstPass 升级一步：删除函数里没有任何 user 且无副作用的指令——这正是第 2 章动手环节和 DCE 的内核。关键在于改了 IR 后，收据必须写对：

C++
PreservedAnalyses run(Function &F, FunctionAnalysisManager &AM) {
  SmallVector<Instruction *, 8> Dead;
  for (BasicBlock &BB : F)
    for (Instruction &I : BB)
      // 没人用它的结果（use 链为空）、且删掉它是安全的（无副作用）
      if (I.use_empty() && isInstructionTriviallyDead(&I))
        Dead.push_back(&I);

  if (Dead.empty())
    return PreservedAnalyses::all();     // 什么都没删 → 一切照旧

  for (Instruction *I : Dead)
    I->eraseFromParent();                // 从基本块链表里摘除并释放（第 2 章的 ilist O(1) 删除）

  PreservedAnalyses PA;
  PA.preserveSet<CFGAnalyses>();         // 只删了指令、没动控制流 → CFG 类分析仍有效
  return PA;
}

这段把第 4 章的收据规则用活了：删指令不改变基本块之间的跳转，所以可以诚实地 preserveSet<CFGAnalyses>()，让支配树等基于 CFG 的分析免于重算。use_empty() 就是第 2 章讲的「查 def-use 图的 use 链是否为空」，eraseFromParent() 就是那条侵入式链表的 O(1) 删除。

边遍历边删会踩迭代器失效 注意上面先收集到 Dead、再统一删，而不是在 for (Instruction &I : BB) 循环体里直接 I.eraseFromParent()。因为删除会让当前迭代器失效，边遍历边删是经典的 use-after-free 崩溃来源。真实 LLVM Pass 里到处是这种「两段式」写法：遍历阶段只标记 / 收集，改动阶段再统一执行。删除顺序也有讲究——若指令间还有 use 依赖，要按「用者先删、被用者后删」的顺序，否则删被用者时它还有 user。

用 .ll 写一个 LIT 测试

写完 Pass 得验证它真的干了活，而且以后别人改代码不会把它搞坏。LLVM 的做法是 LIT 测试：一个 .ll 文件，头部用注释写 RUN: 命令跑 opt，再用 CHECK: 断言输出里该出现/不该出现什么。第 3 章讲的「.ll diff 友好」在这里兑现。

LLVM IR
; dce-test.ll —— 验证我们的 DCE 版 Pass 删掉了死指令
; RUN: opt -load-pass-plugin=%pluginpath -passes=count-inst %s -S | FileCheck %s

define i32 @f(i32 %a, i32 %b) {
entry:
  %dead = add i32 %a, %b        ; 没人用 %dead → 应被删除
  %r = mul i32 %a, %a
  ret i32 %r
}

; CHECK-LABEL: @f(
; CHECK-NOT:   %dead = add       ; 断言：输出里不再有这条死指令
; CHECK:       %r = mul
; CHECK:       ret i32 %r

FileCheck（LLVM 自带工具）把 opt 的输出和 CHECK 行逐条比对：CHECK-NOT: %dead 断言那条死加法被删了，CHECK: %r = mul 断言有用的指令还在。LLVM 自己的 llvm/test/ 下成千上万个测试都是这个套路。你把这个文件放进测试目录、跑 llvm-lit dce-test.ll，绿了就说明 Pass 行为符合预期——这是给编译器改动"上保险"的标准手段。

动手：模拟 opt 的插件加载与管线解析

把这一章的机制——「插件注册名字 → opt 解析 -passes 字符串 → 依次构造并运行 Pass」——浓缩成一个浏览器里能跑的小模型。点 ▶ 运行。

Python 
# 模拟 opt 的插件机制：注册回调认领名字 → 解析 -passes → 依次跑 Pass

# ---- 插件侧：registerPipelineParsingCallback 注册的回调 ----
# 每个回调：给它一个名字，它要么"认领"并返回一个 Pass 函数，要么返回 None
registry = []

def register(name, pass_fn):
    def callback(requested):
        return pass_fn if requested == name else None   # 名字匹配才认领
    registry.append(callback)

# ---- 三个 Pass（简化成"对指令列表的变换"，返回 (新列表, 是否保留CFG)） ----
def count_inst(instrs):
    print(f"    [count-inst] 当前 {len(instrs)} 条指令")
    return instrs, True                     # 只读，保留一切

def dce(instrs):
    kept = [i for i in instrs if not i.get("dead")]
    removed = len(instrs) - len(kept)
    print(f"    [dce] 删除 {removed} 条死指令")
    return kept, True                       # 删指令但不动 CFG

def simplifycfg(instrs):
    print(f"    [simplifycfg] 合并基本块（改动 CFG）")
    return instrs, False                    # 动了 CFG

# 插件把这三个名字接进 PassBuilder
register("count-inst", count_inst)
register("dce", dce)
register("simplifycfg", simplifycfg)

# ---- opt 侧：解析 -passes=... 字符串，用回调把名字变成 Pass ----
def parse_pipeline(passes_str):
    pipeline = []
    for name in passes_str.split(","):
        name = name.strip()
        fn = None
        for cb in registry:                 # 挨个问每个回调："这名字你认领吗？"
            fn = cb(name)
            if fn:
                break
        if not fn:
            raise ValueError(f"unknown pass '{name}'（没有回调认领这个名字）")
        pipeline.append((name, fn))
    return pipeline

# ---- 运行：opt -passes='dce,count-inst,simplifycfg,count-inst' ----
instrs = [
    {"name": "%a"}, {"name": "%b"},
    {"name": "%dead", "dead": True},        # 死指令
    {"name": "%r"},
]
pipeline = parse_pipeline("dce,count-inst,simplifycfg,count-inst")
print(f"解析出管线：{[n for n, _ in pipeline]}\n")

preserved_cfg = True
for name, fn in pipeline:
    print(f"运行 {name}：")
    instrs, keeps_cfg = fn(instrs)
    if not keeps_cfg and preserved_cfg:
        print("      → CFG 被改动，依赖 CFG 的分析缓存作废")
    preserved_cfg = preserved_cfg and keeps_cfg

运行结果：

解析出管线：['dce', 'count-inst', 'simplifycfg', 'count-inst']

运行 dce：
    [dce] 删除 1 条死指令
运行 count-inst：
    [count-inst] 当前 3 条指令
运行 simplifycfg：
    [simplifycfg] 合并基本块（改动 CFG）
      → CFG 被改动，依赖 CFG 的分析缓存作废
运行 count-inst：
    [count-inst] 当前 3 条指令

这个模型把 opt 的两段机制演清楚了：插件端用 register 把名字和 Pass 绑定（对应 registerPipelineParsingCallback）；opt 端的 parse_pipeline 把 -passes 字符串切开，挨个问回调「这名字你认领吗」，拼出一条 Pass 序列再顺序执行。dce 删掉死指令后 count-inst 看到的就只剩 3 条——和你在真实命令行里看到的完全一致。请动手改：在 -passes 字符串里加一个没注册的名字（如 "dce,foobar"），你会看到 parse_pipeline 抛出 unknown pass 'foobar'——这正是 opt 在名字没有任何回调认领时的真实报错。

小结与下一步

你完成了从「读框架」到「用框架」的跨越：

  • 一个 Function Pass = 一个有 PreservedAnalyses run(Function&, FunctionAnalysisManager&) 的类 + PassInfoMixin CRTP 混入。只读就返回 all()，改了 IR 就返回精确的收据（删指令用 preserveSet<CFGAnalyses>()）。
  • 做成插件：导出约定函数 llvmGetPassPluginInfo（返回 PassPluginLibraryInfo，带 LLVM_PLUGIN_API_VERSION），在里面用 registerPipelineParsingCallback 把文本名字翻译成「往管线加哪个 Pass」。
  • 用 opt -load-pass-plugin=./X.so -passes='dce,my-pass' in.ll -S 加载并运行；-passes 里逗号分隔的名字从左到右依次跑，正是第 4 章那个顺序循环。
  • 改 IR 时注意两段式（先收集后删除）避免迭代器失效；用 .ll + RUN:/CHECK: 的 LIT 测试 + FileCheck 给行为上保险。

下一章我们从「自己写一个 Pass」放大到「LLVM 是怎么把上百个 Pass 组织成 -O2 那条流水线的」：PassBuilder（llvm/lib/Passes/PassBuilderPipelines.cpp）如何用第 4 章的适配器把 Module/CGSCC/Function/Loop 层层嵌套，拼出 O0–O3 的默认管线。你这一章手动写在 -passes= 里的东西，PassBuilder 是用 C++ 代码批量搭出来的。

这一章的要点

  • Function Pass 最小实现：struct P : PassInfoMixin<P> { PreservedAnalyses run(Function&, FunctionAnalysisManager&); }；遍历用 for (BasicBlock &BB : F) for (Instruction &I : BB)。
  • 插件入口是硬约定：extern "C" ... llvmGetPassPluginInfo() 返回 {LLVM_PLUGIN_API_VERSION, name, version, registerCB}；在 registerCB 里调 PB.registerPipelineParsingCallback(...) 匹配 -passes 名字、addPass 并返回 true。
  • 运行：clang++ -shared -fPIC X.cpp $(llvm-config --cxxflags) -o X.so 编库；opt -load-pass-plugin=./X.so -passes=name in.ll -S 加载运行；处理该选项的是 llvm/tools/opt/。
  • 改 IR 的 Pass：用 I.use_empty()（查 def-use）+ isInstructionTriviallyDead 判死、eraseFromParent() 删除；两段式（先收集后删）避免迭代器失效；收据据实写（删指令 → preserveSet<CFGAnalyses>()）。
  • 验证用 LIT：.ll 顶部 ; RUN: opt ... | FileCheck %s，配 CHECK/CHECK-NOT/CHECK-LABEL 断言输出。
  • 树内登记 Pass 走 llvm/lib/Passes/PassRegistry.def 并重编；外部实验用插件，免重编。
