第 15 章 属性与 attributes：[[sycl::…]] 如何落到 IR

这一章想让你带走什么

你在 SYCL kernel 上写过 [[sycl::reqd_work_group_size(16, 16)]] 或 [[intel::reqd_sub_group_size(32)]] 吗？这些方括号里的标注，是你向编译器和硬件传达"这个 kernel 必须用这样的执行配置"的方式。这一章追踪它们的完整旅程：从源码里的 [[sycl::…]] 标注，经前端属性系统的定义（Attr.td）与语义检查（SemaSYCLDeclAttr.cpp），最后在 CodeGen 里落成 kernel 函数上的 LLVM metadata（!reqd_work_group_size 等）。

读完你会看清一条贯穿前端的通路：属性 = 声明式描述 → TableGen 生成解析代码 → Sema 检查合法性并挂到 AST → CodeGen 翻成 IR metadata → 后端/驱动读 metadata 定制代码生成。这既是理解 SYCL 性能调优标注如何真正生效的钥匙，也是第 22 章"需要编译器配合的扩展"的直接预演——很多扩展做的第一件事，就是加一个新属性。

属性是什么：一条从源码到硬件的旁路信息

先厘清 attribute 在编译里扮演的角色。普通代码告诉编译器"算什么"；属性告诉编译器"怎么算"——一些无法从代码逻辑本身推导、但影响代码生成或运行配置的元信息。SYCL/DPC++ 里最常见的几个：

属性 含义 影响
[[sycl::reqd_work_group_size(x,y,z)]] kernel 必须用这个 work-group 大小 后端据此定制寄存器分配/局部内存布局
[[sycl::work_group_size_hint(x,y,z)]] 建议的 work-group 大小（非强制） 优化提示
[[intel::reqd_sub_group_size(n)]] kernel 必须用这个 sub-group 大小 影响 SIMD 宽度选择
[[intel::kernel_args_restrict]] kernel 参数指针互不别名（restrict） 允许更激进的优化
[[sycl::device_has(aspect…)]] kernel 依赖某些设备特性（aspect） 编译期/运行期能力检查

关键在于：这些信息必须一路穿过整个前端，最终以某种后端能读的形式附到 kernel 上。SYCL 走的形式是 LLVM IR 上的 metadata——一种挂在函数/指令上的、不影响语义但携带额外信息的注解（第 2、3 章你见过 IR 的结构，metadata 就是挂在上面的"便签"）。

第一站：Attr.td —— 用 TableGen 声明一个属性

和第 7 章的后端描述、第 10 章的 liboffload API 一样，clang 的属性也是声明式描述 + TableGen 生成代码。所有属性定义在 clang/include/clang/Basic/Attr.td。每个属性是一条记录，声明它的拼写（spelling）、能作用在什么实体上（函数？变量？）、带几个参数。本章相关的记录：

属性 Attr.td 记录 行号
reqd_work_group_size SYCLReqdWorkGroupSize 3927
work_group_size_hint SYCLWorkGroupSizeHint 3974
reqd_sub_group_size IntelReqdSubGroupSize 2373
kernel_args_restrict SYCLIntelKernelArgsRestrict 1966
device_has SYCLDeviceHas 1839
（内部）sycl_kernel SYCLKernel 1738

TableGen 在构建时把每条记录生成成一堆 C++：属性类（存参数）、解析器钩子、诊断信息等。所以你在源码树里搜 class SYCLReqdWorkGroupSizeAttr 找不到手写定义——它是从这条 .td 记录生成的。加一个新属性的第一步，永远是往 Attr.td 加一条记录（第 22 章会真的做一次）。

拼写前缀 sycl:: vs intel:: 只是命名空间 你会注意到有的属性是 [[sycl::…]]、有的是 [[intel::…]]。这只是属性拼写所在的命名空间不同——sycl:: 是 SYCL 标准/oneAPI 通用的，intel:: 是 Intel 扩展的。在 Attr.td 里，一个属性记录可以声明多个拼写（比如同时接受 [[intel::reqd_sub_group_size]] 和别的形式）。记录名（如 IntelReqdSubGroupSize）是编译器内部用的，和你写的拼写不必字面一致。
FPGA 属性已从这条分支移除——别照旧资料抄 不少 SYCL 老资料会列 [[intel::num_simd_work_items]]、[[intel::max_work_group_size]] 之类的 FPGA 专属属性。在本书对应的 intel/llvm sycl 分支上，随着 FPGA 支持的下线，这些属性已经被删除——你在当前 Attr.td 里搜不到它们。本章只讲这条分支上确实存在的属性（上表那几个）。这也是"源码为锚"的意义：以你手上这棵树的实际代码为准，而不是过时文档。

第二站：SemaSYCLDeclAttr.cpp —— 语义检查与挂到 AST

TableGen 生成了属性的"骨架"，但属性用得对不对要靠语义分析（Sema）来判断，这些手写的检查逻辑集中在 clang/lib/Sema/SemaSYCLDeclAttr.cpp。它做的事：

  • 参数合法性：reqd_work_group_size 的三个维度必须是正整数常量；reqd_sub_group_size 的值得是设备支持的 SIMD 宽度之一。
  • 实体合法性：这个属性只能贴在 kernel/函数上，贴到别处要报错。
  • 冲突检测：同一个 kernel 上 reqd_work_group_size 和 work_group_size_hint 若矛盾要诊断；重复标注要合并或报错。
  • 挂到 AST：检查通过后，构造出对应的 *Attr 对象、挂到该函数声明（FunctionDecl）的属性列表上。

这一步之后，属性就成了 AST 的一部分——后续的 CodeGen 遍历到这个函数时，能从它的 AST 节点上读到这些属性。这就是前端"检查 + 记录"的经典分工：Sema 负责判断合法并挂到 AST，CodeGen 负责把 AST 翻成 IR。

第三站：CodeGen —— 属性落成 IR metadata

真正把属性"变成 IR"的是 CodeGen。当 CodeGen 为一个 kernel 函数生成 LLVM IR 时，会检查它 AST 上挂的属性，翻成对应的 metadata 挂到 IR 函数上。核心入口是 clang/lib/CodeGen/CodeGenFunction.cpp:649 的 EmitKernelMetadata。对应关系：

属性 生成的 IR metadata 来源
reqd_work_group_size !reqd_work_group_size EmitKernelMetadata
work_group_size_hint !work_group_size_hint EmitKernelMetadata
reqd_sub_group_size !intel_reqd_sub_group_size EmitKernelMetadata
（ESIMD kernel） !sycl_explicit_simd EmitKernelMetadata
device_has 相关 !sycl_declared_aspects / !sycl_used_aspects CodeGenModule.cpp 的 applySYCLAspectsMD

比如 [[sycl::reqd_work_group_size(16, 16)]] 会让 kernel 函数带上：

LLVM IR
define spir_kernel void @_ZTS8MyKernel(...) !reqd_work_group_size !5 {
  ; …
}
!5 = !{i32 16, i32 16, i32 1}   ; 三个维度

后端（或 llvm-spirv 翻译成 SPIR-V 的执行模式 LocalSize）读这条 metadata，就知道要为这个 work-group 大小定制代码。

两个特别的属性

两个属性的落地方式和上面略有不同，值得单独看：

kernel_args_restrict → noalias：它不是变成一条 kernel metadata，而是在生成 kernel 函数参数时，给那些指针参数加上 LLVM 的 noalias 属性（逻辑在 clang/lib/CodeGen/CGCall.cpp:3720）。noalias 是 LLVM 优化器认得的标准指针属性，告诉优化器"这些指针不会互相别名"，从而允许更激进的重排和向量化。这是一个漂亮的例子：SYCL 层的 restrict 语义，直接映射到 LLVM 已有的优化基础设施，不用发明新东西。

device_has → aspect metadata：device_has 声明 kernel 用到哪些设备 aspect（如 fp64、atomic64）。它经 CodeGenModule.cpp 的 applySYCLAspectsMD 落成 !sycl_declared_aspects（显式声明的）和 !sycl_used_aspects（编译器推断出用到的）。运行时/编译期据此做能力检查：把 kernel 派到不支持该 aspect 的设备上就能提前报错，而不是运行时崩溃。

属性的四段旅程：定义 → 检查 → 落 IR → 后端读 ① Attr.td 用 TableGen 声明属性（拼写、作用实体、参数），生成解析骨架；② SemaSYCLDeclAttr.cpp 检查合法性并把 *Attr 挂到 FunctionDecl；③ CodeGen 的 EmitKernelMetadata（CodeGenFunction.cpp:649）把属性翻成 IR metadata（!reqd_work_group_size 等）；④ 后端 / llvm-spirv 读 metadata 定制代码生成。特例：kernel_args_restrict → 参数上的 noalias（CGCall.cpp:3720）；device_has → !sycl_used_aspects（applySYCLAspectsMD）做能力检查。

spir_kernel：kernel 函数的调用约定

顺带认识一个反复出现的标记：上面 IR 里那个 spir_kernel。SYCL 的 kernel 入口函数用一个特殊的调用约定（calling convention） SPIR_KERNEL（设置逻辑在 clang/lib/CodeGen/Targets/SPIR.cpp:594）。它区别于普通函数的 spir_func：spir_kernel 标记的是"可以从主机端 enqueue 的设备入口点"，llvm-spirv 会把它翻成 SPIR-V 里的 OpEntryPoint。这呼应了第 14 章——ConstructOpenCLKernel 生成的那个 kernel 函数，正是带 spir_kernel 约定、并挂着本章这些 metadata 的函数。调用约定 + metadata，共同定义了"一个 SYCL kernel 在 IR 层长什么样"。

亲手看属性变成 metadata

这条通路完全可观察——让 clang 停在 IR 这一层，直接看属性落成了什么：

终端
# 只编设备端、输出 LLVM IR，观察 kernel 上的 metadata
clang++ -fsycl -fsycl-device-only -S -emit-llvm kernel.cpp -o kernel.ll

# 找 kernel 函数上的 metadata 与参数属性
grep -n 'reqd_work_group_size\|intel_reqd_sub_group_size\|noalias\|sycl_used_aspects\|spir_kernel' kernel.ll

给 kernel 加上 [[sycl::reqd_work_group_size(16,16,1)]] 前后各编一次、对照 kernel.ll，你能亲眼看到多出来的 !reqd_work_group_size !N 和末尾的 !N = !{i32 16, i32 16, i32 1}。这是"属性真的生效了"的最直接证据，也是调优时确认"我的标注有没有被编译器吃进去"的手段。

动手：模拟属性从源码到 IR metadata 的四段旅程

把这一章的核心——「属性经定义→检查→落 IR→后端读四段变成 metadata」——用一个浏览器里能跑的小模型演出来。点 ▶ 运行。

Python 
# 模拟 SYCL 属性的旅程：Attr.td 定义 → Sema 检查 → CodeGen 落 metadata → 后端读

# ---- ① Attr.td：声明属性（拼写、作用实体、参数个数、落成的 metadata 名）----
ATTR_DEFS = {
    "reqd_work_group_size":  {"applies": "kernel", "nargs": 3, "md": "!reqd_work_group_size"},
    "reqd_sub_group_size":   {"applies": "kernel", "nargs": 1, "md": "!intel_reqd_sub_group_size"},
    "kernel_args_restrict":  {"applies": "kernel", "nargs": 0, "md": "(参数 noalias)"},
}

# ---- ② Sema：检查合法性，通过则"挂到 AST" ----
def sema_check(attr, args):
    d = ATTR_DEFS.get(attr)
    if not d:
        return None, f"未知属性 [[sycl::{attr}]]"
    if len(args) != d["nargs"]:
        return None, f"[[sycl::{attr}]] 需要 {d['nargs']} 个参数，给了 {len(args)}"
    if any((not isinstance(a, int)) or a <= 0 for a in args):
        return None, f"[[sycl::{attr}]] 参数必须是正整数常量"
    return {"attr": attr, "args": args}, None      # 合法：挂到 FunctionDecl

# ---- ③ CodeGen：把挂在 AST 上的属性翻成 IR metadata ----
def codegen(kernel_name, ast_attrs):
    print(f"define spir_kernel void @{kernel_name}(...) ", end="")
    md_lines = []
    for i, a in enumerate(ast_attrs):
        d = ATTR_DEFS[a["attr"]]
        if a["attr"] == "kernel_args_restrict":
            print("  ; 参数指针加 noalias", end="")
        else:
            vals = a["args"] + [1] * (3 - len(a["args"])) if a["attr"] == "reqd_work_group_size" else a["args"]
            print(f"{d['md']} !{i} ", end="")
            md_lines.append(f"!{i} = !{{{', '.join('i32 '+str(v) for v in vals)}}}")
    print("{ ... }")
    for m in md_lines:
        print("  " + m)

# ---- ④ 后端 / llvm-spirv：读 metadata 定制代码生成 ----
def backend_read(ast_attrs):
    for a in ast_attrs:
        if a["attr"] == "reqd_work_group_size":
            print(f"  [后端] 读到 !reqd_work_group_size → 锁定 work-group = {a['args']}")
        elif a["attr"] == "reqd_sub_group_size":
            print(f"  [后端] 读到 !intel_reqd_sub_group_size → SIMD 宽度 = {a['args'][0]}")
        elif a["attr"] == "kernel_args_restrict":
            print(f"  [优化器] 参数 noalias → 允许更激进的向量化/重排")

# 源码里写的属性标注（属性名, 参数）
source_attrs = [("reqd_work_group_size", [16, 16]), ("kernel_args_restrict", [])]

ast = []
print("=== ② Sema 检查 ===")
for name, args in source_attrs:
    node, err = sema_check(name, args)
    if err:
        print(f"  ✗ {err}")
    else:
        print(f"  ✓ [[sycl::{name}({', '.join(map(str,args))})]] 合法 → 挂到 AST")
        ast.append(node)
print("\n=== ③ CodeGen 落成 IR ===")
codegen("_ZTS8MyKernel", ast)
print("\n=== ④ 后端读 metadata ===")
backend_read(ast)

运行结果：

=== ② Sema 检查 ===
  ✓ [[sycl::reqd_work_group_size(16, 16)]] 合法 → 挂到 AST
  ✓ [[sycl::kernel_args_restrict()]] 合法 → 挂到 AST

=== ③ CodeGen 落成 IR ===
define spir_kernel void @_ZTS8MyKernel(...) !reqd_work_group_size !0   ; 参数指针加 noalias { ... }
  !0 = !{i32 16, i32 16, i32 1}

=== ④ 后端读 metadata ===
  [后端] 读到 !reqd_work_group_size → 锁定 work-group = [16, 16]
  [优化器] 参数 noalias → 允许更激进的向量化/重排

看清这条流水线：源码里的 [[sycl::reqd_work_group_size(16,16)]] 先过 Sema 检查（参数个数、是否正整数）→ 合法后挂到 AST → CodeGen 把它翻成 !reqd_work_group_size !0 和末尾的 !0 = !{i32 16, i32 16, i32 1}（注意第三维被自动补成 1）→ 后端读这条 metadata 锁定 work-group 大小。而 kernel_args_restrict 走的是另一条路——变成参数上的 noalias。请动手改：把 source_attrs 里的 reqd_work_group_size 参数改成 [16, 0]（含非正数），重新运行——你会看到 Sema 在第二段就拒绝它、根本走不到 CodeGen。这正是"属性合法性由 Sema 把关"的意义。

小结与下一步

这一章把 SYCL 属性的完整通路走通了：

  • 属性是从源码到硬件的旁路元信息——告诉编译器"怎么算"（执行配置、别名、能力依赖），最终以 IR metadata 的形式附到 kernel 上。
  • 四段旅程：① Attr.td（clang/include/clang/Basic/Attr.td）用 TableGen 声明属性，生成解析骨架；② SemaSYCLDeclAttr.cpp 检查合法性并把 *Attr 挂到 FunctionDecl；③ CodeGen 的 EmitKernelMetadata（CodeGenFunction.cpp:649）落成 metadata；④ 后端 / llvm-spirv 读 metadata。
  • 对应表：reqd_work_group_size→!reqd_work_group_size、reqd_sub_group_size→!intel_reqd_sub_group_size、ESIMD→!sycl_explicit_simd。特例：kernel_args_restrict→参数 noalias（CGCall.cpp:3720，复用 LLVM 优化基础设施）；device_has→!sycl_used_aspects/!sycl_declared_aspects（applySYCLAspectsMD）做能力检查。
  • kernel 函数用 spir_kernel 调用约定（SPIR.cpp:594）标记设备入口点，llvm-spirv 翻成 OpEntryPoint；调用约定 + metadata 共同定义"IR 层的一个 SYCL kernel"。
  • 注意：FPGA 专属属性（num_simd_work_items 等）已从这条分支移除，别照旧资料抄。

第三部分的"设备端编译"两章（14、15）到此完成——你已经看清一份 .cpp 的设备侧如何被前端识别、命名、标注并落成带 metadata 的 IR。下一章转向运行时：queue/context/device/handler 这些第 13 章见过的 pImpl 句柄，在运行时到底是怎样的对象模型、q.submit(...) 一路怎么走到向设备发起启动。第三部分的重心从"编译期"转到"运行期"。
