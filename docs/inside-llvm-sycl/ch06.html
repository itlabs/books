第 6 章 优化管线：PassBuilder 与 O0–O3 默认流水线

这一章想让你带走什么

第 4 章讲了 Pass 框架的设计、第 5 章你亲手写了一个 Pass 挂进 opt。但 -O2 背后不是一个 Pass，而是上百个 Pass 按精心设计的顺序、层层嵌套组成的一条流水线。这一章回答：这条流水线是谁、按什么逻辑搭出来的？答案是 PassBuilder（llvm/lib/Passes/PassBuilder.cpp 与 PassBuilderPipelines.cpp）。

你会看清三件事：-O0 到 -O3（还有 -Os/-Oz）到底差在哪；PassBuilder 如何用第 4 章的适配器把 Module/CGSCC/Function/Loop 四层嵌套成一棵管线树；以及为什么 Pass 的顺序如此关键（简化为别的 Pass 创造机会、清理收拾别的 Pass 留下的烂摊子）。这一章让你能读懂 opt -passes='default<O2>' -print-pipeline-passes 打印出来的那一长串，也理解 SYCL/设备端编译为什么要在标准管线里插自己的 Pass。

从 -O2 说起：一个开关，一整条流水线

你敲 clang -O2 时，到底发生了什么？-O2 不是"某一个优化"，而是一个优化级别（optimization level）——它选定一整套 Pass 和它们的排列。级别有六档：

级别 取向 大致特征
O0 不优化 几乎不跑优化 Pass，编译最快、调试最友好
O1 轻度 便宜、收益高的优化
O2 标准 生产默认档，激进但稳妥的全套优化
O3 激进 在 O2 基础上更aggressive（更多内联/向量化，可能增大体积）
Os 优化体积 像 O2 但避免增大代码尺寸的变换
Oz 极限压体积 比 Os 更进一步，牺牲速度换最小体积

选定级别后，真正把「这个级别该跑哪些 Pass、什么顺序」翻译成一条具体管线的，是 PassBuilder。clang、opt 都通过它来构建管线——所以 clang -O2 和 opt -passes='default<O2>' 得到的是同一条流水线。

O0 也不是「什么都不做」 即使 -O0，管线里也有少量必须跑的 Pass——比如 always_inline（处理 __attribute__((always_inline))）、以及把 alloca 相关内建、协程等降级的强制变换。这正是第 4 章 RequiredPassInfoMixin（不可跳过）与 OptionalPassInfoMixin（O0/调试时可跳过）区分的用武之地：O0 下 Optional 的 Pass 被跳过，Required 的照跑。所以 O0 IR 依然是「能正确运行」的，只是没被优化。

PassBuilder：把级别翻译成管线树

PassBuilder 的核心职责是提供一族 build...Pipeline 方法，输入一个优化级别，输出一个装好 Pass 的 ModulePassManager。最常用的入口是：

C++
// PassBuilder 构建一条完整的 module 级优化管线（简化签名）
ModulePassManager
PassBuilder::buildPerModuleDefaultPipeline(OptimizationLevel Level);

它内部（实现在 llvm/lib/Passes/PassBuilderPipelines.cpp）把管线拆成几个阶段，一层层调更小的构建函数：

  • buildModuleSimplificationPipeline —— 简化阶段：内联、把代码整理成利于后续优化的形状。
  • buildModuleOptimizationPipeline —— 优化阶段：向量化、循环优化等更重的变换。
  • 其下又有 buildFunctionSimplificationPipeline（函数级简化：InstCombine、GVN、SimplifyCFG…）、循环管线等。

而把这些不同 IR 层级 的管线拼到一起，靠的正是第 4 章那套适配器——因为「PassManager / Adaptor 本身也是 Pass」，四层可以像积木一样嵌套：

C++
// PassBuilderPipelines.cpp 里的典型嵌套（示意）
ModulePassManager MPM;
// Module 层：先跑一些全局 Pass……
// 然后用适配器下潜到 CGSCC 层做内联，CGSCC 里再下潜到 Function 层
MPM.addPass(createModuleToPostOrderCGSCCPassAdaptor(   // Module → CGSCC
    createCGSCCToFunctionPassAdaptor(                  // CGSCC → Function
        std::move(FunctionSimplificationPM))));        // 一整条 Function 简化管线
// Function 管线内部又可能用 createFunctionToLoopPassAdaptor 下潜到 Loop 层

于是整条 -O2 管线在结构上是一棵嵌套树，形如 module(..., cgscc(inline, function(instcombine, ..., loop(licm, ...))), ...)。每一层用它自己的 PassManager 装本层的 Pass，用适配器接到下一层。第 4 章那句「你写 Pass 只管作用在哪一层，跨层驱动交给适配器」，在 PassBuilder 这里被用到极致。

图 6-1　-O2 管线的嵌套结构。外层 Module 管线用适配器下潜到 CGSCC 层（做内联），CGSCC 再下潜到 Function 层（InstCombine/GVN 等），Function 再下潜到 Loop 层（LICM/展开）。每层管理自己的 Pass，适配器负责「对上层的每个单位，把下层管线跑一遍」——这就是 PassBuilder 搭出来的东西

顺序为什么是命根子

一条优化管线里，Pass 的顺序和选择本身就是最难的工程。同样一批 Pass，排列不同，效果可能天差地别。背后有两条反复出现的模式：

① 简化为别的 Pass「创造机会」。 很多强力优化只有在 IR 被整理成特定形状后才能触发。例子：mem2reg（把栈上的 alloca 提升成 SSA 寄存器值，插入必要的 phi）几乎是所有后续优化的前提——只有变量成了 SSA 值，常量传播、GVN 才认得出「这两处是同一个值」。所以简化管线里它排得很靠前。类似地，内联把被调函数的函数体展开到调用处，为调用处的常量折叠、死代码消除创造了大量新机会——这就是为什么内联之后总要再跑一轮 InstCombine/SimplifyCFG。

② 清理别的 Pass「留下的烂摊子」。 变换 Pass 常常在简化的同时制造出新的冗余：内联后会留下一批可以合并的指令、可以删的死代码；循环展开后会有重复的计算。所以像 InstCombine（指令组合）、SimplifyCFG（控制流图简化）、DCE 这类"清理工"会在管线里反复出现多次，穿插在重型变换之间收拾残局。你打印 -O2 管线会看到 instcombine 出现好几次，不是 bug，是设计。

这两条结合，就是为什么 PassBuilder 的管线是「简化 → 重型变换 → 清理 → 再简化 → 再变换……」的多轮交替，而不是每个 Pass 各跑一次了事。管线的具体顺序是 LLVM 社区多年调优、用大量 benchmark 喂出来的经验结晶——PassBuilderPipelines.cpp 里那些看似随意的排列，几乎每一处都有历史原因。

别自己拍脑袋重排管线 既然顺序这么关键，一个常见的坑是：新手为了"多优化一点"往 -passes 里手动堆一堆 Pass，或调换顺序，结果反而更慢、甚至触发本不会出现的 bug（比如把某个依赖 SSA 形态的 Pass 放到了 mem2reg 之前）。默认管线是经过海量测试的；要加自定义 Pass，正确姿势是用扩展点插入，而不是重写整条管线。见下一节。

扩展点：往标准管线里「插」自己的 Pass

真实需求里，你常常想在标准 O2 管线的某个位置插入自己的 Pass，而不破坏其余部分——这正是 SYCL、各种 sanitizer、插桩工具要做的事。PassBuilder 为此提供**扩展点（extension point）**注册接口：

C++
// 在管线的特定位置注册回调，插入自己的 Pass
PB.registerPipelineStartEPCallback(          // 管线最开始
    [](ModulePassManager &MPM, OptimizationLevel Level) {
      MPM.addPass(MyEarlyPass());
    });
PB.registerOptimizerLastEPCallback(          // 优化管线最末尾
    [](ModulePassManager &MPM, OptimizationLevel Level) {
      MPM.addPass(MyCleanupPass());
    });

常见扩展点有 PipelineStartEP（最前）、PeepholeEP（每轮 peephole 清理处）、ScalarOptimizerLateEP、OptimizerLastEP（最后）等。回调还能拿到当前 OptimizationLevel，让你按级别决定插不插、插什么。第 5 章那个插件的 registerCB 里，除了用 registerPipelineParsingCallback 认领 -passes 名字，也可以用这些 EP 回调把 Pass 自动插进 default<O2>——这就是 sanitizer 插件"开了 -O2 也能自动生效"的机制。

PassBuilder = 优化级别 → 管线树的编译器 把这一章浓缩成一句：PassBuilder 是一台"元编译器"——输入优化级别（O0–O3/Os/Oz），用第 4 章的适配器把上百个 Pass 编译成一棵 Module/CGSCC/Function/Loop 嵌套的管线树，顺序经过精心调优（简化创造机会、清理收拾残局，多轮交替）。你既能用 opt -passes='default<O2>' 复用它，也能通过扩展点往里插自己的 Pass 而不动其余——SYCL 的设备端 Pass 就是这么进管线的。

打印真实管线看一眼

opt 能把某个级别的管线展开成扁平的 Pass 名字串打印出来，这是理解和调试管线最好的工具：

终端
# 把 default<O2> 这条管线里所有 Pass 按顺序、按嵌套打印出来
opt -passes='default<O2>' -print-pipeline-passes in.ll -disable-output

输出是一长串（截取示意），你能一眼认出这一章讲的嵌套和交替：

annotation2metadata,forceattrs,inferattrs,...,
cgscc(devirt<4>(inline,function(sroa,early-cse,...,instcombine,...))),
...,function(...,loop(licm,...),instcombine,...),...

看这几个特征：cgscc(...) 里套着 inline 和 function(...)——正是图 6-1 的嵌套；function(...) 里套着 loop(...)；instcombine、sroa、simplifycfg 在不同位置反复出现——正是"清理工穿插在变换之间"。这一串就是 PassBuilder::buildPerModuleDefaultPipeline(O2) 在内存里搭出来那棵树的文本投影。你甚至可以把这串复制出来、手动改几个 Pass、再用 -passes='<改过的串>' 喂回 opt，做管线实验。

动手：模拟「简化—变换—清理」的多轮管线

用一个浏览器里能跑的小模型演示管线为什么要多轮交替：一段代码里，内联会暴露出新的死代码和可折叠常量，需要后续的清理 Pass 才能收割。我们跑两种管线——「只跑一遍每个 Pass」vs「变换后接清理、多轮」——对比最终剩下多少条指令。点 ▶ 运行。

Python 
# 模拟优化管线：为什么要"简化—重型变换—清理"多轮交替

# 一段代码，用指令条数 + 隐藏的"可优化机会"表示
# opportunities 表示"内联后才会暴露出来、需要清理才能删掉"的冗余指令数
def make_code():
    return {"instrs": 20, "inlined": False, "hidden_dead": 0}

def inline(code):
    # 内联：把被调函数展开进来，指令变多，但也"暴露"出一批新的死代码机会
    if not code["inlined"]:
        code["instrs"] += 12          # 展开带来更多指令
        code["hidden_dead"] += 8      # 其中 8 条内联后其实是死的（暴露出机会）
        code["inlined"] = True
        print(f"    [inline] 展开被调函数：+12 指令，暴露 8 条潜在死代码")
    return code

def instcombine_dce(code):
    # 清理工：把已暴露的死代码/可折叠指令收割掉
    if code["hidden_dead"] > 0:
        print(f"    [instcombine+dce] 收割 {code['hidden_dead']} 条已暴露的冗余")
        code["instrs"] -= code["hidden_dead"]
        code["hidden_dead"] = 0
    else:
        print(f"    [instcombine+dce] 没有可收割的冗余")
    return code

print("=== 管线 A：只内联，不清理（顺序没安排好）===")
a = make_code()
inline(a)
print(f"最终指令数：{a['instrs']}（8 条死代码没人收割，白白留着）\n")

print("=== 管线 B：内联后接清理（PassBuilder 的做法）===")
b = make_code()
pipeline = [("inline", inline), ("instcombine+dce", instcombine_dce)]
for name, fn in pipeline:
    print(f"  运行 {name}:")
    fn(b)
print(f"最终指令数：{b['instrs']}（内联创造的机会被清理工收割了）")

运行结果：

=== 管线 A：只内联，不清理（顺序没安排好）===
    [inline] 展开被调函数：+12 指令，暴露 8 条潜在死代码
最终指令数：32（8 条死代码没人收割，白白留着）

=== 管线 B：内联后接清理（PassBuilder 的做法）===
  运行 inline:
    [inline] 展开被调函数：+12 指令，暴露 8 条潜在死代码
  运行 instcombine+dce:
    [instcombine+dce] 收割 8 条已暴露的冗余
最终指令数：24（内联创造的机会被清理工收割了）

这就是「顺序是命根子」的可运行版：内联本身先让代码变大（32 条），它的价值一部分要靠后续的清理 Pass 才能兑现（收割到 24 条）。如果管线里内联后不接 InstCombine/DCE，那 8 条被暴露出来的死代码就白白留着——这正是 PassBuilder 要把清理工穿插在重型变换之后、并多轮交替的原因。请动手改：给管线 B 再加一个 ("inline", inline) 放在最后（模拟对新函数再内联一轮），你会看到它因为 inlined 已为真而不再展开——真实管线里对"是否还有新机会"的判断，正是决定要不要再来一轮的依据。

小结与下一步

你现在理解了 -O2 背后那条流水线是怎么来的：

  • 优化级别（O0–O3、Os、Oz）是一个开关，选定一整套 Pass 及其顺序；O0 也仍跑少量 Required Pass。
  • PassBuilder（PassBuilder.cpp / PassBuilderPipelines.cpp）把级别翻译成管线：buildPerModuleDefaultPipeline → 简化管线 + 优化管线 → 函数/循环子管线。
  • 管线是一棵 Module/CGSCC/Function/Loop 嵌套树，靠第 4 章的适配器拼接（createModuleToPostOrderCGSCCPassAdaptor 等）。
  • 顺序是命根子：简化为后续 Pass 创造机会（mem2reg、内联在前），清理工（InstCombine/SimplifyCFG/DCE）反复穿插收拾残局——所以管线是多轮交替，不是各跑一次。
  • 加自定义 Pass 用扩展点（registerPipelineStartEPCallback / OptimizerLastEP 等），别手动重排默认管线。
  • 用 opt -passes='default<O2>' -print-pipeline-passes 打印真实管线来学习和调试。

第 4–6 章合起来，你已经把 LLVM 中端（Pass 框架 + 管线）从设计到动手到组织全部走通。下一章是第一部分的收尾——后端与代码生成：优化好的 IR 如何经过指令选择、寄存器分配、指令调度，变成某款具体芯片（llvm/lib/Target/<芯片>/）的机器码。这一步之后，我们就带着完整的 LLVM 心智模型进入第二部分的异构世界（SPIR-V、offload、PTX）。

这一章的要点

  • 优化级别 O0–O3/Os/Oz 选定一整套 Pass 与顺序；clang -O2 与 opt -passes='default<O2>' 得到同一条管线。O0 仍跑 Required Pass（如 always_inline）。
  • PassBuilder（llvm/lib/Passes/PassBuilder.cpp、PassBuilderPipelines.cpp）用 buildPerModuleDefaultPipeline 等构建管线，分简化（内联、整形）与优化（向量化、循环）阶段。
  • 管线是 Module/CGSCC/Function/Loop 嵌套树，靠适配器拼接；cgscc(inline, function(instcombine, loop(licm))) 是典型形状。
  • 顺序关键：简化创造机会（mem2reg、内联靠前），InstCombine/SimplifyCFG/DCE 等清理工反复出现收拾残局；管线多轮交替。
  • 加 Pass 用扩展点（registerPipelineStartEPCallback、OptimizerLastEP 等），SYCL/sanitizer 靠它进管线；勿手改默认顺序。
  • 调试利器：opt -passes='default<O2>' -print-pipeline-passes 打印展开后的完整管线。
