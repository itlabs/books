第 17 章 运行时（二）：命令组、调度器与依赖图

这一章想让你带走什么

第 16 章那条脊柱停在了 Scheduler::addCG——一个命令组（command group, CG）交给了调度器。这一章接着往下：调度器拿到 CG 之后，怎么把它挂进一张命令依赖图、怎么自动算出命令之间的读写依赖（RAW / WAR / WAW）、在数据不在目标设备时怎么插入内存拷贝、最后怎么按拓扑序把命令 enqueue 到设备。

这是 SYCL 运行时最精巧的一块——也是 SYCL "你只描述数据依赖、运行时自动排出正确执行顺序"这一承诺的兑现处。读完你会看清：一个 buffer 的每次访问如何变成图里的一个节点、accessor 的访问模式（读/写）如何决定要不要连一条依赖边、以及为什么 out-of-order 队列能自动并行无关的 kernel、串行有依赖的 kernel。理解这张图，你就理解了 SYCL 内存模型的运行时内核。

命令组：一次提交打包成的那个对象

先把"命令组"这个词钉死。你在 q.submit 里那个 lambda 做的所有事（选一个 kernel、绑定几个 accessor、给出 NDRange），被第 16 章的 handler::finalize 打包成一个 CG 对象（基类 detail::CG，sycl/source/detail/cg.hpp:55）。CG 有很多子类对应不同操作，最重要的是执行 kernel 的 CGExecKernel（cg.hpp:144），其他还有 CGCopy（拷贝）、CGFill（填充）、CGHostTask（主机任务）、CGBarrier（屏障）等，用一个枚举 CGType（sycl/include/sycl/detail/cg_types.hpp:26：Kernel、CopyUSM、Fill、CodeplayHostTask…）区分。

每个 CG 都携带一样至关重要的东西——requirements（需求）：它要读写哪些内存。这存在 CG 的 MRequirements 里（std::vector<AccessorImplHost *>）。这里有个关键的类型别名（accessor_impl.hpp:184）：

C++
using Requirement = AccessorImplHost;   // 一个"需求"就是一个 accessor 的实现对象

也就是说，每个绑定到命令组的 accessor，就是一条 requirement。它记录了访问哪个内存对象（MSYCLMemObj）、以什么模式（MAccessMode：read / write / read_write…）、访问范围/偏移。accessor 变成 requirement 的地方在 handler::associateWithHandlerCommon（handler.cpp:855）：它把 accessor 的 impl 指针 push 进 CG 的 MRequirements。这些 requirement 就是调度器算依赖的全部输入——SYCL 的整个自动依赖机制，就建立在"每个 CG 声明了它读写哪些内存、怎么读写"之上。

调度器：一张图，两个半区

调度器是个单例（Scheduler::getInstance()，sycl/source/detail/scheduler/scheduler.hpp:369）。它维护一张无环的命令依赖图，图里的节点是各种"命令"（Command），表示四类底层动作：分配内存、释放内存、执行命令组、拷贝内存。

调度器在设计上被切成两个半区，各用不同的锁，这个划分很重要：

  • GraphBuilder（图构建器） —— 修改图：加节点、连依赖边。它要拿写锁。
  • GraphProcessor（图处理器） —— enqueue：把命令推到设备执行。它只拿读锁。

Scheduler::addCG（scheduler.cpp:105）就体现了这个两阶段：① 先拿写锁，调 MGraphBuilder.addCG(...) 把 CG 包成一个 ExecCGCommand 节点、连好依赖边；② 放写锁，再拿读锁调 enqueueCommandForCG（GraphProcessor::enqueueCommand）把新命令及其依赖 enqueue。图构建和图执行分离、用不同锁模式——这让多线程提交时构建互斥、而执行可并发。

两个"图"别搞混 sycl/source/detail/scheduler/ 里这张核心调度依赖图，和 sycl/source/detail/graph/ 那个 ext_oneapi_graph 扩展（用户显式录制的命令图，exec_graph_impl/node_impl）是两回事。本章讲的是前者——每次 submit 都在用、你看不见但一直在跑的那张运行时依赖图。后者是第四部分扩展话题里用户主动构建的图，二者通过 addCommandGraphUpdate 等接口衔接，但概念上要分开。

命令与依赖边：节点怎么连起来

图的节点基类是 Command（commands.hpp:109）。它的 enum CommandType 列出了所有节点种类：RUN_CG（执行命令组）、ALLOCA（分配）、RELEASE（释放）、COPY_MEMORY、MAP_MEM_OBJ / UNMAP_MEM_OBJ、EMPTY_TASK、HOST_TASK 等。执行你 kernel 的那个节点是 ExecCGCommand（commands.hpp:628），它裹着第 16 章打包的那个 CG。

节点之间的边是双向记录的：

  • std::vector<DepDesc> MDeps（commands.hpp:309）—— 出边：我依赖谁。
  • std::unordered_set<Command *> MUsers（commands.hpp:311）—— 反向边：谁依赖我。

边的结构 DepDesc（commands.hpp:83）记三样：MDepCommand（被依赖的命令）、MDepRequirement（是哪条 requirement 导致了这条依赖）、MAllocaCmd（对应内存对象的分配命令）。连边靠 Command::addDep（commands.cpp:786）：它把 DepDesc 推进 MDeps，同时调被依赖命令的 addUser(this)——一次调用建好双向边。

每个节点还绑着一个 event：Command 构造时创建 MEvent 并 MEvent->setCommand(this)（commands.cpp:534），event 反过来存 Command *MCommand。这就是为什么用户拿到的 sycl::event 能映射回图里的节点——第 16 章 addCG 返回的那个 event，正是这个 ExecCGCommand 的 event。

依赖是怎么算出来的：RAW / WAR / WAW

这是全章的核心。每个内存对象（buffer/image）在调度器里有一条 MemObjRecord（scheduler.hpp:200），记录着"这个内存对象当前的最新状态"——特别是两组叶子（leaves）：

  • MWriteLeaves —— 最近的写命令。
  • MReadLeaves —— 最近的只读命令。

当一个新 CG 带着对某内存对象的 requirement 进来，GraphBuilder::findDepsForReq（graph_builder.cpp:575）算它该依赖谁，规则就是经典的三种数据冒险：

  • 新 requirement 是只读（read）：只需依赖 MWriteLeaves（前面的写者）——这是 RAW（read-after-write，读后写）。读之间不互相依赖（两个读可以并行）。
  • 新 requirement 是写（write/read_write/…）：既要依赖 MWriteLeaves（WAW，write-after-write），也要依赖 MReadLeaves（WAR，write-after-read，得等前面的读完成才能覆盖）。

代码里这个"只读只看写叶子、写要看读写两组叶子"的逻辑一目了然（graph_builder.cpp:581-587）。另外有个旁路优化（CanBypassDep，graph_builder.cpp:599）：若两条 requirement 都是只读、或它们的访问范围不重叠（doOverlap 比较偏移和大小，graph_builder.cpp:43），就不必连边——两个写不同区域的 kernel 依然能并行。

算完依赖，新命令通过 addDep 连边，然后 addNodeToLeaves 把自己装成这个内存对象的新叶子（写命令进 MWriteLeaves，读命令进 MReadLeaves），供后续的 CG 继续算依赖。整张图就这样一次提交一次提交地长出来。

SYCL 自动依赖 = 每个内存对象维护读/写叶子，按 RAW/WAR/WAW 连边 每个 buffer 有一条 MemObjRecord（scheduler.hpp:200），维护 MWriteLeaves/MReadLeaves。新 CG 的每条 requirement 经 findDepsForReq（graph_builder.cpp:575）算依赖：只读 → 依赖写叶子（RAW）；写 → 依赖写叶子（WAW）+ 读叶子（WAR）。两个只读、或访问范围不重叠（doOverlap）则旁路不连边。你只在 accessor 上声明读/写，运行时据此自动排出正确顺序——无关的并行、有依赖的串行。这就是 SYCL 内存模型的运行时内核。
图 17-1　依赖图按 accessor 读/写模式自动连边。CG1 写 B → CG2 读 B 连 RAW 边；CG2 读 → CG3 写连 WAR 边；CG3 写还依赖 CG1 这个写叶子（WAW）。每个 buffer 的 MemObjRecord 维护读/写叶子，新命令算完依赖后把自己设为新叶子。只读之间、访问不重叠则旁路不连边

数据搬运：不在目标设备就自动拷

依赖图不只排顺序，还管数据在哪。MemObjRecord 有个 MCurContext——记录这个内存对象的最新数据当前在哪个 context（设备）。当新 CG 要在另一个 context 上访问它，GraphBuilder::addCG（graph_builder.cpp:922）就得插入内存搬运命令：

  • 若目标 context 和 MCurContext 一致 → 无需拷贝（顶多做个 remap）。
  • 若不一致 → 调 insertMemoryMove（graph_builder.cpp:329）。因为很多后端不能设备直连设备拷贝，运行时可能先拷回主机、再拷到目标设备（两次 MemCpyCommand）。若 src/dst 分配是"链接"的，则用 MapMemObject/UnMapMemObject 一对映射命令。搬完更新 Record->setCurContext(目标)。

一个小而美的优化：若访问模式是 discard_write/discard_read_write（丢弃原有内容），就跳过拷贝，直接切 context——反正旧数据不要了。这些搬运命令和 kernel 命令一样是图里的节点，一样参与依赖排序。所以 SYCL 里你从不手写"把 buffer 拷到 GPU"——你只声明访问，调度器根据数据当前位置自动插入拷贝节点。

enqueue：按拓扑序推上设备

图建好后，GraphProcessor（scheduler.hpp:815）负责把命令 enqueue 到设备。核心是 enqueueCommand（graph_processor.cpp:68）——它先递归 enqueue 所有依赖（遍历命令的 MPreparedDepsEvents），再 enqueue 命令自己。这个"先依赖后自己"的递归，正是拓扑排序：保证一个命令被推上设备时，它依赖的命令都已经在跑或已完成。

等待也走这里：GraphProcessor::waitForEvent（graph_processor.cpp:20）拿到 event 对应的命令，以阻塞模式 enqueue 它（连同依赖），然后放锁、调 Event.waitInternal() 真正等硬件完成。这就是你 q.wait() 或 event.wait() 背后发生的事。

host task 是"阻塞型"节点 普通 kernel 命令 enqueue 后就交给设备异步跑了。但 host task（CGHostTask，你用 h.host_task(...) 提交的、在主机上跑的一段代码）不同——它对它的后继是阻塞的：Command::isBlocking() = isHostTask() && !MEvent->isCompleted()（commands.hpp:171）。调度器用 handleBlockingCmd 把依赖它的命令记进 MBlockedUsers，等 host task 在主机线程上跑完（DispatchHostTask）再放行。这是"设备命令 + 主机命令"混在一张图里必须处理的细节——主机任务没有设备 event 可等，只能靠这套阻塞/解阻塞机制。

动手：模拟依赖图的自动连边

把这一章的核心——「每个 buffer 维护读/写叶子，按 RAW/WAR/WAW 给新命令连边」——用一个浏览器里能跑的小模型演出来。点 ▶ 运行。

Python 
# 模拟 SYCL 调度器：按 accessor 读/写模式，用读/写叶子自动算依赖

class MemObjRecord:                 # 每个 buffer 一条：维护读叶子和写叶子
    def __init__(self):
        self.write_leaves = []      # 最近的写命令
        self.read_leaves = []       # 最近的读命令

class Scheduler:
    def __init__(self):
        self.records = {}           # buffer 名 -> MemObjRecord
        self.n = 0
    def add_cg(self, name, reqs):
        # reqs: [(buffer, mode)]，mode 是 'read' 或 'write'
        self.n += 1
        cmd = f"{name}#{self.n}"
        deps = set()
        for buf, mode in reqs:
            rec = self.records.setdefault(buf, MemObjRecord())
            read_only = (mode == "read")
            # 只读：只依赖写叶子(RAW)；写：依赖写叶子(WAW)+读叶子(WAR)
            deps.update(rec.write_leaves)                 # RAW / WAW
            if not read_only:
                deps.update(rec.read_leaves)              # WAR
        # 连边后，把自己设为对应叶子
        for buf, mode in reqs:
            rec = self.records[buf]
            if mode == "read":
                rec.read_leaves.append(cmd)               # 读命令不清空写叶子的历史……
            else:
                rec.write_leaves = [cmd]                  # 新写成为唯一写叶子
                rec.read_leaves = []                      # 写之后清空读叶子
        why = "无依赖(可并行)" if not deps else "依赖 " + ", ".join(sorted(deps))
        print(f"  {cmd:<10} reqs={reqs}")
        print(f"             → {why}")
        return cmd

s = Scheduler()
print("场景：CG1 写B，CG2 读B，CG3 读B，CG4 写B，另有 CG5 写C（无关）")
s.add_cg("writeB",  [("B", "write")])   # 无依赖
s.add_cg("readB",   [("B", "read")])    # RAW: 依赖 writeB
s.add_cg("readB2",  [("B", "read")])    # 只读，和 readB 并行；仍 RAW 依赖 writeB
s.add_cg("writeB2", [("B", "write")])   # WAW+WAR: 依赖 writeB 之后的两个读
s.add_cg("writeC",  [("C", "write")])   # 访问不同 buffer C，完全无关 → 可并行

运行结果：

场景：CG1 写B，CG2 读B，CG3 读B，CG4 写B，另有 CG5 写C（无关）
  writeB#1   reqs=[('B', 'write')]
             → 无依赖(可并行)
  readB#2    reqs=[('B', 'read')]
             → 依赖 writeB#1
  readB2#3   reqs=[('B', 'read')]
             → 依赖 writeB#1
  writeB2#4  reqs=[('B', 'write')]
             → 依赖 readB#2, readB2#3
  writeC#5   reqs=[('C', 'write')]
             → 无依赖(可并行)

看清调度器的自动排序：writeB 无依赖先跑；两个读 readB/readB2 都只 RAW 依赖 writeB、彼此不依赖（可并行）；writeB2 要覆盖 B，得等两个读都完成（WAR，所以依赖 readB 和 readB2）；而 writeC 访问的是另一个 buffer，完全无关，可以和上面任何命令并行。你全程只声明了每个命令读还是写哪个 buffer，正确的执行顺序是运行时算出来的——这正是 findDepsForReq 在真实调度器里做的事。请动手改：把 writeC 改成也访问 B（[("B","read")]），重新运行——你会看到它立刻长出对 writeB2 的依赖，从"可并行"变成"要排队"。

小结与下一步

你现在看清了 SYCL 运行时自动排序的内核：

  • 命令组（CG）（cg.hpp:55，kernel 是 CGExecKernel）携带 requirements（Requirement = AccessorImplHost，accessor_impl.hpp:184）——每个绑定的 accessor 就是一条 requirement，声明读写哪个内存、什么模式。这是调度器算依赖的全部输入。
  • 调度器（单例，scheduler.hpp:369）维护一张无环依赖图，切成 GraphBuilder（改图，写锁）和 GraphProcessor（enqueue，读锁）；addCG（scheduler.cpp:105）先建图节点 ExecCGCommand、连边，再 enqueue。
  • 依赖计算（findDepsForReq，graph_builder.cpp:575）：每个内存对象一条 MemObjRecord（scheduler.hpp:200）维护 MWriteLeaves/MReadLeaves；只读依赖写叶子（RAW），写依赖写叶子（WAW）+读叶子（WAR）；只读之间、访问不重叠（doOverlap）旁路不连边。
  • 数据搬运：MemObjRecord::MCurContext 记数据在哪；跨 context 访问时 insertMemoryMove（graph_builder.cpp:329）自动插 MemCpy/Map 命令，discard_* 模式跳过拷贝。
  • enqueue：GraphProcessor::enqueueCommand（graph_processor.cpp:68）先递归 enqueue 依赖再 enqueue 自己（拓扑序）；waitForEvent 支撑 q.wait()。host task 是阻塞型节点，靠 MBlockedUsers 解阻塞。

下一章补上第 16、17 章反复出现却一直没展开的那层——Unified Runtime（UR）。每个 impl 握着的 ur_*_handle_t、每次 adapter.call<UrApiKind::…>、以及最终把命令推上设备的 urEnqueueKernelLaunchWithArgsExp，都在 UR 这层。第 18 章讲 SYCL 如何通过 UR 对接 Level Zero / CUDA / OpenCL——你会认出它正是第 10 章"接口 + 插件"母题在 SYCL 侧的化身。

这一章的要点

  • 命令组 CG（cg.hpp:55；kernel=CGExecKernel cg.hpp:144；类型枚举 cg_types.hpp:26）带 requirements（Requirement = AccessorImplHost，accessor_impl.hpp:184），每个 accessor 一条，声明读写哪个内存对象及模式——调度器算依赖的唯一输入。
  • 调度器单例（scheduler.hpp:369）分 GraphBuilder（改图/写锁）+ GraphProcessor（enqueue/读锁）；addCG（scheduler.cpp:105）两阶段：建图 ExecCGCommand（commands.hpp:628）连双向边（MDeps/MUsers，addDep commands.cpp:786）→ enqueue。
  • RAW/WAR/WAW：MemObjRecord（scheduler.hpp:200）维护 MWriteLeaves/MReadLeaves；findDepsForReq（graph_builder.cpp:575）只读依赖写叶子、写依赖读写两组叶子；只读之间/访问不重叠（doOverlap graph_builder.cpp:43）旁路。
