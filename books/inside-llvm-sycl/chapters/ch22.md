# 第 22 章 需要编译器配合的扩展：SemaSYCL 与前端支持

## 这一章想让你带走什么

<div class="goal-box">
<p>第 21 章的 free function kernels 用了个巧劲：它<strong>没有</strong>新增专用编译器属性，而是复用了一套通用的 <code>__sycl_detail__</code> IR 属性机制——头文件贴记号、前端认记号。这一章把这条"最难的一环"彻底讲透：什么样的扩展<strong>必须</strong>动编译器前端、动哪些文件、以及 DPC++ 一个精心的设计如何让<strong>大多数</strong>扩展<strong>免于</strong>改前端。</p>
<p>读完你会掌握两条路线：<strong>通用路线</strong>——用 <code>add_ir_attributes_*</code> 把"属性名+值"机械地灌进 IR，前端不认识具体属性也照样处理（这是 <code>sycl_ext_oneapi_properties</code> 的设计核心，也是 free function kernels、device_global 等一大批扩展的共同底座）；<strong>专用路线</strong>——当扩展需要真正的语义检查或代码生成时，怎么端到端加一个新属性：<code>Attr.td</code> 定义 → <code>AttrDocs.td</code> 文档 → <code>SemaSYCLDeclAttr.cpp</code> 检查 → CodeGen 落 IR。你会用真实的 <code>reqd_work_group_size</code>、<code>device_has</code> 追出全链路。这是"写扩展"里技术门槛最高、也最能看清 DPC++ 前端设计哲学的一章。</p>
</div>

## 什么时候必须改编译器？

先建立判断力。一个 SYCL 扩展**不是**总要改编译器——大量扩展纯粹是"库扩展"（新类、新函数，全在头文件和运行时里实现，编译器毫不知情）。只有当扩展需要下面这类能力时，才要动前端：

1. **要把用户意图带到设备 IR / SPIR-V 上**：比如某个 kernel 或变量要携带一条只有后端/驱动才读的元信息（work-group 大小、aspect 依赖、编译期 property）。
2. **要做源码层面的语义检查/诊断**：比如"这个属性只能贴在 kernel 上""参数不能用某类型"，需要在编译期报错。
3. **要识别一种新的 kernel 形态**：比如第 21 章的 free function kernels——前端得认出"这个普通函数其实是个 kernel"并为它生成入口。
4. **要加一个驱动开关或语言选项**：`-fsycl-*` 之类。

关键洞察是：DPC++ 的前端设计**刻意压低了第 1、2 类的门槛**——它提供了一套通用机制，让"把属性带到 IR"这件最常见的事**不需要**每次都改前端。先讲这条通用路线。

## 通用路线：__sycl_detail__ 与"前端不必懂属性"

第 21 章见过那个 `add_ir_attributes_function`。它其实是一整套机制的一员。这套机制的**设计目标**写在 `sycl/doc/design/CompileTimeProperties.md`（实现 `sycl_ext_oneapi_properties` 扩展的设计文档）里，原话点破了要害：

> "One of the goals of this design is to **avoid changes to the front-end each time we add a new property**, so the front-end is not required to understand each property it consumes. Instead, it follows a **mechanical process** for converting properties listed in the C++ annotation into LLVM IR."

翻译过来：**加一个新 property，不该改前端**。前端不需要"认识"每个具体属性，它只做一件机械的事——把 C++ 注解里列出的一串"名字+值"对，原样搬到 LLVM IR 上。这套 `__sycl_detail__` 属性有四个成员，覆盖四种载体：

| 属性拼写（`__sycl_detail__::`） | TableGen 记录（`Attr.td`） | 行号 | 贴在哪 |
|---|---|---|---|
| `add_ir_attributes_function` | `SYCLAddIRAttributesFunction` | 2250 | 函数 |
| `add_ir_attributes_kernel_parameter` | `SYCLAddIRAttributesKernelParameter` | 2260 | kernel 参数 |
| `add_ir_attributes_global_variable` | `SYCLAddIRAttributesGlobalVariable` | 2271 | 全局变量 |
| `add_ir_annotations_member` | `SYCLAddIRAnnotationsMember` | 2282 | 结构体成员 |

四个记录共享一段 `AdditionalMembers`（`Attr.td:2014` 的 `SYCLAddIRAttrCommonMembers`），核心是 `getFilteredAttributeNameValuePairs(...)`（`Attr.td:2205`）——**把可变参数的 `("name", value, ...)` 拆成一串 `pair<string,string>`**。这就是"机械搬运"的执行者。它们都带 `let LangOpts = [SYCLIsDevice, SilentlyIgnoreSYCLIsHost];`——**只在设备编译趟生效，主机趟静默忽略**（呼应第 21 章那个 `#ifdef __SYCL_DEVICE_ONLY__`）。

### 它们在 CodeGen 里落成什么

四种载体在 CodeGen 里落成不同形态的 IR：

- **函数属性** → LLVM **字符串函数属性**（`CodeGenFunction.cpp:1199`）：读 `SYCLAddIRAttributesFunctionAttr`，调 `getFilteredAttributeNameValuePairs`，`FnAttrBuilder.addAttribute(name, value)`。于是 IR 函数上多出 `"sycl-nd-range-kernel"="3"` 这样的属性。
- **kernel 参数** → 参数属性（`CGCall.cpp:3484/3572`）：特别地，`"sycl-unaliased"` 被识别（`hasSYCLRestrictPropertyIRAttr`，`CGCall.cpp:3482`）并落成 `noalias`——第 15 章 `kernel_args_restrict` 那条路的通用版。
- **成员** → `llvm.ptr.annotation`，注解名 `"sycl-properties"`（`EmitSYCLAnnotationCall`，`CodeGenFunction.cpp:3116/3133`）。
- **全局变量** → 全局上的属性（`CodeGenModule.cpp:4544`），如 `device_global` 的 `"sycl-unique-id"`（`:4553`）。

落进 IR 之后，这些字符串属性由后续的 **`sycl-post-link`** Pass 或 `llvm-spirv` 翻译阶段消费（第 8、9 章）。整条链里**前端从头到尾没"理解"过任何具体 property 的语义**——它只搬字符串。这就是为什么加一个新的编译期 property 通常一行前端代码都不用改，只在头文件里把它编码成 `add_ir_attributes_*` 的参数即可。

<div class="keypoint">
<strong>通用路线：__sycl_detail__ 让"把属性带到 IR"不必改前端</strong>
<strong>DPC++ 前端提供 4 个通用属性（<code>add_ir_attributes_function</code>/<code>_kernel_parameter</code>/<code>_global_variable</code>/<code>add_ir_annotations_member</code>，<code>Attr.td:2250–2282</code>），把头文件里编码的"名字+值"对<strong>机械地</strong>搬到 IR（函数字符串属性 / 参数属性 / <code>llvm.ptr.annotation "sycl-properties"</code> / 全局属性）。前端不理解具体 property 的语义（设计目标见 <code>CompileTimeProperties.md</code>），由后续 <code>sycl-post-link</code> / <code>llvm-spirv</code> 消费。</strong>所以大多数"要把意图带到 IR"的扩展<strong>免改前端</strong>——只在头文件里用宏展开成 <code>add_ir_attributes_*</code>。free function kernels、device_global、compile-time properties 都走这条路。
</div>

## 专用路线：端到端加一个新属性

当通用机制不够用时——比如你需要**真正的语义检查**（参数必须是正整数、只能贴在 kernel 上、和别的属性冲突要诊断），或需要落成一条结构化的 **metadata**（而非字符串属性）——就得走专用路线：加一个 clang 属性。用最经典的 `reqd_work_group_size`（第 15 章见过它的落地）端到端追一遍，四步：

### ① Attr.td：定义属性

`clang/include/clang/Basic/Attr.td:3927`：

```llvm-norun
def SYCLReqdWorkGroupSize : InheritableAttr, LanguageOptionsSpecificAttr {
  let Spellings = [GNU<"reqd_work_group_size">,
                   CXX11<"cl", "reqd_work_group_size">,
                   CXX11<"sycl", "reqd_work_group_size">];
  let Args = [ExprArgument<"XDim">, ExprArgument<"YDim", 1>, ExprArgument<"ZDim", 1>];
  let Subjects = SubjectList<[Function], ErrorDiag>;
  let LangOpts = [SYCLIsDevice, SilentlyIgnoreSYCLIsHost];
  // AdditionalMembers: getXDimVal()/getYDimVal()/getZDimVal()
}
```

`Spellings` 声明用户能写的拼写（`[[sycl::reqd_work_group_size]]` / `[[cl::...]]`）；`Args` 是三个表达式参数；`Subjects=[Function]` 限定只能贴在函数上（贴别处报 `ErrorDiag`）。TableGen 据此生成 `SYCLReqdWorkGroupSizeAttr` 类。

### ② AttrDocs.td：文档

`clang/include/clang/Basic/AttrDocs.td:3301` 的 `ReqdWorkGroupSizeAttrDocs`——每个面向用户的属性都要在这里写文档，`clang -Wdocumentation` 和官方属性文档都从这来。**别忘了这一步**，CI 会检查属性有没有对应文档。

### ③ SemaSYCLDeclAttr.cpp：语义检查

真正的合法性判断在 `clang/lib/Sema/SemaSYCLDeclAttr.cpp`，通常是"handle + add + merge"三个函数配套：

- `handleSYCLReqdWorkGroupSizeAttr`（`:1064`）—— 从 `ParsedAttr` 解析出 X/Y/Z 三个表达式参数。
- `addSYCLReqdWorkGroupSizeAttr`（`:956`）—— 核心校验：每个维度用 `VerifyIntegerConstantExpression` 求常量、要求**严格为正**（否则 `err_attribute_requires_positive_integer`）、查重（`err_duplicate_attribute`），通过后 `new SYCLReqdWorkGroupSizeAttr(...)` 挂到 `Decl` 上（`:1013`）。
- `mergeSYCLReqdWorkGroupSizeAttr`（`:1086`）—— 处理同一声明多次出现时的合并/冲突。

这一步是专用路线相对通用路线**多花的力气**，也是它的**价值**：能给出精准的编译期诊断。诊断消息本身定义在 `clang/include/clang/Basic/DiagnosticSemaKinds.td`。

### ④ CodeGen：落成 metadata

`clang/lib/CodeGen/CodeGenFunction.cpp:738`：读 `FD->getAttr<SYCLReqdWorkGroupSizeAttr>()`，求出各维值，`Fn->setMetadata("reqd_work_group_size", ...)`（`:767`，还设 `"work_group_num_dim"`）。注意 SYCL 目标下**维度是反序**发射的。至此属性变成 IR 上的结构化 metadata，后端 / `llvm-spirv` 读它定制代码生成（第 15 章讲过下游）。

<div class="note">
<strong>专用路线的第二个例子：<code>device_has</code></strong>
想再验证一遍这个四步模式，看 <code>device_has</code>（第 15 章的 aspect 依赖）：① <code>Attr.td:1839</code> <code>def SYCLDeviceHas</code>（<code>VariadicExprArgument&lt;"Aspects"&gt;</code>）；② <code>AttrDocs.td:941</code> <code>SYCLDeviceHasDocs</code>；③ <code>SemaSYCLDeclAttr.cpp</code> 的 <code>handleSYCLDeviceHasAttr</code>（<code>:1441</code>）/ <code>addSYCLDeviceHasAttr</code>（<code>:389</code>）/ <code>mergeSYCLDeviceHasAttr</code>（<code>:1454</code>）；④ CodeGen <code>CodeGenModule.cpp:3893</code> 的 <code>applySYCLAspectsMD(A, ..., "sycl_declared_aspects")</code>。同一套 <strong>Attr.td → AttrDocs.td → Sema(handle/add/merge) → CodeGen</strong> 骨架，换个属性照搬。认得这个骨架，你就能读懂几乎所有 SYCL 专用属性。
</div>

## 识别新 kernel 形态：SemaSYCL 的重活

第三类需求——**认出一种新 kernel 形态**——是最重的前端活，第 21 章的 free function kernels 就是例子。它虽然用通用属性贴记号（`"sycl-nd-range-kernel"`），但"认记号并把普通函数当 kernel 处理"这件事要动 `clang/lib/Sema/SemaSYCL.cpp` 的实打实逻辑：

- `SemaSYCL::isFreeFunction(const FunctionDecl *FD)`（`:1174`）—— 扫函数所有重声明上的 `SYCLAddIRAttributesFunctionAttr`，找 `"sycl-nd-range-kernel"` / `"sycl-single-task-kernel"`；若后续声明带属性而**首个声明没带**，报 `err_free_function_first_occurrence_missing_attr`。
- `getFreeFunctionRangeDim(...)`（`:1209`）—— 从属性取维数（nd-range 取 1/2/3，single-task 取 0）。
- `constructFreeFunctionKernelName(...)`（`:1238`）—— 构造 `__sycl_kernel_` 前缀的修饰名（返回 {新名, 稳定名}），写进集成头文件。
- `FreeFunctionKernelBodyCreator`（`:4323`）—— 生成 kernel 入口函数体。
- 参数合法性由 `unsupportedFreeFunctionParamType()`（`:1094`）在一堆调用点把关。

配套诊断在 `DiagnosticSemaKinds.td`：`err_not_sycl_free_function`（`:13613`）、`err_free_function_with_default_arg`（`:13615`）、`err_free_function_variadic_args`（`:13617`）、`err_free_function_return_type`（`:13619`）、`err_free_function_first_occurrence_missing_attr`（`:13621`）、`err_free_function_class_method`（`:13623`）。**规范文档里写的每一条"必须/不能"，在这里都对应一条 `err_*` 诊断**——这是"实现一个 kernel 形态扩展"的主要工作量所在。

## 驱动开关与语言选项

第四类——加一个 `-fsycl-*` 开关或语言选项——涉及两处文件：

- **语言选项** `clang/include/clang/Basic/LangOptions.def`：SYCL 相关的如 `LANGOPT(SYCLIsDevice, ...)`（`:265`）、`LANGOPT(SYCLIsHost, ...)`（`:266`）、`ENUM_LANGOPT(SYCLVersion, ...)`（`:274`）。前面那些属性的 `let LangOpts = [SYCLIsDevice, ...]` 引用的正是这里。
- **驱动/前端开关** `clang/include/clang/Options/Options.td`（注意：本分支在 `Options/` 下，不是老资料里的 `Driver/`）。例如 `def fsycl : Flag<["-"], "fsycl">`（`:7680`）、`fno_sycl`（`:7682`）、`fsycl_device_only`（`:7684`）、`fsycl_targets=`（`:7736`）。

大多数扩展**不需要**新开关（能力协商靠 feature-test 宏，第 20 章）；只有当扩展要改变编译模式或需要显式启停时才加。

<div class="warn">
<strong>路径会漂移：<code>Options.td</code> 现在在 <code>clang/include/clang/Options/</code></strong>
不少资料（含 clang 上游历史）把驱动选项写成 <code>clang/include/clang/Driver/Options.td</code>。在本书对应的 <code>intel/llvm</code> <code>sycl</code> 分支上，它已经<strong>移到</strong> <code>clang/include/clang/Options/Options.td</code>。这类"文件搬家"正是"源码为锚"要防的坑——引用路径前先在你手上的树里 <code>find</code> 一下。
</div>

## 亲手追前端改动

在源码里把这两条路线的锚点都确认一遍：

```bash
# 通用路线：四个 add_ir_attributes_* 记录
grep -n 'SYCLAddIRAttributes\|SYCLAddIRAnnotationsMember' clang/include/clang/Basic/Attr.td

# 通用机制的设计目标（"避免每加一个 property 就改前端"）
grep -n 'avoid changes to the front-end\|mechanical' sycl/doc/design/CompileTimeProperties.md

# 专用路线：一个属性的四步
grep -n 'SYCLReqdWorkGroupSize' clang/include/clang/Basic/Attr.td clang/include/clang/Basic/AttrDocs.td
grep -n 'ReqdWorkGroupSizeAttr' clang/lib/Sema/SemaSYCLDeclAttr.cpp clang/lib/CodeGen/CodeGenFunction.cpp

# 识别 kernel 形态：free function 前端逻辑与诊断
grep -n 'isFreeFunction\|constructFreeFunctionKernelName' clang/lib/Sema/SemaSYCL.cpp
grep -n 'err_free_function' clang/include/clang/Basic/DiagnosticSemaKinds.td
```

想直观感受两条路线的差别：拿一个 compile-time property（通用路线）和 `reqd_work_group_size`（专用路线）各写个小 kernel，`-fsycl-device-only -S -emit-llvm` 出 IR——前者落成 `"sycl-..."` 字符串**函数属性**，后者落成 `!reqd_work_group_size` **metadata**。字符串属性那条链里前端一行没改，metadata 那条链里前端改了四处。这就是"能免则免、必要才加"的取舍。

把这一章的核心——「通用路线机械搬运、专用路线四步落地」——用一个能在浏览器里跑的小模型演出来。点 **▶ 运行**。

```python
# 模拟 SYCL 扩展的两条编译器路线：通用(免改前端) vs 专用(加新属性)

# ========== 通用路线：add_ir_attributes_* 机械搬运，前端不懂语义 ==========
def generic_route(fn_name, name_value_pairs):
    # 前端只做一件事：把 (name, value) 对搬成 IR 字符串属性。不校验、不理解。
    print(f"[通用] define spir_kernel void @{fn_name}(...) ", end="")
    print(" ".join(f'"{n}"="{v}"' for n, v in name_value_pairs), "{ ... }")
    print(f"       前端改动: 0 行（新增任何 property 都走这条，不碰前端）\n")

# ========== 专用路线：加一个 clang 属性，四步 ==========
def dedicated_route(attr, args):
    print(f"[专用] 属性 [[sycl::{attr}]]({', '.join(map(str,args))}) 的四步：")
    # ① Attr.td 定义（这里用一张表模拟"记录"）
    ATTR_TD = {"reqd_work_group_size": {"subject": "Function", "nargs": 3, "md": "reqd_work_group_size"}}
    rec = ATTR_TD.get(attr)
    if not rec:
        print("   ① Attr.td: 无此记录 → 需要先添加\n"); return
    print(f"   ① Attr.td: 记录已定义（贴在 {rec['subject']}，{rec['nargs']} 参数）")
    print(f"   ② AttrDocs.td: 文档已写")
    # ③ Sema 语义检查（专用路线的价值：精准诊断）
    vals = list(args) + [1] * (rec["nargs"] - len(args))
    for i, v in enumerate(vals):
        if not isinstance(v, int) or v <= 0:
            print(f"   ③ Sema: ✗ 第{i}维={v} 非正整数 → err_attribute_requires_positive_integer（编译期报错）\n")
            return
    print(f"   ③ Sema: ✓ 校验通过（正整数、无重复）→ 挂到 FunctionDecl")
    # ④ CodeGen 落 metadata（注意 SYCL 目标维度反序）
    rev = list(reversed(vals))
    print(f"   ④ CodeGen: !{rec['md']} = !{{{', '.join('i32 '+str(v) for v in rev)}}}（反序发射）\n")

print("=== 场景1：一个编译期 property（通用路线）===")
generic_route("_ZTS7MyKernel", [("sycl-nd-range-kernel", "3"), ("sycl-my-new-prop", "42")])

print("=== 场景2：reqd_work_group_size 合法（专用路线）===")
dedicated_route("reqd_work_group_size", [16, 16])

print("=== 场景3：reqd_work_group_size 非法参数（专用路线的诊断价值）===")
dedicated_route("reqd_work_group_size", [16, 0])
```

运行结果：

看清两条路线的取舍：**场景 1** 加了一个全新 property `sycl-my-new-prop`，`generic_route` 一行没改——因为它只机械搬字符串，这正是 `CompileTimeProperties.md` 那句"避免每加一个 property 就改前端"的意义；新属性想加就加，前端毫不知情。**场景 2/3** 走专用路线，多了 `Attr.td / AttrDocs.td / Sema / CodeGen` 四步，但换来了**场景 3** 那种精准诊断（第二维为 0 直接编译期报错）和反序的结构化 metadata。请动手改：给场景 1 再塞一对全新的 `("sycl-whatever", "7")`，你会看到它照样被搬进 IR、前端依然零改动——这就是通用机制"能免则免"的威力；只有当你需要场景 3 那样的校验时，才值得走专用路线。

## 小结与下一步

这一章讲清了"需要编译器配合的扩展"到底要动什么：

- **先判断要不要改前端**：纯库扩展不用。只有要「把意图带到 IR」「做语义检查」「认新 kernel 形态」「加驱动开关」才动前端。
- **通用路线（首选，多数免改前端）**：`__sycl_detail__` 的四个 `add_ir_attributes_*`（`Attr.td:2250–2282`）把头文件编码的"名字+值"**机械**搬到 IR（函数字符串属性 / 参数 `noalias` / `llvm.ptr.annotation "sycl-properties"` / 全局属性），前端不懂语义，下游 `sycl-post-link`/`llvm-spirv` 消费。设计目标见 `CompileTimeProperties.md`：**避免每加一个 property 就改前端**。
- **专用路线（要语义检查/结构化 metadata 才走）**：四步——① `Attr.td` 定义（`SYCLReqdWorkGroupSize` `:3927`）② `AttrDocs.td` 文档（`:3301`）③ `SemaSYCLDeclAttr.cpp` 的 handle/add/merge 校验（`:1064/956/1086`）④ CodeGen 落 metadata（`CodeGenFunction.cpp:738`，`!reqd_work_group_size`，维度反序）。`device_has` 是同骨架的第二例。
- **识别新 kernel 形态**（最重的活）：`SemaSYCL.cpp` 的 `isFreeFunction`（`:1174`）/ `getFreeFunctionRangeDim`（`:1209`）/ `constructFreeFunctionKernelName`（`:1238`）/ `FreeFunctionKernelBodyCreator`（`:4323`），规范里每条"必须/不能"对应一条 `err_free_function_*` 诊断（`DiagnosticSemaKinds.td:13613+`）。
- **驱动/语言选项**：`LangOptions.def`（`SYCLIsDevice` `:265`）+ `clang/include/clang/Options/Options.td`（`fsycl` `:7680`，注意路径已从 `Driver/` 搬到 `Options/`）。开发者文档 `clang/docs/SYCLSupport.rst`。

下一章从"改编译器"回到"用扩展"——**第 23 章 案例巡礼**：拿两个分量很重的真实扩展做样本，**bindless images**（用句柄而非 accessor 访问图像，突破编译期数量限制）与 **kernel_compiler**（运行时把源码字符串编译成 kernel）。你会看到前者主要靠运行时 + UR，后者动用了顶层 `sycl-jit/` 在进程内调 clang 库——两个扩展、两种截然不同的实现重心，正好检验你这一部分学到的"扩展解剖学"。

**这一章的要点**

- 判断要不要改前端：要「带意图到 IR」「语义检查」「认新 kernel 形态」「加驱动开关」才动；纯库扩展不动。
- **通用路线**：`__sycl_detail__` 四属性 `add_ir_attributes_function/_kernel_parameter/_global_variable` + `add_ir_annotations_member`（`Attr.td:2250–2282`，共享 `SYCLAddIRAttrCommonMembers` `:2014` 的 `getFilteredAttributeNameValuePairs` `:2205`），只在 `SYCLIsDevice` 生效；CodeGen 落字符串属性/`noalias`/`"sycl-properties"`/全局属性。**前端不懂语义、加 property 免改前端**（`CompileTimeProperties.md`）。

- **选项**：`LangOptions.def`（`SYCLIsDevice:265`）+ `clang/include/clang/Options/Options.td`（`fsycl:7680`——路径已从 `Driver/` 移到 `Options/`）。文档 `clang/docs/SYCLSupport.rst`、`sycl/doc/design/CompileTimeProperties.md`。
