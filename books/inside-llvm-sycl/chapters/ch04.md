# 第 4 章 Pass 框架（一）：新 PassManager 的设计

## 这一章想让你带走什么

<div class="goal-box">
<p>第 1 章说过，中端优化是一次次「IR → 更好的 IR」的变换，每个变换叫一个 <strong>Pass</strong>。这一章我们打开 LLVM 组织这些 Pass 的框架——<strong>新 PassManager（new PM）</strong>。它有三个让老手都觉得优雅的设计：Pass <strong>不靠继承</strong>（没有 <code>Pass</code> 基类，任何有 <code>run()</code> 方法的类都能当 Pass）；分析结果由 <strong>AnalysisManager 按需计算、缓存、失效</strong>；Pass 用 <strong><code>PreservedAnalyses</code></strong> 精确声明「我动了什么、没动什么」，框架据此决定哪些缓存要作废。</p>
<p>读完你能对着 <code>llvm/include/llvm/IR/PassManager.h</code> 把这套机制讲清楚：一个 Pass 到底要满足什么「契约」、四层 IR 单位（Module/CGSCC/Function/Loop）如何用适配器嵌套、分析缓存怎么失效。这是第 5 章「亲手写一个 Pass 并挂进 <code>opt</code>」的理论基础。</p>
</div>

## 一个 Pass 长什么样：没有基类，只有「契约」

先破除一个来自教科书的直觉：在新 PM 里，**没有一个叫 `Pass` 的基类要你去继承**。`llvm/include/llvm/IR/PassManager.h` 开头就写着这句设计宣言（原话大意）：

> LLVM 里没有所谓「pass 接口」。任何一个类，只要它支持一个方法，能把自己「跑」在某个 IR 单位上，就能当一个 pass 用。

这里的「某个方法」是一个固定签名的 `run`：

```cpp-norun
// 一个作用在 Function 上的 Pass 必须提供的方法——这就是全部契约
PreservedAnalyses run(Function &F, FunctionAnalysisManager &AM);
```

只要你的类有这么一个 `run`，它就是一个 Function Pass。作用在 Module 上就把 `Function`/`FunctionAnalysisManager` 换成 `Module`/`ModuleAnalysisManager`。**没有虚函数、没有继承、没有注册宏是必需的**——框架靠 C++ 模板做「基于概念的多态」（type erasure），把各种不同的 Pass 类型统一擦除成一个内部接口。这套擦除机制藏在 `llvm/include/llvm/IR/PassManagerInternal.h` 里的 `PassConcept` / `PassModel`，你平时写 Pass 完全不用碰它。

那 Pass 之间的共性代码（比如打印名字）怎么办？用一个 **CRTP 混入类**，而不是基类：

```cpp-norun
// DCE（死代码消除）Pass 的真实声明
//   —— llvm/include/llvm/Transforms/Scalar/DCE.h
class DCEPass : public OptionalPassInfoMixin<DCEPass> {
public:
  PreservedAnalyses run(Function &F, FunctionAnalysisManager &AM);
};
```

`OptionalPassInfoMixin<DCEPass>`（还有 `PassInfoMixin`、`RequiredPassInfoMixin`）只提供 `name()`、`printPipeline()` 这类样板，**不提供 `run`**。`run` 完全由你自己写。`Required` 和 `Optional` 的区别只有一个：`-O0` 或 opt-bisect 调试时，`Optional` 的 Pass 可以被跳过，`Required` 的不能。

<div class="note">
<strong>为什么不用继承？</strong>
新 PM 的注释直接引用了 Sean Parent 的名言「继承是万恶之源（Inheritance Is The Base Class of Evil）」。用「有 <code>run</code> 方法就行」的鸭子类型 + 模板擦除，好处是 Pass 变成<strong>值语义</strong>的普通对象：能栈上创建、能拷贝进管线、能内联，没有虚表开销，也不必被单一继承层级绑死在某一个 IR 层级上。这是新旧 PM 最根本的分野。
</div>

## 三个词就能概括：变换、分析、保留集

新 PM 的运转围绕三个概念。理解了它们的关系，整个框架就通了。

- **变换 Pass（transform）**：真正改 IR 的家伙。DCE、内联、常量折叠都是。它的 `run` 返回一个 `PreservedAnalyses`。
- **分析 Pass（analysis）**：不改 IR，只**计算关于 IR 的情报**并缓存起来，供变换 Pass 查询。比如支配树（Dominator Tree）、循环信息（LoopInfo）、别名分析。分析的结果由 **AnalysisManager** 管理。
- **保留集 `PreservedAnalyses`（PA）**：变换 Pass 跑完后交回的一张「收据」，声明「哪些分析结果我没破坏、还能接着用；哪些我动了、请作废」。

三者的协作是这样一个循环：变换 Pass 需要情报 → 找 AnalysisManager 要（要不到就现算并缓存）→ 变换 Pass 改 IR、返回 PA → 框架拿 PA 去让**过时的**分析缓存失效。下面逐个拆开。

## PassManager：一串 Pass，本身也是一个 Pass

`PassManager` 就是「按顺序装一串 Pass 并挨个跑」的容器。它是个模板，按作用的 IR 单位实例化（`PassManager.h`）：

```cpp-norun
template <typename IRUnitT,
          typename AnalysisManagerT = AnalysisManager<IRUnitT>,
          typename... ExtraArgTs>
class PassManager : public RequiredPassInfoMixin<PassManager<...>> {
  PreservedAnalyses run(IRUnitT &IR, AnalysisManagerT &AM, ExtraArgTs...);
  template <typename PassT> void addPass(PassT &&Pass);
  std::vector<std::unique_ptr<PassConceptT>> Passes;   // 有序的 Pass 列表
};

using ModulePassManager   = PassManager<Module>;    // 常用别名
using FunctionPassManager = PassManager<Function>;
```

注意它自己也继承了 `RequiredPassInfoMixin`——换句话说，**PassManager 本身就是一个合法的 Pass**。这不是巧合，而是刻意的：正因为「一串 Function Pass 打包起来还是个 Function Pass」，我们才能把它整个塞进更上层的容器里嵌套（下一节的适配器就靠这个）。

它的 `run()` 循环是整套机制的心脏，实现在 `llvm/include/llvm/IR/PassManagerImpl.h`。剥掉仪表化（instrumentation）代码后，逻辑是这样：

```cpp-norun
// PassManager::run 的核心逻辑（简化）
PreservedAnalyses PA = PreservedAnalyses::all();   // 先乐观假设"什么都没坏"
for (auto &Pass : Passes) {
    PreservedAnalyses PassPA = Pass->run(IR, AM);  // 跑这个 Pass，拿到它的收据
    AM.invalidate(IR, PassPA);                     // 立刻按收据作废过时缓存
    PA.intersect(std::move(PassPA));               // 汇总：取所有收据的"交集"
}
return PA;
```

三个细节值得记住：

1. **起点是 `PreservedAnalyses::all()`**——乐观假设没有任何分析被破坏，让后续的 Pass 去「打破」它。
2. **失效是逐个 Pass 立刻做的**（`AM.invalidate` 在循环内）。这样下一个 Pass 看到的缓存已经是干净的。
3. **汇总用 `intersect`（交集）**——只有**所有** Pass 都保留的分析，整个 PassManager 才算保留。任何一个 Pass 破坏了某分析，对外就得算破坏。

## PreservedAnalyses：那张收据怎么写

`PreservedAnalyses`（定义在 `llvm/include/llvm/IR/Analysis.h`）是变换 Pass 和框架之间的合同。它有几种典型写法：

```cpp-norun
// 情况一：我压根没改 IR（比如分析发现无事可做）
return PreservedAnalyses::all();     // 一切照旧，所有缓存继续有效

// 情况二：我大改了 IR，谁也别信了
return PreservedAnalyses::none();    // 全部作废，后续要用的分析统统重算

// 情况三（最常见）：我改了指令，但没动控制流图的形状
PreservedAnalyses PA;
PA.preserveSet<CFGAnalyses>();       // 声明"CFG 类的分析我没破坏"
return PA;
```

第三种是精髓。像 DCE 这样的 Pass 只删除无用指令、不改变基本块之间的跳转关系，所以它可以诚实地说「CFG 我没动」，那些建立在 CFG 之上的分析（支配树等）就**不必重算**。看 DCE 的真实实现（`llvm/lib/Transforms/Scalar/DCE.cpp`）：

```cpp-norun
PreservedAnalyses DCEPass::run(Function &F, FunctionAnalysisManager &AM) {
  if (!eliminateDeadCode(F, &AM.getResult<TargetLibraryAnalysis>(F)))
    return PreservedAnalyses::all();   // 没删掉任何东西 → 什么都没坏
  PreservedAnalyses PA;
  PA.preserveSet<CFGAnalyses>();       // 删了死代码，但 CFG 形状不变
  return PA;
}
```

短短几行，把这一章的要点全演示了：`OptionalPassInfoMixin`、固定签名的 `run`、用 `AM.getResult<...>(F)` 向 AnalysisManager 要情报、返回一张诚实的 `PreservedAnalyses`。第 5 章你写的第一个 Pass 就长这样。

<div class="warn">
<strong>收据写错了会怎样？</strong>
<code>PreservedAnalyses</code> 是<strong>你对框架的承诺</strong>，框架完全相信它、不会去核对。如果你的 Pass 明明改了控制流，却返回 <code>preserveSet&lt;CFGAnalyses&gt;()</code>，框架就不会作废支配树缓存，后续 Pass 会拿着一棵<strong>过时的、和当前 IR 不符的</strong>支配树去做变换——这类 bug 极其隐蔽，往往表现为「某个优化组合下偶发崩溃或生成错误代码」。写 Pass 时，返回的保留集必须<strong>严格不超过</strong>你真正保留的东西；拿不准就返回 <code>none()</code>，慢一点但绝不会错。
</div>

## AnalysisManager：按需计算、缓存、失效

分析结果由 `AnalysisManager<IRUnitT>` 掌管（`PassManager.h`）。它对变换 Pass 暴露两个主要方法：

- `getResult<AnalysisT>(IR)`：要某个分析的结果。**没缓存就现算一遍**（惰性），算完存下来再返回。
- `getCachedResult<AnalysisT>(IR)`：只查缓存，**从不触发计算**，没有就返回空指针。

缓存怎么存的？每种分析类型有一个全局唯一的 `AnalysisKey*` 作为身份（来自 `AnalysisInfoMixin::ID()`，返回 `&DerivedT::Key`）。AnalysisManager 内部用两张表：`AnalysisResultLists` 把「IR 单位指针」映射到该单位上所有分析结果的链表，`AnalysisResults` 把「(分析 Key, IR 单位) 对」映射到链表里的具体位置。惰性计算的实现（`getResultImpl`）很直白：缓存未命中，就找到注册的分析 Pass、调它的 `run` 算出结果、存进表、返回。

失效（`invalidate`）就是把 `PreservedAnalyses` 兑现的地方。它的逻辑：

1. 如果收据说「和本层所有分析相关的都保留了」（`allAnalysesInSetPreserved`），直接返回，一个都不动——**最快路径**。
2. 否则遍历本单位上每个缓存的分析结果，问它「按这张收据，你还有效吗？」（`Result.invalidate(IR, PA, Inv)`），失效的就删掉。

第 2 步里那个「问它还有效吗」很关键：**是每个分析结果自己判断**的，而不是框架硬删。因为分析之间会互相依赖（比如 A 分析内部持有 B 分析的结果），框架提供一个 `Invalidator` 做深度优先的传递失效——B 失效了，依赖 B 的 A 也跟着失效。这套精细的失效，正是新 PM 能安全地「只重算该重算的」的底气。

## 四层 IR 单位与「适配器」

第 2 章讲过 IR 的层级：Module ⊃ Function ⊃ BasicBlock。优化也按层级组织，新 PM 有**四种** IR 单位，各有自己的 PassManager 和 AnalysisManager：

| 层级 | PassManager 别名 | 作用对象 | 典型优化 |
|---|---|---|---|
| **Module** | `ModulePassManager` | 整个模块 | 全局优化、跨函数决策 |
| **CGSCC** | `CGSCCPassManager` | 调用图的强连通分量 | **函数内联** |
| **Function** | `FunctionPassManager` | 单个函数 | InstCombine、GVN、DCE |
| **Loop** | `LoopPassManager` | 单个循环 | 循环不变量外提、展开 |

（CGSCC = Call Graph Strongly Connected Component，调用图强连通分量，是做内联这类过程间优化的天然单位——一组互相递归调用的函数会落在同一个 SCC 里。）

问题来了：一个 `FunctionPass` 只会处理「一个函数」，但优化的入口是整个 `Module`。怎么让一个 Module 级的管线去驱动一个 Function Pass 跑遍每个函数？答案是**适配器（adaptor）**：

```cpp-norun
// 把一个 Function Pass 包装成一个 Module Pass：
//   它会遍历 Module 里每个 Function，各跑一遍内层的 Function Pass
ModulePassManager MPM;
MPM.addPass(createModuleToFunctionPassAdaptor(DCEPass()));
//          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^  适配器：Module → Function
```

`createModuleToFunctionPassAdaptor`（`PassManager.h`）返回的 `ModuleToFunctionPassAdaptor` 本身是个合法的 Module Pass（又是「PassManager/Adaptor 也是 Pass」的好处），它的 `run(Module&, ...)` 内部就是一个 for 循环，对模块里每个函数调用内层 Function Pass 的 `run`。同族的适配器还有：

- `createModuleToPostOrderCGSCCPassAdaptor` —— Module → CGSCC。
- `createCGSCCToFunctionPassAdaptor` —— CGSCC → Function。
- `createFunctionToLoopPassAdaptor` —— Function → Loop。

一层层套起来，就构成了真实优化管线的嵌套结构，比如 `module(cgscc(inline, function(instcombine, loop(licm))))`。第 6 章会看到 `PassBuilder` 正是这样用适配器把 O2 管线搭出来的。

<div class="keypoint">
<strong>适配器让「层级」和「Pass 复用」两不误</strong>
因为「PassManager 和 Adaptor 本身都是 Pass」，四个层级能像积木一样任意嵌套：一个 Function Pass 写一次，既能被 Module 管线通过适配器驱动、也能嵌在 CGSCC 内联管线里复用。<strong>你写 Pass 时只需关心「我作用在哪一层」，跨层的驱动交给适配器</strong>——这正是新 PM 用组合（composition）取代继承（inheritance）换来的灵活性。
</div>

## 动手：一个新 PM 的极简模拟器

把上面的机制浓缩成一个能在浏览器里跑的小模型：一个 `AnalysisManager` 负责惰性计算 + 缓存 + 按 `PreservedAnalyses` 失效；两个变换 Pass，一个诚实保留 CFG、一个大改一切。点 **▶ 运行**，看缓存何时命中、何时被作废重算。

```python
# 一个极简的"新 PassManager"模型：惰性分析缓存 + 按保留集失效

class AnalysisManager:
    def __init__(self):
        self.cache = {}          # 分析名 -> 结果
        self.compute_count = {}  # 统计每个分析被"现算"了几次

    def get_result(self, name):
        # 惰性：缓存没有才现算（对应 getResult 的 cache-miss 分支）
        if name not in self.cache:
            self.compute_count[name] = self.compute_count.get(name, 0) + 1
            print(f"    [现算] {name}（缓存未命中）")
            self.cache[name] = f"{name}-result"
        else:
            print(f"    [命中] {name}（用缓存）")
        return self.cache[name]

    def invalidate(self, preserved):
        # preserved 是"保留集"：不在里面的分析统统作废
        if preserved == "all":
            return
        for name in list(self.cache):
            if name not in preserved:
                print(f"    [失效] {name}（不在保留集里，删除）")
                del self.cache[name]

def instcombine(am):
    # 变换 Pass：查询支配树做优化，改了指令但没动 CFG
    print("  运行 InstCombine：")
    am.get_result("DominatorTree")
    return {"DominatorTree", "LoopInfo"}   # 收据：CFG 相关的都保留

def dce(am):
    print("  运行 DCE：")
    am.get_result("DominatorTree")
    return {"DominatorTree", "LoopInfo"}   # 同样保留 CFG

def simplifycfg(am):
    # 变换 Pass：合并/删除基本块，CFG 变了！
    print("  运行 SimplifyCFG：")
    am.get_result("DominatorTree")
    return "none"                          # 收据：全作废

am = AnalysisManager()
pipeline = [("InstCombine", instcombine), ("DCE", dce),
            ("SimplifyCFG", simplifycfg), ("InstCombine", instcombine)]

for i, (name, fn) in enumerate(pipeline, 1):
    print(f"[{i}] {name}")
    preserved = fn(am)
    am.invalidate(preserved)

print("\n各分析被现算的次数：", am.compute_count)
```

```
[2] DCE

[3] SimplifyCFG
  运行 SimplifyCFG：
    [命中] DominatorTree（用缓存）
    [失效] DominatorTree（不在保留集里，删除）
[4] InstCombine

各分析被现算的次数： {'DominatorTree': 2}
```

看清楚这条时间线：`DominatorTree` 第 1 步现算一次，第 2、3 步都命中缓存**不重算**——因为 InstCombine 和 DCE 都诚实地保留了 CFG；直到第 3 步 SimplifyCFG 返回 `none()` 把它作废，第 4 步才不得不重算。整份管线里支配树只算了 **2 次**而不是 4 次。请动手改：把 `simplifycfg` 的收据从 `"none"` 改成 `{"DominatorTree", "LoopInfo"}`（谎称保留了 CFG），你会看到第 4 步变成缓存命中、`DominatorTree` 只算 1 次——**但这在真实编译器里就是那个隐蔽 bug**：后续 Pass 拿到的是一棵和已被改动的 CFG 不符的支配树。这就是上面「收据写错了会怎样」那段的可运行版。

## 小结与下一步

你现在理解了新 PassManager 的设计哲学：

- Pass **没有基类**，只有一个「有 `PreservedAnalyses run(IRUnit&, AnalysisManager&)` 方法」的契约；共性代码靠 `PassInfoMixin` 这类 **CRTP 混入**提供，框架用模板做类型擦除。用组合与值语义取代继承。
- 三个核心概念：**变换 Pass**（改 IR）、**分析 Pass**（算情报、由 AnalysisManager 缓存）、**`PreservedAnalyses`**（变换跑完交回的收据，声明保留了什么）。
- `PassManager::run` 从 `PreservedAnalyses::all()` 出发，逐个 Pass 跑、逐个立即 `invalidate`、用 `intersect` 汇总。
- `AnalysisManager` **惰性计算 + 缓存 + 精确失效**（每个分析自己判断是否还有效，依赖靠 `Invalidator` 传递）——这是新 PM 比旧 PM 快的根源。
- 四层 IR 单位（Module / CGSCC / Function / Loop）用**适配器**（`createModuleToFunctionPassAdaptor` 等）任意嵌套，因为「PassManager / Adaptor 本身也是 Pass」。

下一章我们从「读框架」转到「用框架」：亲手写一个 Function Pass，编译成一个**插件**（`llvmGetPassPluginInfo` + `registerPipelineParsingCallback`），用 `opt -load-pass-plugin=... -passes=你的pass` 挂进管线跑起来——把这一章的每个抽象概念落到能编译、能运行的代码上。

---

**这一章的要点**

- 新 PM 里 Pass **无基类**：任何有 `PreservedAnalyses run(IRUnitT&, AnalysisManager<IRUnitT>&)` 的类都是 Pass；`PassInfoMixin` / `RequiredPassInfoMixin` / `OptionalPassInfoMixin` 只是 CRTP 混入的样板。核心头文件 `llvm/include/llvm/IR/PassManager.h`。
- **`PreservedAnalyses`** 是变换 Pass 交回的收据：`all()`（啥都没坏）/ `none()`（全作废）/ `preserveSet<CFGAnalyses>()`（保留某类）。它是承诺，框架不核对，写错会导致隐蔽 bug。
- **`AnalysisManager`** 惰性算分析（`getResult` 未命中即计算）、按 `AnalysisKey` 缓存、按收据精确失效（`invalidate`，含依赖传递）。
- `PassManager::run` 逐 Pass 跑 → 立即 `invalidate` → `intersect` 汇总；起点是 `all()`。
- 四层单位 **Module / CGSCC / Function / Loop**，靠 `createXToYPassAdaptor` 适配器嵌套；`ModuleToFunctionPassAdaptor` 等本身都是 Pass。
- 最小真实范例：`DCEPass`（`llvm/lib/Transforms/Scalar/DCE.cpp`），一屏代码演示了全部要点。
