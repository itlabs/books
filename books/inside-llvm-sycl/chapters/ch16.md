# 第 16 章 运行时（一）：queue / context / device / handler 对象模型

## 这一章想让你带走什么

<div class="goal-box">
<p>第 13 章你知道了 <code>queue</code>/<code>context</code>/<code>device</code> 这些类是 pImpl 句柄——公开壳里只有一个指向 <code>*_impl</code> 的指针。这一章翻到指针指向的另一侧：这些 <code>*_impl</code> 真身里装了什么、它们彼此怎么引用、谁拥有谁。然后走通那条最核心的路径——<strong><code>q.submit([](handler&amp; h){ h.parallel_for(...); })</code> 从公开 API 一路到向设备发起启动，中间经过哪些对象、每一跳做什么</strong>。</p>
<p>读完你会有一张 SYCL 运行时的<strong>对象关系图</strong>：<code>queue_impl</code> 持有 UR 队列句柄、引用一个 <code>context_impl</code> 和一个 <code>device_impl</code>；<code>platform_impl</code> 拥有所有 <code>device_impl</code>；<code>handler</code> 是每次提交时临时建的"命令组构建器"，把你的 lambda 打包成一个<strong>命令组（command group, CG）</strong>交给调度器。这张图是第 17 章（调度器与依赖图）和第 18 章（UR）的地基。</p>
</div>

## pImpl 的另一侧：impl 类住在哪、装了什么

第 13 章讲过公开类和 impl 类的目录分工。这里把五个核心类的两侧对应关系钉死（都在 `intel/llvm` 里）：

| 公开类 | 公开头 | impl 类 | impl 位置 |
|---|---|---|---|
| `sycl::queue` | `sycl/include/sycl/queue.hpp` | `detail::queue_impl` | `sycl/source/detail/queue_impl.{hpp,cpp}` |
| `sycl::context` | `sycl/include/sycl/context.hpp` | `detail::context_impl` | `sycl/source/detail/context_impl.{hpp,cpp}` |
| `sycl::device` | `sycl/include/sycl/device.hpp` | `detail::device_impl` | `sycl/source/detail/device_impl.{hpp,cpp}` |
| `sycl::platform` | `sycl/include/sycl/platform.hpp` | `detail::platform_impl` | `sycl/source/detail/platform_impl.{hpp,cpp}` |
| `sycl::handler` | `sycl/include/sycl/handler.hpp` | `detail::handler_impl` | `sycl/source/detail/handler_impl.hpp` |

每个 impl 类的核心，是它持有的那个 **UR 句柄**（第 18 章的主角）加上对其他 impl 的引用：

- **`queue_impl`**（`queue_impl.hpp:71`）：`ur_queue_handle_t MQueue`（真正的 UR 队列）、`device_impl &MDevice`（引用，不拥有）、`std::shared_ptr<context_impl> MContext`（共享拥有上下文）、`async_handler MAsyncHandler`、`property_list MPropList`、`bool MIsInorder`，以及一组依赖追踪状态（记录"这个队列上最后一个事件是什么"，供排序用）。
- **`context_impl`**（`context_impl.hpp:43`）：`ur_context_handle_t MContext`、`std::vector<device_impl *> MDevices`（上下文里的设备）、`platform_impl &MPlatform`、以及一个重量级成员 `KernelProgramCache MKernelProgramCache`——**编译好的 program/kernel 缓存**（同一个 kernel 不必每次都重新构建）。
- **`device_impl`**（`device_impl.hpp:139`）：`ur_device_handle_t MDevice`、`platform_impl &MPlatform`。
- **`platform_impl`**（`platform_impl.hpp:34`）：`ur_platform_handle_t MPlatform`、`adapter_impl *MAdapter`，以及关键的 `std::vector<std::unique_ptr<device_impl>> MDevices`——**平台拥有它的所有设备**。

<div class="note">
<strong>不是所有 pImpl 都是 <code>shared_ptr</code>——<code>device</code> 是个例外</strong>
第 13 章说公开类"持有 <code>shared_ptr&lt;X_impl&gt;</code>"，这是主流形态（<code>queue</code>/<code>context</code>/<code>platform</code> 都是）。但 <code>device</code> 是<strong>非拥有的裸指针</strong>：<code>device.hpp:414</code> 是 <code>detail::device_impl *impl = nullptr;</code>。原因就在上面——<code>device_impl</code> 由它的父 <code>platform_impl</code> 用 <code>unique_ptr</code> 拥有（<code>platform_impl.hpp:215</code>），所以用户手里的 <code>device</code> 只是<strong>借用</strong>一个指针，生命周期挂靠在 platform 上。这是 pImpl 的一个合理变体：当实现对象已有明确的所有者时，句柄退化成"借用指针"而非"共享所有权"。<code>handler</code> 则更特殊——热路径上它的 <code>handler_impl</code> 是<strong>栈上</strong>建的、按引用传入（见下文），公开壳里的 <code>impl</code> 指向那个栈对象。
</div>

## 引用语义与 impl_utils：句柄怎么和 impl 互相转换

第 13 章说这些类是引用语义句柄——`operator==` 直接比较 `impl` 指针（如 `queue.hpp:381` 的 `return impl == RHS.impl;`）。运行时内部经常要在"公开句柄"和"impl 指针"之间来回转换，这套工具在 `sycl/include/sycl/detail/impl_utils.hpp`：

- `detail::getSyclObjImpl(obj)` —— 从公开对象取出它的 `impl`。它被特意做成一个**函数对象**（而非普通自由函数），目的是把它挡在 ADL（参数依赖查找）之外，防止这个内部 API 泄漏到用户命名空间（这是为修 intel/llvm issue #20820 加的防线）。
- `detail::createSyclObjFromImpl<T>(impl)` —— 反向：从一个 impl 造出公开对象。若目标的 `impl` 是 `shared_ptr`，它用 `shared_from_this()` 构造；若是裸指针（`device` 那种），直接转发。

这就是为什么每个公开类都要 `friend sycl::detail::ImplUtils;`——让这套工具能碰到私有的 `impl` 成员。你读运行时源码时会反复看到 `getSyclObjImpl(q)`，意思就是"把这个 `queue` 句柄拆成它的 `queue_impl`"。

## 核心路径：q.submit 一路走到发起启动

现在走通那条脊柱。用户写：

```cpp-norun
q.submit([&](sycl::handler &h) {
  sycl::accessor a{buf, h};
  h.parallel_for<class VecAdd>(range<1>{N}, [=](id<1> i){ a[i] += 1; });
});
```

它在运行时里的旅程（每一跳都标了源码位置）：

1. **`queue::submit`**（`queue.hpp:433`）—— 公开模板入口。它把你的 command-group 函数（CGF）**类型擦除**成 `detail::type_erased_cgfo_ty`，转发到 `submit_with_event`。
2. **`queue::submit_with_event_impl`**（`queue.cpp:172`）—— 公开壳的薄转发层，调 `impl->submit_with_event(...)`。这就是第 13 章那条"头→source→detail/impl"三跳的中间站。
3. **`queue_impl::submit_impl`**（`queue_impl.cpp:310`）—— 真正干活的地方：
   - 在**栈上**建一个 `handler_impl HandlerImplVal(*this, ...)` 和 `handler Handler(HandlerImplVal)`（这就是前面说的"handler 是临时的命令组构建器"）。
   - **执行你的 CGF**：`CGF(Handler)`——你的 lambda 在这里被调用，`h.parallel_for<VecAdd>(...)` 把 kernel 信息记录进 handler。
   - 读出 `handler_impl` 里记录的命令组类型 `MCGType`，按队列是否 in-order 分派到不同的 finalize helper。
4. **`handler::parallel_for`**（`handler.hpp:1294`）—— 在 CGF 执行期间被调用，最终经 `StoreLambda` 把你的 kernel 函子捕获下来、`setType(CGType::Kernel)` 标记这是个 kernel 命令组，把 NDRange、参数等记进 `handler_impl->MKernelData`。
5. **`handler::finalize`**（`handler.cpp:444`）—— 提交的终点：把记录的信息打包成一个具体的**命令组对象** `detail::CGExecKernel`（`cg.hpp:144`，装着 NDRange、host kernel、参数列表、kernel 名等），然后交给调度器：`Scheduler::getInstance().addCG(std::move(CommandGroup), *Queue, ...)`（`handler.cpp:845`）。

第 17 章从 `addCG` 接着讲（调度器怎么把这个 CG 挂进依赖图）；第 18 章讲最终 `urEnqueueKernelLaunchWithArgsExp` 那一下真正的 UR 调用。这一章你只要记住这条脊柱的**形状**：`submit` → `queue_impl::submit_impl` →（建 handler、跑你的 CGF）→ `handler::finalize` →（打包成 CG）→ `Scheduler::addCG`。

<div class="keypoint">
<strong>submit 的脊柱：queue 壳转发 → queue_impl 建 handler 跑 CGF → finalize 打包 CG → 交调度器</strong>
<strong><code>q.submit(CGF)</code>（<code>queue.hpp:433</code>）→ <code>queue::submit_with_event_impl</code>（<code>queue.cpp:172</code>，壳转发）→ <code>queue_impl::submit_impl</code>（<code>queue_impl.cpp:310</code>）在栈上建 <code>handler</code>、执行你的 CGF；你的 <code>h.parallel_for</code>（<code>handler.hpp:1294</code>）把 kernel 记进 <code>handler_impl</code>；<code>handler::finalize</code>（<code>handler.cpp:444</code>）打包成 <code>CGExecKernel</code>（<code>cg.hpp:144</code>）交给 <code>Scheduler::addCG</code>（<code>handler.cpp:845</code>）。</strong>handler 是"每次提交临时建的命令组构建器"——这是理解 SYCL 提交模型的关键。
</div>

## in-order vs out-of-order：一个 property 决定排序

`queue_impl::submit_impl` 里那个"按是否 in-order 分派"的分支，背后是 SYCL 的队列排序语义。`queue::is_in_order()`（`queue.cpp:242`）的实现极简单：

```cpp-norun
bool queue::is_in_order() const {
  return has_property<property::queue::in_order>();
}
```

也就是说，**队列是否顺序执行，完全由构造时传入的 `property_list` 里有没有 `property::queue::in_order` 决定**。所有 queue 构造函数都接受一个尾随的 `const property_list &PropList = {}`，`queue_impl` 把它存成 `MPropList`，创建 UR 队列时翻译成 UR 的队列标志。

这个区别对第 17 章很重要：**out-of-order 队列**里，两个提交之间要不要排序，取决于它们有没有数据依赖（调度器算依赖图）；**in-order 队列**则强制每个提交依赖前一个，`queue_impl` 用它的依赖追踪状态（`MDefaultGraphDeps.LastEventPtr` 之类）把它们串成一条链。同一套 `submit_impl`，靠这个 property 走出两种排序行为。

## 每个 impl 都通向 UR：adapter.call

这一章反复出现的 `ur_*_handle_t` 不是摆设——impl 类通过一个**适配器（adapter）**把这些句柄驱动起来。模式统一是：从 impl 拿到 `adapter_impl`（`getAdapter()`），调用模板方法 `Adapter.call<UrApiKind::xxx>(...)`。几个确凿的调用点：

- 建队列：`queue_impl` 里 `getAdapter().call<UrApiKind::urQueueCreate>(Context, Device, &Properties, ...)`（`queue_impl.hpp:543`）。
- 建上下文：`context_impl.cpp:55` 的 `getAdapter().call<UrApiKind::urContextCreate>(...)`。
- 查设备信息：`device_impl` 里遍地是 `call<UrApiKind::urDeviceGetInfo>`。

这就是运行时对象模型和第 18 章的接缝：**每个 SYCL 运行时对象，本质是一个 UR 句柄 + 一层 C++ 生命周期/缓存管理**。`queue`/`context`/`device` 的"创建"最终都落成一次 `urXxxCreate`。第 18 章会把 UR 这层单独展开——它对接 Level Zero / CUDA / OpenCL 的方式，正是第 10 章"接口 + 插件"母题的又一次化身。

## 动手：模拟 submit 的脊柱与 in-order 排序

把这一章的核心——「submit 建 handler 跑 CGF、finalize 打包成 CG 交调度器，in-order 与否决定排序」——用一个浏览器里能跑的小模型演出来。点 **▶ 运行**。

```python
# 模拟 SYCL submit 脊柱：queue → handler(命令组构建器) → finalize 打包 CG → 调度器

class CG:                       # 命令组：finalize 的产物
    def __init__(self, kind, kernel, ndrange, deps):
        self.kind, self.kernel, self.ndrange, self.deps = kind, kernel, ndrange, deps
    def __repr__(self):
        return f"CG(kernel={self.kernel}, nd={self.ndrange}, deps={self.deps})"

class Handler:                  # 每次 submit 临时建的命令组构建器（栈上）
    def __init__(self):
        self.cgtype = None; self.kernel = None; self.ndrange = None
    def parallel_for(self, kernel_name, ndrange):
        self.cgtype = "Kernel"                    # setType(CGType::Kernel)
        self.kernel = kernel_name; self.ndrange = ndrange   # 记进 MKernelData
    def finalize(self, deps):                     # 打包成 CGExecKernel
        return CG(self.cgtype, self.kernel, self.ndrange, deps)

class Scheduler:                # 第 17 章的主角，这里只收 CG
    def __init__(self): self.graph = []
    def addCG(self, cg):
        self.graph.append(cg)
        print(f"    [调度器] addCG → {cg}")
        return f"event#{len(self.graph)}"

class Queue:
    def __init__(self, in_order):
        self.in_order = in_order                  # 由 property::queue::in_order 决定
        self.sched = Scheduler()
        self.last_event = None                    # in-order 的依赖追踪
    def submit(self, cgf):
        h = Handler()                             # queue_impl::submit_impl 建 handler
        cgf(h)                                    # 执行用户 CGF：h.parallel_for(...)
        # in-order：强制依赖上一个提交；out-of-order：交给调度器算数据依赖（这里留空）
        deps = [self.last_event] if (self.in_order and self.last_event) else []
        cg = h.finalize(deps)                     # handler::finalize
        ev = self.sched.addCG(cg)                 # → Scheduler::addCG
        self.last_event = ev
        return ev

for mode in [True, False]:
    print(f"=== {'in-order' if mode else 'out-of-order'} 队列 ===")
    q = Queue(in_order=mode)
    q.submit(lambda h: h.parallel_for("VecAdd", (1024,)))
    q.submit(lambda h: h.parallel_for("VecMul", (1024,)))
    print()
```

运行结果：

```
=== in-order 队列 ===
    [调度器] addCG → CG(kernel=VecAdd, nd=(1024,), deps=[])
    [调度器] addCG → CG(kernel=VecMul, nd=(1024,), deps=['event#1'])

=== out-of-order 队列 ===
    [调度器] addCG → CG(kernel=VecAdd, nd=(1024,), deps=[])
    [调度器] addCG → CG(kernel=VecMul, nd=(1024,), deps=[])
```

看清两件事：① **每次 `submit` 都新建一个 `Handler`**、跑你的 CGF、`finalize` 成一个 `CG` 交给调度器——这就是那条脊柱的形状。② **in-order 队列**里第二个提交自动带上了 `deps=['event#1']`（依赖前一个），**out-of-order** 则两个都 `deps=[]`（要不要排序留给第 17 章的调度器按数据依赖决定）。一个 `in_order` 布尔就分出了两种排序行为，正如真实代码里那个 `property::queue::in_order`。请动手改：把 `submit` 里 out-of-order 分支也加上依赖（模拟两个 kernel 共享同一 buffer），看它如何变成一条链——这正是第 17 章调度器要自动完成的工作。

## 小结与下一步

你现在有了 SYCL 运行时的对象关系图和提交脊柱：

- **impl 类**（`queue_impl`/`context_impl`/`device_impl`/`platform_impl`，都在 `sycl/source/detail/`）各持有一个 **UR 句柄**（`ur_*_handle_t`）加对其他 impl 的引用；`context_impl` 还扛着 `KernelProgramCache`。
- **所有权**：`platform_impl` 用 `unique_ptr` **拥有**所有 `device_impl`——所以公开的 `device` 句柄是**非拥有裸指针**（pImpl 的合理变体），而 `queue`/`context`/`platform` 是 `shared_ptr`。`handler` 的 impl 在热路径上是**栈上**临时对象。
- 句柄↔impl 的转换靠 `impl_utils.hpp` 的 `getSyclObjImpl`（做成函数对象挡 ADL）/ `createSyclObjFromImpl`。
- **提交脊柱**：`queue::submit`（`queue.hpp:433`）→ `submit_with_event_impl`（`queue.cpp:172`，壳转发）→ `queue_impl::submit_impl`（`queue_impl.cpp:310`，建 handler 跑 CGF）→ `handler::parallel_for`（`handler.hpp:1294`，记 kernel）→ `handler::finalize`（`handler.cpp:444`，打包成 `CGExecKernel`）→ `Scheduler::addCG`（`handler.cpp:845`）。
- **in-order vs out-of-order** 由 `property::queue::in_order` 决定（`queue.cpp:242`）：in-order 强制串链，out-of-order 交调度器按数据依赖排。
- 每个 impl 经 `adapter.call<UrApiKind::xxx>(...)` 通向 UR（如 `urQueueCreate`/`urContextCreate`）——接缝在第 18 章展开。

下一章接着 `Scheduler::addCG` 往下讲：命令组进了调度器之后，**调度器怎么把它挂进依赖图、怎么算出命令之间的读写依赖（RAW/WAR/WAW）、怎么在需要时插入内存拷贝、最后按拓扑序 enqueue 到设备**。第 17 章是 SYCL 运行时最精巧的一块——命令组、调度器与依赖图。
