# 第 10 章 运行时 liboffload 与 plugins-nextgen：kernel 怎么上设备

## 这一章想让你带走什么

<div class="goal-box">
<p>第 9 章把设备镜像打进了胖二进制，程序启动时那段注册代码把镜像登记到了 offload 运行时。这一章讲"最后一步"：<strong>运行时如何拿着这些镜像，把 kernel 真正加载到某块具体设备上、分配显存、拷数据、发起执行</strong>。主角是顶层 <code>offload/</code> 目录下的两块东西——<strong><code>liboffload</code></strong>（一层设备无关的统一 API）和 <strong><code>plugins-nextgen</code></strong>（针对 CUDA / AMDGPU / Level Zero / Host 各写一份的插件）。</p>
<p>你会看清一个贯穿本书的架构母题的又一次出现：<strong>用一层设备无关的抽象接口，把上层运行时和下层各家 GPU 的驱动 API 解耦</strong>——和第 1 章 IR 解耦前后端、第 8 章 SPIR-V 解耦编译器与驱动是同一种思路。理解了 offload 运行时的"接口 + 插件"结构，你就为第 18 章（SYCL 的 Unified Runtime）打好了底——它们是同一个模式在不同层的化身。</p>
</div>

## 问题：kernel 已在身边，可设备千差万别

程序跑起来后，情况是这样的：胖二进制里带着编好的设备镜像（SPIR-V / PTX / …），运行时也已经通过启动注册代码知道"我有哪些镜像"。现在用户代码要求"把这个 kernel 跑在 GPU 上"。运行时要完成的动作其实很固定：

1. **发现并选设备**：机器上有哪些设备（NVIDIA GPU？Intel GPU？就用 CPU？），选一个。
2. **加载镜像**：把对应目标的设备镜像加载进该设备、拿到可执行的 kernel 句柄。
3. **分配设备内存**：在设备上开显存，把输入数据从主机拷过去。
4. **发起执行**：按指定的线程组织（多少个 work-item）启动 kernel。
5. **取回结果并同步**：把输出拷回主机、等执行完成。

问题在于：**每家设备的驱动 API 完全不同**。NVIDIA 用 CUDA Driver API（`cuMemAlloc`、`cuLaunchKernel`…），AMD 用 HSA/HIP，Intel 用 Level Zero（`zeMemAllocDevice`、`zeCommandListAppendLaunchKernel`…）。上面那五步动作**语义相同、API 迥异**。如果让上层运行时直接去调各家 API，代码会被 `#ifdef CUDA / #ifdef LEVEL_ZERO` 撕成碎片。

解法你已经在第 1、8 章见过两次：**插一层设备无关的接口**。上层运行时只对着这层统一接口编程，每家设备各提供一个**插件**去实现这层接口、内部翻译成自己的驱动调用。

## offload/ 目录：运行时住在哪

先纠正一个常见的旧印象：这套 offload 运行时**不在** `openmp/libomptarget` 下——在本书代码库（intel/llvm `sycl` 分支）里，它已经被搬到**顶层的 `offload/` 目录**。结构大致是：

```
offload/
├── liboffload/        # 新的设备无关统一 API（本章主角之一）
│   ├── API/           # 用 TableGen (.td) 描述 API，构建时生成头文件
│   └── src/           # OffloadImpl.cpp 等实现
├── plugins-nextgen/   # 各设备插件（本章主角之二）
│   ├── cuda/
│   ├── amdgpu/
│   ├── level_zero/
│   ├── host/          # CPU 上"假装设备"跑，便于调试
│   └── common/        # 插件共享的基础设施
├── libomptarget/      # OpenMP target offload 运行时（历史主体）
└── include/           # omptarget.h、PluginManager.h、rtl.h 等共享头
```

这里有个历史层次要理清：`libomptarget` 是**为 OpenMP `target` offload** 服务的运行时，是这块代码最初的主体；`liboffload` 则是一层**更通用的新 API**——它的 README 说得明确：「liboffload 提供一个适合实现多种 offload 语言运行时的抽象层，而不只是 OpenMP」。也就是说，offload 社区正在把"设备无关的运行时能力"从 OpenMP 专用中提炼成一个通用层。而两者都共享同一套 **`plugins-nextgen`** 插件——插件层是设备无关的公共资产。

<div class="note">
<strong>"nextgen"是相对旧插件而言</strong>
<code>plugins-nextgen</code> 的"下一代"是相对早期 libomptarget 那套旧插件接口而言的重写：它把"一个设备插件要实现什么"抽象成一组清晰的 C++ 基类（在 <code>plugins-nextgen/common/</code> 里），各设备（cuda/amdgpu/level_zero/host）继承并实现之。这种"公共基类定义契约、各设备子类实现"的组织，让新增设备、共享通用逻辑（内存管理、镜像加载框架）都更干净——本质还是"接口与实现分离"。
</div>

## liboffload 的 API 是「生成」出来的

`liboffload` 有个值得一看的工程细节：它的**公共 API 头文件不是手写的，而是从 TableGen 描述生成的**。在 `offload/liboffload/API/` 下是一堆 `.td` 文件——`OffloadAPI.td`、`Device.td`、`Memory.td`、`Kernel.td`、`Queue.td`、`Program.td` 等，每个描述一类 API（设备管理、内存、kernel、队列……）。构建时用 offload 版的 TableGen 把它们生成成：

- `OffloadAPI.h` —— 公共 API 头（`-gen-api`）。
- `OffloadPrint.hpp` —— 打印/追踪辅助（`-gen-print-header`）。

所以你在源码树里**找不到** `OffloadAPI.h`——它只在构建目录里出现。这和第 7 章后端用 TableGen 描述芯片、生成 C++ 是同一种手法：**用声明式描述批量生成本该重复的样板代码**。好处是 API 的形状（函数签名、参数、错误码、枚举）单一来源、一致、可同时生成文档和追踪代码。

<div class="warn">
<strong>在源码里 grep 不到 <code>olMemAlloc</code> 别惊讶</strong>
既然 API 是生成的，你直接搜某个 API 函数名可能只搜到它在 <code>.td</code> 里的<strong>声明</strong>，而非 <code>.h</code>。想看完整 API，要么去<strong>构建目录</strong>找生成的 <code>OffloadAPI.h</code>，要么读 <code>offload/liboffload/API/*.td</code> 这些"源头"。本章正文里的 <code>olMemAlloc</code>/<code>olLaunchKernel</code> 是<strong>示意名</strong>，帮助理解 API 的形状；请以你构建出的 <code>OffloadAPI.h</code> 为准来查确切拼写。启用 <code>OFFLOAD_TRACE</code> 环境变量可以在运行时打印每次 API 调用，是观察这层接口最直接的方式。
</div>

## 一个插件要实现什么

站在 `plugins-nextgen/common/` 定义的契约角度，一个设备插件大致要提供这几类能力（对应本章开头那五步）：

- **平台/设备管理**：初始化驱动、枚举设备、查询设备属性。cuda 插件里就是 `cuInit`、`cuDeviceGet`、`cuDeviceGetAttribute` 之类。
- **镜像加载**：接收一份设备镜像（对 NVIDIA 是 PTX/cubin），加载成设备上的模块，从中取出 kernel 函数句柄。cuda 用 `cuModuleLoadData` + `cuModuleGetFunction`。
- **内存管理**：设备内存分配/释放、主机↔设备数据拷贝。cuda 用 `cuMemAlloc`/`cuMemcpyHtoD`/`cuMemcpyDtoH`。
- **kernel 启动**：按给定的线程网格（grid/block 维度）和参数发起执行。cuda 用 `cuLaunchKernel`。
- **同步与事件**：等待队列/流完成、事件依赖。cuda 用 stream/event API。

上层运行时通过 `liboffload` 统一 API 说"分配 1MB、把这段数据拷过去、用这些参数启动这个 kernel、等它跑完"，插件层把每一句翻译成对应设备驱动的真实调用。**`host` 插件**是个特别有用的存在：它不对接任何 GPU，而是在 CPU 上"假装"成一个设备把 kernel 跑起来——这让你在没有 GPU 的机器上也能调试整条 offload 链路，是排查"到底是运行时的问题还是驱动的问题"的利器。

<div class="keypoint">
<strong>offload 运行时 = 统一 API（liboffload）+ 各设备插件（plugins-nextgen）</strong>
<strong>上层运行时对着 <code>liboffload</code> 的设备无关 API（从 <code>offload/liboffload/API/*.td</code> 生成）编程；<code>plugins-nextgen</code> 下每家设备（cuda/amdgpu/level_zero/host）实现同一套契约，把"选设备、加载镜像、分配内存、启动 kernel、同步"翻译成各自的驱动调用。</strong>加一款设备只需加一个插件，上层不改——这就是第 1 章 N+M 思路在运行时层的再现。记住这个"接口+插件"骨架，第 18 章的 SYCL Unified Runtime 你会一眼认出：它是同一个模式，只是服务于 SYCL。
</div>

## liboffload 与 SYCL 的关系：别混淆两层

这里要划清一条界线，避免和第 18 章打架。本书语境里有**两套**"设备无关运行时 API"：

- **`liboffload`（本章）**：offload 项目的通用运行时 API，脱胎于 OpenMP offload，目标是服务多种 offload 语言运行时。
- **Unified Runtime（UR，第 18 章）**：SYCL/DPC++ 运行时对接各后端（Level Zero / CUDA / OpenCL）用的抽象层。

两者**理念完全一致**（都是"统一接口 + 后端适配器"），也都对接 CUDA/Level Zero 这些驱动，但它们是**不同的层、服务不同的上层**：SYCL 运行时（第 16–17 章）走的是 UR。你现在只需记住："接口+插件/适配器"这个模式在 LLVM 异构栈里出现了不止一次；`liboffload` 是 offload 侧的那个，UR 是 SYCL 侧的那个。第 18 章会专门讲 UR，届时可回头对照本章。

## 亲手观察：让运行时把动作打印出来

`liboffload` 内建了追踪能力。跑一个 offload 程序时打开追踪，就能看到运行时对着统一 API 依次做了哪些动作：

```bash
# 打开 liboffload 追踪，观察运行时的每一步 API 调用
OFFLOAD_TRACE=1 ./my_offload_app
```

你会看到类似"发现设备 → 加载镜像 → 分配内存 → 拷数据 → 启动 kernel → 同步"的一串调用日志。对 CUDA 后端，还能用 NVIDIA 自己的工具交叉验证插件到底调了哪些底层 API：

```bash
# 用 nsys 追踪底层 CUDA Driver API 调用，和 OFFLOAD_TRACE 对照
nsys profile --trace=cuda ./my_offload_app
```

把两份日志对照着看，你能亲眼确认那层翻译：`liboffload` 的一次 `launch`，在 cuda 插件里变成了一次 `cuLaunchKernel`。这种"上层统一动作 ↔ 下层驱动调用"的对照，是理解和调试运行时最有效的手段——也再次体现"每一层都可观察"这个贯穿本书的性质。

## 动手：模拟统一 API 分派到各设备插件

把这一章的机制——「上层对统一 API 编程 → 运行时按选中的设备分派到对应插件 → 插件翻译成自己的驱动调用」——浓缩成一个浏览器里能跑的小模型。点 **▶ 运行**。

```python
# 模拟 offload 运行时：统一 API + 各设备插件（plugins-nextgen）

# ---- 每个插件实现同一套契约：加载镜像 / 分配内存 / 启动 kernel ----
# 用一个基类表达"契约"，各设备子类把统一动作翻译成自己的驱动调用
class Plugin:
    name = "base"
    def load_image(self, image): raise NotImplementedError
    def alloc(self, nbytes):     raise NotImplementedError
    def launch(self, kernel, grid): raise NotImplementedError

class CudaPlugin(Plugin):
    name = "cuda"
    def load_image(self, image): return f"cuModuleLoadData({image})"
    def alloc(self, n):          return f"cuMemAlloc({n})"
    def launch(self, k, grid):   return f"cuLaunchKernel({k}, grid={grid})"

class LevelZeroPlugin(Plugin):
    name = "level_zero"
    def load_image(self, image): return f"zeModuleCreate({image})"
    def alloc(self, n):          return f"zeMemAllocDevice({n})"
    def launch(self, k, grid):   return f"zeCommandListAppendLaunchKernel({k}, grid={grid})"

class HostPlugin(Plugin):        # 在 CPU 上"假装"设备，便于调试
    name = "host"
    def load_image(self, image): return f"dlopen({image})  # 当普通函数加载"
    def alloc(self, n):          return f"malloc({n})"
    def launch(self, k, grid):   return f"直接在 CPU 线程池跑 {k}, grid={grid}"

# ---- 运行时：注册插件，按选中的设备分派（上层完全不碰具体驱动 API）----
PLUGINS = {p.name: p for p in [CudaPlugin(), LevelZeroPlugin(), HostPlugin()]}

def run_kernel_on(device, image, kernel, nbytes, grid):
    plugin = PLUGINS.get(device)
    if not plugin:
        print(f"[运行时] 没有 {device} 的插件"); return
    print(f"[运行时] 选中设备 {device}，分派给 {plugin.name} 插件：")
    print(f"    加载镜像 → {plugin.load_image(image)}")
    print(f"    分配内存 → {plugin.alloc(nbytes)}")
    print(f"    启动kernel→ {plugin.launch(kernel, grid)}")
    print()

# 上层运行时说的都是同一句"抽象的话"，只是 device 不同
for dev, img in [("cuda", "VecAdd.ptx"),
                 ("level_zero", "VecAdd.spv"),
                 ("host", "VecAdd.so")]:
    run_kernel_on(dev, img, kernel="VecAdd", nbytes=1024, grid=(256, 1, 1))
```

运行结果：

```
[运行时] 选中设备 cuda，分派给 cuda 插件：
    加载镜像 → cuModuleLoadData(VecAdd.ptx)
    分配内存 → cuMemAlloc(1024)
    启动kernel→ cuLaunchKernel(VecAdd, grid=(256, 1, 1))

[运行时] 选中设备 level_zero，分派给 level_zero 插件：
    加载镜像 → zeModuleCreate(VecAdd.spv)
    分配内存 → zeMemAllocDevice(1024)
    启动kernel→ zeCommandListAppendLaunchKernel(VecAdd, grid=(256, 1, 1))

[运行时] 选中设备 host，分派给 host 插件：
    加载镜像 → dlopen(VecAdd.so)  # 当普通函数加载
    分配内存 → malloc(1024)
    启动kernel→ 直接在 CPU 线程池跑 VecAdd, grid=(256, 1, 1)
```

看清这层分派：上层 `run_kernel_on` 说的永远是同一句抽象的话（"加载镜像、分配内存、启动 kernel"），运行时按 `device` 分派给对应插件，**每个插件把同一动作翻译成自己驱动 API 的真实调用**——CUDA 的 `cuLaunchKernel`、Level Zero 的 `zeCommandListAppendLaunchKernel`、host 的线程池。上层代码对这些差异一无所知，这正是"统一 API + 插件"解耦的价值。请动手改：加一个 `class AmdgpuPlugin(Plugin)` 实现那三个方法（用 HSA 风格的名字），注册进 `PLUGINS`，再 `run_kernel_on("amdgpu", ...)`——你会发现**上层的 `run_kernel_on` 一个字都不用改**就支持了新设备。这就是"加一款设备 = 加一个插件"。

## 小结与下一步

你现在补齐了异构编译"运行期"的关键一环：

- 运行时要完成的动作很固定（**发现设备 → 加载镜像 → 分配内存/拷数据 → 启动 kernel → 同步**），但各家设备的驱动 API 迥异。
- 解法是**统一接口 + 各设备插件**：**`liboffload`**（顶层 `offload/liboffload/`）提供设备无关的 API，**`plugins-nextgen`**（`offload/plugins-nextgen/`，含 `cuda`/`amdgpu`/`level_zero`/`host`/`common`）各实现一份。
- `liboffload` 的 API **由 TableGen（`API/*.td`）生成**（`OffloadAPI.h` 在构建目录，不在源码树）——和第 7 章后端描述芯片同法。`libomptarget` 是 OpenMP 专用运行时，`liboffload` 是提炼出的通用层，二者共享 `plugins-nextgen`。
- **`host` 插件**在 CPU 上模拟设备，是无 GPU 调试的利器；`OFFLOAD_TRACE=1` 可观察运行时的每步 API 调用。
- 这个"接口+插件"骨架和第 1 章 IR、第 8 章 SPIR-V 是同一种解耦思路，也预告了第 18 章 SYCL 的 **Unified Runtime**（别把 `liboffload` 和 UR 混为一层——理念相同，服务不同上层）。

**第二部分的通用异构基础（offload 工具链 + 运行时）到此打通。** 下一章聚焦一个具体后端家族——**NVIDIA**：设备端 IR 如何经 **NVPTX 后端**（第 7 章那个特殊后端）降级成 **PTX**、**libclc** 如何为设备端提供数学函数与内建实现、以及 SYCL 怎么通过 `-fsycl-targets=nvptx64-nvidia-cuda` 选中这条 NVIDIA 路径。之后我们就带着完整的异构栈，进入第三部分的 SYCL 实现。

---

**这一章的要点**

- offload 运行时固定动作：**发现/选设备 → 加载镜像 → 分配内存与拷数据 → 启动 kernel → 同步**；难点是各设备驱动 API 完全不同。
- 解法 = **统一接口 + 插件**：**`liboffload`**（`offload/liboffload/`，设备无关 API）+ **`plugins-nextgen`**（`offload/plugins-nextgen/`，子目录 `cuda`/`amdgpu`/`level_zero`/`host`/`common`）。
- 运行时在**顶层 `offload/`**（不在旧的 `openmp/libomptarget`）；`libomptarget` 是 OpenMP 专用运行时，`liboffload` 是面向多语言的通用层，二者共用插件。
- `liboffload` 公共 API **由 TableGen 生成**（`offload/liboffload/API/*.td` → 构建目录的 `OffloadAPI.h`/`OffloadPrint.hpp`）；源码树里 grep 不到生成后的头。
- 一个插件实现平台/设备管理、镜像加载、内存管理、kernel 启动、同步；**`host` 插件**在 CPU 模拟设备便于调试；`OFFLOAD_TRACE=1` 追踪 API 调用。
- "接口+插件"是贯穿本书的解耦母题（IR/SPIR-V/UR）；**`liboffload` ≠ Unified Runtime（第 18 章）**，理念同、层次异。
