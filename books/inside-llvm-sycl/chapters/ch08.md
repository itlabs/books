# 第 8 章 SPIR-V：异构世界的可移植中间表示与 llvm-spirv

## 这一章想让你带走什么

<div class="goal-box">
<p>第 1 章说 LLVM IR 是一层「统一的中间表示」，让前后端解耦。但 LLVM IR 有个天生的局限：它<strong>不是一个稳定的、跨厂商的标准</strong>——它的位码格式随版本变、它假设了很多 LLVM 自己的约定。当你要把一段 kernel 交给「别人家的驱动」（Intel、AMD 的 GPU 驱动）去最终编译时，需要一种<strong>由标准组织定义、版本稳定、厂商中立</strong>的中间表示。这就是 <strong>SPIR-V</strong>。</p>
<p>这一章讲清楚：SPIR-V 是什么、为什么异构计算需要它、它和 LLVM IR 是什么关系；以及 intel/llvm 里那个把 LLVM IR 和 SPIR-V 双向互译的关键工具 <strong><code>llvm-spirv</code></strong>（源码在 <code>llvm-spirv/</code>）。这是理解后面 SYCL 设备端编译（第 12、14 章）「一份源码怎么跑到 Intel/AMD GPU 上」的必备一环。</p>
</div>

## 为什么 LLVM IR 还不够

回忆第 1 章那条链：源码 → IR →（优化）→ IR → 机器码。对于 CPU，这条链在**你自己的编译器进程里**一次走完，最后直接吐出 x86 机器码，没问题。

但异构计算（把 kernel 送上 GPU）多了一个残酷的现实：**最后一步「翻成 GPU 原生指令」不是由你的编译器做的，而是由 GPU 厂商的驱动做的**。Intel 的 GPU 驱动、AMD 的 GPU 驱动各有自己闭源的编译器，把某种中间表示翻成各自的原生 ISA（Intel 的 Gen ISA、AMD 的 GCN/RDNA 指令）。你的编译器和厂商驱动之间，需要一份**双方都认的合同**——一种交接用的中间表示。

LLVM IR 能当这份合同吗？不能，原因很实在：

- **不稳定**：LLVM 位码格式几乎每个大版本都会变，没有向后兼容承诺。驱动没法保证认得你这版 LLVM 吐的 IR。
- **不中立**：LLVM IR 是 LLVM 项目的内部表示，夹带大量 LLVM 特有的假设。让每个 GPU 厂商都去内嵌一个特定版本的 LLVM、并跟着升级，不现实。
- **信息不对口**：LLVM IR 是为「编译 CPU 代码」设计的，缺少描述并行 kernel 所需的一等概念（工作项、地址空间语义、执行模型、能力声明等）。

于是 Khronos Group（定义 OpenGL、Vulkan、OpenCL、SYCL 的那个标准组织）定义了 **SPIR-V（Standard Portable Intermediate Representation）**：一种**由标准组织维护、版本化且稳定、厂商中立、专为并行/图形 kernel 设计**的二进制中间表示。它就是那份合同。

## SPIR-V 与 LLVM IR：像，但不是一回事

SPIR-V 和 LLVM IR 长得很像——都是 **SSA 形式**、都有类型、都由指令（在 SPIR-V 里叫 op，如 `OpFAdd`、`OpLoad`、`OpFunctionCall`）组成。这不是巧合：SPIR-V 的设计本来就借鉴了 LLVM IR，所以两者之间做**近乎一一对应的翻译**是可行的。这是 `llvm-spirv` 能存在的前提。

但它们服务于不同目的，有几个关键差异：

| | LLVM IR | SPIR-V |
|---|---|---|
| **定义者** | LLVM 项目 | Khronos 标准组织 |
| **稳定性** | 随版本变，无兼容承诺 | 版本化、稳定 |
| **面向** | 编译器内部、CPU 为主 | 跨厂商交接、并行/图形 kernel |
| **能力模型** | 无 | 有 **Capability** 声明（如 `Kernel`、`Addresses`） |
| **典型形态** | `.ll` / `.bc` | `.spv` 二进制 |

那个 **Capability（能力）** 机制值得单独提：SPIR-V 模块开头会声明自己用到了哪些能力，比如计算 kernel 会声明 `Kernel` 能力。`llvm-spirv/` 这个翻译器目前主要处理的就是声明了 `Kernel` 能力的**计算味 SPIR-V**（OpenCL/SYCL 用的那种），而不是图形着色器那种。这解释了它为什么和 SYCL、OpenCL 绑得这么紧。

<div class="note">
<strong>两个「SPIR-V」，别搞混</strong>
intel/llvm 里其实有<strong>两套</strong>和 SPIR-V 相关的东西：<br>
① <code>llvm-spirv/</code> —— 就是本章主角，Khronos 官方的 <strong>SPIRV-LLVM-Translator</strong>，一个<strong>库 + 命令行工具</strong>，做 LLVM IR ↔ SPIR-V 的<strong>双向</strong>翻译。SYCL 工具链现在默认走它。<br>
② <code>llvm/lib/Target/SPIRV/</code> —— 一个 LLVM <strong>后端</strong>（和 X86、NVPTX 平级，见第 1 章），把 LLVM IR <strong>单向</strong>降级成 SPIR-V。<br>
两者目标相似、历史不同。这一章讲 ①（translator），因为它是当前 SYCL 设备端编译链路上的实际组件。
</div>

## llvm-spirv：双向翻译器

`llvm-spirv/` 的 README 把它自己称为「LLVM/SPIR-V 双向翻译器（Bi-Directional Translator）」。**双向**是重点：

- **正向（LLVM IR → SPIR-V）**：编译时用。SYCL 把设备端代码编成 LLVM IR、优化完，再用它翻成 `.spv`，嵌进最终的 fat binary（第 9 章）交给运行时。
- **反向（SPIR-V → LLVM IR）**：消费别人给的 SPIR-V 时用。比如运行时拿到一段 SPIR-V，想再做点基于 LLVM 的处理，可以先翻回 IR。

顶层结构（都在 `llvm-spirv/` 下）：

- `include/` —— 公开头文件，核心是 `LLVMSPIRVLib.h`（库的入口 API）。
- `lib/SPIRV/` —— 翻译器主体。里面有：
  - `libSPIRV/` —— SPIR-V 模块的内存模型（SPIR-V 自己那套「Module / Function / 指令」对象）。
  - `Mangler/` —— 名字修饰处理。
  - 一批预处理/降级 Pass，比如 `OCLToSPIRV.cpp`（把 OpenCL 味的 IR 调整成适合翻成 SPIR-V 的形状）、`SPIRVLowerBool.cpp`、`SPIRVLowerLLVMIntrinsic.cpp`。
- `tools/llvm-spirv/` —— `llvm-spirv` 命令行工具本体。

命令行用起来很直观：

```bash
# 正向：把设备端的 LLVM 位码翻成 SPIR-V 二进制
llvm-spirv device.bc -o device.spv

# 反向：把 SPIR-V 翻回 LLVM 位码
llvm-spirv -r device.spv -o back.bc

# 顺便：把 SPIR-V 二进制反汇编成可读文本（调试常用）
llvm-spirv --to-text device.spv -o device.spt
```

<div class="keypoint">
<strong>SPIR-V 在这本书里的位置</strong>
把第 1 章那条链接到异构场景，就得到 SYCL 设备端的真实链路：<strong>设备代码 →（clang 前端）→ LLVM IR →（opt 优化）→ LLVM IR →（llvm-spirv）→ SPIR-V →（厂商驱动）→ GPU 原生指令</strong>。<code>llvm-spirv</code> 就是倒数第二个箭头。而 NVIDIA 是个例外——它不吃 SPIR-V，走的是 PTX（第 11 章）。「Intel/AMD 走 SPIR-V、NVIDIA 走 PTX」这条分叉，你在第 12 章 SYCL 全景里会再见到。
</div>

## 亲手看一眼：从 IR 到 SPIR-V 的对应

SPIR-V 是二进制，但它的文本反汇编（`--to-text`）能让你看清它和 LLVM IR 的对应。还是第 1 章那个加法，设备端版本的 LLVM IR 片段大概是：

```llvm-norun
; LLVM IR：一个把两数相加的 kernel 片段
define spir_kernel void @add(float %a, float %b, float* %out) {
entry:
  %sum = fadd float %a, %b       ; 浮点加法
  store float %sum, float* %out  ; 写回结果
  ret void
}
```

经 `llvm-spirv` 翻成 SPIR-V 后，反汇编出来的结构（大幅简化、示意）：

```spirv-norun
; SPIR-V（--to-text 反汇编，示意）
              OpCapability Kernel          ; 声明：我是计算 kernel
              OpCapability Addresses
   %void   =  OpTypeVoid
   %float  =  OpTypeFloat 32
   %fnty   =  OpTypeFunction %void %float %float ...
   %add    =  OpFunction %void None %fnty       ; 对应 define @add
   %a      =  OpFunctionParameter %float
   %b      =  OpFunctionParameter %float
              OpLabel                            ; 对应 entry: 基本块
   %sum    =  OpFAdd %float %a %b                ; 对应 fadd
              OpStore %out %sum                  ; 对应 store
              OpReturn                           ; 对应 ret void
              OpFunctionEnd
```

一眼就能连线：`define @add` ↔ `OpFunction`、`fadd` ↔ `OpFAdd`、`store` ↔ `OpStore`、`ret void` ↔ `OpReturn`、`entry:` 基本块 ↔ `OpLabel`。多出来的是开头那两句 `OpCapability`——这正是 SPIR-V 比 LLVM IR 「多」的那层信息：**显式声明自己需要哪些能力**，好让驱动提前判断「我这块硬件支持不支持」。因为结构如此对应，双向翻译才成为可能。

## 动手：一个「能力协商」模拟器

SPIR-V 的 `OpCapability` 不只是装饰——它让「模块声明需求、驱动检查支持」这个协商能在**真正编译到硬件之前**发生。下面这个浏览器里能跑的小模型，模拟三块声明了不同能力的 SPIR-V 模块，去三种支持不同能力集的设备上「落地」，看谁能跑、谁被拒。点 **▶ 运行**。

```python
# 模拟 SPIR-V 的 OpCapability 协商：模块声明需求，驱动检查是否满足

# 每个设备（驱动）支持的能力集合
devices = {
    "Intel GPU":  {"Kernel", "Addresses", "Float16", "Int64"},
    "AMD GPU":    {"Kernel", "Addresses", "Int64"},
    "老旧 GPU":    {"Kernel", "Addresses"},
}

# 三个 SPIR-V 模块，各自开头 OpCapability 声明的需求
modules = {
    "普通加法 kernel":   {"Kernel", "Addresses"},
    "用了半精度 kernel": {"Kernel", "Addresses", "Float16"},
    "用了 64 位整数":    {"Kernel", "Addresses", "Int64"},
}

def can_run(need, have):
    # 模块需要的每一项能力，设备都得支持
    return need <= have   # 集合的子集判断

for mod_name, need in modules.items():
    print(f"\n模块《{mod_name}》声明能力 {sorted(need)}")
    for dev_name, have in devices.items():
        if can_run(need, have):
            print(f"  ✓ {dev_name}：能力满足，可交给驱动编译执行")
        else:
            missing = need - have
            print(f"  ✗ {dev_name}：缺少 {sorted(missing)}，编译前即被拒绝")
```

运行结果：

```
模块《普通加法 kernel》声明能力 ['Addresses', 'Kernel']
  ✓ Intel GPU：能力满足，可交给驱动编译执行
  ✓ AMD GPU：能力满足，可交给驱动编译执行
  ✓ 老旧 GPU：能力满足，可交给驱动编译执行

模块《用了半精度 kernel》声明能力 ['Addresses', 'Float16', 'Kernel']
  ✓ Intel GPU：能力满足，可交给驱动编译执行
  ✗ AMD GPU：缺少 ['Float16']，编译前即被拒绝
  ✗ 老旧 GPU：缺少 ['Float16']，编译前即被拒绝

模块《用了 64 位整数》声明能力 ['Addresses', 'Int64', 'Kernel']
  ✓ Intel GPU：能力满足，可交给驱动编译执行
  ✓ AMD GPU：能力满足，可交给驱动编译执行
  ✗ 老旧 GPU：缺少 ['Int64']，编译前即被拒绝
```

这就是 `OpCapability` 的价值：一个可移植的中间表示不能假设「所有硬件都一样」，于是它让每个模块**显式声明自己用到的能力**，运行时/驱动据此做一次快速的「合同检查」——不满足就在真正编译前干净地拒绝，而不是等编到一半崩溃。这也解释了为什么 SPIR-V 适合当异构世界的交接格式，而只为单一 CPU 设计的 LLVM IR 没有这层机制。请动手改：给「老旧 GPU」的能力集加上 `"Int64"`，看第三个模块在它上面从 ✗ 变 ✓。

## 小结与下一步

你现在理解了异构编译里那层「多出来的」中间表示：

- LLVM IR 虽好，但**不稳定、不中立、缺并行语义**，不适合当编译器与 GPU 厂商驱动之间的交接合同。
- **SPIR-V** 是 Khronos 定义的中间表示：**标准、版本稳定、厂商中立、为并行 kernel 设计**，并有 **`OpCapability` 能力协商**机制。它就是那份合同。
- SPIR-V 与 LLVM IR **结构相似**（都是 SSA、有类型、有指令），所以能近乎一一对应地翻译——这是翻译器存在的前提。
- **`llvm-spirv`**（`llvm-spirv/`，即 Khronos 的 SPIRV-LLVM-Translator）做 LLVM IR ↔ SPIR-V 的**双向**翻译，是当前 SYCL 设备端编译链路上的实际组件；注意别和 `llvm/lib/Target/SPIRV/`（一个单向后端）搞混。

下一章我们往「链路的下游」走一步：设备端代码编成了 SPIR-V，可它得和主机端代码**打包进同一个可执行文件**里，运行时才能在启动 kernel 时把它取出来交给驱动。这套「把主机对象和多个设备镜像捆成一个胖二进制」的工具链——`clang-offload-bundler`、`clang-linker-wrapper` 等——就是第 9 章的主题。

---

**这一章的要点**

- 异构编译的最后一步「翻成 GPU 原生指令」由**厂商驱动**完成，所以编译器和驱动之间需要一份**厂商中立、版本稳定**的中间表示合同。
- LLVM IR 当不了这份合同：不稳定、不中立、缺并行/能力语义。**SPIR-V**（Khronos 标准）可以。
- SPIR-V 也是 **SSA + 类型 + 指令**（`OpFAdd`/`OpLoad`/…），和 LLVM IR 结构对应；额外有 **`OpCapability`** 声明，支持「编译前的能力协商」。
- **`llvm-spirv`**（`llvm-spirv/`，SPIRV-LLVM-Translator）双向互译 LLVM IR ↔ SPIR-V；命令 `llvm-spirv x.bc -o x.spv` / `-r` 反向 / `--to-text` 反汇编。
- 记住链路分叉：**Intel/AMD 走 SPIR-V，NVIDIA 走 PTX**（第 11 章）。这条分叉在第 12 章 SYCL 全景里会再出现。
