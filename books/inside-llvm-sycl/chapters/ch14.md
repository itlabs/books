# 第 14 章 设备端编译：integration header/footer 与 kernel 命名

## 这一章想让你带走什么

<div class="goal-box">
<p>第 12 章讲过 SYCL "两趟编译"里那座关键的桥——<strong>集成头文件（integration header）</strong>：设备趟编译时前端顺手生成一个 <code>.h</code>，主机趟再把它 <code>#include</code> 进来，于是主机端代码就"认识"了设备端 kernel 的名字、参数个数与顺序。这一章把这座桥拆开看：这个 <code>.h</code> 里到底写了什么、它的孪生兄弟<strong>集成 footer</strong> 又管什么、以及那个贯穿始终的谜题——<strong>kernel 的名字（那串 <code>_ZTS…</code>）究竟是怎么算出来的</strong>。</p>
<p>读完你会知道：主机端 <code>q.submit([](handler&amp; h){ h.parallel_for&lt;MyKernel&gt;(...); })</code> 里的 <code>MyKernel</code> 类型，如何在编译期被 <strong>Itanium C++ 名字修饰（name mangling）</strong>变成一个字符串常量，写进集成头的 <code>KernelInfo</code> 特化，从而让主机端"喊出"和设备端镜像里完全一致的名字。这是 SYCL "单一源码"魔法的技术核心——理解它，你就理解了主机与设备两趟编译如何"对暗号"。</p>
</div>

## 回顾：为什么需要这座桥

先把第 12 章的困境重述清楚。SYCL 是"单一源码"：主机代码和设备 kernel 写在同一个 `.cpp` 里。但它们**分两趟编译**——设备趟只编 kernel、产出设备镜像（SPIR-V/PTX）；主机趟编 CPU 代码、产出可执行文件里驱动运行时的那部分。

问题来了：主机趟要发起"启动名为 X 的 kernel"，它必须知道**这个 kernel 叫什么名字、有几个参数、每个参数多大、是什么类型**——可这些信息是设备趟才算出来的。两趟编译各跑各的，怎么共享？

SYCL 的答案是**让设备趟把这些信息写成一个 C++ 头文件**，主机趟包含它。这就是**集成头文件**。它是一份由编译器自动生成的、连接两趟编译的"契约"。

## 打开集成头：里面写了什么

集成头由前端在设备趟生成，核心逻辑在 `clang/lib/Sema/SemaSYCL.cpp` 里的 `SYCLIntegrationHeader` 类（声明在 `clang/include/clang/Sema/SemaSYCL.h:53`）。它发射（emit）出的内容大致是这几块：

- **`kernel_names[]`** —— 每个 kernel 的修饰名字符串（就是那串 `_ZTS…`）。
- **`kernel_args_sizes[]`** —— 每个 kernel 的参数总数 / 各参数信息条数。
- **`kernel_signatures[]`** —— 一个 `kernel_param_desc_t` 数组，逐个描述每个参数：**种类**（是普通标量？accessor？还是别的）、**大小**、**在参数列表里的偏移**。
- **`KernelInfo<T>` / `KernelInfoData<char...>` 特化** —— 把上面这些信息绑定到具体的 kernel 类型 `T` 上，让主机端能用 `T` 反查到它的名字和参数布局。

这些类型的定义不在生成的头里，而在一个**随 SYCL 头发布**的固定头文件 `sycl/include/sycl/detail/kernel_desc.hpp`（还记得第 13 章那个"随头发布的 `detail/`"吗？就是它）。关键定义：

- `enum class kernel_param_kind_t`（`kernel_desc.hpp:54-66`）—— 参数种类：`kind_accessor`、`kind_std_layout`（普通标量/结构体）、`kind_pointer`、`kind_specialization_constants_buffer` 等。
- `struct kernel_param_desc_t`（`kernel_desc.hpp:69-80`）—— 一个参数的描述：`kind` + `info`（大小或类型信息）+ `offset`。
- `template <class KernelNameType> struct KernelInfo`（`kernel_desc.hpp:98-113`）—— 按 kernel 类型索引的元信息：`getName()`、`getNumParams()`、`getParamDesc(i)` 等。

<div class="note">
<strong>集成头是编译产物，不是你写的代码</strong>
你在源码树里<strong>找不到</strong>某个 kernel 的集成头——它是设备趟编译时临时生成的。想亲眼看它，加编译选项让 clang 把它落盘：<code>-fsycl-int-header=out.h</code>（见后文），编出来的 <code>out.h</code> 就是这份契约的真容。里面全是 <code>KernelInfo</code> 特化、一个个 <code>kernel_param_desc_t</code> 的字面量数组，以及一串串 <code>_ZTS…</code> 名字。它长得不像"人写的代码"，因为它本就是给编译器和运行时对暗号用的。
</div>

## kernel 的名字从哪来：_ZTS 与名字修饰

SYCL 里你这样命名一个 kernel：

```cpp-norun
q.submit([&](sycl::handler &h) {
  sycl::accessor a{buf, h};
  h.parallel_for<class VecAdd>(range<1>{N}, [=](sycl::id<1> i) {
    a[i] = a[i] + 1;
  });
});
```

那个 `class VecAdd` 是一个**类型**，被当作 `parallel_for` 的模板参数（"kernel name"）。SYCL 规范要求 kernel name 全局唯一，用它来把主机端的启动请求和设备端的 kernel 对应起来。那么这个类型怎么变成设备镜像里和集成头里那串字符串的？

答案是 **Itanium C++ ABI 的名字修饰（name mangling）**，具体走的是 RTTI 名字那条路。前端计算 kernel 名的入口是 `SemaSYCL.cpp:1282` 的 `constructKernelName`，它最终调用到 `clang/lib/AST/ItaniumMangle.cpp:7610` 的 `mangleCXXRTTIName`——也就是给类型生成 `typeid` 名字的那套逻辑。产物就带着标志性的 **`_ZTS` 前缀**（`_Z` 是 Itanium 修饰的起手式，`TS` 表示 "type name string"，即 RTTI 的类型名）。所以 `class VecAdd` 会变成形如 `_ZTS6VecAdd` 的字符串（`6` 是名字长度）。

这解释了一个你可能踩过的坑：**为什么 kernel name 类型必须在全局可见、不能是局部 lambda 的匿名类型**——因为要能被稳定地修饰成一个跨两趟编译一致的名字。

<div class="keypoint">
<strong>kernel 名 = 用 RTTI 名字修饰把 kernel-name 类型变成字符串</strong>
<strong>你给 <code>parallel_for&lt;T&gt;</code> 的类型参数 <code>T</code>，被前端 <code>constructKernelName</code>（<code>SemaSYCL.cpp:1282</code>）经 <code>mangleCXXRTTIName</code>（<code>ItaniumMangle.cpp:7610</code>）修饰成一个带 <code>_ZTS</code> 前缀的字符串。</strong>这个字符串同时写进①设备镜像里 kernel 函数的符号名、②集成头里的 <code>KernelInfo&lt;T&gt;::getName()</code>。两趟编译各自独立地对同一个类型跑同一套修饰算法，得到<strong>逐字节相同</strong>的名字——这就是主机端和设备端"对上暗号"的根基。
</div>

## SYCL 前端如何"看见"一个 kernel

集成头里的信息不是凭空来的——前端要先在设备趟**识别出哪些是 kernel、逐个参数分析它们**。这套机制值得一看，因为它是第 22 章"需要编译器配合的扩展"的基础。

一切从一个内部属性开始：`sycl_kernel`（定义在 `clang/include/clang/Basic/Attr.td:1738`）。SYCL 头里的 `parallel_for`/`single_task` 最终会调用一个被 `__attribute__((sycl_kernel))` 标注的模板函数，前端看到这个属性，就知道"这里要生成一个 OpenCL/SPIR-V kernel 入口"。核心动作在 `SemaSYCL.cpp:5372` 的 `ConstructOpenCLKernel`：它为用户的 kernel lambda 生成一个真正的 kernel 函数（`spir_kernel` 调用约定，见第 15 章）。

生成过程用了一个优雅的**访问者（visitor）框架**——前端遍历 kernel lambda 的每个捕获参数，三个协作的 visitor 各干一摊：

- **`SyclKernelDeclCreator`** —— 构造 kernel 函数的形参列表（accessor 会被拆成指针+范围等多个实参）。
- **`SyclKernelBodyCreator`** —— 生成 kernel 函数体：把传进来的实参重新组装回 lambda 的捕获、再调用原 lambda。
- **`SyclKernelIntHeaderCreator`** —— **一边遍历一边往集成头里写** `kernel_param_desc_t`：每遇到一个 accessor 参数就记 `kind_accessor`、每个标量记 `kind_std_layout`。

<div class="note">
<strong>accessor 是怎么"拆开"过设备边界的</strong>
一个 <code>accessor</code> 在主机端是个复杂对象（带指针、范围、偏移）。它不能整个塞进 kernel 参数——设备端 kernel 收到的是一个裸指针加几个整数。这套拆解由 kernel 对象里约定的特殊成员方法完成：<code>__init</code>（把传入的指针/范围<strong>装配</strong>成设备端的 accessor）和 <code>__finalize</code>（收尾，如 stream 的刷新）。visitor 遇到有 <code>__init</code> 的类型，就按它的形参列表把参数摊平。这就是为什么集成头里一个 accessor 会对应<strong>多条</strong> <code>kernel_param_desc_t</code>——你在生成的头里能直接数出来。
</div>

## 集成 footer：头的孪生兄弟

设备趟其实生成**两个**文件：集成头（integration header）和**集成 footer（integration footer）**。footer 的类是 `SYCLIntegrationFooter`（`SemaSYCL.h:228`），发射逻辑在 `SemaSYCL.cpp:7821`（头在 `:7206`）。

两者的差别在**怎么被包含进主机趟**，这也揭示了它们各自的职责：

- **集成头**：主机趟编译时，driver 用 `-include-internal-header` 把它**强制包含在源文件最前面**（driver 逻辑在 `clang/lib/Driver/ToolChains/Clang.cpp:5814`）。因为它要提供的是 kernel 元信息，得在用户代码之前就位。
- **集成 footer**：用 `-include-internal-footer` 强制包含在源文件**最后**（`Clang.cpp:5829`）。因为它要注册的东西——**spec constant（特化常量）、`device_global`、host pipe**——必须在用户已经定义完这些全局对象**之后**才能引用它们、生成注册用的 shim 函数。

换句话说：**头在前，负责"声明 kernel 长什么样"；footer 在后，负责"把用户定义的设备端全局对象登记到运行时"**。footer 里那些 shim 函数把 `device_global` 变量、spec constant 之类和它们的字符串标识符绑定，运行时据此在设备镜像里找到对应的符号。

控制这两个文件的编译选项（定义在 `clang/include/clang/Options/Options.td`）：

| 选项 | 作用 | 位置 |
|---|---|---|
| `-fsycl-int-header=<file>` | 指定集成头输出路径 | `Options.td:9454` |
| `-fsycl-int-footer=<file>` | 指定集成 footer 输出路径 | `Options.td:9459` |
| `-include-internal-header` | 主机趟：头强制包含在**最前** | `Clang.cpp:5814` |
| `-include-internal-footer` | 主机趟：footer 强制包含在**最后** | `Clang.cpp:5829` |

这套 emit 由 `clang/lib/Sema/Sema.cpp:1297`（头）/`:1294`（footer）在编译结束时触发。

## 亲手把集成头逼出来看

这套机制平时藏在 driver 背后，但你可以让 clang 把集成头落盘、亲眼读它：

```bash
# 只跑设备趟，把集成头写到 vecadd-int.h
clang++ -fsycl -fsycl-device-only \
    -fsycl-int-header=vecadd-int.h vecadd.cpp -c -o /dev/null

# 看它生成了什么——找 kernel 名字和参数描述
grep -n '_ZTS\|kernel_param_desc_t\|KernelInfo' vecadd-int.h
```

你会看到一串 `KernelInfo<...>` 特化、`getName()` 返回的 `_ZTS…` 字符串、以及描述每个参数的 `kernel_param_desc_t` 字面量数组。把它和设备镜像里的符号对照（`llvm-spirv` 反汇编或 `nm` 看 PTX），会发现 kernel 名**逐字节相同**——这就是"对暗号"成功的直接证据。这也是调试"kernel 找不到 / 参数对不上"这类问题的第一现场：十有八九是集成头里的名字或参数布局和设备镜像不一致。

## 动手：模拟设备趟如何生成集成头、主机趟如何读它

把这一章的核心——「设备趟对 kernel-name 类型做名字修饰、把参数摊平写进集成头；主机趟读它拿到同一个名字去启动」——用一个浏览器里能跑的小模型演出来。点 **▶ 运行**。

```python
# 模拟集成头：设备趟"写"、主机趟"读"，两边靠名字修饰对上暗号

# ---- 名字修饰：模拟 mangleCXXRTTIName 的 _ZTS 前缀（_ZTS + 长度 + 名字）----
def mangle_rtti_name(type_name):
    return f"_ZTS{len(type_name)}{type_name}"

# ---- 设备趟：识别 kernel，把参数"摊平"，生成集成头（一个 dict 模拟 KernelInfo 表）----
def device_pass(kernel_name_type, captures):
    # captures: [(名字, 种类)]，accessor 会摊成 指针+范围 多个参数
    param_descs = []
    for cap_name, cap_kind in captures:
        if cap_kind == "accessor":
            # __init 把 accessor 拆成 指针 + 访问范围（示意：2 条）
            param_descs.append({"kind": "kind_accessor", "of": cap_name, "info": "ptr"})
            param_descs.append({"kind": "kind_std_layout", "of": cap_name, "info": "range"})
        else:
            param_descs.append({"kind": "kind_std_layout", "of": cap_name, "info": cap_kind})
    return {                        # 这就是集成头里的 KernelInfo<T> 特化
        mangle_rtti_name(kernel_name_type): {
            "name": mangle_rtti_name(kernel_name_type),
            "num_params": len(param_descs),
            "params": param_descs,
        }
    }

# ---- 主机趟：拿到同一个 kernel-name 类型，独立算出同一个修饰名，去集成头里查 ----
def host_pass_launch(kernel_name_type, integration_header):
    key = mangle_rtti_name(kernel_name_type)     # 主机趟独立跑同一套修饰
    info = integration_header.get(key)
    if not info:
        print(f"[主机趟] 找不到 kernel {key} —— 暗号没对上！"); return
    print(f"[主机趟] 要启动 kernel-name 类型 = {kernel_name_type}")
    print(f"         独立算出修饰名 = {key}")
    print(f"         在集成头里查到：{info['num_params']} 个参数")
    for i, p in enumerate(info["params"]):
        print(f"           arg[{i}] {p['kind']:<16} (来自 {p['of']}, {p['info']})")
    print(f"         → 发起 urEnqueueKernelLaunch(\"{info['name']}\", ...)")

# 设备趟编译这个 kernel：VecAdd，捕获一个 accessor a 和一个标量 n
int_header = device_pass("VecAdd", captures=[("a", "accessor"), ("n", "int")])
print("设备趟生成的集成头（KernelInfo 表）：")
for k, v in int_header.items():
    print(f"  {k} → {v['num_params']} params")
print()
# 主机趟：用同一个类型名去启动
host_pass_launch("VecAdd", int_header)
```

```
         → 发起 urEnqueueKernelLaunch("_ZTS6VecAdd", ...)
```

看清这条对称：设备趟和主机趟**各自独立**对 `"VecAdd"` 跑同一套 `mangle_rtti_name`，都得到 `_ZTS6VecAdd`——名字一致纯粹因为算法确定、输入相同，**不需要两趟之间直接通信**，中间只隔着一个集成头文件。还要注意那个 accessor `a` 在集成头里变成了**两条**参数描述（指针 + 范围），正是 `__init` 摊平的效果。请动手改：把 `captures` 里再加一个 `("b", "accessor")`，重新运行——你会看到参数条数从 3 涨到 5，主机趟也照样能读出来。这正是真实 SYCL 里加一个 accessor 参数时集成头的变化。

## 小结与下一步

这一章拆开了 SYCL "单一源码"魔法的技术核心：

- **集成头（integration header）** 是设备趟生成、主机趟包含的"契约"：它用 `KernelInfo<T>` 特化记录每个 kernel 的**名字**、**参数个数**、**每个参数的种类/大小/偏移**（`kernel_param_desc_t`，定义在 `sycl/include/sycl/detail/kernel_desc.hpp`）。
- **kernel 名靠 RTTI 名字修饰算出**：`parallel_for<T>` 的类型 `T` 经 `constructKernelName`（`SemaSYCL.cpp:1282`）→ `mangleCXXRTTIName`（`ItaniumMangle.cpp:7610`）变成带 `_ZTS` 前缀的字符串。两趟编译独立跑同一套修饰得到逐字节相同的名字——这是"对暗号"的根基。
- 前端靠内部属性 **`sycl_kernel`**（`Attr.td:1738`）识别 kernel，`ConstructOpenCLKernel`（`SemaSYCL.cpp:5372`）用三个 **visitor**（Decl/Body/IntHeader Creator）生成 kernel 函数并同步写集成头；accessor 经 `__init`/`__finalize` **摊平**成多个参数。
- **集成 footer** 是孪生兄弟：用 `-include-internal-footer` 包含在主机源**最后**，负责注册用户定义的 **spec constant / `device_global` / host pipe**（头在前"声明 kernel"，footer 在后"登记全局对象"）。
- 编译选项：`-fsycl-int-header=`/`-fsycl-int-footer=` 指定输出，driver 用 `-include-internal-header`/`-include-internal-footer` 强制包含。

下一章继续沿设备趟往下走：`[[sycl::reqd_work_group_size(…)]]` 这类 **SYCL 属性（attributes）**是怎么从源码里的标注，一路经 `Attr.td` 定义、`SemaSYCLDeclAttr.cpp` 处理、最后在 CodeGen 里落成 kernel 函数上的 **LLVM metadata**（`!reqd_work_group_size` 等）的——把"设备端编译"这一章的另一半补齐。

**这一章的要点**

- **集成头**是设备趟生成、主机趟强制包含的两趟编译"契约"：`KernelInfo<T>` 特化 + `kernel_param_desc_t[]` 记录每个 kernel 的名字、参数个数与各参数的种类/大小/偏移（类型定义在 `sycl/include/sycl/detail/kernel_desc.hpp`）。
- **kernel 名 = RTTI 名字修饰**：`constructKernelName`（`SemaSYCL.cpp:1282`）→ `mangleCXXRTTIName`（`ItaniumMangle.cpp:7610`），产出 `_ZTS…`；两趟独立修饰得逐字节相同的名字，故主机端能喊出与设备镜像一致的名字。
- 前端识别 kernel 靠 `sycl_kernel` 属性（`Attr.td:1738`）；`ConstructOpenCLKernel`（`SemaSYCL.cpp:5372`）+ 三个 visitor 生成 kernel 函数并写集成头；accessor 经 `__init`/`__finalize` 摊平为多条参数描述。

- 选项：`-fsycl-int-header=`（`Options.td:9454`）/ `-fsycl-int-footer=`（`:9459`）；driver 的 `-include-internal-header`（`Clang.cpp:5814`）/ `-include-internal-footer`（`:5829`）。可用 `-fsycl-device-only -fsycl-int-header=out.h` 把集成头逼出来亲眼看。
