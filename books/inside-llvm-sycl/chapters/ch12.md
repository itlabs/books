# 第 12 章 SYCL 全景：单一源码、两趟编译与规范到实现的映射

## 这一章想让你带走什么

<div class="goal-box">
<p>前 11 章你已经手握 LLVM 的全套地基：IR、Pass、SPIR-V、offload 工具链、PTX。现在把它们用起来。SYCL 是一个由 Khronos 定义的<strong>规范</strong>，它承诺「用标准 C++、写一份 <code>.cpp</code>、同时含主机端和设备端代码，编译到多家 GPU」。这一章要回答一个编译器工程师真正关心的问题：<strong>这个「魔法」在 intel/llvm 里到底是怎么实现的？</strong></p>
<p>核心答案是<strong>两趟编译（two-pass compilation）</strong>：同一份源码被 clang 编译两次——一趟当「设备编译器」抽出 kernel 编成设备镜像，一趟当「主机编译器」编出普通 CPU 程序，两趟之间靠一个<strong>集成头文件（integration header）</strong>对接。读完你会有一张贯穿全书第三部分的地图：知道「规范 vs 实现」的分工，知道设备端那趟走的正是第 8、11 章那条 IR → SPIR-V/PTX 的链路，也知道后面第 13–19 章各自钉在这张图的哪个位置。</p>
</div>

## 先分清两件事：规范 vs 实现

初学 SYCL 最容易犯的错，是把「SYCL」当成一个东西。其实有两层：

- **SYCL 规范**：Khronos 发布的一份文档（如 SYCL 2020），规定了 `sycl::queue`、`sycl::buffer`、`parallel_for` 这些类和函数**应该有什么行为**。它只说「是什么」，不说「怎么实现」。任何人都可以照着它写一个实现。
- **SYCL 实现**：真正把规范变成能用的编译器 + 运行时库的代码。**DPC++（Data Parallel C++）就是 Intel 的实现，源码就在你面前这个 `intel/llvm` 仓库里**。另一个知名实现是 AdaptiveCpp。

这本书讲的是**实现**——`intel/llvm` 里那套真实的头文件、编译器改动、运行时库。规范说「`queue::submit` 提交一个命令组」，我们要看的是 `sycl/source/detail/queue_impl.cpp` 里它**具体干了什么**。

<div class="note">
<strong>DPC++ ≈ Clang + SYCL 支持</strong>
DPC++ 不是从头写的新编译器。它就是把 <strong>SYCL 的语义支持加进了 Clang/LLVM</strong>：前端多认识几个属性（如 <code>sycl_kernel</code>）、多一趟设备编译逻辑（<code>clang/lib/Sema/SemaSYCL.cpp</code>），加上一整套运行时库（<code>sycl/</code>）。所以你前 11 章学的 LLVM 知识，在这里全部用得上——SYCL 不是另起炉灶，是<strong>盖在 LLVM 地基上的一层楼</strong>。
</div>

## 单一源码的难题：主机和设备代码混在一个文件里

SYCL 的卖点是「单一源码」：主机逻辑和设备 kernel 写在同一个 `.cpp` 里。看一个最小例子：

```sycl-norun
#include <sycl/sycl.hpp>
int main() {
    sycl::queue q;                       // ← 主机端代码
    int data[1] = {41};
    {
        sycl::buffer<int, 1> buf(data, sycl::range<1>(1));
        q.submit([&](sycl::handler& h) {
            sycl::accessor a(buf, h, sycl::read_write);
            h.single_task([=]() {        // ← 这个 lambda 是设备端代码！
                a[0] += 1;               //    要编译成 GPU kernel
            });
        });
    }                                    // ← buffer 析构，结果同步回 data
    // 这里 data[0] == 42
}
```

对编译器来说，这一个文件里藏着两种命运截然不同的代码：

- `main`、`queue`、`buffer` 的构造——这些是**主机代码**，要编成普通的 x86 程序，跑在 CPU 上。
- `single_task` 里那个 lambda `[=]() { a[0] += 1; }`——这是**设备代码**，要抽出来、编译成 GPU kernel（走第 8 章的 SPIR-V 或第 11 章的 PTX），最后由 GPU 执行。

一个标准 C++ 编译器只会把它当普通程序编。SYCL 实现的核心工作，就是**把这两种代码分开、各编各的，再让运行时在启动 kernel 时把它们重新对上**。这就引出了两趟编译。

## 两趟编译：同一份源码，编译器跑两遍

DPC++ 的做法是：驱动（driver）让 clang 把同一份 `.cpp` 编译**两趟**，各带不同的使命。

**设备编译趟（device compilation）**：定义了宏 `__SYCL_DEVICE_ONLY__`。这一趟的任务是：

1. **设备代码外提（outlining）**：从 kernel（那个 lambda）出发，找出所有它会用到的代码，把无关的主机代码全部丢掉。
2. **把 lambda 变成 kernel 函数**：SYCL 用一个带 `sycl_kernel` 属性（定义在 `clang/include/clang/Basic/Attr.td`）的模板函数做「入口」，前端在 Sema（`clang/lib/Sema/SemaSYCL.cpp`）里据此生成一个 OpenCL 风格的 kernel 入口——它声明捕获的 accessor、对每个 accessor 调 `__init(...)`、再调用 lambda 体。
3. **设备端诊断**：设备代码里不能用异常、虚函数等 GPU 上做不到的东西，这一趟负责报错。
4. **生成集成头文件**（下一节详述）。
5. 产出：设备端的 **LLVM IR**，随后走第 1、8、11 章那条链——`llvm-link` 合并、`llvm-spirv` 翻成 SPIR-V（Intel/AMD）或经 NVPTX 后端出 PTX（NVIDIA），最终嵌进胖二进制。

**主机编译趟（host compilation）**：不定义那个宏。这一趟就是把文件当普通 C++ 程序编，产出跑在 CPU 上的主机对象。但它需要知道每个 kernel 的名字、参数个数和类型顺序——好在运行时正确地把主机端 `parallel_for<Name>(...)` 的调用和设备镜像里的那个 kernel 对上。这些信息，就来自设备趟生成的集成头文件。

## 集成头文件：两趟之间的合同

集成头文件（integration header）是理解 DPC++ 的关键。它是设备编译趟**自动生成**的一个 `.h`，里面用代码描述了「每个 kernel 叫什么名字、有几个参数、每个参数是什么类型、在内存里怎么排布」。主机编译趟通过驱动注入的 `-fsycl-int-header=` 选项（在 `clang/lib/Driver/ToolChains/Clang.cpp` 里设置）把它 `#include` 进来。

它承载的核心契约在 `sycl/include/sycl/detail/kernel_desc.hpp` 里：

- `enum class kernel_param_kind_t` —— 参数种类（accessor？标量？指针？）。
- `struct kernel_param_desc_t` —— 单个参数的描述（种类、大小、偏移）。
- `template <class KernelNameType> struct KernelInfo` —— 每个 kernel 一份，提供 `getName()`、`getNumParams()`、`getParamDesc(int)`、`isESIMD()`。

其中 `getName()` 返回的是 kernel 名字类型的 **Itanium 修饰名**（形如 `_ZTS14example_kernel`，`_ZTS` 是 typeinfo-name 的修饰前缀）。这正是主机端和设备端能「对上号」的机制：主机端你写 `parallel_for<class MyKernel>(...)`，运行时把 `MyKernel` 修饰成同一个字符串，拿它去设备镜像里查找同名的 kernel 入口。

<div class="warn">
<strong>为什么非得先有集成头文件、主机趟才能编？</strong>
标准 C++ 编译器<strong>看不懂</strong> SYCL 的 kernel lambda 该抽出哪些参数、按什么顺序传。但主机端要生成「启动这个 kernel」的调用代码，就必须知道这些。设计上的选择是：让<strong>更懂 SYCL 的设备趟</strong>把这些信息算出来、写成一个普通 C++ 头文件，这样<strong>任何</strong>标准编译器都能在主机趟消费它。这就是集成头文件必须在主机趟之前产出的原因——它是「让普通编译器也能处理 SYCL 主机代码」的粘合层。第 14 章会拆开它的真实内容。
</div>

## 一份源码，两条设备后端路径

设备趟产出 LLVM IR 之后的下游，正是你在第 8、11 章见过的分叉——这里把它和 SYCL 接起来：

- **Intel / AMD GPU**：LLVM IR →（`llvm-spirv`）→ **SPIR-V** → 厂商驱动 → 原生指令。
- **NVIDIA GPU**：LLVM IR →（NVPTX 后端）→ **PTX** → CUDA 驱动 → SASS。

也就是说，第 8 章那句「Intel/AMD 走 SPIR-V、NVIDIA 走 PTX」，在 SYCL 里就是**同一份 `.cpp` 的设备趟可以产出多个目标镜像**，全部嵌进最终的胖二进制（第 9 章的打包工具链）。运行时启动 kernel 时，根据实际选中的设备挑对应的镜像交给对应的驱动。这就是「单一源码、跨厂商」在实现层面的真相。

## 动手：模拟两趟编译如何「对上号」

两趟编译能配合的关键，是**同一个 kernel 名字类型，在两趟里被修饰成同一个字符串**——主机趟拿它当「查找键」，设备趟拿它当「镜像里 kernel 的名字」。下面这个浏览器里能跑的小模型演示这个对接：设备趟把每个 kernel 编成镜像并记下修饰名；主机趟拿修饰名去查、启动。点 **▶ 运行**。

```python
# 模拟 DPC++ 两趟编译：设备趟建"镜像 + 集成头文件"，主机趟据此查找并启动 kernel

def mangle(kernel_name_type):
    # 极简版"Itanium 修饰"：_ZTS + 长度 + 名字（真实规则复杂得多，这里只示意）
    return f"_ZTS{len(kernel_name_type)}{kernel_name_type}"

# ---- 设备编译趟 ----
# 从源码里外提出两个 kernel（名字类型 -> 参数列表）
kernels_in_source = {
    "VecAdd":   ["accessor a", "accessor b", "accessor c"],
    "AddOne":   ["accessor data"],
}

device_image = {}       # 设备镜像：修饰名 -> 已编译 kernel
integration_header = {} # 集成头文件：修饰名 -> 参数信息
for name_type, params in kernels_in_source.items():
    m = mangle(name_type)
    device_image[m] = f"<{name_type} 的 SPIR-V 二进制>"
    integration_header[m] = params
    print(f"[设备趟] 外提 kernel《{name_type}》→ 修饰名 {m}，编成 SPIR-V，写入集成头文件")

print()

# ---- 主机编译趟 + 运行时 ----
# 主机代码里这样启动：q.parallel_for<VecAdd>(...) —— 运行时用同样的修饰规则查找
def launch(kernel_name_type):
    m = mangle(kernel_name_type)                    # 主机端用同一套修饰规则
    if m not in integration_header:
        print(f"[主机/运行时] 启动《{kernel_name_type}》失败：集成头文件里没有 {m}")
        return
    params = integration_header[m]                  # 从集成头文件知道要传几个参数
    print(f"[主机/运行时] 启动《{kernel_name_type}》：修饰名 {m}")
    print(f"              集成头文件说它需要 {len(params)} 个参数：{params}")
    print(f"              → 在设备镜像里找到并执行：{device_image[m]}")

launch("VecAdd")
launch("AddOne")
```

运行结果：

```
[设备趟] 外提 kernel《VecAdd》→ 修饰名 _ZTS6VecAdd，编成 SPIR-V，写入集成头文件
[设备趟] 外提 kernel《AddOne》→ 修饰名 _ZTS6AddOne，编成 SPIR-V，写入集成头文件

[主机/运行时] 启动《VecAdd》：修饰名 _ZTS6VecAdd
              集成头文件说它需要 3 个参数：['accessor a', 'accessor b', 'accessor c']
              → 在设备镜像里找到并执行：<VecAdd 的 SPIR-V 二进制>
[主机/运行时] 启动《AddOne》：修饰名 _ZTS6AddOne
              集成头文件说它需要 1 个参数：['accessor data']
              → 在设备镜像里找到并执行：<AddOne 的 SPIR-V 二进制>
```

看清这个对接：两趟编译**从不直接通信**，它们只共享一样东西——**同一套名字修饰规则**。设备趟按规则把 kernel 名编成字符串写进集成头文件和设备镜像；主机趟/运行时按同一规则把 `parallel_for<Name>` 里的 `Name` 编成字符串去查。名字一致，就对上了。请动手改：把 `launch("VecAdd")` 改成 `launch("VecMul")`（一个源码里不存在的 kernel），你会看到运行时报「集成头文件里没有」——这正是当主机端和设备端 kernel 名字对不上时，真实 DPC++ 会遇到的链接/查找错误。

## 小结与下一步

你现在有了第三部分的总地图：

- **规范 vs 实现**：SYCL 是 Khronos 的规范；**DPC++ 是 Intel 的实现**，源码在 `intel/llvm`，本质是「Clang/LLVM + SYCL 支持 + 运行时库」。
- 单一源码的难题是**一个文件里混着主机代码和设备 kernel**。DPC++ 用**两趟编译**破解：**设备趟**（`__SYCL_DEVICE_ONLY__`）外提 kernel、编成 SPIR-V/PTX、生成集成头文件；**主机趟**当普通 C++ 编、读取集成头文件。
- **集成头文件**是两趟之间的合同（`kernel_desc.hpp` 里的 `KernelInfo`），靠**一致的名字修饰**（`_ZTS...`）让主机调用和设备镜像对上号。
- 设备趟下游是第 8、11 章那条链的分叉：**Intel/AMD → SPIR-V，NVIDIA → PTX**，多个镜像一起进胖二进制。
- 编译期在第 12–15 章，运行期在第 16–18 章。

下一章我们从「宏观流程」钻进「头文件」：打开 `sycl/include/sycl/sycl.hpp`，看它到底 `#include` 了什么、`queue`/`buffer`/`accessor`/`handler` 这些类各自声明在哪个头里、`detail/` 目录藏着哪些不对用户暴露的实现细节。摸清头文件地图，你才能在读运行时源码时知道每个类型从哪来。

---

**这一章的要点**

- **SYCL = 规范**（Khronos），**DPC++ = 实现**（Intel，即 `intel/llvm`）；DPC++ 是盖在 LLVM 地基上的一层，不是新编译器。
- 单一源码里主机代码与设备 kernel 混在一起，靠**两趟编译**分开：**设备趟**外提 kernel + 编成 SPIR-V/PTX + 生成集成头文件（定义宏 `__SYCL_DEVICE_ONLY__`）；**主机趟**编普通程序 + 读集成头文件。
- kernel 的抽取靠前端 `sycl_kernel` 属性（`clang/.../Attr.td`）与 `clang/lib/Sema/SemaSYCL.cpp`；集成头文件契约在 `sycl/include/sycl/detail/kernel_desc.hpp`（`KernelInfo`、`kernel_param_desc_t`）。
- 主机与设备靠**一致的名字修饰**（Itanium `_ZTS...`）对上号；这也是集成头文件必须先于主机趟生成的原因。
- 设备后端两条路：**SPIR-V（Intel/AMD）** 与 **PTX（NVIDIA）**，多镜像同入胖二进制，运行时按选中设备挑镜像。
