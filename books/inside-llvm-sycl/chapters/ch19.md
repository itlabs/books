# 第 19 章 设备库与 libdevice：数学函数、fallback 与 SPIR-V 内建

## 这一章想让你带走什么

<div class="goal-box">
<p>你在 kernel 里写过 <code>sycl::sqrt(x)</code>、<code>std::sin(x)</code>、甚至 <code>assert(i &lt; N)</code> 吗？在 CPU 上这些靠链接系统的 libm/libc 实现——可 <strong>GPU 上没有系统 libc/libm</strong>。那这些函数的实现从哪来？这一章收掉这个从第 8、11 章就埋下的悬念：<strong><code>libdevice/</code></strong> 提供 C 库/数学/复数/assert 的<strong>设备端 fallback 实现</strong>，<strong><code>libclc/</code></strong> 提供底层 SPIR-V/OpenCL 内建，二者以 LLVM bitcode（<code>.bc</code>）的形态在设备链接时被选择性地链进设备镜像。</p>
<p>读完你会看清一条完整的解析链：<code>sycl::sqrt</code> → <code>__spirv_ocl_sqrt</code>（SPIR-V 扩展指令）→ 在原生支持 SPIR-V 的后端直接用、在 NVIDIA 等后端由 libclc 兜底；<code>std::ceil</code> → libdevice 的 <code>__devicelib_ceilf</code> → 还是落到 <code>__spirv_ocl_ceil</code>。你还会理解那个精妙的 <strong>weak 符号 fallback 机制</strong>——设备驱动若原生支持某功能就用原生的，否则用编进 <code>.bc</code> 的 fallback，以及 driver 如何按目标选择该链哪些设备库。这是第三部分的收尾，把"设备端标准库从哪来"彻底讲透。</p>
</div>

## 问题：GPU 上没有 libc/libm

先把困境摆清。一个设备 kernel 里可能有：

- **数学函数**：`sycl::sqrt`、`sycl::sin`、`sycl::fma`……
- **C 标准库**：`memcpy`、`memset`、`assert`……
- **复数运算**：`std::complex` 的乘除。
- **Intel 数学函数（IMF）**：一批 Intel 特有的高性能/特定舍入数学函数。

在主机上这些都靠链接器把系统的 libm.so / libc.so 链进来解决。但设备端**没有这些系统库**——GPU 不跑操作系统、没有 libc。所以 SYCL 必须**自己提供一套面向设备的实现**，在设备代码链接时链进设备镜像。这套实现分两层，各司其职：**`libdevice`（C 库/数学的 fallback）** 和 **`libclc`（底层 SPIR-V 内建）**。

## libdevice：设备端的 C 库/数学 fallback

`libdevice/` 是一个**顶层目录**（和 `sycl/`、`llvm/` 平级，不在 `sycl/` 下）。它的定位，`CMakeLists.txt` 开头的注释一句话讲明："为**没有原生支持这些功能**的 SPIR-V 设备提供各种功能的工具项目"。里面是一批 wrapper 源文件，编译成设备端 bitcode 库：

| 源文件 | 内容 | 编成的库 |
|---|---|---|
| `crt_wrapper.cpp` | C 运行时：`memcpy`/`memset`/`memcmp`、`assert` 支持 | `libsycl-crt.bc` |
| `cmath_wrapper.cpp` | C 数学：`fabsf`/`ceilf`/`logf`/`rint`… | `libsycl-cmath.bc` |
| `imf_wrapper.cpp` | Intel 数学函数（IMF） | `libsycl-imf.bc` |
| `fallback-bfloat16.cpp` | bfloat16 转换（emulated） | `libsycl-fallback-bfloat16.bc` |
| `itt_*.cpp` | ITT 性能instrumentation | `libsycl-itt-*.bc` |
| `sanitizer/*.cpp` | 设备端 asan/msan/tsan 运行时 | `libsycl-asan.bc` 等 |

### fallback 的核心：weak 符号 + 转发到 SPIR-V 内建

libdevice 最精巧的地方是它的 **fallback 机制**，理解它就理解了 libdevice 的全部设计。以 `ceilf` 为例，链条是三层：

```cpp-norun
// ① wrapper（cmath_wrapper.cpp）：暴露标准 libc 名字，转发到 __devicelib_*
DEVICE_EXTERN_C_INLINE float ceilf(float x) { return __devicelib_ceilf(x); }

// ② fallback 定义（fallback-cmath.hpp）：__devicelib_* 落到 SPIR-V 内建
static inline float __devicelib_ceilf(float x) { return __spirv_ocl_ceil(x); }

// ③ 所有 wrapper 符号被标成 weak（device.h）
#define DEVICE_EXTERNAL SYCL_EXTERNAL __attribute__((weak))
```

关键就在那个 **`__attribute__((weak))`**（弱符号）。弱符号的语义是"**如果别处有同名的强符号，就用别处的；否则才用我这个**"。于是：

- **如果设备驱动/后端原生提供了 `ceilf` 的实现**（强符号），链接时那个原生实现胜出，libdevice 的 fallback 被丢弃——用上硬件最优的版本。
- **如果后端没有原生实现**，libdevice 编进 `.bc` 的这个 weak fallback 就被链进去兜底——保证 kernel 至少能跑。

这就是 libdevice 名字里"fallback"的含义：**它是保底实现，原生实现优先**。一个 weak 属性就优雅地表达了"能用原生就用原生、不行我兜底"的策略——不需要运行时判断，链接器自动完成。

<div class="note">
<strong>fallback 实现现在多是 <code>.hpp</code>，不是 <code>fallback-*.cpp</code></strong>
老资料常提 <code>fallback-cmath.cpp</code>/<code>fallback-complex.cpp</code>/<code>fallback-cstring.cpp</code> 这些独立源文件。在当前分支上它们已经变成<strong>头文件</strong>（<code>fallback-cmath.hpp</code>/<code>fallback-complex.hpp</code>/<code>fallback-cstring.hpp</code>），被对应的 wrapper <code>.cpp</code> 包含。<code>fallback-bfloat16.cpp</code> 是仅剩的独立 <code>fallback-*.cpp</code>。assert 没有独立 fallback 文件——它在 <code>crt_wrapper.cpp</code> + <code>wrapper.h</code> 里（<code>__devicelib_assert_fail</code>）。又一次"以你手上的树为准"的提醒。
</div>

## libclc：底层 SPIR-V/OpenCL 内建

上面 fallback 的终点是 `__spirv_ocl_ceil` 这样的 **SPIR-V 内建**。这些内建本身的实现，是**另一层**——`libclc/`（也是顶层目录，第 11 章见过）。分工是这样的：

- **libclc** 提供 **SPIR-V/OpenCL 内建原语本身**：`__spirv_ocl_*`（数学扩展指令）、work-item 查询（`__spirv_BuiltInGlobalInvocationId` 等取 global id）、原子、group 操作。核心在 `libclc/libspirv/`。它尤其重要于 **NVPTX/AMDGPU** ——那些后端没有 SPIR-V 消费者，`__spirv_*` 得靠 libclc 提供具体实现。
- **libdevice** 提供**更高层的 C/C++ 标准库表面**（`<cmath>`/`<complex>`/`<cstring>`/`assert`）和 **IMF**，实现成薄薄的 `__devicelib_*` fallback，**它们自己再调 libclc 的 `__spirv_ocl_*` 内建**。

一句话对比：**libclc = 各目标的 SPIR-V/OpenCL 内建原语；libdevice = 建在这些原语之上的 C/C++ 标准库与 Intel 数学 fallback**（外加 assert/sanitizer/ITT/bfloat16）。

<div class="keypoint">
<strong>两层设备库：libclc 提供 SPIR-V 内建原语，libdevice 提供建于其上的 C/C++ 标准库 fallback</strong>
<strong>设备端没有 libc/libm。<code>libclc/</code>（顶层，<code>libspirv/</code>）提供底层 <code>__spirv_ocl_*</code> 内建与 work-item 查询；<code>libdevice/</code>（顶层）提供 <code>&lt;cmath&gt;</code>/<code>&lt;complex&gt;</code>/<code>&lt;cstring&gt;</code>/<code>assert</code>/IMF 的 <code>__devicelib_*</code> fallback，转发到 libclc 的内建。</strong>fallback 用 <strong>weak 符号</strong>（<code>SYCL_EXTERNAL __attribute__((weak))</code>）：驱动原生支持则用原生、否则用编进 <code>.bc</code> 的 fallback。两层都以 LLVM bitcode 在设备链接时链入。
</div>

## sycl::sqrt 的完整解析链

把这一章串起来，看 `sycl::sqrt(x)` 从 SYCL 头到设备内建的完整旅程：

1. **`sycl::sqrt`**（公开 API，`sycl/include/sycl/builtins.hpp`）——经 `sycl/include/sycl/detail/builtins/math_functions.hpp` 里的宏（`BUILTIN_GENF_NATIVE_OPT(..., sqrt)`）展开。
2. **`__spirv_ocl_sqrt`**——宏展开后调用的 SPIR-V 扩展指令（对应 OpenCL.std 扩展指令集里的 `sqrt`）。这些 `__spirv_ocl_*` 是**编译器声明的内建**，由 driver 的 `-fdeclare-spirv-builtins` 开启（对应 LangOption `DeclareSPIRVBuiltins`），内建表从 `clang/lib/Sema/SPIRVBuiltins.td` 生成。
3. **落地分两条路**：
   - **原生支持 SPIR-V 的后端**（Intel GPU 经 Level Zero、OpenCL）：`__spirv_ocl_sqrt` 由 `llvm-spirv` 翻成 SPIR-V 的 `OpExtInst`（`OpenCL.std sqrt`），交给驱动，用硬件的 sqrt。
   - **NVPTX/AMDGPU**（无 SPIR-V 消费者）：`__spirv_ocl_sqrt` 由 **libclc** 的对应目标实现（`libclc/libspirv/lib/nvptx/math/` 等）提供。

而 `std::ceil` / C 库那条路则多绕一层 libdevice：`std::ceil` → libdevice 的 `ceilf`(weak) → `__devicelib_ceilf` → `__spirv_ocl_ceil` → 同样落到上面两条路之一。

## driver 如何选择并链接设备库

这些 `.bc` 库不是全部无脑链进去——driver 按**目标**挑选。逻辑在 `clang/lib/Driver/ToolChains/SYCL.cpp` 的 `getDeviceLibNames()`：

- **SPIR/SPIR-V 目标**：链基础集 `libsycl-crt`、`libsycl-cmath`、`libsycl-imf`（Windows 还有 `libsycl-msvc-math`），除非你传 `--no-offloadlib`。
- **NVPTX 目标** → 单个大 bitcode `devicelib-nvptx64-nvidia-cuda.bc`；**AMDGCN** → `devicelib-amdgcn-amd-amdhsa.bc`（第 11 章提过）。
- **按需追加**：`-fsycl-instrument-device-code` 加 ITT 库；bfloat16 按设备能力选 `libsycl-native-bfloat16` 或 `libsycl-fallback-bfloat16`；`-fsanitize=` 加对应的 asan/msan/tsan 设备库。

链接方式是 `-mlink-builtin-bitcode` 且默认 **internalize**（`--internalize --only-needed`）——只把 kernel **实际引用到**的函数链进去，没用到的 fallback 不进镜像，保持设备镜像精简。开关是 `--no-offloadlib` / `--offloadlib`（双横线、无 `f`；`-nogpulib`/`-nocudalib` 是它的别名。不是想象中的 `-fsycl-device-lib`）。

至于 **bfloat16 的原生 vs fallback** 决策，还有一层运行时判断：program manager（`sycl/source/detail/program_manager/program_manager.cpp`）读设备库镜像的元信息（版本 `0`=fallback、`1`=native），查后端能力，支持就用 native 镜像、否则 fallback——这是 weak 符号 fallback 思想在运行时的又一次体现（第 8、14 章的 property set 机制）。

## 动手：模拟 weak 符号 fallback 的选择

把这一章的核心——「weak fallback：原生实现优先，否则用编进 .bc 的兜底」——用一个浏览器里能跑的小模型演出来。点 **▶ 运行**。

```python
# 模拟 libdevice 的 weak 符号 fallback：链接器如何在原生 vs fallback 间选择

# ---- libdevice 提供的 weak fallback 实现（编进 .bc）----
FALLBACK_SYMBOLS = {
    "ceilf":  ("weak", "__devicelib_ceilf → __spirv_ocl_ceil"),
    "sqrtf":  ("weak", "__devicelib_sqrtf → __spirv_ocl_sqrt"),
    "sinf":   ("weak", "__devicelib_sinf → __spirv_ocl_sin"),
    "assert": ("weak", "__devicelib_assert_fail"),
}

def link_device_image(used_symbols, native_strong_symbols):
    """模拟设备链接：用到的符号，优先用后端提供的强符号，否则用 weak fallback。
       internalize：只链 kernel 实际用到的符号。"""
    print(f"kernel 用到的符号: {used_symbols}")
    print(f"后端原生提供(强符号): {native_strong_symbols or '（无）'}\n")
    linked = []
    for sym in used_symbols:
        if sym in native_strong_symbols:
            linked.append((sym, "原生强符号", native_strong_symbols[sym]))
        elif sym in FALLBACK_SYMBOLS:
            _, impl = FALLBACK_SYMBOLS[sym]
            linked.append((sym, "libdevice weak fallback", impl))
        else:
            linked.append((sym, "!! 未解析 !!", "链接错误"))
    for sym, kind, impl in linked:
        print(f"  {sym:<8} → [{kind}] {impl}")
    # internalize：FALLBACK 里没被用到的不进镜像
    unused = set(FALLBACK_SYMBOLS) - set(used_symbols)
    print(f"\n  (internalize：未用到的 fallback 不链入 → {sorted(unused)})")

print("=== 后端 A：驱动原生支持 sqrtf（如某 GPU 有硬件 sqrt）===")
link_device_image(
    used_symbols=["sqrtf", "ceilf", "assert"],
    native_strong_symbols={"sqrtf": "硬件 sqrt.f32 指令"},   # 原生强符号胜出
)
print("\n=== 后端 B：驱动啥数学都不原生提供 ===")
link_device_image(
    used_symbols=["sqrtf", "ceilf"],
    native_strong_symbols={},                                # 全部用 fallback 兜底
)
```

```
  (internalize：未用到的 fallback 不链入 → ['assert', 'sinf'])
```

看清 weak fallback 的两种结局：**后端 A** 原生提供 `sqrtf`（强符号），链接器就用硬件版、丢弃 libdevice 的 weak fallback；而 `ceilf`/`assert` 没有原生实现，就用 libdevice 兜底。**后端 B** 什么都不提供，全部落到 fallback。注意 `internalize` 的效果：没被 kernel 用到的 `sinf`（和后端 A 的场景）根本不链进镜像——这就是 `--only-needed` 保持设备镜像精简的作用。请动手改：给后端 B 的 `native_strong_symbols` 加上 `{"ceilf": "硬件 ceil"}`，重新运行——你会看到 `ceilf` 立刻从 fallback 切成原生。一个 weak 属性就实现了这套"能用原生就用原生"的策略，全程无需运行时判断。

## 小结与下一步

第三部分的收尾——设备端"标准库从哪来"讲透了：

- 设备端**没有系统 libc/libm**，SYCL 靠两层设备库补齐，都以 **LLVM bitcode（`.bc`）** 在设备链接时链入：
  - **`libdevice/`**（顶层）：`<cmath>`/`<complex>`/`<cstring>`/`assert`/**IMF** 的设备端实现，加 bfloat16/ITT/sanitizer。编成 `libsycl-crt.bc`/`libsycl-cmath.bc`/`libsycl-imf.bc` 等。
  - **`libclc/`**（顶层，`libspirv/`）：底层 `__spirv_ocl_*` 内建与 work-item 查询，尤其为 NVPTX/AMDGPU 提供 SPIR-V 内建实现。
- **fallback 机制**：libdevice 的 wrapper 符号是 **weak**（`SYCL_EXTERNAL __attribute__((weak))`，`device.h`），转发到 `__devicelib_*` → `__spirv_ocl_*`。驱动原生支持则原生强符号胜出，否则用编进 `.bc` 的 weak fallback——链接器自动选择，无需运行时判断。
- **解析链**：`sycl::sqrt`（`math_functions.hpp` 宏）→ `__spirv_ocl_sqrt`（`-fdeclare-spirv-builtins` 开启、`SPIRVBuiltins.td` 生成）→ 原生 SPIR-V 后端经 `llvm-spirv` 出 `OpExtInst` / NVPTX 等由 libclc 实现。`std::ceil` 多绕一层 libdevice。
- **driver 选库**（`SYCL.cpp` 的 `getDeviceLibNames()`）：SPIR-V 目标链 crt/cmath/imf 基础集；NVPTX/AMDGCN 链单个大 `.bc`；按 `-fsycl-instrument-device-code`/`-fsanitize=`/bfloat16 能力追加；`-mlink-builtin-bitcode` + **internalize**（只链用到的）；开关 `--no-offloadlib`。
