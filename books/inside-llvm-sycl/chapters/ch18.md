# 第 18 章 Unified Runtime：SYCL 如何对接 Level Zero / CUDA / OpenCL

## 这一章想让你带走什么

<div class="goal-box">
<p>第 16、17 章里反复出现两样东西却一直没展开：每个 impl 握着的 <code>ur_*_handle_t</code> 句柄、每次 <code>adapter.call&lt;UrApiKind::…&gt;(...)</code> 的调用。它们都指向同一层——<strong>Unified Runtime（UR，统一运行时）</strong>。这一章把这层单独讲清：UR 是什么、它的 API 长什么样、它怎么用<strong>适配器（adapter）</strong>对接 Level Zero / CUDA / OpenCL 等各家后端、以及 SYCL 运行时怎么通过一层薄薄的 glue 接上它。</p>
<p>读完你会认出一个老朋友：UR 就是第 10 章 "统一接口 + 各设备插件" 那个解耦母题在 <strong>SYCL 侧</strong>的化身——上层 SYCL 运行时只对 UR 的设备无关 API 编程，每家 GPU 各写一个 adapter 把 UR 调用翻译成自己的驱动调用（<code>zeCommandListAppendLaunchKernel</code> / <code>cuLaunchKernel</code> / <code>clEnqueueNDRangeKernel</code>）。加一款设备 = 加一个 adapter，SYCL 运行时一行不改。这是 SYCL "一份代码跑遍 Intel/NVIDIA/AMD" 承诺的最后一块拼图。</p>
</div>

## UR 是什么：SYCL 的设备无关运行时层

回顾第 17 章结尾：调度器把命令 enqueue 到设备。但"设备"是抽象的——具体是 Intel GPU（走 Level Zero）、NVIDIA GPU（走 CUDA Driver API）还是 CPU（走 OpenCL）？这些后端的 API 完全不同。如果让 SYCL 运行时直接去调各家 API，代码会被 `#ifdef` 撕碎——这正是第 10 章讲过的困境。

**Unified Runtime（UR）** 就是插在中间的那层设备无关抽象。它在本书代码库里是一个**顶层目录** `unified-runtime/`（注意：**不在** `sycl/` 下面，是和 `sycl/`、`llvm/` 平级的独立组件）。SYCL 运行时只对着 UR 的 C API 编程，说"创建一个队列""分配设备内存""启动这个 kernel"，UR 再通过 adapter 分派到具体后端。

<div class="warn">
<strong>UR ≠ liboffload（第 10 章），别混为一层</strong>
第 10 章的 <code>liboffload</code> 和本章的 UR <strong>理念完全一致</strong>（都是"统一接口 + 后端适配"），也都对接 CUDA/Level Zero，但它们是<strong>两个不同的层、服务不同的上层</strong>：<code>liboffload</code> 脱胎于 OpenMP offload、面向多语言运行时；<strong>UR 是 SYCL/DPC++ 专用</strong>的那一层。SYCL 运行时走的是 UR，不是 liboffload。有意思的是它们之间有座桥——UR 的 <code>offload</code> adapter 能把 UR 调用转发给 liboffload。但请在脑子里把这两层分开：本章只讲 SYCL 侧的 UR。
</div>

## UR 的 API：从 YAML 规范生成

UR 的 API 是一套 C 接口，声明在 `unified-runtime/include/unified-runtime/ur_api.h`。和第 10 章 liboffload、第 7 章后端一样，它也是**声明式描述 + 代码生成**的产物——`ur_api.h` 由 `unified-runtime/scripts/core/*.yml` 这些 YAML 规范生成。YAML 单一来源地描述每个 API 的签名、参数、枚举、错误码，生成出一致的头文件、加载器代码和文档。

UR 用一组**不透明句柄（opaque handle）**对应 SYCL 的核心对象——这正是第 16 章那些 `ur_*_handle_t` 的来源：

| UR 句柄 | 对应 SYCL 概念 |
|---|---|
| `ur_platform_handle_t` | platform |
| `ur_device_handle_t` | device |
| `ur_context_handle_t` | context |
| `ur_queue_handle_t` | queue |
| `ur_program_handle_t` | 编译好的设备程序 |
| `ur_kernel_handle_t` | kernel |
| `ur_mem_handle_t` | buffer/image |
| `ur_event_handle_t` | event |

对应的核心入口点（函数）也很好认，基本是"动词 + 名词"：`urLoaderInit`（初始化加载器）、`urAdapterGet`（枚举 adapter）、`urPlatformGet` / `urDeviceGet` / `urContextCreate` / `urQueueCreate` / `urMemBufferCreate` / `urUSMDeviceAlloc` / `urProgramCreateWithIL`（从 SPIR-V IL 建 program）/ `urKernelCreate`。而启动 kernel 的那一下是 **`urEnqueueKernelLaunchWithArgsExp`**。

<div class="note">
<strong>启动 API 的确切名字有个坑</strong>
你可能想当然地去搜 <code>urEnqueueKernelLaunch</code>——但当前分支上真正用的启动入口是 <strong><code>urEnqueueKernelLaunchWithArgsExp</code></strong>（带 <code>WithArgs</code> 和 <code>Exp</code> 后缀，<code>Exp</code> 表示 experimental 扩展入口）。这类"名字带 Exp 后缀""看起来有个更短的版本其实没有"的情况，在 UR 里不止一处。查 API 时以你构建出的 <code>ur_api.h</code> 为准，别凭记忆拼——这也是"源码为锚"的一个具体教训。
</div>

## adapter：把 UR 调用翻译成各家驱动

UR 的后端实现叫 **adapter（适配器）**，住在 `unified-runtime/source/adapters/` 下，每家后端一个子目录：

```
unified-runtime/source/adapters/
├── level_zero/        # Intel GPU（Level Zero）
│   └── v2/            # Level Zero 的新一代实现（子目录，注意别看漏！）
├── cuda/              # NVIDIA GPU（CUDA Driver API）
├── hip/               # AMD GPU（HIP/ROCm）
├── opencl/            # 任何 OpenCL 设备（含 CPU）
├── native_cpu/        # 直接在 CPU 上跑，无需外部驱动
├── offload/           # 桥接到 liboffload（第 10 章）
└── mock/              # 测试用假 adapter
```

每个 adapter 实现同一套 UR API 契约，内部翻译成自己后端的调用。同一个 `urEnqueueKernelLaunchWithArgsExp`：

- 在 **level_zero** adapter 里 → `zeCommandListAppendLaunchKernel`
- 在 **cuda** adapter 里 → `cuLaunchKernel`
- 在 **opencl** adapter 里 → `clEnqueueNDRangeKernel`

这就是第 10 章那张图的 SYCL 版：**上层对一套 UR API 编程，adapter 把统一动作翻译成各自驱动的真实调用；加一款设备只要加一个 adapter**。`mock` adapter 尤其有用——它不碰任何真实驱动，返回假句柄，让 SYCL 运行时的单元测试能在没有 GPU 的机器上跑。

<div class="note">
<strong>两套 Level Zero 实现不是笔误</strong>
你会在 <code>level_zero/</code> 目录里看到一个 <code>v2/</code> 子目录，里头是<strong>另一套</strong>实现。这是 Intel GPU 后端正在进行的一次重写——<code>v2</code> 是新一代实现（更贴合新的 Level Zero immediate command list 等特性），和旧的并存、逐步切换。读源码遇到两套别以为看错了。这种"新旧实现并存"在活跃演进的运行时里很常见（对比第 16 章 <code>plugins-nextgen</code> 相对旧插件的"nextgen"）。
</div>

## loader：UR 怎么找到并加载 adapter

adapter 是动态库，UR 需要在运行时**发现并加载**它们——这是 **loader（加载器）** 的活，代码在 `unified-runtime/source/loader/`。关键文件 `ur_adapter_registry.hpp`、`ur_manifests.hpp` 描述"有哪些 adapter、去哪找"。`urLoaderInit` 初始化加载器，`urAdapterGet` 枚举出当前环境可用的 adapter。

用户侧控制"用哪个后端/设备"的旋钮是环境变量 **`ONEAPI_DEVICE_SELECTOR`**——比如 `ONEAPI_DEVICE_SELECTOR=level_zero:gpu` 选 Level Zero 的 GPU、`cuda:*` 选所有 CUDA 设备。loader 据此决定加载哪些 adapter、暴露哪些设备。这解释了第 16 章那句"发现并选设备"到底是谁在做——loader + adapter 枚举。

## SYCL 侧的 glue：sycl 怎么接上 UR

UR 是独立组件（顶层 `unified-runtime/`），SYCL 运行时通过一层**薄薄的 glue** 接上它，代码在 `sycl/source/detail/ur.{hpp,cpp}`（还记得第 13 章那个 `source/detail/` 吗？就在那）。这层 glue 把第 16 章那个模板调用 `Adapter.call<UrApiKind::xxx>(...)` 落到实处——`UrApiKind::urQueueCreate` 这样的枚举被映射到真正的 UR 函数指针，`call<>` 还统一做错误检查（UR 返回错误码就抛 SYCL 异常）。

所以第 16 章那条脊柱的最后一跳现在闭合了：
`queue_impl` 建队列 → `getAdapter().call<UrApiKind::urQueueCreate>(...)`（`queue_impl.hpp:543`）→ glue 查到 UR 的 `urQueueCreate` 函数指针 → loader 分派到选中的 adapter（如 level_zero）→ adapter 内部调 `zeCommandListCreate` 之类 → 拿到 `ze_command_queue_handle_t`，包成 `ur_queue_handle_t` 交回来。第 17 章调度器最终 enqueue kernel 时，同样这条链走到 `urEnqueueKernelLaunchWithArgsExp` → adapter → `zeCommandListAppendLaunchKernel`。**整条从 `q.submit` 到硬件启动的路径，到这里全部贯通**。

## 观察 UR：把每次调用打印出来

UR 内建追踪，是观察这层最直接的方式。核心开关是 `SYCL_UR_TRACE`（定义在 `sycl/source/detail/config.def`）：

```bash
# 打印 SYCL 运行时对 UR 的每一次调用（urQueueCreate、urEnqueueKernelLaunch… 全打出来）
SYCL_UR_TRACE=1 ./my_sycl_app

# 选后端/设备：只用 Level Zero 的 GPU
ONEAPI_DEVICE_SELECTOR=level_zero:gpu ./my_sycl_app

# 它是位掩码（bit-mask）：-1 打开所有追踪类型，输出更全
SYCL_UR_TRACE=-1 ./my_sycl_app
```

`SYCL_UR_TRACE=1` 的输出是理解一个 SYCL 程序"到底对运行时下了哪些命令"的最佳窗口——你会看到 `urContextCreate` → `urQueueCreate` → `urProgramCreateWithIL` → `urKernelCreate` → `urEnqueueKernelLaunchWithArgsExp` 这一串，和你源码里的 `context`/`queue`/`parallel_for` 一一对上。再配合第 10 章的 `nsys`（CUDA）或 Level Zero 的追踪工具看底层驱动调用，就能亲眼确认 adapter 那层翻译。调试性能或"kernel 没跑"问题时，这是第一现场。

<div class="keypoint">
<strong>UR = SYCL 的"统一接口 + adapter 插件"层</strong>
<strong>UR 是顶层组件 <code>unified-runtime/</code>（不在 <code>sycl/</code> 下），API（<code>ur_api.h</code>）从 <code>scripts/core/*.yml</code> 生成，用 <code>ur_*_handle_t</code> 不透明句柄对应 SYCL 对象。<code>unified-runtime/source/adapters/</code> 下每家后端一个 adapter（level_zero[_v2]/cuda/hip/opencl/native_cpu/offload/mock），把同一个 <code>urEnqueueKernelLaunchWithArgsExp</code> 翻成 <code>zeCommandListAppendLaunchKernel</code>/<code>cuLaunchKernel</code>/<code>clEnqueueNDRangeKernel</code>。</strong>loader（受 <code>ONEAPI_DEVICE_SELECTOR</code> 控制）发现加载 adapter；SYCL 经 <code>sycl/source/detail/ur.{hpp,cpp}</code> 的 <code>adapter.call&lt;UrApiKind::…&gt;</code> 接上。这是第 10 章解耦母题在 SYCL 侧的化身——加设备只加 adapter。
</div>

## 动手：模拟 UR 分派到各后端 adapter

把这一章的核心——「SYCL 对 UR 统一 API 编程 → loader 按选择器加载 adapter → adapter 翻译成各家驱动调用」——用一个浏览器里能跑的小模型演出来。点 **▶ 运行**。

```python
# 模拟 Unified Runtime：统一 UR API + 各后端 adapter

# ---- 每个 adapter 实现同一套 UR 契约，翻译成自己后端的驱动调用 ----
class Adapter:
    name = "base"
    def urContextCreate(self, dev):      raise NotImplementedError
    def urQueueCreate(self, ctx, dev):   raise NotImplementedError
    def urProgramCreateWithIL(self, il): raise NotImplementedError
    def urEnqueueKernelLaunchWithArgsExp(self, k, nd): raise NotImplementedError

class LevelZeroAdapter(Adapter):
    name = "level_zero"
    def urContextCreate(self, dev):      return "zeContextCreate()"
    def urQueueCreate(self, ctx, dev):   return "zeCommandQueueCreate()"
    def urProgramCreateWithIL(self, il): return f"zeModuleCreate({il})  # 吃 SPIR-V"
    def urEnqueueKernelLaunchWithArgsExp(self, k, nd): return f"zeCommandListAppendLaunchKernel({k}, {nd})"

class CudaAdapter(Adapter):
    name = "cuda"
    def urContextCreate(self, dev):      return "cuCtxCreate()"
    def urQueueCreate(self, ctx, dev):   return "cuStreamCreate()"
    def urProgramCreateWithIL(self, il): return f"cuModuleLoadData({il})  # 吃 PTX"
    def urEnqueueKernelLaunchWithArgsExp(self, k, nd): return f"cuLaunchKernel({k}, {nd})"

class OpenCLAdapter(Adapter):
    name = "opencl"
    def urContextCreate(self, dev):      return "clCreateContext()"
    def urQueueCreate(self, ctx, dev):   return "clCreateCommandQueue()"
    def urProgramCreateWithIL(self, il): return f"clCreateProgramWithIL({il})"
    def urEnqueueKernelLaunchWithArgsExp(self, k, nd): return f"clEnqueueNDRangeKernel({k}, {nd})"

# ---- loader：按 ONEAPI_DEVICE_SELECTOR 挑 adapter ----
ADAPTERS = {a.name: a for a in [LevelZeroAdapter(), CudaAdapter(), OpenCLAdapter()]}

def loader_select(selector):                 # 形如 "level_zero:gpu"
    backend = selector.split(":")[0]
    return ADAPTERS.get(backend)

# ---- SYCL 运行时：只说 UR 的统一动作，不碰具体驱动 ----
def sycl_run(selector, il, kernel, ndrange):
    ad = loader_select(selector)
    if not ad:
        print(f"[loader] 没有 {selector} 对应的 adapter"); return
    print(f"[loader] ONEAPI_DEVICE_SELECTOR={selector} → 加载 {ad.name} adapter")
    print(f"  urContextCreate               → {ad.urContextCreate('dev0')}")
    print(f"  urQueueCreate                 → {ad.urQueueCreate('ctx','dev0')}")
    print(f"  urProgramCreateWithIL         → {ad.urProgramCreateWithIL(il)}")
    print(f"  urEnqueueKernelLaunchWithArgsExp → {ad.urEnqueueKernelLaunchWithArgsExp(kernel, ndrange)}")
    print()

# 同一段 SYCL 逻辑，只因选择器不同就落到不同后端
sycl_run("level_zero:gpu", "VecAdd.spv", "VecAdd", (1024,))
sycl_run("cuda:gpu",       "VecAdd.ptx", "VecAdd", (1024,))
sycl_run("opencl:cpu",     "VecAdd.spv", "VecAdd", (1024,))
```

运行结果：

```
[loader] ONEAPI_DEVICE_SELECTOR=level_zero:gpu → 加载 level_zero adapter
  urContextCreate               → zeContextCreate()
  urQueueCreate                 → zeCommandQueueCreate()
  urProgramCreateWithIL         → zeModuleCreate(VecAdd.spv)  # 吃 SPIR-V
  urEnqueueKernelLaunchWithArgsExp → zeCommandListAppendLaunchKernel(VecAdd, (1024,))

[loader] ONEAPI_DEVICE_SELECTOR=cuda:gpu → 加载 cuda adapter
  urContextCreate               → cuCtxCreate()
  urQueueCreate                 → cuStreamCreate()
  urProgramCreateWithIL         → cuModuleLoadData(VecAdd.ptx)  # 吃 PTX
  urEnqueueKernelLaunchWithArgsExp → cuLaunchKernel(VecAdd, (1024,))

[loader] ONEAPI_DEVICE_SELECTOR=opencl:cpu → 加载 opencl adapter
  urContextCreate               → clCreateContext()
  urQueueCreate                 → clCreateCommandQueue()
  urProgramCreateWithIL         → clCreateProgramWithIL(VecAdd.spv)
  urEnqueueKernelLaunchWithArgsExp → clEnqueueNDRangeKernel(VecAdd, (1024,))
```

看清这层分派：`sycl_run` 说的永远是同一串 UR 动作（`urContextCreate`/`urQueueCreate`/`urProgramCreateWithIL`/`urEnqueueKernelLaunchWithArgsExp`），loader 按 `ONEAPI_DEVICE_SELECTOR` 挑 adapter，**每个 adapter 把同一个 UR 调用翻成自己驱动的真实函数**——Level Zero 的 `ze…`、CUDA 的 `cu…`、OpenCL 的 `cl…`。SYCL 运行时对这些差异一无所知。注意 `urProgramCreateWithIL` 那行还体现了第 8、11 章的分叉：Level Zero/OpenCL 吃 SPIR-V、CUDA 吃 PTX。请动手改：加一个 `class HipAdapter(Adapter)`（用 `hip…` 风格的名字）注册进 `ADAPTERS`，再 `sycl_run("hip:gpu", ...)`——你会发现 `sycl_run` 一个字都不用改就支持了 AMD。这就是"加设备 = 加 adapter"。

## 小结与下一步

第三部分的运行时三章（16-18）到此闭合，从 `q.submit` 到硬件启动的完整路径贯通了：

- **UR（Unified Runtime）** 是 SYCL 的设备无关运行时层，位于**顶层 `unified-runtime/`**（不在 `sycl/` 下）。API（`ur_api.h`）从 `unified-runtime/scripts/core/*.yml` **生成**，用 `ur_*_handle_t` 不透明句柄对应 platform/device/context/queue/program/kernel/mem/event。
- 核心入口：`urLoaderInit`/`urAdapterGet`/`urPlatformGet`/`urDeviceGet`/`urContextCreate`/`urQueueCreate`/`urMemBufferCreate`/`urUSMDeviceAlloc`/`urProgramCreateWithIL`/`urKernelCreate`，启动是 **`urEnqueueKernelLaunchWithArgsExp`**（注意确切名字）。
- **adapter**（`unified-runtime/source/adapters/`）每家后端一个：`level_zero`（内含 `v2/` 子目录，新一代实现）/`cuda`/`hip`/`opencl`/`native_cpu`/`offload`/`mock`，把统一 UR 调用翻成各家驱动（`ze…`/`cu…`/`cl…`）。加设备只加 adapter。
- **loader**（`unified-runtime/source/loader/`，`ur_adapter_registry.hpp`）发现并加载 adapter，受 **`ONEAPI_DEVICE_SELECTOR`** 控制选后端/设备。
- SYCL 经 **`sycl/source/detail/ur.{hpp,cpp}`** 这层 glue（`adapter.call<UrApiKind::…>`）接上 UR——闭合了第 16 章那条脊柱的最后一跳。
- 观察：`SYCL_UR_TRACE=1` 打印每次 UR 调用（`SYCL_UR_TRACE=-1` 位掩码全开，更全）。
- **UR ≠ liboffload（第 10 章）**：理念同、层次异，服务不同上层；UR 的 `offload` adapter 是二者之间的桥。

下一章是第三部分的收尾——**设备库与 libdevice**。设备代码里的 `sycl::sqrt`、`std::sin`、`assert`、复数运算，在 GPU 上没有系统 libc/libm 可用，靠 `libdevice/` 提供的 fallback 实现 + `libclc` 的 SPIR-V 内建补齐。第 19 章讲这些数学函数、fallback 机制和 `__spirv_*` 内建怎么被选择、链接进设备镜像——把设备端"标准库从哪来"这个悬念收掉。

**这一章的要点**

- **UR** 是 SYCL 的设备无关运行时层，顶层组件 `unified-runtime/`（**不在** `sycl/` 下）；`ur_api.h` 从 `scripts/core/*.yml` 生成；用 `ur_*_handle_t` 句柄对应 SYCL 对象。

- SYCL 经 **`sycl/source/detail/ur.{hpp,cpp}`** 的 `adapter.call<UrApiKind::…>` 接上 UR，闭合第 16 章脊柱最后一跳。
- 观察：`SYCL_UR_TRACE=1`（每次 UR 调用，位掩码；`-1` 全开）。**UR ≠ liboffload**（第 10 章）：理念同层次异；UR 的 `offload` adapter 桥接二者。
