## 一条总原则：as-if 规则

在看具体优化前，先建立统领全局的一条规则。编译器凭什么能把你的代码改得面目全非（消除拷贝、删掉循环、跳过函数调用）还不算出错？靠的是 **as-if 规则**：

<div class="keypoint">
<strong>as-if 规则</strong>
编译器可以对你的代码做<strong>任意变换</strong>，只要变换后程序的<strong>可观察行为</strong>不变（输出、对 <code>volatile</code> 变量的访问、文件/IO 等）。<br>
换句话说：编译器不保证"按你写的一步步执行"，只保证"最终看到的结果和你写的一样"。它可以删掉、合并、重排任何不影响可观察结果的操作。这就是它敢做激进优化的法律依据。
</div>

所以你写的 C++ 是"意图的描述"，不是"执行的脚本"。编译器读懂意图，然后生成达到同样效果的最快代码。下面逐个看它的招式。

## 内联：消除函数调用的开销

**内联（inlining）** 是最基础也最重要的优化：把一次函数调用**直接替换成函数体**，省掉调用的开销（跳转、传参、栈帧），更重要的是——内联后，函数体和调用点的代码能**一起被进一步优化**。

```cpp-asm
int add(int a, int b) { return a + b; }

int compute() {
    return add(add(1, 2), add(3, 4));   // 三次调用
}
```

点开 **🔬 看汇编 / 编译器处理**（`-O2`）：`compute` 里**没有任何 `call` 指令**，三次 `add` 全被内联，而且编译器进一步算出 `1+2+3+4 = 10`，直接返回 `10`。三次函数调用 + 三次加法，最终变成一个常量。

<div class="note">
<strong>inline 关键字≠"请内联"</strong>
现代编译器<strong>自主决定</strong>是否内联，<code>inline</code> 关键字对此几乎没有影响（它今天主要的作用是允许函数定义出现在多个翻译单元、绕过 ODR，见第 2 章）。是否内联由编译器权衡：小函数、调用频繁的倾向内联；大函数内联会导致代码膨胀，反而不内联。你不需要手动指挥它。
</div>

## 常量折叠与传播

编译器会在编译期算掉所有能算的常量表达式（**常量折叠**），并把已知的常量值代入后续计算（**常量传播**）：

```cpp-asm
int compute() {
    int a = 10;
    int b = 20;
    int c = a * b + 5;      // 全是编译期已知
    return c * 2;
}
```

看汇编：整个函数体被折叠成 `return 410`——`10*20+5=205`，`205*2=410`，运行时一次运算都不做。这和第 13 章的 `constexpr` 是一回事，区别在于：`constexpr` 是你**强制要求**编译期计算，而常量折叠是编译器**主动**做的优化，连普通变量都会被它看穿。

## 拷贝消除与 RVO：值语义为何不慢

回扣第 3、7 章的核心承诺——值语义看似到处拷贝，实际上编译器把不必要的拷贝**消除**了。最重要的是**返回值优化（RVO）**：函数返回一个局部对象时，编译器直接在调用者的内存里构造它，不产生任何拷贝或移动。

```cpp
#include <iostream>
#include <string>

struct Tracker {
    Tracker() { std::cout << "构造\n"; }
    Tracker(const Tracker&) { std::cout << "拷贝\n"; }
    Tracker(Tracker&&) noexcept { std::cout << "移动\n"; }
};

Tracker make() {
    Tracker t;         // 构造
    return t;          // 看似要拷贝/移动出去
}

int main() {
    Tracker result = make();   // 看似又要拷贝/移动一次
    std::cout << "完成\n";
}
```

输出：

```
构造
完成
```

只有一次"构造"！按值语义的字面意思本该有若干次拷贝/移动，但编译器用 RVO 把 `t` 直接构造在了 `result` 的位置——**零拷贝、零移动**。

<div class="keypoint">
<strong>RVO 是"零成本抽象"的关键支柱</strong>
正因为有 RVO 和拷贝消除，你才能<strong>放心地按值返回大对象</strong>（第 4 章说"返回值而非返回引用"就是基于此）——写起来是清晰的值语义，跑起来没有拷贝开销。C++17 起，对"返回纯右值"（如 <code>return Tracker();</code>）的拷贝消除甚至是<strong>强制</strong>的，编译器必须省掉。这就是为什么现代 C++ 的惯用法是大大方方按值传递和返回，把优化交给编译器。
</div>

## 去虚化：连虚函数都能优化掉

第 18 章说虚调用要查 vtable、通常不能内联。但当编译器能**确定对象的实际类型**时，它会做**去虚化（devirtualization）**——把虚调用变回直接调用，甚至内联：

```cpp-asm
struct Base { virtual int f() const { return 1; } };
struct Derived : Base { int f() const override { return 42; } };

int known() {
    Derived d;
    return d.f();       // 类型确定是 Derived → 去虚化，直接内联返回 42
}
```

看汇编：`known` 直接返回 `42`，没有 vtable 查找——编译器看出 `d` 确定是 `Derived`，把虚调用去虚化并内联了。这说明第 18 章"虚函数有开销"的说法有前提：**只有当编译器无法确定实际类型时（如通过基类指针），才真正付查表代价**；类型已知时，开销可能被优化掉。

## 死代码消除与循环优化

编译器还会删掉**永远不会执行或结果没被用到的代码**（死代码消除），并对循环做各种变换（展开、向量化、外提不变量）：

```cpp-asm
int deadCode(int x) {
    int unused = x * 999;   // 结果没被用到 → 删除
    if (false) {            // 永远不成立 → 整块删除
        return -1;
    }
    return x + 1;
}

long loopSum() {
    long sum = 0;
    for (int i = 0; i < 100; ++i) sum += i;   // 编译期可算 → 可能直接算出 4950
    return sum;
}
```

看汇编：`deadCode` 里 `unused` 那行和 `if (false)` 块消失得无影无踪，只剩 `return x + 1`；`loopSum` 的循环常常被编译器**直接算出结果 4950**（循环边界和运算都是编译期已知的）。你写的循环，编译器可能根本不跑。

<div class="warn">
<strong>不要为了"帮编译器"而牺牲可读性</strong>
知道了这些优化，一个常见的误区是开始手写"微优化"——手动内联、用位运算代替乘除、缓存中间变量……<strong>绝大多数情况下这是徒劳甚至有害的</strong>：编译器做这些比你在行，而扭曲的代码反而可能<em>妨碍</em>它优化。正确姿势是：<strong>写清晰、直接、符合惯用法的代码，信任编译器</strong>。只有当性能剖析（profiling）确认某处是瓶颈时，才针对性优化——而且先看汇编确认编译器没做到你要的，再动手。
</div>

## 优化级别：-O0 到 -O3

以上优化大多只在开启优化时才发生。编译器的优化级别用 `-O` 系列开关控制：

| 级别 | 含义 |
|---|---|
| `-O0` | 不优化（默认）。编译快、汇编忠实对应源码，适合**调试**。 |
| `-O1` | 基础优化。 |
| `-O2` | **推荐的发布级别**。开启绝大多数优化（内联、常量折叠、去虚化等），且不显著增加编译时间/体积。 |
| `-O3` | 更激进（更多循环展开、向量化）。有时更快，偶尔因代码膨胀反而变慢——需实测。 |
| `-Os` | 优化**体积**（嵌入式常用）。 |

<div class="keypoint">
<strong>关键认知：调试版和发布版是两个世界</strong>
<code>-O0</code> 下你看到的汇编几乎逐行对应源码——所有拷贝都真的发生、所有函数都真的调用。<code>-O2</code> 下则如前面所见，大量代码被消除、合并、算掉。<br>
所以：<strong>测性能一定要用 <code>-O2</code>（或更高）</strong>，用 <code>-O0</code> 测出来的"慢"毫无意义；而<strong>调试用 <code>-O0</code></strong>（或 <code>-Og</code>），因为优化后的代码和源码对不上，单步调试会乱跳。本书所有"看汇编"示例默认用 <code>-O2</code>，正是为了让你看到编译器的真实手笔。
</div>

## 动手：亲手验证"零成本抽象"

这一章最好的练习就是对比。下面这个函数用了一整套高层抽象（lambda、algorithm、值语义），看起来"开销很大"。把它放进 Compiler Explorer，用 **🔬 看汇编** 并确保是 `-O2`：

```cpp-asm
#include <vector>
#include <numeric>
#include <algorithm>

// 高层写法：抽象层层叠叠
int sumOfSquaresOfEvens() {
    std::vector<int> v{1, 2, 3, 4, 5, 6};
    int total = 0;
    for (int x : v) {
        if (x % 2 == 0) total += x * x;
    }
    return total;   // 2² + 4² + 6² = 56
}
```

看汇编你会发现：尽管源码里有 `vector`（堆分配）、循环、条件判断，编译器在 `-O2` 下能看穿整个计算是编译期确定的，很可能**直接返回常量 `56`**——vector 的堆分配、循环、判断统统消失。

<div class="note">
<strong>这就是"零成本抽象"的证据</strong>
第 1 章承诺过：C++ 的高层抽象编译后往往和手写底层代码一样快。这里你亲眼看到了——写得清晰（用容器、循环表达意图）和跑得快（汇编里只剩一个常量）在 C++ 里不是二选一。前提是：<strong>开优化</strong>，并<strong>信任编译器</strong>。
</div>

理解了编译器这一层，你对 C++ 的心智模型就完整了：从源码、值语义、生命周期，到模板实例化、再到编译器的最终优化。第五部分还剩两块——并发（下一章），以及把全书知识熔于一炉的压轴项目：从零实现一个 `vector`。

---

**这一章的要点**

- **as-if 规则**：编译器可任意变换代码，只要**可观察行为**不变——这是一切激进优化的依据。
- **内联**消除调用开销并让调用点与函数体一起优化；`inline` 关键字不再是给编译器的内联指令。
- **常量折叠/传播**在编译期算掉已知计算；**拷贝消除/RVO** 消除按值返回的拷贝，是"零成本抽象"和"放心按值返回"的基础（C++17 起部分强制）。
- **去虚化**在类型可知时把虚调用变直接调用甚至内联——虚函数的开销只在类型不可知时才真正付出。
- **死代码消除、循环优化**（展开/向量化/直接算出结果）进一步压缩代码。
- 用 **`-O2`** 测性能和看真实汇编，用 **`-O0`/`-Og`** 调试；**写清晰代码、信任编译器**，只在 profiling 确认瓶颈后才手动优化。
