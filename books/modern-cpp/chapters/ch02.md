# 第 2 章 从源码到可执行：编译、链接与翻译单元

## 这一章想让你带走什么

<div class="goal-box">
<p>在 Java / Python 里，"把代码跑起来"基本是个黑箱：<code>import</code> 一下就能用别的模块，运行时自会找到它们。C++ 的模型完全不同——它把工作切成<strong>编译</strong>和<strong>链接</strong>两个独立阶段，而这两个阶段各有各的报错。</p>
<p>读完这一章，你会理解一段 <code>.cpp</code> 是怎么变成可执行文件的，能一眼分清"编译错误"和"链接错误"，也会明白为什么 C++ 有头文件、以及 <code>#include</code> 到底干了什么。这些知识能帮你读懂 C++ 里一大半令人困惑的报错。</p>
</div>

## 你熟悉的模型 vs C++ 的模型

先对照一下。在 Python 里，你 `import utils`，解释器在运行时找到 `utils.py` 并加载它。在 Java 里，你用 `import com.foo.Bar`，编译器和 JVM 靠类路径（classpath）在编译期和运行期解析符号。两种语言里，**"我引用的东西定义在哪"这件事，工具链在很晚的时候才需要拍板**。

C++ 走的是一条更古老、更"分而治之"的路：

<div class="keypoint">
<strong>C++ 的核心事实：每个 .cpp 文件是独立编译的</strong>
编译器一次只看<strong>一个</strong> <code>.cpp</code> 文件，把它单独翻译成一个目标文件（<code>.o</code>），<strong>完全不知道其他 .cpp 的存在</strong>。等所有 <code>.cpp</code> 都各自编译完，再由<strong>链接器</strong>把这些目标文件拼到一起、解析彼此的引用，最终生成一个可执行文件。<br>
"编译"和"链接"是两个独立的阶段，由不同的工具负责，产生不同类型的错误。
</div>

1. **预处理（preprocessing）**：处理所有 `#` 开头的指令。`#include` 被**原样替换**成对应头文件的全部内容，宏被展开。产物是一个巨大的、纯粹的 `.cpp`。
2. **编译（compilation）**：把预处理后的源码翻译成汇编代码。语法错误、类型错误都在这一步被抓出来。
3. **汇编（assembly）**：把汇编代码转成机器码，打包成**目标文件**（`.o` / `.obj`）。
4. **链接（linking）**：把所有目标文件、以及用到的库拼在一起，解析每个"用到但不在本文件定义"的符号，生成最终可执行文件。

一条 `g++ main.cpp -o app` 命令其实一口气跑完了这四步。用 godbolt 或加 `-S` 参数，你能看到第 ② 步产出的汇编——这正是本书"🔬 看汇编"按钮做的事。

## 翻译单元：编译器眼里的"一个文件"

第 ① 步预处理之后得到的那个"展开了所有 include 的完整源码"，有个正式名字叫**翻译单元（translation unit）**。这是编译器处理的基本单位。

<div class="keypoint">
<strong>一个 .cpp + 它 include 的所有内容 = 一个翻译单元</strong>
编译器每次只吃一个翻译单元，把它编译成一个 <code>.o</code>。它对其他翻译单元里有什么<strong>一无所知</strong>。<br>
这就带来一个根本问题：如果 <code>a.cpp</code> 想调用定义在 <code>b.cpp</code> 里的函数，编译 <code>a.cpp</code> 时编译器根本看不到那个函数——它怎么知道该函数存不存在、参数对不对？
</div>

答案就是 C++ 的**声明 / 定义分离**机制，也是头文件存在的全部理由。

## 声明 vs 定义：C++ 报错的分水岭

这是本章最重要的一对概念，请务必分清：

- **声明（declaration）**：告诉编译器"有这么个东西，它长这样"——名字、类型、函数签名。不产生实体。
- **定义（definition）**：真正给出实体——函数的函数体、变量的存储。

```cpp-norun
int add(int a, int b);            // 声明：只说"有个 add 函数，签名如此"
int add(int a, int b) { return a + b; }   // 定义：给出函数体
```

编译一个翻译单元时，**只要有声明，编译器就满意了**——它相信"这东西的定义在别处，链接器会找到"。于是编译能通过。真正去核对"定义到底在不在"的，是**链接器**。

这就精确地划出了两类错误的分界：


<div class="note">
<strong>一眼分辨两类报错</strong>
· <strong>编译错误</strong>："expected ';'"、"'x' was not declared"、"no matching function"、"cannot convert"——都是<em>单个文件内部</em>说不通，在编译阶段报出。<br>
· <strong>链接错误</strong>：最典型的就是 <code>undefined reference to 'foo'</code>（找不到定义）和 <code>multiple definition of 'foo'</code>（定义重复了）。它们在<em>拼接多个文件</em>时才暴露，来自链接器（报错里常出现 <code>ld</code> 或 <code>collect2</code>）。<br>
搞混这两者，是初学 C++ 调试时最大的时间黑洞。附录 C 会给出一份完整的报错速查。
</div>

## 头文件：把声明"共享"给多个翻译单元

现在能理解头文件了。如果 `a.cpp` 要调用 `b.cpp` 里的 `add`，`a.cpp` 必须先**看到 `add` 的声明**。与其在每个用到它的文件里手抄一遍声明，不如把声明写进一个**头文件**，让大家 `#include` 它——`#include` 会在预处理阶段把那份声明原样粘贴进来。

典型的三文件结构是这样分工的：

```cpp-norun
// ---- math.h：放声明，给别人 include ----
int add(int a, int b);      // 只有声明

// ---- math.cpp：放定义，编译成 math.o ----
#include "math.h"
int add(int a, int b) { return a + b; }   // 唯一的定义

// ---- main.cpp：include 头文件拿到声明，编译成 main.o ----
#include "math.h"
int main() { return add(1, 2); }
```

编译时 `math.cpp` 和 `main.cpp` **各自独立**编译成 `.o`：`main.o` 里有一个"对 `add` 的未解析引用"，`math.o` 里有 `add` 的定义。链接器把两者一拼，引用就解析上了。这正是命令 `g++ main.cpp math.cpp -o app` 背后发生的事。

<div class="warn">
<strong>ODR：同一个定义，全程序只能有一份</strong>
C++ 有条铁律叫<strong>单一定义规则（One Definition Rule, ODR）</strong>：一个函数或全局变量，在整个程序里<strong>只能被定义一次</strong>。<br>
这解释了头文件的黄金准则：<strong>头文件里放声明，不放定义</strong>（<code>inline</code> 函数、模板、类定义除外——它们有特殊豁免）。如果你把一个普通函数的<em>定义</em>写进头文件，然后两个 <code>.cpp</code> 都 <code>#include</code> 它，链接器就会看到<em>两份</em>定义，报 <code>multiple definition</code>。这也是为什么头文件几乎总要配一个 <code>#pragma once</code>（或 include guard）——防止同一个头在一个翻译单元里被展开多次。
</div>

## 亲手制造、再读懂一个链接错误

单文件也能触发链接错误——只声明、不定义即可。下面这段能编译、但**不能链接**：

```cpp-norun
// 声明了一个函数，但整个程序里没有它的定义
int mysteryNumber();

int main() {
    return mysteryNumber();   // 编译没问题（有声明），链接失败（没定义）
}
```

把它放进 Compiler Explorer 并开启执行，你会看到类似这样的链接器报错（不是编译器报错）：

```
/usr/bin/ld: undefined reference to `mysteryNumber()'
collect2: error: ld returned 1 exit status
```

注意报错来自 `ld`（链接器）而非编译器——因为编译阶段一切合法，只是链接时找不到 `mysteryNumber` 的实体。补上定义，问题就消失。下面这个完整版就能正常编译、链接、运行：

```cpp
#include <iostream>

int mysteryNumber();          // 声明

int main() {
    std::cout << mysteryNumber() << '\n';
}

int mysteryNumber() {         // 定义（本例中放在同一个文件里）
    return 42;
}
```

编译器很宽容地允许"先用后定义"（只要用之前有声明），因为它知道链接器最后会来收尾。运行输出 `42`。

## 为什么值得理解这套模型

这套"独立编译 + 统一链接"的机制，是 C++ 很多特性的根源：

- **编译快、可增量**：改一个 `.cpp` 只需重编它自己，再重新链接，不必重编整个项目——大型项目靠这个才能维持可接受的构建时间。
- **头文件的一切规矩**（放声明、`#pragma once`、ODR）都是它的直接推论。
- **模板为什么通常整个写在头文件里**（第 11 章）——因为编译器必须在使用处看到模板的完整定义才能实例化，这也是这套模型下的必然结果。

<div class="note">
<strong>C++20 的模块（modules）</strong>
头文件这套机制有它的历史包袱：<code>#include</code> 本质是文本粘贴，同一个头被成百上千个翻译单元反复展开、反复编译，既慢又容易出微妙问题。C++20 引入了<strong>模块（modules）</strong>作为现代替代——用 <code>import</code> 取代 <code>#include</code>，语义更接近你熟悉的 Java / Python 模块，编译也更快。但截至目前工具链支持仍在成熟中，头文件在可见的未来仍是主流，所以本书以头文件模型为主来讲解。
</div>

理解了"源码如何变成程序"，我们就有了读懂报错的基础。接下来的第二部分（从下一章的值语义开始）会深入 C++ 的核心——对象、内存与生命周期。

**这一章的要点**

- C++ 每个 `.cpp` **独立编译**成目标文件，再由**链接器**拼接、解析符号——编译和链接是两个独立阶段。
- 一段旅程分四步：**预处理**（展开 `#include`）→ **编译**（→汇编）→ **汇编**（→`.o`）→ **链接**（→可执行）。
- 一个 `.cpp` 加它 include 的全部内容，构成一个**翻译单元**，是编译的基本单位。
- **声明**说"有这东西、长这样"（编译器满意即可），**定义**给出实体（链接器负责核对）。
- 由此分出两类报错：**编译错误**（单文件内部说不通）和**链接错误**（`undefined reference` / `multiple definition`，拼接时才暴露）。
- **头文件放声明**供多方 `#include`；受 **ODR** 约束，普通函数/变量全程序只能定义一次，故头文件配 `#pragma once`。
