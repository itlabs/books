## 启动线程

`std::thread` 启动一个线程，构造时传入要执行的可调用体（函数、lambda——第 16 章）。主线程要么 `join()`（等它结束）、要么 `detach()`（放任它独立运行）：

```cpp
#include <iostream>
#include <thread>

void worker(int id) {
    std::cout << "线程 " << id << " 工作中\n";
}

int main() {
    std::thread t1(worker, 1);              // 传函数 + 参数
    std::thread t2([]{ std::cout << "lambda 线程\n"; });   // 传 lambda

    t1.join();      // 等 t1 结束
    t2.join();      // 等 t2 结束
    std::cout << "主线程结束\n";
}
```

<div class="warn">
<strong>std::thread 析构前必须 join 或 detach</strong>
如果一个 <code>std::thread</code> 对象销毁时它仍"可 join"（既没 join 也没 detach），程序会直接 <code>std::terminate</code> 崩溃。这是个容易忘的坑——尤其在有异常时（异常跳过了你的 <code>join()</code>）。C++20 给了更安全的选择：<strong><code>std::jthread</code></strong>，它的析构函数会自动 join（RAII！），下面就用它。
</div>

## jthread：RAII 版的线程

`std::jthread`（C++20）是 `std::thread` 的 RAII 升级版——析构时自动 `join()`，你再也不会忘：

```cpp
#include <iostream>
#include <thread>
#include <vector>

int main() {
    std::vector<std::jthread> workers;
    for (int i = 0; i < 3; ++i) {
        workers.emplace_back([i]{ std::cout << "jthread " << i << " 完成\n"; });
    }
    std::cout << "已启动所有线程\n";
    // 不用手写 join：workers 里的 jthread 逐个析构时自动 join
}
```

`workers` 离开作用域时，每个 `jthread` 析构自动 join——这就是 RAII（第 5 章）应用到线程：**"清理"（join）绑定到对象生命周期，绝不遗漏。** 注意各线程打印的**顺序不确定**（谁先跑由调度器定），但程序保证所有线程都完成后才退出。

## 数据竞争：并发最危险的坑

现在讲并发的核心危险。当**多个线程同时访问同一块数据、且至少一个在写**，就构成**数据竞争（data race）**——在 C++ 里这是**未定义行为**，不是"结果可能不对"，而是"什么都可能发生"。

看一个经典错误：两个线程同时给一个计数器加数。

```cpp
#include <iostream>
#include <thread>
#include <vector>

int main() {
    long counter = 0;                        // 共享数据

    auto task = [&counter] {
        for (int i = 0; i < 100000; ++i) ++counter;   // ❌ 多线程同时 ++，数据竞争
    };

    std::vector<std::jthread> threads;
    for (int i = 0; i < 4; ++i) threads.emplace_back(task);
    threads.clear();                         // 等所有线程结束（析构即 join）

    // 期望 400000，但因数据竞争，实际几乎总是小于它，且每次不同
    std::cout << "counter = " << counter << " (期望 400000)\n";
}
```

`++counter` 看起来是一步，其实是"读—改—写"三步。两个线程交错执行时会互相覆盖，导致计数丢失。运行结果几乎总是小于 400000，且**每次都不一样**——这正是数据竞争的典型症状。

<div class="keypoint">
<strong>为什么 <code>++counter</code> 不安全</strong>
<code>++counter</code> 在机器层面是：① 从内存读到寄存器 → ② 寄存器 +1 → ③ 写回内存。两个线程可能都读到了同一个旧值（比如都读到 100），各自 +1 后都写回 101——本该是 102，却丢了一次。这类丢失在高频并发下大量发生。<strong>并发下，对共享可变数据的任何非同步访问都是错的。</strong>
</div>

解决数据竞争有两条路：**互斥锁**（保护临界区）和**原子操作**（让操作本身不可分割）。

## 互斥锁 + RAII：lock_guard

第 5 章你已经见过 `std::lock_guard`——用 RAII 管理互斥锁。这里正式用它保护共享数据：**同一时刻只有一个线程能进入被锁保护的临界区**。

```cpp
#include <iostream>
#include <thread>
#include <vector>
#include <mutex>

int main() {
    long counter = 0;
    std::mutex m;

    auto task = [&counter, &m] {
        for (int i = 0; i < 100000; ++i) {
            std::lock_guard<std::mutex> lock(m);   // 构造：加锁；析构：自动解锁
            ++counter;                             // 临界区：受保护，一次只有一个线程
        }
    };

    std::vector<std::jthread> threads;
    for (int i = 0; i < 4; ++i) threads.emplace_back(task);
    threads.clear();                               // 等全部结束

    std::cout << "counter = " << counter << " (期望 400000)\n";
}
```

输出：

```
counter = 400000 (期望 400000)
```

这次结果**确定是 400000**。`lock_guard` 构造时锁住 `m`，离开 `{}` 时自动解锁——即使临界区里抛异常也保证解锁（第 5 章）。**你不可能忘记解锁**，这就是 RAII 在并发里的价值。

<div class="note">
<strong>其他锁工具</strong>
· <code>std::unique_lock</code>：比 <code>lock_guard</code> 灵活（能手动 lock/unlock、能配合条件变量），但稍重。<br>
· <code>std::scoped_lock</code>（C++17）：能<strong>一次锁住多个互斥量</strong>且避免死锁，锁多个资源时用它。<br>
默认用最简单的 <code>lock_guard</code>，需要额外能力时才升级。
</div>

## 原子操作：无锁的轻量方案

对"计数器 +1"这种简单操作，加锁有点重。**`std::atomic<T>`** 提供**原子操作**——保证"读—改—写"作为一个不可分割的整体完成，无需显式加锁：

```cpp
#include <iostream>
#include <thread>
#include <vector>
#include <atomic>

int main() {
    std::atomic<long> counter{0};            // 原子计数器

    auto task = [&counter] {
        for (int i = 0; i < 100000; ++i) ++counter;   // 原子自增，无数据竞争
    };

    std::vector<std::jthread> threads;
    for (int i = 0; i < 4; ++i) threads.emplace_back(task);
    threads.clear();

    std::cout << "counter = " << counter << " (期望 400000)\n";
}
```

输出：

```
counter = 400000 (期望 400000)
```

同样得到正确的 400000，但代码更简洁、通常也更快——`++counter` 现在是一条原子指令，硬件保证它不可分割。

<div class="keypoint">
<strong>互斥锁 vs 原子操作</strong>
<ul>
<li><strong>原子操作</strong>：适合<strong>单个变量</strong>的简单操作（计数器、标志位、指针交换）。轻量、无锁、通常最快。</li>
<li><strong>互斥锁</strong>：适合保护<strong>一组相关数据</strong>或<strong>复合操作</strong>（比如"检查再修改"、更新多个字段）——这些没法用单个原子操作表达。</li>
</ul>
一句话：<strong>一个变量的简单更新用 atomic，一段复合逻辑或多个变量用 mutex。</strong>
</div>

## 内存模型：为什么"顺序"在并发里不可靠

最后简单认识"内存模型"这个概念。它回答一个反直觉的问题：**一个线程写入的顺序，另一个线程看到的会是同一个顺序吗？** 答案是——**不保证**。

<div class="keypoint">
<strong>编译器和 CPU 都会重排指令</strong>
第 19 章讲过 as-if 规则：编译器可以重排指令，只要<strong>单线程</strong>可观察行为不变。CPU 为了性能也会乱序执行、缓存写入。在单线程里这些重排你永远察觉不到；但在<strong>多线程</strong>里，线程 A 眼中"先写 x 再写 y"，线程 B 看到的可能是"先 y 后 x"——因为那些优化只保证单线程的一致性。<br>
<strong>内存模型</strong>就是一套规则，规定"在什么同步措施下，一个线程的写入才保证对另一个线程可见、且顺序可靠"。
</div>

好消息是：**只要你正确使用 mutex 和 atomic，就不用直接和内存模型的复杂细节打交道**——它们默认提供了足够强的顺序保证（顺序一致性）。`atomic` 默认的操作会建立线程间的"同步点",确保数据可见性和顺序。

<div class="warn">
<strong>不要碰 memory_order 的弱序，除非你真的知道自己在做什么</strong>
<code>std::atomic</code> 允许你指定更弱的内存序（<code>memory_order_relaxed</code> 等）以榨取性能，这是并发编程里<strong>最难、最容易出错</strong>的领域——连专家都常写错。<strong>默认（顺序一致）几乎总是对的选择</strong>，只有在性能剖析证明它是瓶颈、且你透彻理解内存模型时，才考虑弱序。本书的立场：知道它存在，敬而远之。
</div>

## 一个务实的并发建议

<div class="keypoint">
<strong>写并发代码的优先级</strong>
<ol>
<li><strong>最好不共享可变状态</strong>：能用消息传递、任务返回值（<code>std::future</code>）、或让每个线程处理独立数据，就别共享——没有共享就没有数据竞争。这也是值语义（独立副本）在并发里的价值。</li>
<li><strong>必须共享时，用最简单的工具</strong>：简单计数/标志用 <code>atomic</code>，复合逻辑用 <code>mutex</code> + <code>lock_guard</code>。</li>
<li><strong>用 RAII 管理一切</strong>：<code>jthread</code> 自动 join，<code>lock_guard</code> 自动解锁——让生命周期机制替你保证正确性。</li>
<li><strong>远离手动内存序</strong>：默认顺序一致，别碰弱序。</li>
</ol>
</div>

## 动手：数据竞争 vs 正确同步

下面把"未同步"和"用 atomic 同步"放在一起对比。未同步版的结果每次运行都可能不同（且通常错误）；同步版稳定正确。运行几次感受差异：

```cpp
#include <iostream>
#include <thread>
#include <vector>
#include <atomic>

int main() {
    const int N = 4, PER = 50000;

    // 未同步：数据竞争，结果不可靠
    long racy = 0;
    {
        std::vector<std::jthread> ts;
        for (int i = 0; i < N; ++i)
            ts.emplace_back([&racy]{ for (int j = 0; j < PER; ++j) ++racy; });
    }   // 等全部结束

    // 同步：atomic，结果确定
    std::atomic<long> safe{0};
    {
        std::vector<std::jthread> ts;
        for (int i = 0; i < N; ++i)
            ts.emplace_back([&safe]{ for (int j = 0; j < PER; ++j) ++safe; });
    }

    std::cout << "期望值:   " << (long)N * PER << '\n';
    std::cout << "atomic:   " << safe << "  (总是正确)\n";
    std::cout << "racy:     " << racy << "  (通常偏小，每次不同)\n";
}
```

`atomic` 那行**总是**打印 200000；`racy` 那行则通常小于 200000、且每次运行的值都不一样——**你亲眼看到了未定义行为的数据竞争与正确同步的区别**。（注：`racy` 是真正的数据竞争、属未定义行为，这里仅用于演示危害，真实代码绝不能这么写。）

---

**这一章的要点**

- C++ 并发是真并行（无 GIL）、无运行时兜底：**数据竞争是未定义行为**。
- `std::thread` 析构前必须 `join`/`detach`；优先用 **`std::jthread`**（C++20），析构自动 join（RAII）。
- **数据竞争**：多线程同时访问共享数据且至少一个在写。`++counter` 非原子，会丢更新。
- 两种解法：**`mutex` + `lock_guard`**（RAII 自动解锁，保护临界区/复合操作）和 **`std::atomic`**（单变量简单操作，无锁更轻）。
- **内存模型**：编译器/CPU 会重排，跨线程的顺序与可见性不保证——但正确用 mutex/atomic（默认顺序一致）就无需操心细节；**远离手动弱内存序**。
- 优先级：**尽量不共享可变状态** > 用最简单的同步工具 > RAII 管理 > 远离手动内存序。
