# 第 18 章 多态：虚函数、vtable 与静态多态

## 这一章想让你带走什么

<div class="goal-box">
<p>你在 Java 里写多态是本能：方法默认就是虚的，子类重写、父类引用调用，运行时分派到正确的实现。C++ 也能做同样的事，但有两个关键不同：多态<strong>不是默认的</strong>（要显式 <code>virtual</code>），而且 C++ 还有一种 Java 没有的<strong>静态多态</strong>（编译期完成，零开销）。</p>
<p>读完你会理解虚函数背后的 <strong>vtable</strong> 机制（配汇编看个究竟）、为什么基类析构函数必须是虚的、以及运行时多态和编译期多态各自的取舍。</p>
</div>

## 运行时多态：virtual 是显式的

先看 C++ 的动态多态，形态和 Java 类似——基类定义 `virtual` 方法，派生类 `override` 重写，通过基类指针/引用调用时分派到实际类型的实现：

```cpp
#include <iostream>
#include <memory>
#include <vector>

struct Shape {
    virtual double area() const = 0;      // 纯虚函数：= 0 表示"抽象，没有实现"
    virtual ~Shape() = default;           // 虚析构（下文解释为何必须）
};

struct Circle : Shape {
    double r;
    Circle(double r) : r(r) {}
    double area() const override { return 3.14159 * r * r; }   // override 重写
};

struct Rectangle : Shape {
    double w, h;
    Rectangle(double w, double h) : w(w), h(h) {}
    double area() const override { return w * h; }
};

int main() {
    std::vector<std::unique_ptr<Shape>> shapes;
    shapes.push_back(std::make_unique<Circle>(2.0));
    shapes.push_back(std::make_unique<Rectangle>(3.0, 4.0));

    for (const auto& s : shapes) {
        std::cout << "面积: " << s->area() << '\n';   // 分派到实际类型
    }
}
```

输出：

```
面积: 12.5664
面积: 12
```

通过 `Shape*` 调用 `area()`，运行时自动分派到 `Circle::area` 或 `Rectangle::area`——这就是**动态多态**。几个 C++ 特有的点：

<div class="keypoint">
<strong>C++ 多态与 Java 的关键差异</strong>
<ul>
<li><strong>多态不是默认的</strong>：只有标了 <code>virtual</code> 的函数才参与动态分派。Java 所有方法默认虚；C++ 默认<strong>非虚</strong>（普通函数调用，编译期定死，更快）。你要多态就得显式声明。</li>
<li><strong>纯虚函数</strong> <code>= 0</code>：等价 Java 的抽象方法，使类成为抽象类，不能实例化。</li>
<li><strong><code>override</code> 关键字</strong>：显式标注"我在重写基类虚函数"。强烈建议每次重写都写上——如果签名写错（没真正重写到），编译器会报错，挡住一类隐蔽 bug。</li>
<li><strong>多态必须通过指针或引用</strong>（这里用 <code>unique_ptr</code>）——按值持有会"切片"，下文详述。</li>
</ul>
</div>

## vtable：虚函数背后的机制

"运行时如何知道该调 `Circle::area` 还是 `Rectangle::area`"？答案是 **vtable（虚函数表）**。理解它，你就理解了 C++（以及大多数语言）实现多态的底层原理。

<div class="keypoint">
<strong>vtable 机制一句话</strong>
每个<strong>有虚函数的类</strong>，编译器为它生成一张 <strong>vtable</strong>——一个函数指针数组，装着该类各个虚函数的实际地址。每个<strong>对象</strong>则多存一个隐藏指针 <code>vptr</code>，指向自己类的 vtable。<br>
调用 <code>s->area()</code> 时，运行时做的是：<strong>顺着对象的 vptr 找到 vtable，从中取出 area 的地址，跳过去执行</strong>。所以调哪个实现，取决于对象的 vptr 指向哪张表——而这在对象构造时就按其实际类型设好了。
</div>

这套机制的**代价**是：虚调用比普通调用多一次间接寻址（查表），且通常无法被内联（编译期不知道实际类型）。这就是为什么 C++ 不把虚设为默认——**你只在需要多态时才付这个代价**。

用汇编看虚调用和普通调用的区别。下面 `viaVirtual` 通过基类指针调虚函数，`direct` 直接调具体类型：

```cpp-asm
struct Base { virtual int f() const; };
struct Derived : Base { int f() const override; };

int viaVirtual(const Base& b) { return b.f(); }   // 虚调用：查 vtable
int direct(const Derived& d)   { return d.f(); }   // 也可能被去虚化
```

点开 **🔬 看汇编 / 编译器处理**：`viaVirtual` 里能看到通过 vtable 的**间接调用**（先加载 vptr、再从表里取地址、`call` 那个地址）。而 `direct` 因为编译器确切知道类型是 `Derived`，常常能**去虚化（devirtualization）**成直接调用甚至内联——这是编译器的一项重要优化（第 19 章）。

## 为什么基类析构函数必须是 virtual

这是 C++ 多态里最重要的一条安全规则，也是最容易被忽略的坑。看前面 `Shape` 我特意写了 `virtual ~Shape()`。如果**不写** virtual 会怎样？

<div class="warn">
<strong>通过基类指针 delete 派生对象，基类析构必须是 virtual</strong>
当你 <code>delete</code> 一个 <code>Shape*</code>（实际指向 <code>Circle</code>）时：<br>
· 若 <code>~Shape</code> 是 <strong>virtual</strong>：经 vtable 分派，先调 <code>~Circle</code> 再调 <code>~Shape</code>，正确清理。<br>
· 若 <code>~Shape</code> <strong>非 virtual</strong>：只调用 <code>~Shape</code>，<code>Circle</code> 特有的成员<strong>不会被析构</strong>——资源泄漏，未定义行为。<br>
铁律：<strong>一个类只要打算被继承并通过基类指针多态使用，它的析构函数就必须是 virtual。</strong>（用 <code>unique_ptr&lt;Shape&gt;</code> 时同理——它内部也是通过 <code>Shape*</code> 删除。）
</div>

## 对象切片：值语义与多态的冲突

第 3 章讲过 C++ 默认值语义。多态却要求指针/引用——这两者相撞会产生一个 Java 程序员完全陌生的 bug：**对象切片（slicing）**。

```cpp
#include <iostream>

struct Shape {
    virtual double area() const { return 0; }
    virtual ~Shape() = default;
};
struct Circle : Shape {
    double r;
    Circle(double r) : r(r) {}
    double area() const override { return 3.14159 * r * r; }
};

void byValue(Shape s) {                     // ❌ 按值接收基类：发生切片
    std::cout << "按值: " << s.area() << '\n';
}
void byRef(const Shape& s) {                // ✅ 按引用：保留多态
    std::cout << "按引用: " << s.area() << '\n';
}

int main() {
    Circle c(2.0);
    byValue(c);    // Circle 被"切"成 Shape，area() 调的是 Shape 版本 → 0
    byRef(c);      // 多态保留，调 Circle::area → 12.5664
}
```

输出：

```
按值: 0
按引用: 12.5664
```

`byValue(c)` 把一个 `Circle` **拷贝成了 `Shape`**——派生部分（`r` 和 `Circle` 的行为）被"切掉"了，只留下基类的空壳，`area()` 返回基类版本的 `0`。

<div class="keypoint">
<strong>切片：多态类型永远别按值传递</strong>
按值传递基类会截断派生部分。所以<strong>多态对象一律通过指针或引用使用</strong>——<code>const Base&amp;</code> 传参、<code>unique_ptr&lt;Base&gt;</code> 存储。这也是为什么前面的 <code>shapes</code> 用的是 <code>vector&lt;unique_ptr&lt;Shape&gt;&gt;</code> 而不是 <code>vector&lt;Shape&gt;</code>。Java 没有这个问题，因为它的对象变量本就全是引用；C++ 的值语义让切片成为一个真实的陷阱。
</div>

## 静态多态：编译期的多态，零开销

C++ 还有一种 Java 没有的多态——**静态多态**，用模板（第 11 章）在**编译期**实现"同一段代码适配不同类型"，完全不需要 vtable、不付运行时代价。

回想第 11 章的鸭子类型：模板只要求类型支持用到的操作。所以"对任何有 `area()` 的类型都能工作"根本不需要继承：

```cpp
#include <iostream>

// 不需要继承、不需要 virtual：编译期多态
struct Circle    { double area() const { return 3.14159 * r * r; } double r; };
struct Rectangle { double area() const { return w * h; } double w, h; };

// 模板：对任何"有 area()"的类型都适用，编译期为每种类型生成专属代码
template <typename Shape>
void printArea(const Shape& s) {
    std::cout << "面积: " << s.area() << '\n';   // 直接调用，无 vtable，可内联
}

int main() {
    Circle c{.r = 2.0};
    Rectangle r{.w = 3.0, .h = 4.0};
    printArea(c);    // 编译期确定调 Circle::area
    printArea(r);    // 编译期确定调 Rectangle::area
}
```

输出：

```
面积: 12.5664
面积: 12
```

`Circle` 和 `Rectangle` **没有共同基类、没有 virtual**，但 `printArea` 照样对两者都工作——因为模板在编译期为每种类型生成专属版本，`s.area()` 是直接调用，能被内联，**零运行时开销**。

<div class="keypoint">
<strong>动态多态 vs 静态多态，怎么选？</strong>
<ul>
<li><strong>动态多态（virtual）</strong>：类型集合在<strong>运行时</strong>才确定时用——比如一个 <code>vector&lt;unique_ptr&lt;Shape&gt;&gt;</code> 要装运行时才决定的各种形状、或插件式架构。代价是虚调用开销。</li>
<li><strong>静态多态（模板）</strong>：类型在<strong>编译期</strong>就已知时用——追求极致性能、能内联。代价是代码膨胀、错误信息复杂（第 11、12 章）。</li>
</ul>
一句话：<strong>运行时才知道类型 → 虚函数；编译期就知道类型 → 模板。</strong>
</div>

<div class="note">
<strong>CRTP：静态多态的进阶招式</strong>
有一种叫 <strong>CRTP</strong>（奇异递归模板模式，Curiously Recurring Template Pattern）的技巧——让派生类把自己作为模板参数传给基类：<code>class Derived : Base&lt;Derived&gt;</code>。它能在编译期实现"基类调用派生类方法"的效果，达到虚函数的多态形态却零开销。这是高性能 C++ 库（如一些数学库、Eigen）的常见手法，属于进阶话题，知道有这么个东西即可。
</div>

## 动手：对比切片与正确的多态

下面这段程序把动态多态、切片陷阱、正确用法放在一起。运行前先预测每行输出——尤其 `slice` 那次：

```cpp
#include <iostream>
#include <memory>
#include <vector>

struct Animal {
    virtual std::string sound() const { return "..."; }
    virtual ~Animal() = default;
};
struct Dog : Animal { std::string sound() const override { return "汪"; } };
struct Cat : Animal { std::string sound() const override { return "喵"; } };

void bySlice(Animal a) { std::cout << "切片: " << a.sound() << '\n'; }   // 切片
void byRef(const Animal& a) { std::cout << "引用: " << a.sound() << '\n'; }

int main() {
    Dog d;
    bySlice(d);      // 切片 → Animal::sound → "..."
    byRef(d);        // 多态 → Dog::sound → "汪"

    // 正确的多态容器
    std::vector<std::unique_ptr<Animal>> zoo;
    zoo.push_back(std::make_unique<Dog>());
    zoo.push_back(std::make_unique<Cat>());
    for (const auto& a : zoo) std::cout << "动物园: " << a->sound() << '\n';
}
```

预期输出：

```
切片: ...
引用: 汪
动物园: 汪
动物园: 喵
```

第一行 `切片: ...` 就是把 `Dog` 按值传给 `Animal` 参数时派生部分被切掉的后果；改用引用（第二行）多态才保留。而 `zoo` 用 `unique_ptr<Animal>` 存储，遍历时正确分派到各自的 `sound()`——**这是 C++ 里做运行时多态的标准姿势**。

第五部分继续深入底层。下一章我们把前面反复提到的"编译器优化"正式展开——内联、去虚化、RVO、循环优化，看编译器到底为你的代码做了多少事。

---

**这一章的要点**

- C++ 动态多态需**显式 `virtual`**（默认非虚，更快）；纯虚 `= 0` 是抽象方法；重写加 `override` 让编译器帮你查错。
- 虚函数靠 **vtable** 实现：每个多态对象带 `vptr` 指向其类的虚函数表，虚调用经表间接跳转——代价是多一次寻址、通常不能内联。
- **基类析构函数必须是 `virtual`**（当通过基类指针删除派生对象时），否则派生部分不被析构，泄漏。
- **对象切片**：多态类型按值传递会截断派生部分——多态对象一律用**指针或引用**（`const Base&`、`unique_ptr<Base>`）。
- **静态多态**（模板）在编译期实现多态，无 vtable、可内联、零开销——类型编译期已知时用；运行时才知类型则用虚函数。
- 进阶手法 **CRTP** 能以静态多态达到虚函数的形态而零开销。
