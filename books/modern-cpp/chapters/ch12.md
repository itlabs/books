# 第 12 章 concepts 与约束：模板的类型系统

## 这一章想让你带走什么

<div class="goal-box">
<p>上一章结尾我们看到模板的两大痛点：约束是<strong>隐式的</strong>（藏在函数体里），报错是<strong>灾难性的</strong>（在实例化深处爆炸）。C++20 的 <strong>concepts</strong> 就是来治这两个病的。</p>
<p>读完这一章，你会知道如何给模板参数写上<strong>显式的、编译器能检查的约束</strong>，让"用错类型"在调用处就被清晰地拒绝——报错信息从"一百行天书"变成"一句人话"。这让 C++ 泛型第一次拥有了像样的"类型系统"。</p>
</div>

## 问题重现：没有约束的模板

回忆上一章的 `maxOf`。它要求 `T` 支持 `>`，但这个要求没写在任何地方：

```cpp-norun
template <typename T>
T maxOf(T a, T b) { return a > b ? a : b; }   // "T 要支持 >" 是隐藏的
```

当有人传了不支持 `>` 的类型，编译器会一路深入实例化、直到 `a > b` 那行才失败，然后吐出一大坨指向标准库内部的错误。**问题的本质是：约束没有出现在接口上，编译器无法在调用处就拦住你。**

Java 用**有界泛型**解决这个：`<T extends Comparable<T>>`，把约束写进签名。C++20 的 concepts 做的是同样的事，但更强大——它不要求类型去继承或实现某个接口，而是直接描述"这个类型需要支持哪些操作"。

## 第一个 concept：给模板加上约束

`concept` 是一个**编译期的布尔条件**，描述"一个类型要满足什么"。我们定义一个"可比较大小"的 concept，然后用它约束 `maxOf`：

```cpp
#include <iostream>

// 定义一个 concept：T 要支持 a > b，且结果能当 bool 用
template <typename T>
concept Comparable = requires(T a, T b) {
    { a > b } -> std::convertible_to<bool>;
};

// 用 concept 约束模板参数：把 typename 换成 concept 名
template <Comparable T>
T maxOf(T a, T b) {
    return a > b ? a : b;
}

int main() {
    std::cout << maxOf(3, 7) << '\n';        // int 满足 Comparable ✅
    std::cout << maxOf(2.5, 1.5) << '\n';     // double 满足 ✅
}
```

`template <Comparable T>` 读作"对任何满足 `Comparable` 的类型 T"。约束现在**写在了签名上**，一目了然。`requires(T a, T b) { ... }` 块描述具体要求：里面的 `{ a > b } -> std::convertible_to<bool>` 意思是"表达式 `a > b` 必须合法，且其结果能转换成 `bool`"。

<div class="keypoint">
<strong>concept 把隐式约束变成显式契约</strong>
以前 "T 要支持 &gt;" 藏在函数体里，靠实例化失败来"事后发现"。现在它是签名的一部分，编译器在<strong>调用处</strong>就检查：类型满足 → 放行；不满足 → 立刻报错，且报错直指"你的类型不满足 Comparable"，而不是标准库内部的天书。
</div>

## 见证报错的天壤之别

这是 concepts 最直观的价值。对一个不支持 `>` 的类型调用受约束的 `maxOf`：

```cpp-norun
#include <iostream>

template <typename T>
concept Comparable = requires(T a, T b) {
    { a > b } -> std::convertible_to<bool>;
};

template <Comparable T>
T maxOf(T a, T b) { return a > b ? a : b; }

struct Widget {};   // 不支持 >

int main() {
    Widget x, y;
    maxOf(x, y);    // ❌ 报错，但这次是"人话"
}
```

把它放进 Compiler Explorer。这次的报错会清晰地说明：

```
error: no matching function for call to 'maxOf(Widget&, Widget&)'
note: candidate: template<class T> requires Comparable<T> T maxOf(T, T)
note: constraints not satisfied
note: the required expression '(a > b)' is invalid
```

对比上一章那段深入标准库内部的冗长错误——现在编译器直接告诉你：**`maxOf` 需要 `Comparable`，而 `Widget` 不满足，因为 `a > b` 不合法。** 一句话点破。这就是把约束提到接口层面的回报。

## 标准库自带一大批现成 concepts

你不必事事自己定义 concept。`<concepts>` 头文件提供了一整套常用约束，覆盖了绝大多数日常需要：

| concept | 含义 |
|---|---|
| `std::integral<T>` | T 是整数类型 |
| `std::floating_point<T>` | T 是浮点类型 |
| `std::convertible_to<From, To>` | From 能转换成 To |
| `std::same_as<T, U>` | T 和 U 是同一类型 |
| `std::totally_ordered<T>` | T 支持全套比较 `< <= > >= == !=` |
| `std::copyable<T>` / `std::movable<T>` | T 可拷贝 / 可移动 |

用标准 concept 约束一个"只接受数值类型"的函数：

```cpp
#include <iostream>
#include <concepts>

// 只接受整数或浮点，拒绝其他一切
template <typename T>
    requires std::integral<T> || std::floating_point<T>
T square(T x) {
    return x * x;
}

int main() {
    std::cout << square(5) << '\n';       // int ✅ → 25
    std::cout << square(2.5) << '\n';      // double ✅ → 6.25
    // square("hi");                       // ❌ string 不满足约束，调用处即报错
}
```

这里用了另一种写法：**`requires` 子句**（写在模板参数列表后面）。它比"把 concept 名当类型前缀"更灵活——能用 `||`、`&&` 组合多个约束。两种写法可以混用，看哪个更清晰。

<div class="note">
<strong>三种等价的约束写法</strong>
下面三种都表达"T 必须是整数"，选最顺眼的即可：
<pre>template &lt;std::integral T&gt; void f(T x);              // 前缀写法
template &lt;typename T&gt; requires std::integral&lt;T&gt;      // requires 子句
void f(std::integral auto x);                        // 简写：auto 前加 concept</pre>
第三种（<code>std::integral auto</code>）最简洁，适合简单约束的函数参数。
</div>

## 用约束做重载：编译期分派

concepts 还解锁了一个优雅的能力：**根据类型是否满足某约束，选择不同的实现**。以前要用复杂的模板技巧（SFINAE）才能做到的事，现在直白得多。

下面写两个同名函数，一个专门处理整数、一个处理浮点，编译器按类型自动选对版本：

```cpp
#include <iostream>
#include <concepts>

// 整数版本
template <std::integral T>
void describe(T x) {
    std::cout << x << " 是整数\n";
}

// 浮点版本
template <std::floating_point T>
void describe(T x) {
    std::cout << x << " 是浮点数\n";
}

int main() {
    describe(42);       // 选整数版本
    describe(3.14);     // 选浮点版本
    describe(100L);     // long 也是整数 → 整数版本
}
```

预期输出：

```
42 是整数
3.14 是浮点数
100 是整数
```

编译器根据实参类型满足哪个 concept，自动挑选对应的 `describe`。这叫**约束下的重载决议**——两个候选各自带着不同的 concept 约束，谁匹配就用谁。这比运行时的 `if (type == ...)` 判断更高效（编译期就定了）也更清晰。

<div class="keypoint">
<strong>concepts 让泛型代码"可读、可诊断、可分派"</strong>
<ul>
<li><strong>可读</strong>：约束写在签名上，读者一眼看懂这个模板要什么类型。</li>
<li><strong>可诊断</strong>：违反约束在调用处报错，信息清晰。</li>
<li><strong>可分派</strong>：不同约束选不同实现，编译期完成，零运行时开销。</li>
</ul>
它给了 C++ 模板一套真正的"类型系统"，把上一章那种"无约束的鸭子类型"驯化成了"有契约的泛型"。
</div>

## 自定义 concept 组合复杂需求

`requires` 块能描述相当细致的要求——不只是"支持某运算符"，还能要求"有某个成员函数""某表达式返回特定类型"。定义一个"像容器"的 concept：

```cpp
#include <iostream>
#include <vector>
#include <string>
#include <concepts>

// 要求：有 size() 返回可转 size_t，有 begin()/end() 能遍历
template <typename C>
concept Container = requires(C c) {
    { c.size() } -> std::convertible_to<std::size_t>;
    c.begin();
    c.end();
};

template <Container C>
void report(const C& c) {
    std::cout << "容器有 " << c.size() << " 个元素\n";
}

int main() {
    std::vector<int> v{1, 2, 3};
    std::string s = "hello";
    report(v);      // vector 满足 Container ✅ → 3
    report(s);      // string 也满足 ✅ → 5
    // report(42);  // ❌ int 没有 size()/begin()/end()，调用处即报错
}
```

`Container` 这个 concept 把"是个容器"的模糊概念，变成了三条**机器可检查**的具体要求。任何满足这三条的类型——无论是标准库的 `vector`、`string`，还是你自定义的类型——都自动满足 `Container`,无需声明"我实现了容器接口"。**这就是 concepts 版的鸭子类型：既保留了灵活，又补上了契约。**

## 动手：用 concept 约束把错误挡在门口

下面这段程序定义一个"可打印"concept，并用它约束打印函数。正常部分能编译运行；注释里的 `// ❌` 行放开后会在**调用处**被清晰拒绝——动手试试，感受和上一章无约束模板报错的区别：

```cpp
#include <iostream>
#include <concepts>
#include <string>

// 可打印：要求 std::cout << x 合法
template <typename T>
concept Printable = requires(std::ostream& os, T x) {
    { os << x } -> std::same_as<std::ostream&>;
};

template <Printable T>
void logValue(const T& x) {
    std::cout << "值 = " << x << '\n';
}

struct Opaque {};   // 没有定义 operator<<

int main() {
    logValue(42);           // int 可打印 ✅
    logValue(3.14);         // double 可打印 ✅
    logValue(std::string("hi"));   // string 可打印 ✅

    // Opaque o;
    // logValue(o);         // ❌ 放开：编译器直接说 Opaque 不满足 Printable
}
```

预期输出：

```
值 = 42
值 = 3.14
值 = hi
```

`logValue` 的签名 `template <Printable T>` 就是它的契约：只接受能被 `std::cout` 打印的类型。传入 `Opaque` 会在调用处立刻失败，理由清楚——**约束把"能不能用"的判断从实例化深处提到了接口表面。**

到这里，泛型的"是什么"（模板）和"要什么"（concepts）就都讲清楚了。下一章我们转向编译期的另一个维度：不只让类型在编译期确定，还要让**计算**本身在编译期就完成——`constexpr` 与 `consteval`。

---

**这一章的要点**

- **concept** 是描述"类型需满足什么"的编译期布尔条件，把模板的**隐式约束**变成写在签名上的**显式契约**。
- 用 `template <ConceptName T>`、`requires` 子句、或 `ConceptName auto` 三种等价方式给模板加约束。
- 最大的实际收益是**报错**：违反约束在**调用处**被清晰拒绝，而非在实例化深处爆炸出天书。
- 标准库 `<concepts>` 提供大量现成约束（`std::integral`、`std::floating_point`、`std::convertible_to` 等）。
- 不同 concept 约束的同名函数可做**编译期重载分派**，零运行时开销。
- `requires` 块能描述细致要求（成员函数、表达式返回类型等），实现"有契约的鸭子类型"。
