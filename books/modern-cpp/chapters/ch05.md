# 第 5 章 RAII：没有 GC，如何自动清理

## 这一章想让你带走什么

<div class="goal-box">
<p>上一章你确认了 C++ 的铁律：<strong>对象离开作用域时，析构函数确定地、立即地被调用。</strong>这一章讲 C++ 如何把这条铁律变成一件威力惊人的武器——<strong>RAII</strong>：把资源的获取与释放绑定到对象的生命周期上。</p>
<p>读完你会明白，为什么没有 GC 的 C++ 反而能做到"几乎不用手动释放任何资源"，以及为什么 RAII 被认为是 C++ 最重要的设计思想。</p>
</div>

## 先看没有 RAII 的世界有多痛

假设你要打开一个文件、用完关掉。用最原始的手动方式写，会是这样：

```cpp-norun
void process() {
    FILE* f = std::fopen("data.txt", "r");
    if (!f) return;

    doSomething(f);      // 如果这里抛异常呢？
    if (error) return;   // 如果从这里提前返回呢？

    std::fclose(f);      // 只有"一路顺利走到这"才会执行
}
```

`fclose` 只在函数正常执行到最后一行时才被调用。但凡中途 `return`、或者 `doSomething` 抛了异常，`fclose` 就被跳过——文件句柄泄漏了。你熟悉这个问题：

<div class="compare">
<div class="known">
<div class="compare-head">Java：try-with-resources</div>

```java
try (var f = new FileReader("data.txt")) {
    doSomething(f);
}   // 无论正常还是异常
    // f.close() 都会被调用
```

</div>
<div class="cpp-side">
<div class="compare-head">Python：with 语句</div>

```python
with open("data.txt") as f:
    do_something(f)
# 无论正常还是异常
# 文件都会被关闭
```

</div>
</div>

Java 和 Python 都发明了**专门的语法**（`try-with-resources`、`with`）来解决"无论如何都要清理"。C++ 的答案不一样——它不需要专门语法，因为**每个对象的析构函数本来就"无论如何都会被调用"**。

## RAII：资源的生命周期 = 对象的生命周期

RAII 的全称是 **Resource Acquisition Is Initialization**（资源获取即初始化），但这个名字起得不好，容易让人抓不住重点。它真正的意思是：

<div class="keypoint">
<strong>RAII 一句话</strong>
把资源<strong>获取</strong>写进构造函数，把资源<strong>释放</strong>写进析构函数。于是资源的生命周期就和对象的生命周期<strong>绑定</strong>在了一起——对象一诞生就持有资源，对象一销毁就归还资源。而对象的销毁（上一章的铁律）是<strong>自动且确定</strong>的，所以资源的释放也就自动且确定了。
</div>

我们亲手写一个管理文件的 RAII 类，把上面那段脆弱的代码变得万无一失：

```cpp
#include <cstdio>
#include <iostream>

// 一个最小的 RAII 文件包装：构造时打开，析构时关闭
class File {
    std::FILE* f_;
public:
    File(const char* path, const char* mode) : f_(std::fopen(path, mode)) {
        std::cout << "  [File] 打开\n";
    }
    ~File() {
        if (f_) { std::fclose(f_); std::cout << "  [File] 关闭\n"; }
    }
    std::FILE* get() const { return f_; }
    bool ok() const { return f_ != nullptr; }
};

void mayThrow(bool boom) {
    File f("/dev/null", "w");        // 构造：获取资源
    std::cout << "  使用文件…\n";
    if (boom) throw std::runtime_error("出错了");
    std::cout << "  正常用完\n";
}   // 无论是否抛异常，f 离开作用域都会析构 → 文件被关闭

int main() {
    std::cout << "正常路径：\n";
    mayThrow(false);

    std::cout << "异常路径：\n";
    try { mayThrow(true); }
    catch (const std::exception& e) { std::cout << "  捕获：" << e.what() << '\n'; }
}
```

运行输出：

```
正常路径：
  [File] 打开
  使用文件…
  正常用完
  [File] 关闭
异常路径：
  [File] 打开
  使用文件…
  [File] 关闭
  捕获：出错了
```

盯住**异常路径**：`mayThrow(true)` 在中途抛了异常，"正常用完"那行没执行——但 `[File] 关闭` **照样打印了**。这就是 RAII 的魔力：异常在栈上向上传播时，会把沿途每一层作用域里的局部对象逐个析构，这个过程叫**栈展开（stack unwinding）**。文件在栈展开时被自动关闭，然后异常才继续往上抛。

## RAII 不只是关文件

一旦你抓住"释放写进析构函数"这个模式，会发现它能统一管理**一切成对的获取/释放**操作：

| 资源 | 获取 | 释放 | 标准库里的 RAII 类型 |
|---|---|---|---|
| 堆内存 | `new` | `delete` | `std::unique_ptr` / `std::vector` |
| 文件 | `fopen` | `fclose` | `std::fstream` |
| 互斥锁 | `lock` | `unlock` | `std::lock_guard` |
| 线程 | 启动 | `join` | `std::jthread` |

这也解释了上一章的伏笔：`std::vector` 就是一个管理"堆上元素数组"的 RAII 类——构造/扩容时向堆要内存，析构时还回去。**你早就在用 RAII 了，只是现在知道了它的名字。**

看标准库的 RAII 如何让"加锁"绝不会忘记解锁：

```cpp
#include <iostream>
#include <mutex>

std::mutex m;

void safe(int id) {
    std::lock_guard<std::mutex> guard(m);   // 构造：加锁
    std::cout << "线程 " << id << " 进入临界区\n";
    // ……即使这里抛异常，guard 析构时也会自动解锁
}   // guard 离开作用域：自动解锁

int main() {
    safe(1);
    safe(2);
    std::cout << "完成\n";
}
```

`std::lock_guard` 构造时锁住互斥量，析构时解锁。你**不可能忘记解锁**，因为解锁不是你写的一句话，而是对象销毁的必然结果。对比 Java 里 `synchronized` 块或手动 `lock()`/`finally{unlock()}`——RAII 把这件事变成了不需要你操心的默认行为。

## 为什么这比 GC 更适合"资源"

这里要点破一个关键区别。GC 擅长管理**内存**,但对**其他资源**（文件、锁、连接、句柄）力不从心,原因是**时机**：

<div class="note">
<strong>GC 回收是"未来某时"，RAII 释放是"此时此刻"</strong>
Java 的 <code>finalize</code>、Python 的 <code>__del__</code> 都不保证<em>何时</em>运行——对象可能在最后一次使用后很久才被回收。对内存，这无所谓（反正内存够用）。但对<strong>稀缺资源</strong>就是灾难：一个数据库连接如果要等 GC 才释放，你的连接池很快就会枯竭。这正是 Java / Python 不得不额外发明 <code>try-with-resources</code> / <code>with</code> 的原因——GC 帮不上忙。<br>
RAII 的释放和作用域结束是<strong>同一时刻</strong>，精确可预测。这就是为什么 C++ 用一套机制（对象生命周期）就统一解决了内存和所有其他资源的管理。
</div>

## 一条现代 C++ 的铁律

RAII 直接推导出现代 C++ 最重要的一条编码准则：

<div class="keypoint">
<strong>不要裸写 <code>new</code> / <code>delete</code></strong>
每一次手动 <code>new</code> 都要求你在<em>所有</em>可能的退出路径上都记得 <code>delete</code>——包括异常路径，这几乎不可能做对。正确做法是让 RAII 对象替你持有资源：用 <code>std::vector</code>、<code>std::string</code> 管理动态数组，用 <code>std::unique_ptr</code>（第 8 章）管理单个堆对象。<strong>你负责表达"我要什么资源"，析构函数负责"什么时候还"。</strong>
</div>

## 动手：亲眼看栈展开清理资源

下面这段程序有**嵌套的 RAII 对象**，并在最内层抛出异常。运行前先预测：异常抛出后，哪些"释放"会被打印，顺序如何？

```cpp
#include <iostream>
#include <stdexcept>

struct Guard {
    std::string name;
    Guard(std::string n) : name(std::move(n)) { std::cout << "获取 " << name << '\n'; }
    ~Guard() { std::cout << "释放 " << name << '\n'; }
};

void inner() {
    Guard a("资源A");
    Guard b("资源B");
    std::cout << "即将抛出异常\n";
    throw std::runtime_error("boom");
    std::cout << "这行永远不会执行\n";
}

int main() {
    try {
        Guard outer("资源Outer");
        inner();
    } catch (const std::exception& e) {
        std::cout << "捕获异常：" << e.what() << '\n';
    }
    std::cout << "程序继续，没有泄漏\n";
}
```

预期输出：

```
获取 资源Outer
获取 资源A
获取 资源B
即将抛出异常
释放 资源B
释放 资源A
释放 资源Outer
捕获异常：boom
程序继续，没有泄漏
```

异常从 `inner` 抛出，栈开始展开：先析构 `inner` 里的 `b`、`a`（逆序），再析构 `main` 里的 `outer`，然后异常才被 `catch` 捕获。**三个资源全部被干净释放，没有一句手动清理代码。** 这就是 C++ 在没有 GC 的情况下依然安全的根本原因。

下一章我们回到第 6 章埋下的线索：当一个 RAII 对象被拷贝或移动时，它持有的资源该怎么办?这就引出了拷贝、移动与所有权。

---

**这一章的要点**

- **RAII**：资源获取写进构造函数，资源释放写进析构函数——资源生命周期绑定到对象生命周期。
- 因为析构在离开作用域时**自动且确定**地发生（哪怕是异常导致的**栈展开**），资源释放也就万无一失。
- RAII 统一管理一切成对的获取/释放：内存、文件、锁、线程、连接……你其实一直在用（`vector`、`string` 都是）。
- 相比 GC，RAII 的释放时机**精确可预测**（"此时此刻"而非"未来某时"），所以更适合管理稀缺资源——这正是 Java / Python 还要额外提供 `try-with-resources` / `with` 的原因。
- 现代 C++ 铁律：**别裸写 `new`/`delete`**，让 RAII 对象（容器、智能指针）替你持有资源。
