# 第 13 章 编译期计算：constexpr 与 consteval

## 这一章想让你带走什么

<div class="goal-box">
<p>第 1 章我给你看过一个让你印象深刻的例子：<code>square(9)</code> 编译后汇编里直接是 <code>81</code>，运行时一次乘法都没有。这一章讲清楚这背后的机制——<strong>编译期计算</strong>：让计算在<em>编译代码时</em>就完成，运行时直接用现成结果。</p>
<p>Java / Python 里没有对应的东西（它们的常量折叠很有限，且不可编程）。读完你会理解 <code>constexpr</code>、<code>consteval</code> 到底在做什么，以及"把计算从运行期搬到编译期"能带来什么。</p>
</div>

## 一个新维度：代码可以在编译期"跑"

先建立最核心的直觉。到目前为止，你默认所有代码都在**运行时**执行。但 C++ 有一条独特的能力：**只要一个计算的输入在编译期就能确定，这个计算本身也可以在编译期完成。**

看第 9 章埋下的伏笔——`const` 和 `constexpr` 的区别，现在展开：

```cpp
#include <iostream>

int main() {
    const int a = 10;             // 只读，但值可能来自运行时
    constexpr int b = 3 * 4;      // 编译期就算出 12

    int arr[b];                   // ✅ b 是编译期常量，能当数组大小
    // int bad[a];                // 这行是否合法取决于 a 的来源；constexpr 才保证可以

    std::cout << b << ' ' << sizeof(arr)/sizeof(int) << '\n';   // 12 12
}
```

<div class="keypoint">
<strong>const vs constexpr，一句话说清</strong>
<ul>
<li><code>const</code>：<strong>运行期</strong>的只读承诺——"这个值定了就不改"，但它<em>什么时候</em>定下来可以是运行时。</li>
<li><code>constexpr</code>：<strong>编译期</strong>的常量——"这个值在编译时就必须能算出来"。因此它能用在需要编译期常量的地方：数组大小、模板参数、<code>case</code> 标签等。</li>
</ul>
所有 <code>constexpr</code> 变量都隐含 <code>const</code>，反之不然。
</div>

## constexpr 函数：既能编译期跑，也能运行期跑

真正强大的是 **`constexpr` 函数**——把一个函数标成 `constexpr`，它就获得了"双重身份"：**输入是编译期常量时，在编译期求值；输入是运行时值时，就当普通函数在运行期跑。**

```cpp
#include <iostream>

constexpr long factorial(int n) {      // 一个可用于编译期的函数
    long result = 1;
    for (int i = 2; i <= n; ++i) result *= i;
    return result;
}

int main() {
    constexpr long f5 = factorial(5);   // 编译期算出 120，塞进程序
    std::cout << "5! = " << f5 << '\n';

    int n = 6;
    std::cout << "6! = " << factorial(n) << '\n';   // n 是运行时值 → 运行期计算
}
```

同一个 `factorial`，第一次调用在编译期就算成了 `120`（存进可执行文件的常量区），第二次因为参数 `n` 是运行时变量，就退化成普通函数在运行时算。**一份代码，两种执行时机。**

<div class="note">
<strong>现代 C++ 的 constexpr 已经很能干</strong>
早期（C++11）的 <code>constexpr</code> 函数限制极严，几乎只能写一个 return 语句。如今（C++14 起逐步放开）它支持循环、局部变量、分支、甚至在编译期使用 <code>std::vector</code> 和 <code>std::string</code>（C++20）。上面的 <code>factorial</code> 用了普通的 <code>for</code> 循环——这在编译期照样能跑。你可以像写普通函数一样写 constexpr 函数。

## 眼见为实：汇编里没有计算

用汇编印证"编译期算完了"。下面两个函数，一个传编译期常量、一个传运行时参数，对比它们生成的代码：

```cpp-asm
constexpr long factorial(int n) {
    long r = 1;
    for (int i = 2; i <= n; ++i) r *= i;
    return r;
}

long compileTime() { return factorial(10); }   // 编译期：直接返回 3628800
long runTime(int n) { return factorial(n); }    // 运行期：真的循环相乘
```

点开 **🔬 看汇编 / 编译器处理**：

- `compileTime` 的汇编里**没有任何循环或乘法**，直接 `mov` 一个常量 `3628800` 就返回——整个阶乘在编译时算完了。
- `runTime` 里则有实实在在的循环和乘法指令——因为 `n` 要到运行时才知道。

这就是第 1 章那个"81"现象的完整机制:**当编译器能确定输入,它就替你把计算做掉,运行时零成本。**

## consteval：强制"必须"在编译期

`constexpr` 是"**可以**在编译期"，但不保证——传运行时参数它就退回运行期。有时你想要更强的保证：**这个函数只允许在编译期调用**。这就是 C++20 的 **`consteval`**（也叫"立即函数"）：

```cpp
#include <iostream>

consteval int alwaysCompileTime(int x) {   // 必须在编译期求值
    return x * x;
}

int main() {
    constexpr int a = alwaysCompileTime(5);   // ✅ 编译期，OK → 25
    std::cout << a << '\n';

    // int n = 5;
    // int b = alwaysCompileTime(n);          // ❌ n 是运行时值 → 编译错误
}
```

`consteval` 函数如果拿不到编译期常量，直接编译失败——它**拒绝**在运行期执行。

<div class="keypoint">
<strong>constexpr 与 consteval 的分工</strong>
<ul>
<li><code>constexpr</code>：<strong>能</strong>编译期就编译期，<strong>不能</strong>就运行期。灵活，是日常首选。</li>
<li><code>consteval</code>：<strong>必须</strong>编译期，否则报错。用于你想强制保证"这个计算绝不拖到运行时"的场景（如生成编译期查找表、编译期校验的工厂函数）。</li>
</ul>
还有个 <code>constinit</code>（较少用）：保证一个变量在编译期完成初始化，但它本身<em>不是</em> const，之后可以修改——用来避免"静态初始化顺序问题"。
</div>

## static_assert：把检查也搬到编译期

编译期计算配上 **`static_assert`**，能让你在**编译时**就验证条件——不满足直接编译失败，根本不会生成有问题的程序。这比运行时的 `assert` 早得多、也彻底得多：

```cpp
#include <iostream>

constexpr int bufferSize = 1024;

// 编译期就检查约束，不满足则编译不过
static_assert(bufferSize >= 256, "缓冲区太小");
static_assert(bufferSize % 2 == 0, "缓冲区大小必须是偶数");

constexpr int square(int x) { return x * x; }
static_assert(square(8) == 64, "square 实现有误");   // 编译期验证函数正确性

int main() {
    std::cout << "所有编译期检查通过，bufferSize = " << bufferSize << '\n';
}
```

如果把第一个断言改成 `bufferSize >= 4096`，程序**根本编译不过**，编译器会打印你写的错误信息 `"缓冲区太小"`。这是一种强大的"左移"——把本该运行时才暴露的问题，提前到编译期拦截。

<div class="note">
<strong>你其实见过它的近亲</strong>
第 12 章的 concepts，本质也是编译期检查（检查"类型满足约束"）。<code>static_assert</code> 则检查"编译期常量表达式为真"。它们共同体现了 C++ 的一个哲学：<strong>能在编译期确定的事，就别拖到运行期</strong>——错误越早发现越好，计算越早完成越省。
</div>

## 编译期计算能做什么、不能做什么

<div class="keypoint">
<strong>编译期计算的价值</strong>
<ul>
<li><strong>零运行时成本</strong>：结果在编译时算好，运行时直接用——查找表、数学常量、配置推导等尤其受益。</li>
<li><strong>更早的错误</strong>：配合 <code>static_assert</code>，把校验提前到编译期。</li>
<li><strong>用在必须是常量的地方</strong>：数组大小、模板参数、枚举值等。</li>
</ul>
</div>

但它有天然边界：编译期计算**不能做依赖运行时信息的事**——不能读文件、不能获取用户输入、不能调用非 `constexpr` 的函数、不能做真正的动态内存分配残留到运行期（编译期分配的必须在编译期释放）。道理很简单：编译时这些信息还不存在。

<div class="warn">
<strong>别把什么都塞进编译期</strong>
编译期计算虽好，但过度使用会拖慢<strong>编译</strong>速度（把运行时负担转移成了编译时负担），复杂的 constexpr/模板元编程还会让代码难读。经验法则：<strong>对"输入天然是编译期常量、且反复用到"的计算</strong>（查找表、固定配置、维度换算），编译期计算收益最大；对普通的运行时逻辑，老老实实运行期跑就好。
</div>

## 动手：编译期生成一张查找表

编译期计算最经典的应用之一，是**在编译时就把一张表算好**，运行时零成本查表。下面用 constexpr 生成前 N 个平方数的表：

```cpp
#include <iostream>
#include <array>

// 编译期生成：前 N 个数的平方表
template <int N>
constexpr std::array<int, N> makeSquareTable() {
    std::array<int, N> table{};
    for (int i = 0; i < N; ++i) table[i] = i * i;
    return table;
}

// 整张表在编译期就算好，存进程序的常量区
constexpr auto squares = makeSquareTable<10>();

int main() {
    // 编译期验证：这张表确实对
    static_assert(squares[5] == 25);
    static_assert(squares[9] == 81);

    std::cout << "编译期算好的平方表：";
    for (int x : squares) std::cout << x << ' ';
    std::cout << '\n';
}
```

预期输出：

```
编译期算好的平方表：0 1 4 9 16 25 36 49 64 81
```

这张 `squares` 表**在编译时就完全算好了**，直接嵌进可执行文件——运行时那个循环只是在打印现成的值，一次乘法都没算。而两个 `static_assert` 保证了表的正确性在编译期就被验证。**"把工作从运行期搬到编译期"，这就是最典型的样子。**

第三部分（泛型与编译期）到此完整：模板给了你编译期的类型抽象，concepts 给了它约束，constexpr/consteval 给了你编译期的计算能力。接下来第四部分转向日常最高频的工具——标准库，从容器与迭代器开始。

---

**这一章的要点**

- **编译期计算**：只要输入在编译期确定，计算就能在编译时完成，运行时直接用结果——零运行时成本。
- **`const`**（运行期只读）vs **`constexpr`**（编译期常量）：后者能用于数组大小、模板参数等需要编译期常量的地方。
- **`constexpr` 函数**有双重身份：输入是编译期常量则编译期求值，否则退化为普通运行期函数。现代 constexpr 支持循环、分支、局部变量等。
- **`consteval`** 强制函数只在编译期求值（拿不到编译期常量就报错）。
- **`static_assert`** 把条件检查搬到编译期，不满足直接编译失败——错误"左移"。
- 编译期计算不能依赖运行时信息（文件、输入等）；过度使用会拖慢编译，适合"输入天然是常量且反复用到"的计算。
