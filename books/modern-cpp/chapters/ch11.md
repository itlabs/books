# 第 11 章 模板：编译期的泛型

## 这一章想让你带走什么

<div class="goal-box">
<p>你会用 Java 泛型 <code>List&lt;T&gt;</code>、也见过 Python 的"鸭子类型"。C++ 的<strong>模板</strong>看起来像 Java 泛型，本质却完全不同——它不是运行时的类型参数，而是<strong>编译期的代码生成器</strong>。</p>
<p>读完这一章，你会理解模板到底是"怎么工作的"：为什么它能做到零开销、为什么模板代码几乎都写在头文件里、以及为什么模板的报错信息那么吓人（下一章的 concepts 正是来治这个的）。</p>
</div>

## 先看一个模板长什么样

假设你要写一个"取两者较大值"的函数。不用模板，你得为每种类型各写一份：

```cpp-norun
int    maxOf(int a, int b)       { return a > b ? a : b; }
double maxOf(double a, double b) { return a > b ? a : b; }
// string、自定义类型……还要各写一遍，函数体一模一样
```

函数模板让你只写一份"模具"，类型留作参数：

```cpp
#include <iostream>
#include <string>

template <typename T>          // T 是类型参数
T maxOf(T a, T b) {
    return a > b ? a : b;
}

int main() {
    std::cout << maxOf(3, 7) << '\n';           // T = int
    std::cout << maxOf(2.5, 1.5) << '\n';        // T = double
    std::cout << maxOf<std::string>("apple", "pear") << '\n';   // T = string
}
```

一份 `maxOf` 模板，服务了 `int`、`double`、`string`。这一步和 Java 泛型看起来一样。但接下来就分道扬镳了。

## 核心区别：模板是"编译期代码生成"

这是本章最重要的一点，请务必建立正确直觉：

<div class="keypoint">
<strong>模板不是一个函数，而是一个"生成函数的模具"</strong>
当你写 <code>maxOf(3, 7)</code>，编译器看到实参是 <code>int</code>，就用 <code>T = int</code> <strong>生成一份真正的 <code>maxOf</code> 函数</strong>——就好像你手写了 <code>int maxOf(int, int)</code>。写 <code>maxOf(2.5, 1.5)</code> 时，它<strong>再生成一份</strong> <code>double</code> 版本。<br>
这个"按用到的类型生成具体代码"的过程叫<strong>实例化（instantiation）</strong>。模板本身不产生任何机器码；只有被实例化出来的具体版本才会。
</div>

这套机制有个正式名字叫**单态化（monomorphization）**——每个用到的类型都生成一份专属代码。它和 Java 的做法恰好相反：

<div class="compare">
<div class="known">
<div class="compare-head">Java 泛型：类型擦除</div>

```java
// 编译后 T 被"擦除"成 Object
// 运行时只有一份字节码
List<Integer> a = ...;
List<String>  b = ...;
// a 和 b 在运行时是同一个类
// 泛型信息在运行期基本不存在
```

</div>
<div class="cpp-side">
<div class="compare-head">C++ 模板：单态化</div>

```cpp-norun
// 编译期为每个类型生成专属代码
std::vector<int>    a;
std::vector<string> b;
// 生成两份不同的、
// 各自优化过的机器码
// 运行期没有任何"泛型"痕迹
```

</div>
</div>

用汇编亲眼验证"生成了两份专属代码"。下面的 `maxOf` 被 `int` 和 `double` 各实例化一次，编译器为它们生成了**不同的指令**（整数比较 vs 浮点比较）：

```cpp-asm
template <typename T>
T maxOf(T a, T b) { return a > b ? a : b; }

int    useInt(int a, int b)          { return maxOf(a, b); }
double useDouble(double a, double b) { return maxOf(a, b); }
```

点开 **🔬 看汇编 / 编译器处理**：`useInt` 里是整数比较指令、`useDouble` 里是浮点比较指令(如 `maxsd`)——两份独立的、各自最优的代码。没有 Java 那种"擦除成 Object 再装箱拆箱"的运行时开销。这就是模板的招牌特性——**零开销抽象**。

<div class="note">
<strong>单态化的代价</strong>
凡事有两面：为每个类型生成一份代码，意味着用得越多、生成的代码越多，可能导致可执行文件膨胀（code bloat）、编译变慢。这是 C++ 编译普遍比 Java 慢的重要原因之一。但换来的是运行期的极致性能——对 C++ 的目标领域（性能敏感场景），这个交易通常划算。
</div>

## 模板参数不止是类型

模板参数除了类型（`typename T`），还能是**编译期常量**（非类型参数）。这在 Java 泛型里是没有的能力：

```cpp
#include <iostream>
#include <array>

// N 是编译期常量参数，不是类型
template <typename T, std::size_t N>
T sum(const std::array<T, N>& arr) {
    T total{};
    for (const T& x : arr) total += x;
    return total;
}

int main() {
    std::array<int, 4> a{1, 2, 3, 4};
    std::array<double, 3> b{1.5, 2.5, 3.0};
    std::cout << sum(a) << '\n';   // 10
    std::cout << sum(b) << '\n';   // 7
}
```

`N` 是数组大小，在编译期就固定。`std::array<int, 4>` 和 `std::array<int, 8>` 是**两个不同的类型**——大小是类型的一部分。这种"值也能当模板参数"的能力，是 C++ 编译期计算（第 13 章）的基础。

## 类模板：给整个类做泛型

不只函数，整个类也能是模板。你天天用的 `std::vector<T>`、`std::unique_ptr<T>` 都是类模板。我们自己写一个极简的"装一个值的盒子"：

```cpp
#include <iostream>
#include <string>

template <typename T>
class Box {
    T value_;
public:
    explicit Box(T v) : value_(std::move(v)) {}
    const T& get() const { return value_; }
    void set(T v) { value_ = std::move(v); }
};

int main() {
    Box<int> bi(42);
    Box<std::string> bs("hello");

    std::cout << bi.get() << '\n';      // 42
    std::cout << bs.get() << '\n';      // hello

    bi.set(100);
    std::cout << bi.get() << '\n';      // 100
}
```

<div class="note">
<strong>C++17 起：类模板参数常常能自动推导</strong>
现代 C++ 里，很多时候连 <code>&lt;T&gt;</code> 都不用写。<code>Box bi(42);</code>（省略 <code>&lt;int&gt;</code>）编译器能从构造参数 <code>42</code> 推导出 <code>T = int</code>，这叫 CTAD（类模板参数推导）。函数模板则从一开始就支持这种推导——<code>maxOf(3, 7)</code> 不用写 <code>maxOf&lt;int&gt;</code>。
</div>

## 隐式约束：C++ 版的"鸭子类型"

这里有个 Python 程序员会觉得眼熟、Java 程序员会觉得陌生的点。看回 `maxOf`：

```cpp-norun
template <typename T>
T maxOf(T a, T b) { return a > b ? a : b; }
```

它对 `T` 有什么要求？没有显式写任何约束，但代码里用了 `a > b`——所以 `T` **必须支持 `>` 运算**。这个要求是**隐式**的：只要你传进来的类型能用 `>` 比较，模板就能实例化；不能比较，就报错。

<div class="keypoint">
<strong>模板是"编译期的鸭子类型"</strong>
模板不要求 <code>T</code> 实现某个接口或继承某个基类（不像 Java 的有界泛型 <code>&lt;T extends Comparable&gt;</code>）。它只要求：用到的操作 <code>T</code> 得支持。<strong>能嘎嘎叫，就是鸭子。</strong><br>
好处是极其灵活——任何支持 <code>&gt;</code> 的类型都能用 <code>maxOf</code>，包括你自己定义的类型，无需事先声明"我实现了某接口"。坏处是：约束是隐式的、藏在函数体里，一旦用错类型，错误信息会非常糟糕。
</div>

亲眼看看"用错类型"有多惨。下面这段试图对一个没有 `>` 运算的类型调用 `maxOf`：

```cpp-norun
struct Widget {};   // 没有定义 operator>

template <typename T>
T maxOf(T a, T b) { return a > b ? a : b; }

int main() {
    Widget x, y;
    maxOf(x, y);    // ❌ 编译错误，但信息极其冗长难懂
}
```

把它放进 Compiler Explorer 编译，你会看到一大坨错误——编译器在深入实例化 `maxOf<Widget>` 后，才在 `a > b` 那里发现 `Widget` 不支持 `>`，于是从实例化的最深处往外吐一长串上下文。**报错发生在使用处的深层，而非"你违反了约定"的那一刻。** 这正是模板长期以来最大的痛点，也是下一章 concepts 要解决的核心问题。

## 为什么模板通常整个写在头文件里

回忆第 2 章：每个 `.cpp` 独立编译，编译器只看当前翻译单元。现在把它和"模板要在使用处实例化"放在一起想：

<div class="keypoint">
<strong>编译器必须在使用处看到模板的完整定义</strong>
要实例化 <code>maxOf&lt;int&gt;</code>，编译器需要 <code>maxOf</code> 的<strong>完整函数体</strong>（才能生成代码）。如果模板的定义藏在另一个 <code>.cpp</code> 里，当前翻译单元只有声明、看不到函数体，就<strong>无法实例化</strong>——会得到链接错误。<br>
所以模板的定义（不只是声明）通常<strong>整个放在头文件里</strong>，让每个使用它的翻译单元都能看到完整定义。这就是为什么打开 STL 头文件，满眼都是实现代码——它们全是模板，不得不如此。
</div>

这也解释了第 2 章埋的伏笔:普通函数"声明在头文件、定义在 .cpp"的分离模式,对模板不适用。模板是"头文件里的完整实现",且受 ODR 的模板豁免(同一模板可在多个翻译单元实例化,链接器会合并)。

## 动手：观察实例化生成多份独立代码

下面这个模板类带一个静态计数器。因为每个 `T` 都生成一份**独立**的类，它们的静态成员也是**各自独立**的——这能直观证明"每个类型一份专属代码"：

```cpp
#include <iostream>
#include <string>

template <typename T>
struct Counter {
    static inline int count = 0;    // 每个实例化的 T 各有一份
    Counter() { ++count; }
};

int main() {
    Counter<int> a, b, c;           // Counter<int>::count 涨到 3
    Counter<std::string> x, y;      // Counter<string>::count 涨到 2

    std::cout << "Counter<int>    创建了 " << Counter<int>::count << " 个\n";
    std::cout << "Counter<string> 创建了 " << Counter<std::string>::count << " 个\n";
}
```

预期输出：

```
Counter<int>    创建了 3 个
Counter<string> 创建了 2 个
```

两个计数器互不相干——因为 `Counter<int>` 和 `Counter<std::string>` 是编译器生成的**两个完全不同的类**，各有各的静态成员。这就是单态化最直观的证据：模板不是一个类，而是"生成了多个类的模具"。

模板给了你无约束的灵活性，但也带来了"隐式约束 + 恐怖报错"的代价。下一章的 **concepts** 正是现代 C++ 的解药：把那些藏在函数体里的隐式要求，变成写在签名上的、编译器能友好检查的显式约束。

---

**这一章的要点**

- 模板是**编译期的代码生成器**，不是运行时的类型参数：用到某类型时，编译器**实例化**出一份该类型的专属代码。
- 这叫**单态化**，与 Java 泛型的**类型擦除**相反：C++ 为每个类型生成独立、各自优化的机器码，**零运行时开销**（代价是代码膨胀、编译变慢）。
- 模板参数可以是类型（`typename T`），也可以是**编译期常量**（如 `std::array<T, N>` 的 `N`）。
- 类和函数都能做模板；C++17 起类模板参数常能自动推导（CTAD）。
- 模板对类型的约束是**隐式的鸭子类型**：用到什么操作，类型就得支持什么——灵活，但用错时报错冗长难懂。
- 因为要在使用处实例化，模板的**完整定义通常放在头文件里**。
