# 第 8 章 引用、指针与智能指针

## 引用：对象的别名

**引用（reference）** `T&` 是一个已存在对象的**别名**——它不是新对象，只是给同一个对象起了第二个名字。你在第 3 章见过它用于传参"改到外面的对象"。

```cpp
#include <iostream>

int main() {
    int x = 10;
    int& ref = x;      // ref 是 x 的别名，不是新对象

    ref = 20;          // 通过 ref 改，等于改 x
    std::cout << "x = " << x << '\n';        // 20
    std::cout << "&x  = " << &x  << '\n';
    std::cout << "&ref= " << &ref << '\n';    // 和 &x 完全相同
}
```

引用有两条铁规矩，正好和它"别名"的身份对应：

<div class="keypoint">
<strong>引用的两条铁律</strong>
<ul>
<li><strong>必须初始化</strong>：引用一诞生就必须绑定到某个对象——没有"空引用"这回事。别名总得是"谁的"别名。</li>
<li><strong>不能改绑</strong>：一旦绑定，终生指向同一个对象。<code>ref = y</code> 不是让 ref 改指向 y，而是把 y 的值赋给 ref 所绑定的那个对象（即 x）。</li>
</ul>
</div>

这让引用非常安全，但也不够灵活：它不能为空、不能中途换目标、不能放进容器里"先占位后赋值"。需要这些能力时，就要用指针。

## 裸指针：可以为空、可以改指向的地址

**指针（pointer）** `T*` 保存的是一个对象的**地址**。和引用相比，它更接近你在 Java 里对"引用变量"的直觉——可以为空、可以重新指向别的对象：

```cpp
#include <iostream>

int main() {
    int a = 1, b = 2;
    int* p = &a;       // p 保存 a 的地址

    std::cout << "*p = " << *p << '\n';   // 解引用：1
    p = &b;            // 改指向 b（引用做不到）
    std::cout << "*p = " << *p << '\n';   // 2

    p = nullptr;       // 空指针（引用做不到）
    if (!p) std::cout << "p 现在为空\n";
}
```

`*p` 是**解引用**（取出指针指向的对象），`&a` 是**取地址**。`nullptr` 是 C++11 的空指针字面量——请始终用它，别用老代码里的 `NULL` 或 `0`。

<div class="note">
<strong>引用 vs 指针，怎么选？</strong>
经验法则：<strong>能用引用就用引用</strong>（更安全、语法更干净），当你确实需要"可空"或"可改指向"时才用指针。<br>
但请注意：无论引用还是裸指针，都<strong>只是"指向"对象，并不拥有它</strong>——它们不负责在对象消亡时释放资源。谁来负责所有权？这就是智能指针要解决的问题。
</div>

## 裸指针管理堆内存的老办法（以及它为什么危险）

如果直接用裸指针管理堆对象，你得手动 `new` 和 `delete`——回到第 5 章批判过的世界：

```cpp-norun
void oldWay() {
    int* p = new int(42);   // 手动分配
    doSomething(p);         // 若这里抛异常……
    delete p;               // ……这行就被跳过 → 内存泄漏
}
```

问题和第 5 章的文件句柄一模一样：任何提前退出（异常、`return`）都会跳过 `delete`。而且你还得时刻警惕：delete 了没？delete 两次没？delete 之后还有没有人在用这个悬垂指针？这些正是 C++ "难且危险"的刻板印象的来源。

现代 C++ 的答案是把裸指针**包进一个 RAII 对象**里——这就是智能指针。它是"栈上的小管家管理堆资源"这一模式（第 4 章）的标准实现。

## unique_ptr：独占所有权

`std::unique_ptr<T>` 表达**独占所有权**：这个堆对象归我一个人，我析构时就 `delete` 它。它是最常用、开销最低的智能指针——**几乎应该是你管理单个堆对象的默认选择**。

```cpp
#include <iostream>
#include <memory>

struct Widget {
    int id;
    Widget(int i) : id(i) { std::cout << "  构造 Widget#" << id << '\n'; }
    ~Widget()             { std::cout << "  析构 Widget#" << id << '\n'; }
};

void useWidget() {
    auto w = std::make_unique<Widget>(1);   // 分配 + 拥有
    std::cout << "  使用 Widget#" << w->id << '\n';   // 用 -> 访问成员
}   // w 离开作用域：自动 delete，无需手写

int main() {
    std::cout << "进入 useWidget:\n";
    useWidget();
    std::cout << "返回 main，Widget 已被自动释放\n";
}
```

运行输出：

```
进入 useWidget:
  构造 Widget#1
  使用 Widget#1
  析构 Widget#1
返回 main，Widget 已被自动释放
```

注意两点：用 `std::make_unique<Widget>(1)` 创建（而不是裸 `new`），用 `->` 像普通指针一样访问成员。析构在 `useWidget` 结束时**自动**发生——你没写 `delete`，也不会泄漏,哪怕中途抛异常。

**"独占"意味着不能拷贝**——否则就有两个拥有者、会双重释放（第 7 章的老问题）。`unique_ptr` 禁用了拷贝，只允许**移动**（转移所有权）：

```cpp
#include <iostream>
#include <memory>

int main() {
    auto a = std::make_unique<int>(42);

    // auto b = a;                 // ❌ 编译错误：unique_ptr 不能拷贝
    auto b = std::move(a);         // ✅ 移动：所有权从 a 转给 b

    std::cout << "b 拥有: " << *b << '\n';
    std::cout << "a 现在为空? " << (a == nullptr ? "是" : "否") << '\n';   // 是
}
```

这就是第 7 章"移动转移所有权"的最纯粹体现:`std::move(a)` 之后 `a` 变空、`b` 接手,任一时刻堆对象只有一个拥有者。

## shared_ptr：共享所有权

有时一个对象确实需要**多个拥有者**，谁都可能是最后一个用它的人。`std::shared_ptr<T>` 用**引用计数**解决："还有几个人拥有我？"计数归零时才释放。

```cpp
#include <iostream>
#include <memory>

struct Resource {
    Resource()  { std::cout << "  资源诞生\n"; }
    ~Resource() { std::cout << "  资源释放\n"; }
};

int main() {
    auto a = std::make_shared<Resource>();
    std::cout << "计数 = " << a.use_count() << '\n';   // 1
    {
        auto b = a;                                    // 拷贝 shared_ptr：共享，计数+1
        std::cout << "计数 = " << a.use_count() << '\n';   // 2
    }   // b 离开作用域：计数-1，但资源还没释放（a 还在）
    std::cout << "计数 = " << a.use_count() << '\n';   // 1
    std::cout << "main 即将结束\n";
}   // a 离开作用域：计数归零 → 资源释放
```

运行输出：

```
  资源诞生
计数 = 1
计数 = 2
计数 = 1
main 即将结束
  资源释放
```

和 `unique_ptr` 不同，`shared_ptr` **可以拷贝**——每次拷贝就多一个共享者、计数加一;每个 `shared_ptr` 析构则计数减一。最后一个走的人负责"关灯"（释放资源）。

<div class="warn">
<strong>别过度使用 shared_ptr</strong>
共享所有权听起来省心，但它有代价：引用计数需要额外内存和原子操作（线程安全的加减），比 <code>unique_ptr</code> 慢；更麻烦的是它让"谁拥有这个对象、它何时释放"变得难以推理。<br>
经验法则：<strong>默认用 <code>unique_ptr</code>，只在确实需要多个拥有者共享同一对象、且生命周期无法明确归属某一方时，才升级到 <code>shared_ptr</code>。</strong>共享是例外，不是默认。
</div>

## weak_ptr：打破循环引用

`shared_ptr` 有一个经典陷阱：**循环引用**。如果 A 用 `shared_ptr` 持有 B，B 又用 `shared_ptr` 持有 A，那么两者的计数永远不会归零——即使外部再没人用它们，它们互相"拽住"对方，谁也不释放，造成内存泄漏。

`std::weak_ptr<T>` 用来打破这个循环:它指向一个 `shared_ptr` 管理的对象,但**不增加引用计数**——它是"观察者"而非"拥有者"。用之前要先 `lock()` 提升成 `shared_ptr`,借此顺便检查对象是否还活着:

```cpp
#include <iostream>
#include <memory>

int main() {
    auto sp = std::make_shared<int>(42);
    std::weak_ptr<int> wp = sp;          // 观察，不拥有（不增加计数）

    std::cout << "计数 = " << sp.use_count() << '\n';   // 1，weak 不计入

    if (auto locked = wp.lock()) {       // 提升为 shared_ptr（对象还活着）
        std::cout << "对象还在: " << *locked << '\n';
    }

    sp.reset();                          // 释放唯一的拥有者，对象销毁

    if (auto locked = wp.lock()) {
        std::cout << "还在\n";
    } else {
        std::cout << "对象已销毁，weak_ptr 感知到了\n";
    }
}
```

运行输出：

```
计数 = 1
对象还在: 42
对象已销毁，weak_ptr 感知到了
```

典型用法:双向关系里,一方用 `shared_ptr`(拥有),另一方用 `weak_ptr`(观察),循环就断开了。父节点用 `shared_ptr` 持有子节点、子节点用 `weak_ptr` 回指父节点,是最常见的模式。

## 三种指针工具的分工

<div class="keypoint">
<strong>该用哪个？一张决策表</strong>
<ul>
<li>只是<strong>指向、访问</strong>一个已有对象，不涉及所有权 → <strong>引用 <code>T&</code></strong>（或需要可空/可改指向时用裸指针 <code>T*</code>）。</li>
<li><strong>独占</strong>一个堆对象（默认选择）→ <strong><code>std::unique_ptr</code></strong>，用 <code>make_unique</code> 创建。</li>
<li>确实需要<strong>多个拥有者共享</strong> → <strong><code>std::shared_ptr</code></strong>，用 <code>make_shared</code> 创建。</li>
<li>观察一个 <code>shared_ptr</code> 对象但<strong>不参与所有权</strong>（打破循环）→ <strong><code>std::weak_ptr</code></strong>。</li>
</ul>
一条总原则：<strong>用智能指针表达所有权，用引用/裸指针表达"仅访问"。裸 <code>new</code> / <code>delete</code> 不该出现在现代 C++ 代码里。</strong>
</div>

## 动手：观察所有权的转移与共享

下面这段把 `unique_ptr` 的移动和 `shared_ptr` 的计数放在一起。运行前先预测每个输出：

```cpp
#include <iostream>
#include <memory>
#include <vector>

struct Item {
    int id;
    Item(int i) : id(i) { std::cout << "  Item#" << id << " 构造\n"; }
    ~Item()             { std::cout << "  Item#" << id << " 析构\n"; }
};

int main() {
    std::cout << "== unique_ptr 转移进容器 ==\n";
    std::vector<std::unique_ptr<Item>> box;
    auto p = std::make_unique<Item>(1);
    box.push_back(std::move(p));        // 所有权转移进 vector
    std::cout << "  p 空? " << (p == nullptr ? "是" : "否") << '\n';

    std::cout << "== shared_ptr 共享计数 ==\n";
    auto s1 = std::make_shared<Item>(2);
    auto s2 = s1;                        // 共享
    std::cout << "  计数 = " << s1.use_count() << '\n';   // 2

    std::cout << "== 作用域结束，逆序清理 ==\n";
}
```

预期输出：

```
== unique_ptr 转移进容器 ==
  Item#1 构造
  p 空? 是
== shared_ptr 共享计数 ==
  Item#2 构造
  计数 = 2
== 作用域结束，逆序清理 ==
  Item#2 析构
  Item#1 析构
```

`Item#1` 的所有权移进了 `vector`，`p` 变空；`Item#2` 被 `s1`、`s2` 共享，计数为 2。`main` 结束时，两个智能指针容器/变量逆序销毁，各自持有的堆对象被自动、准确地释放——**全程没有一句 `new` 或 `delete`**。

这正是现代 C++ 管理堆对象的日常：所有权由智能指针明确表达，释放由 RAII 自动完成。下一章我们转向另一个维度——`const`，看 C++ 如何在类型系统层面表达"不可修改"。

---

**这一章的要点**

- **引用 `T&`** 是对象的别名：必须初始化、不能改绑、不能为空——安全但不灵活。
- **裸指针 `T*`** 保存地址：可空（`nullptr`）、可改指向；但引用和裸指针都只"指向"、**不拥有**对象。
- **`unique_ptr`**：独占所有权，析构自动 `delete`，不能拷贝只能移动——**管理单个堆对象的默认选择**，用 `make_unique` 创建。
- **`shared_ptr`**：引用计数的共享所有权，可拷贝，计数归零才释放——有开销，是例外而非默认，用 `make_shared` 创建。
- **`weak_ptr`**：不增加计数的观察者，`lock()` 后使用，用于打破 `shared_ptr` 的**循环引用**。
- 总原则：**智能指针表达所有权，引用/裸指针表达仅访问；裸 `new`/`delete` 不该出现在现代代码里。**
