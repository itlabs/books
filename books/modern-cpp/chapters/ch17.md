# 第 17 章 错误处理：异常、错误码与 expected

## 这一章想让你带走什么

<div class="goal-box">
<p>在 Java 里错误处理基本就是异常（还分受检/非受检），Python 里"用了就抛"。C++ 的选择更多元：<strong>异常</strong>、<strong>错误码</strong>、以及现代的 <strong><code>std::optional</code> / <code>std::expected</code></strong>——而且社区对"该用哪个"有真实的分歧。</p>
<p>读完你会理解每种方式的取舍：异常如何与 RAII（第 5 章）配合、为什么有些 C++ 项目禁用异常、以及 C++23 的 <code>std::expected</code> 如何把"可能失败"编码进类型、逼你处理错误。</p>

## 异常：和你熟悉的机制很像

C++ 有异常，`throw` / `try` / `catch` 的形态和 Java / Python 几乎一样：

```cpp
#include <iostream>
#include <stdexcept>
#include <string>

int divide(int a, int b) {
    if (b == 0) throw std::runtime_error("除数不能为零");
    return a / b;
}

int main() {
    try {
        std::cout << divide(10, 2) << '\n';   // 5
        std::cout << divide(10, 0) << '\n';   // 抛异常
    } catch (const std::exception& e) {        // 按 const 引用捕获
        std::cout << "捕获错误: " << e.what() << '\n';
    }
    std::cout << "程序继续\n";
}
```

输出：

```
5
捕获错误: 除数不能为零
程序继续
```

几个和 Java / Python 的差异值得点名：

<div class="keypoint">
<strong>C++ 异常的几个特点</strong>
<ul>
<li><strong>可以抛任何类型</strong>——但约定俗成抛<strong>继承自 <code>std::exception</code></strong> 的对象（<code>runtime_error</code>、<code>logic_error</code> 等），这样能用 <code>e.what()</code> 拿到描述。</li>
<li><strong>按 <code>const&</code> 捕获</strong>：<code>catch (const std::exception& e)</code>。按引用避免切片（第 18 章讲），基类引用能捕获所有派生异常。</li>
<li><strong>没有"受检异常"</strong>：不像 Java 要在签名里声明 <code>throws</code>。C++ 不强制，好处是灵活，坏处是光看签名不知道一个函数会不会抛。</li>
</ul>
</div>

## 异常与 RAII：为什么 C++ 的异常是安全的

这是全章最重要的连接点，回扣第 5 章。你可能听过"C++ 异常危险"的说法——那是针对**不用 RAII**的老式代码。**只要资源都由 RAII 对象持有，异常就是安全的**：栈展开时，沿途每个局部对象的析构函数都会被调用，资源自动释放。

```cpp
#include <iostream>
#include <vector>
#include <memory>
#include <stdexcept>

struct Resource {
    std::string name;
    Resource(std::string n) : name(std::move(n)) { std::cout << "  获取 " << name << '\n'; }
    ~Resource() { std::cout << "  释放 " << name << '\n'; }
};

void risky() {
    Resource a("文件");
    auto b = std::make_unique<Resource>("内存");    // RAII 持有
    std::cout << "  即将抛异常\n";
    throw std::runtime_error("出错了");
    std::cout << "  这行不执行\n";
}   // 栈展开：b、a 逆序自动释放

int main() {
    try {
        risky();
    } catch (const std::exception& e) {
        std::cout << "捕获: " << e.what() << '\n';
    }
    std::cout << "没有泄漏\n";
}
```

输出：

```
  获取 文件
  获取 内存
  即将抛异常
  释放 内存
  释放 文件
  捕获: 出错了
  没有泄漏
```

异常抛出后，`b`（内存）、`a`（文件）逆序释放，一句清理代码都没写。**这就是为什么"RAII + 异常"是 C++ 推荐的错误处理组合**——异常负责传播错误，RAII 负责保证无论如何资源都被清理。

<div class="warn">
<strong>析构函数绝不要抛异常</strong>
如果一个异常正在栈展开，而此时某个析构函数又抛出<strong>第二个</strong>异常，程序会直接 <code>std::terminate</code> 崩溃。所以析构函数应当是 <code>noexcept</code> 的（默认就是），里面绝不能让异常逃逸。这是 C++ 异常安全的一条铁律。
</div>

## noexcept：承诺不抛异常

`noexcept` 关键字标记一个函数**不会抛异常**。它既是给调用者的承诺，也是给编译器的优化线索：

```cpp
#include <iostream>

int safeAdd(int a, int b) noexcept {   // 承诺不抛
    return a + b;
}

int main() {
    std::cout << safeAdd(3, 4) << '\n';   // 7
}
```

<div class="note">
<strong>noexcept 为什么重要（回扣第 7 章）</strong>
还记得第 7 章移动构造函数写的 <code>noexcept</code> 吗？那不是装饰。<code>std::vector</code> 扩容时要把元素搬到新内存：如果元素的移动构造是 <code>noexcept</code> 的，它就放心用<strong>移动</strong>（快）；如果移动<em>可能</em>抛异常，vector 为了保证异常安全，只好退而用<strong>拷贝</strong>（慢）——因为移动到一半抛异常会导致数据处于损坏状态，无法回滚。所以：<strong>移动构造/移动赋值应尽量标 <code>noexcept</code></strong>，这直接影响容器性能。
</div>

## 另一条路：把错误编码进返回值

异常适合"真正异常"的情况（罕见、严重）。但对**预期内的失败**（查找没命中、解析格式不对、文件可能不存在），每次都抛异常既慢又啰嗦。这时更好的方式是**把"可能失败"编码进返回类型**。

### std::optional：可能有值，也可能没有

`std::optional<T>` 表示"一个也许存在的 T"——对应"查找可能没结果"这类场景，比返回特殊值（`-1`、`nullptr`）清晰得多：

```cpp
#include <iostream>
#include <optional>
#include <string>

// 返回 optional：找到返回值，找不到返回"空"
std::optional<int> parsePositive(const std::string& s) {
    try {
        int n = std::stoi(s);
        if (n > 0) return n;        // 有值
    } catch (...) {}
    return std::nullopt;            // 无值
}

int main() {
    for (const std::string& s : {std::string("42"), std::string("-3"), std::string("abc")}) {
        auto result = parsePositive(s);
        if (result) {                              // 检查是否有值
            std::cout << s << " -> " << *result << '\n';   // *result 取值
        } else {
            std::cout << s << " -> 解析失败\n";
        }
    }
}
```

输出：

```
42 -> 42
-3 -> 解析失败
abc -> 解析失败
```

`optional` 就像"可能为空的盒子":用 `if (result)` 检查有没有值,用 `*result` 取值。它对应 Java 的 `Optional<T>`、以及 Python 里返回 `None` 的惯用法——但类型系统会提醒你"这里可能没值"。

### std::expected：要么成功值，要么错误原因

`optional` 的短板是:失败时只告诉你"没有值",**说不出为什么失败**。C++23 的 `std::expected<T, E>` 补上了这点——它要么装着成功的 `T`,要么装着错误信息 `E`:

```cpp
#include <iostream>
#include <expected>
#include <string>

// 返回 expected：成功给 int，失败给错误描述
std::expected<int, std::string> parsePositive(const std::string& s) {
    try {
        int n = std::stoi(s);
        if (n > 0) return n;                              // 成功
        return std::unexpected("不是正数");                // 失败，带原因
    } catch (...) {
        return std::unexpected("不是合法数字");
    }
}

int main() {
    for (const std::string& s : {std::string("42"), std::string("-3"), std::string("abc")}) {
        auto result = parsePositive(s);
        if (result) {
            std::cout << s << " -> " << *result << '\n';           // 取成功值
        } else {
            std::cout << s << " -> 错误: " << result.error() << '\n';  // 取错误
        }
    }
}
```

输出：

```
42 -> 42
-3 -> 错误: 不是正数
abc -> 错误: 不是合法数字
```

`expected` 兼得两个世界:像返回值一样高效（无异常开销）、又能携带失败原因。用 `if (result)` 判断成败,`*result` 取成功值,`result.error()` 取错误。这是现代 C++ 处理"预期内失败"的首选,思路接近 Rust 的 `Result<T, E>`。

<div class="keypoint">
<strong>三种"可能失败"的返回类型</strong>
<ul>
<li><code>std::optional&lt;T&gt;</code>：有值 / 无值。失败原因不重要时用。</li>
<li><code>std::expected&lt;T, E&gt;</code>（C++23）：成功值 / 错误值。需要知道失败原因时用。</li>
<li>裸的错误码（返回 <code>int</code>、<code>bool</code> + 输出参数）：老式 C 风格，现代代码尽量用上面两个替代——它们类型安全、无法被忽略。</li>
</ul>
</div>

<div class="keypoint">
<strong>选择的经验法则</strong>
<ul>
<li><strong>预期内的、常见的失败</strong>（查找未命中、输入格式错、文件可能不存在）→ 用 <code>optional</code> / <code>expected</code>。这些不是"异常"，是正常的程序分支。</li>
<li><strong>罕见的、严重的、破坏了前提条件的错误</strong>（内存耗尽、不变量被违反、构造函数无法建立有效对象）→ 用<strong>异常</strong>。构造函数尤其只能靠异常报错（它没有返回值）。</li>
<li><strong>性能极敏感或禁用异常的环境</strong>（游戏引擎、嵌入式、某些库）→ 全程用返回值。异常虽然"零成本"（不抛时无开销），但抛出时较慢，且有些团队因二进制体积、确定性等原因整体禁用它（<code>-fno-exceptions</code>）。</li>
</ul>
一句话：<strong>正常流程里可能发生的"失败"，用返回值；真正意外的"错误"，用异常。</strong>
</div>

<div class="note">
<strong>为什么 C++ 没有 Java 那样的"受检异常"</strong>
Java 强制你在签名声明 <code>throws</code> 并处理。C++ 刻意没这么做——经验表明受检异常常导致"到处 catch 了又原样重抛"的噪音。C++ 的方向反而是：用 <code>expected</code> 把"可预期的失败"变成<strong>返回值</strong >（编译器逼你检查返回值），把异常留给"真正异常"的情况。这是两种不同的错误处理哲学。
</div>

## 动手：同一个操作，两种错误处理风格

下面把"安全除法"分别用异常和 `expected` 实现，直观对比两种风格。运行前先预测输出：

```cpp
#include <iostream>
#include <expected>
#include <stdexcept>
#include <string>

// 风格一：异常
int divideThrow(int a, int b) {
    if (b == 0) throw std::runtime_error("除零");
    return a / b;
}

// 风格二：expected
std::expected<int, std::string> divideExpected(int a, int b) {
    if (b == 0) return std::unexpected("除零");
    return a / b;
}

int main() {
    // 异常风格：正常流程被 try/catch 包裹
    try {
        int r1 = divideThrow(10, 2);        // 先求值，再打印
        std::cout << "异常风格: " << r1 << '\n';
        int r2 = divideThrow(10, 0);        // 这里抛出，下一行不执行
        std::cout << "异常风格: " << r2 << '\n';
    } catch (const std::exception& e) {
        std::cout << "异常风格捕获: " << e.what() << '\n';
    }

    // expected 风格：错误就在返回值里，线性处理
    for (auto [a, b] : {std::pair{10, 2}, std::pair{10, 0}}) {
        auto r = divideExpected(a, b);
        if (r) std::cout << "expected 风格: " << *r << '\n';
        else   std::cout << "expected 风格错误: " << r.error() << '\n';
    }
}
```

预期输出：

```
异常风格: 5
异常风格捕获: 除零
expected 风格: 5
expected 风格错误: 除零
```

注意两种风格的**代码形态差异**：异常风格里，正常流程被 `try` 包住、错误跳到 `catch`（控制流有跳转）；`expected` 风格里，成功和失败都是普通返回值，处理是线性的、就在调用点旁边。对"预期内可能失败"的操作，后者往往更易读、也更高效。

**这一章的要点**

- C++ 异常形态类似 Java / Python，但**约定抛 `std::exception` 派生类**、**按 `const&` 捕获**、**无受检异常**。
- **RAII + 异常**是安全组合：栈展开时局部对象自动析构，资源不泄漏（第 5 章）。**析构函数绝不能抛异常**。
- **`noexcept`** 承诺不抛，是优化线索——移动构造标 `noexcept` 才能让容器扩容用移动而非拷贝（第 7 章）。
- 对**预期内的失败**，把结果编码进返回类型：**`std::optional<T>`**（有/无值）、**`std::expected<T, E>`**（成功值/错误因，C++23）。
- 选择准则：**正常流程的"失败"用返回值，真正意外的"错误"用异常**；构造失败只能用异常；性能敏感/禁异常环境全用返回值。
