## 从 CPU 芯片的样子说起

你对 CPU 芯片有个大致印象：几个（到几十个）**又大又复杂**的核心，每个核心有自己的 L1/L2 缓存，还有巨大的乱序执行引擎、分支预测器；几个核心共享一块 L3 缓存；芯片外是主存（DRAM）。晶体管的预算，大量花在了**让单个核心变聪明**上——缓存、预测、调度，全为降低单线程延迟。

GPU 把同样的晶体管预算花在了**完全不同**的地方：几乎全砸在**算术单元**上，只留很少给控制和缓存。结果是一颗 GPU 上有**成千上万个**极其简单的计算核心，但它们必须共享控制逻辑（这正是上一章 SIMT 的物理原因）。

## GPU 的基本积木：SM（流多处理器）

一颗 GPU 不是一整块，而是由几十个（旗舰卡上百个）相同的单元拼起来，NVIDIA 叫它 **SM（Streaming Multiprocessor，流多处理器）**。SM 是 GPU 真正的"干活单元"，你可以把它类比成 **CPU 的一个物理核心**——只不过它内部宽得多。一个 SM 里有：

- 一堆 **CUDA 核心**（算术单元），还有专门的单元如 **Tensor Core**（做矩阵乘，第 19 章）；
- 一个巨大的**寄存器堆**，分给驻留在这个 SM 上的所有线程；
- 一块 **shared memory / L1**（片上高速内存，几十到几百 KB，由你的程序显式管理，第 7 章的主角）；
- 若干 **warp 调度器**：负责在众多就绪的 warp 之间快速切换，用别的 warp 的计算填满某个 warp 等内存的空档——这就是第 1 章说的"用并行掩盖延迟"的硬件实现。

## 两种内存，两个数量级的速度差

GPU 上最影响性能的硬件事实，是**内存的层级**。粗分两大类：

- **片上内存**：寄存器、shared memory/L1、L2。就在 SM 里或紧挨着，**极快**，但**极小**（KB 到 MB 级）。
- **片外显存（全局内存）**：就是你买卡时看的"24GB 显存"。**大**（GB 级），但相对**慢**得多——访问一次要几百个时钟周期。

这个速度差是理解 GPU 性能的钥匙。回忆第 1 章：GPU 算力严重过剩，所以绝大多数真实 kernel 的瓶颈不是"算得慢"，而是"**数据喂不上来**"——卡在从显存搬数据。于是有了一个衡量 kernel 的核心指标：**算术强度（arithmetic intensity）= 每从显存读一个字节，能做多少次计算**。强度低（读一次只算一两下）的 kernel 叫 **memory-bound（访存受限）**，它的速度由**显存带宽**决定，跟你的 GPU 算力(FLOPS)几乎无关。这就是"GPU 卡在内存带宽上"这句话的确切含义，第 16 章的 Roofline 模型会把它讲成一套可计算的方法。

## 动手：算清"访存受限"到底意味着什么

下面这个程序**能在浏览器里直接跑**——点 **▶ 运行**。它用一块真实旗舰 GPU 的大致参数（算力和带宽），帮你算：像向量加法这种"读得多、算得少"的 kernel，到底是被算力卡住，还是被带宽卡住。这解释了为什么本书后半本花那么多篇幅优化访存，而不是优化计算。

```python
# 用真实量级的 GPU 参数，判断一个 kernel 是"算力受限"还是"带宽受限"
# 参数取自某数据中心级 GPU 的大致公开量级（FP32）
peak_flops   = 60e12      # 峰值算力：60 万亿次浮点运算/秒（60 TFLOP/s）
peak_bw      = 3.0e12     # 显存带宽：3 万亿字节/秒（3 TB/s）

# 机器的"平衡点"：每读 1 字节，需要配多少次计算，才能刚好喂饱算力单元
balance = peak_flops / peak_bw
print(f"这块 GPU 的平衡点：每读 1 字节需要 {balance:.0f} 次浮点运算，才不浪费算力\n")

def analyze(name, flops_per_elem, bytes_per_elem):
    ai = flops_per_elem / bytes_per_elem          # 算术强度
    # 完成 1 个元素，算力路径要多久 vs 访存路径要多久
    t_compute = flops_per_elem / peak_flops
    t_memory  = bytes_per_elem / peak_bw
    bound = "算力受限（compute-bound）" if t_compute > t_memory else "带宽受限（memory-bound）"
    print(f"{name}：")
    print(f"  每元素 {flops_per_elem} 次运算、搬 {bytes_per_elem} 字节 → 算术强度 {ai:.3f} 次/字节")
    print(f"  → {bound}")

# 向量加法 c=a+b：读 a、读 b、写 c 共 3 个 float = 12 字节，只做 1 次加法
analyze("向量加法 c=a+b", flops_per_elem=1, bytes_per_elem=12)
print()
# 一个计算密集的假想 kernel：搬同样的字节，却做 500 次运算
analyze("某计算密集 kernel", flops_per_elem=500, bytes_per_elem=12)
```

运行结果：

```
这块 GPU 的平衡点：每读 1 字节需要 20 次浮点运算，才不浪费算力

向量加法 c=a+b：
  每元素 1 次运算、搬 12 字节 → 算术强度 0.083 次/字节
  → 带宽受限（memory-bound）

某计算密集 kernel：
  每元素 500 次运算、搬 12 字节 → 算术强度 41.667 次/字节
  → 算力受限（compute-bound）
```

结论很扎心：向量加法的算术强度只有 0.083，离平衡点 20 差了两百多倍——它 **99% 的时间在等内存，算力单元基本闲着**。你把这块卡换成算力翻倍的新卡，向量加法**一点都不会变快**，因为瓶颈在带宽。请动手改：把 `analyze("向量加法"...)` 的 `flops_per_elem` 从 1 慢慢调大，看要调到多少它才从"带宽受限"翻转成"算力受限"——那个翻转点，正好就是这块机器的平衡点。

<div class="note">
<strong>这对你写 kernel 意味着什么</strong>
既然多数 kernel 卡在带宽，那么<strong>减少访存、优化访存模式</strong>往往比减少计算更能提速：把数据搬进 shared memory 复用（第 7、11 章）、让一个 warp 的访存能"合并"成一次大传输（第 7、17 章）、把多个算子融合成一个 kernel 少读写几趟显存（第 14 章）。这些都是本书后半本的主线，它们的共同出发点就是这张"两种内存差两个数量级"的图。
</div>

## 一次 kernel 启动，硬件上发生了什么（先建立轮廓）

把这一章串起来，一次 GPU 计算的大致流程是这样的（细节在第 9 章）：

1. CPU（host）把数据从内存拷进 GPU 的**显存**；
2. CPU 发起一次 **kernel 启动**，告诉 GPU："用这么多 block、每 block 这么多线程，去跑这个函数"；
3. GPU 的调度器把这些 **block 分发到各个 SM** 上，一个 SM 可同时驻留多个 block；
4. 每个 SM 把 block 里的线程切成 **warp**，warp 调度器让就绪的 warp 轮流上算术单元，谁等内存就切别人上——**延迟被并行掩盖**；
5. 算完，结果留在显存，CPU 再把它拷回内存。

<div class="warn">
<strong>别忽视第 1 步和第 5 步的搬运成本</strong>
CPU 内存 ↔ GPU 显存 的拷贝要走 PCIe 总线，比显存本身还慢一截。如果你只是为了做一次向量加法就把数据搬上 GPU 再搬回来，光搬运的时间可能就远超省下的计算——这又一次说明第 1 章那句话：<strong>GPU 只在"又多又独立、值得搬过去反复算"的活儿上才划算。</strong>
</div>

## 小结与下一步

你现在有了一张 GPU 硬件地图：芯片由几十个 SM 组成，每个 SM 是个"很宽的核"，内含核心、寄存器、片上高速内存和 warp 调度器；片外是又大又相对慢的显存，而**带宽**常常才是真正的瓶颈。

有了硬件地图，我们终于可以正式讲**软件层级**了。下一章开始进入本书的地基部分：你启动 kernel 时写的 `<<<blocks, threads>>>` 到底是什么——**Grid、Block、Thread** 这套三层结构，以及它们怎么映射到你刚认识的这些 SM 上。

---

**这一章的要点**

- CPU 把晶体管花在**让少数核变聪明**（缓存、预测、乱序）；GPU 把晶体管花在**堆砌海量简单算术单元**，代价是它们共享控制逻辑（SIMT 的物理原因）。
- GPU 的基本积木是 **SM（流多处理器）**，类比一个"很宽的 CPU 核"：内含 CUDA/Tensor 核心、寄存器堆、**shared memory/L1**、和调度 warp 的硬件。
- 软件概念的落点：**block → 一个 SM**，block 内线程切成 **warp** 交给调度器，**grid** 铺满全片 SM。
- 内存分**片上**（寄存器/shared/L2，快而小）和**片外显存**（大而慢，差两个数量级）；**带宽**常是真正瓶颈。
- 用**算术强度**判断 kernel 是**算力受限**还是**带宽受限**——多数真实 kernel 带宽受限，这决定了本书后半本以**优化访存**为主线。
