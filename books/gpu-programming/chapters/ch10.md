## CUDA 是什么，先给它定位

**CUDA（Compute Unified Device Architecture）** 是 NVIDIA 2007 年推出的 GPU 通用计算平台。它有三张牌是别人短期内比不了的：

- **最贴近硬件、给你最多控制**：warp、shared memory、寄存器、异步拷贝、Tensor Core……NVIDIA 硬件有的能力，CUDA 几乎都开了对应的接口。你想榨干一块 NVIDIA 卡，CUDA 是天花板最高的那条路。
- **最成熟、生态最大**：编译器（`nvcc`）、调试器（`cuda-gdb`）、性能分析器（Nsight）、以及一整套高度优化的库——线性代数 **cuBLAS**、深度学习 **cuDNN**、并行算法模板库 **Thrust**、稀疏计算 cuSPARSE……绝大多数 AI 框架的底座就是它们。
- **是事实上的基准**：后面第 12–15 章讲 SYCL、Triton、CuTe DSL，绕不开的一句话都是"它对标 CUDA 的哪一块、又想解决 CUDA 的什么痛点"。

那 CUDA 的短板呢？就一条，但很致命：**只能跑在 NVIDIA 的 GPU 上**。你的代码一旦深度用了 CUDA，就和 NVIDIA 绑定了。这正是后面几种技术存在的理由——它们拿"可移植"来换 CUDA 的一部分"极致控制"。

## 一张图看懂 CUDA 的编程模型

第二部分你已经零散见过这些概念，这里把它们钉在一起。CUDA 程序永远分成两半：跑在 **CPU（host，主机）** 上的普通 C++，和跑在 **GPU（device，设备）** 上的 kernel。host 负责分配显存、搬数据、发起启动、收回结果；device 负责真正的并行计算。二者用三个函数修饰符划清边界：

| 修饰符 | 在哪运行 | 谁能调用 | 典型用途 |
|---|---|---|---|
| `__global__` | device | host（用 `<<<>>>`） | **kernel** 入口，一次启动铺开一大片线程 |
| `__device__` | device | device（kernel 或别的 `__device__`） | kernel 内部复用的辅助函数 |
| `__host__` | host | host | 普通 CPU 函数（不写也是默认） |

> 小技巧：一个函数同时标 `__host__ __device__`，编译器会给 CPU 和 GPU **各编一份**，同一段代码两边都能用——写数学工具函数时很常见。

kernel 用 **执行配置（execution configuration）** `<<<gridDim, blockDim>>>` 启动，这两个数就是第 4 章的 grid 和 block 尺寸。kernel 内部靠四个**内置变量**知道"我是谁"：`gridDim`（grid 有几个 block）、`blockDim`（block 有几个线程）、`blockIdx`（我在第几个 block）、`threadIdx`（我在 block 内第几个）。经典的全局下标公式你已经很熟了：

```cuda-norun
int i = blockIdx.x * blockDim.x + threadIdx.x;   // 一维；二维再加 .y
if (i < n) { /* 只处理我这一个元素 */ }            // 边界检查不能省
```

## CUDA 的编译轨迹：一个 .cu 文件会变成什么

这是 CUDA 最容易含糊、但对理解性能和调试极重要的一环。你写的 `.cu` 文件里 host 和 device 代码**混在一起**，`nvcc` 做的第一件事就是把它俩**拆开**，各走各的编译流水线，最后再缝进同一个可执行文件。

device 代码这条路又分两步，是重点：

1. **`.cu` → PTX**：`nvcc` 先把 device 代码编成 **PTX（Parallel Thread Execution）**。PTX 是一种**虚拟架构**的中间汇编——它不对应任何一块真卡，而是一个稳定的、面向"计算能力 `compute_XX`"的指令集。你在 Compiler Explorer 里看到的就是它。
2. **PTX → SASS**：`ptxas` 再把 PTX 编成 **SASS（Streaming ASSembler）**——某个**真实架构 `sm_XX`** 的机器码，这才是 GPU 真正执行的东西。

`-arch`（或 `-gencode`）就是告诉编译器"目标是哪一代硬件"。这里牵出两个词：

- **AOT（提前编译）**：编译时就把 SASS 生成好嵌进二进制。跑得快、启动无开销，但只认你指定的那几代架构。
- **JIT（运行时即时编译）**：二进制里同时嵌一份 PTX。如果程序在一块**比编译时更新**的 GPU 上运行、找不到匹配的 SASS，驱动会在运行时把 PTX **现编成** SASS。这就是为什么老程序能在新卡上跑——PTX 是那道**向前兼容**的保险。

## Runtime API 还是 Driver API

你写 CUDA 会遇到两套接口，别被绕晕。**Runtime API**（函数以 `cuda...` 开头，如 `cudaMalloc`、`cudaMemcpy`）是高层、好用、绝大多数人用的那套，`<<<>>>` 语法就是它的糖。**Driver API**（以 `cu...` 开头，如 `cuMemAlloc`、`cuLaunchKernel`）是底层，手动管理 context 和模块加载，更啰嗦但更可控，常见于需要动态加载 PTX、或做语言绑定的场景。

**默认用 Runtime API**——本书所有 CUDA 例子都是。知道 Driver API 存在即可，等你要做类似"运行时把一段 PTX 加载进来执行"的高级活儿时再深入。

## 用 stream 把拷贝和计算重叠起来

第 9 章那个向量加法是**串行三步**：把数据拷进显存（H2D）→ 跑 kernel → 把结果拷回（D2H）。这三步用的是**不同的硬件引擎**（拷贝引擎和计算引擎彼此独立），串行做等于让引擎轮流闲着。

**CUDA stream（流）** 就是解药。一个 stream 是一串**按顺序执行**的操作；不同 stream 之间**可以并发**。把数据切成几块、每块丢进不同的 stream，硬件就能一边拷第 2 块、一边算第 1 块、一边把第 0 块的结果拷回——**三个引擎同时忙起来**。这就是"用重叠掩盖拷贝延迟"，和第 1 章"用并行掩盖访存延迟"是同一种思想，只是换了个层级。

```cuda-norun
// 用多个 stream 把 H2D / kernel / D2H 流水线化（示意）
cudaStream_t s[NSTREAMS];
for (int k = 0; k < NSTREAMS; ++k) cudaStreamCreate(&s[k]);
for (int c = 0; c < nchunks; ++c) {
    int st = c % NSTREAMS;                      // 轮流分配到不同 stream
    cudaMemcpyAsync(dx+off, hx+off, sz, cudaMemcpyHostToDevice, s[st]);
    kern<<<blocks, threads, 0, s[st]>>>(dx+off, /* ... */);
    cudaMemcpyAsync(hout+off, dout+off, sz, cudaMemcpyDeviceToHost, s[st]);
}
// 注意：异步拷贝要求 host 内存是 pinned（cudaMallocHost 分配）才能真重叠
```

## 动手：亲眼看 stream 省下多少时间

下面这段程序**能在浏览器里直接跑**——点 **▶ 运行**。它不真用 GPU，而是忠实模拟一条规则：**每个硬件引擎（H2D 拷贝、计算、D2H 拷贝）同一时刻只能干一件事，且同一块数据必须按 H2D→计算→D2H 顺序走**。我们把数据切成几块，对比"全程串行"和"三引擎重叠"两种排法花的时间，并把重叠后的时间线画出来。

```python
# 模拟 CUDA stream：把 H2D → 计算 → D2H 流水线化，看重叠能省多少时间
# 三个引擎各自独立、各自按块顺序排队；同一块数据三个阶段必须依次进行
H2D, COMPUTE, D2H = 2, 3, 2    # 每块数据三个阶段各自的耗时（任意时间单位）
NCHUNKS = 4                     # 把整批数据切成几块

def serial_time():
    # 没有 stream：一块走完三个阶段，才轮下一块，全程串行
    return NCHUNKS * (H2D + COMPUTE + D2H)

def pipelined():
    f_h2d  = [0] * NCHUNKS       # 每块 H2D 结束的时刻
    f_comp = [0] * NCHUNKS       # 每块 计算 结束的时刻
    f_d2h  = [0] * NCHUNKS       # 每块 D2H 结束的时刻
    for i in range(NCHUNKS):
        prev = f_h2d[i-1] if i > 0 else 0        # H2D 引擎上一块何时空出
        f_h2d[i] = prev + H2D
        prev = f_comp[i-1] if i > 0 else 0
        start = max(f_h2d[i], prev)              # 要等本块拷完 且 计算引擎空闲
        f_comp[i] = start + COMPUTE
        prev = f_d2h[i-1] if i > 0 else 0
        start = max(f_comp[i], prev)             # 要等本块算完 且 D2H 引擎空闲
        f_d2h[i] = start + D2H
    return f_h2d, f_comp, f_d2h

def gantt(engines, makespan):
    for name, finishes, cost in engines:
        row = ['.'] * makespan                   # '.' 表示引擎空闲
        for i in range(NCHUNKS):
            for t in range(finishes[i] - cost, finishes[i]):
                row[t] = str(i)                  # 数字=第几块数据占用该引擎
        print(f"  {name}: {''.join(row)}")

s = serial_time()
f_h2d, f_comp, f_d2h = pipelined()
p = f_d2h[-1]
print(f"数据切成 {NCHUNKS} 块，每块 H2D={H2D} 计算={COMPUTE} D2H={D2H}（时间单位）\n")
print(f"不用 stream（全程串行）：{s} 个时间单位")
print(f"用 stream（三引擎重叠）：{p} 个时间单位")
print(f"加速比：{s/p:.2f}×\n")
print("重叠后的时间线（数字=第几块数据，'.'=引擎空闲）：")
gantt([("H2D    ", f_h2d, H2D),
       ("compute", f_comp, COMPUTE),
       ("D2H    ", f_d2h, D2H)], p)
```

运行结果：

```
数据切成 4 块，每块 H2D=2 计算=3 D2H=2（时间单位）

不用 stream（全程串行）：28 个时间单位
用 stream（三引擎重叠）：16 个时间单位
加速比：1.75×

重叠后的时间线（数字=第几块数据，'.'=引擎空闲）：
  H2D    : 00112233........
  compute: ..000111222333..
  D2H    : .....00.11.22.33
```

盯着这三行时间线看：`compute` 引擎从第 2 步就开始干活，一直忙到第 14 步几乎不停——这正是重叠的收益，计算不再干等拷贝。串行要 28，重叠只要 16。请动手改：把 `NCHUNKS` 从 4 调到 8，再运行——你会看到加速比进一步逼近 `(H2D+COMPUTE+D2H)/COMPUTE`（因为计算是这里最慢的引擎，最终整条流水线的节奏由它决定，这就是**瓶颈引擎**）。

<div class="warn">
<strong>重叠不是免费的，有两个前提</strong>
第一，异步拷贝 <code>cudaMemcpyAsync</code> 要真和计算重叠，host 端内存必须是<strong>锁页内存（pinned memory，<code>cudaMallocHost</code> 分配）</strong>，否则驱动会退化成同步拷贝。第二，切块太细会让每块的启动开销占比变大，收益递减。stream 是"值得搬一大批数据反复算"时的优化，不是万能药。
</div>

## 动手：一个比向量加法更丰富的完整 kernel

下面是一个**完整可编译**的 CUDA 程序，做 **SAXPY**（`out = a*x + y`，线性代数里的经典操作），并特意用一个 `__device__` 辅助函数演示 device 端的函数复用。点 **🔗 在 Compiler Explorer 打开** 看它的 PTX，或用下面的 `nvcc` 命令在本地编译运行。

```cuda
#include <cstdio>
#include <cstdlib>
#include <cuda_runtime.h>

// __device__ 函数：只能被 device 上的代码（kernel 或别的 __device__）调用
__device__ float saxpy_one(float a, float x, float y) {
    return a * x + y;                 // 单个元素的 SAXPY
}

// __global__ 函数：kernel 入口，从 host 用 <<<>>> 启动，在 device 上并行执行
__global__ void saxpy(int n, float a, const float* x, const float* y, float* out) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;   // 全局线程编号 = 我负责第几个元素
    if (i < n)                                        // 边界检查：线程数常多于元素数
        out[i] = saxpy_one(a, x[i], y[i]);            // 每个线程只算一个元素
}

int main() {
    const int   N = 1 << 20;          // 约 100 万个元素
    const float a = 2.0f;
    size_t bytes = N * sizeof(float);

    // ---- host 端分配并填充 ----
    float *hx = (float*)malloc(bytes);
    float *hy = (float*)malloc(bytes);
    float *hout = (float*)malloc(bytes);
    for (int i = 0; i < N; ++i) { hx[i] = 1.0f; hy[i] = 2.0f; }

    // ---- device 端分配 ----
    float *dx, *dy, *dout;
    cudaMalloc(&dx, bytes);
    cudaMalloc(&dy, bytes);
    cudaMalloc(&dout, bytes);

    // ---- host → device 拷贝输入 ----
    cudaMemcpy(dx, hx, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(dy, hy, bytes, cudaMemcpyHostToDevice);

    // ---- 执行配置：每 block 256 线程，凑够 N 个线程 ----
    int threads = 256;
    int blocks  = (N + threads - 1) / threads;
    saxpy<<<blocks, threads>>>(N, a, dx, dy, dout);
    cudaDeviceSynchronize();          // 等 kernel 真正跑完

    // ---- device → host 拷回结果 ----
    cudaMemcpy(hout, dout, bytes, cudaMemcpyDeviceToHost);

    // ---- 验证：每个都应是 a*1 + 2 = 4 ----
    bool ok = true;
    for (int i = 0; i < N; ++i)
        if (hout[i] != 4.0f) { ok = false; break; }
    printf("out[0] = %.1f, 校验 %s\n", hout[0], ok ? "OK" : "FAILED");

    cudaFree(dx); cudaFree(dy); cudaFree(dout);
    free(hx); free(hy); free(hout);
    return 0;
}
```

本地编译运行：

```bash
nvcc -arch=sm_80 saxpy.cu -o saxpy && ./saxpy
# 输出：out[0] = 4.0, 校验 OK
```

把上面 kernel 单独拎出来，就能在 Compiler Explorer 里专心看它编成的 PTX——注意找 `mad`（乘加融合）或 `fma` 指令，那就是 `a*x+y` 被编译器合成的一条乘加：

```cuda-ptx
// 贴进 Compiler Explorer，编译器选 nvcc，参数填 -arch=sm_80
// 右侧就是这段 kernel 编成的 PTX（GPU 的虚拟架构汇编）
__device__ float saxpy_one(float a, float x, float y) {
    return a * x + y;
}
__global__ void saxpy(int n, float a, const float* x, const float* y, float* out) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) out[i] = saxpy_one(a, x[i], y[i]);
}
```

<div class="note">
<strong>看 PTX 时留意这几处</strong>
<code>saxpy_one</code> 大概率被<strong>内联</strong>掉了（PTX 里看不到单独的函数调用）——device 端的小函数几乎总会被内联，所以放心把 kernel 拆成小 <code>__device__</code> 函数，可读性上去了，性能不打折。再看那句 <code>@%p bra</code> 之类的条件跳转，就是你写的 <code>if (i &lt; n)</code> 边界检查。
</div>

## 小结与下一步

你现在把 CUDA 的编程模型系统化了：host/device 两半、三个修饰符、执行配置与内置变量；也搞清了 `.cu` 一路编到 PTX 再到 SASS 的轨迹、AOT 与 JIT 的分工，以及 stream 怎么用重叠换速度。这套东西是后面所有技术的参照原点。

但到目前为止，我们的 kernel 都是"每个线程读一次、算一下、写一次"的**访存受限**类型（回忆第 3 章的算术强度）。CUDA 真正拉开差距的地方，是它让你**亲手管理 shared memory**——把数据搬进片上高速内存反复复用，把一个访存受限的问题硬生生推向算力受限。下一章我们就用 GPU 优化里最经典的案例——**用 shared memory 做 tiling 矩阵乘**——把这件事讲透。

---

**这一章的要点**

- CUDA 的定位：**最贴硬件、控制最全、生态最大（cuBLAS/cuDNN/Thrust）**，但**只跑 NVIDIA GPU**；它是本书其它技术的参照原点。
- 编程模型：**host/device 分工**；`__global__`（kernel 入口）、`__device__`（device 内部函数）、`__host__`（CPU 函数）三个修饰符；执行配置 `<<<grid, block>>>` + 内置变量 `blockIdx/blockDim/threadIdx/gridDim`。
- 编译轨迹：`nvcc` 把 `.cu` 拆成 host 与 device 两路；device 走 **PTX（虚拟架构）→ ptxas → SASS（真实架构）**；**AOT** 提前编好 SASS、**JIT** 用嵌入的 PTX 在更新的卡上现编——PTX 是向前兼容的保险。
- **Runtime API**（`cuda*`）好用是默认；**Driver API**（`cu*`）底层可控，知道即可。
- **stream** 让 H2D/计算/D2H 三个引擎重叠流水线化，用重叠掩盖拷贝延迟；前提是 pinned 内存 + 别切太碎。
