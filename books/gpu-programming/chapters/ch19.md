## 回顾：CUDA 核心是"标量的乘加机器"

第 3 章说过，一个 SM 里塞满了 **CUDA 核心**，每个核心一拍做一次标量运算。矩阵乘法的内层就是无数次**乘加（FMA，fused multiply-add）**：`c += a * b`。一个 warp 的 32 个 CUDA 核心，一拍能做 32 个这样的标量 FMA。

问题在于，AI 的核心计算是**矩阵乘**，而且是巨大的矩阵乘。一个 `M×K` 乘 `K×N` 的矩阵乘要做 `M×N×K` 次乘加。用标量 FMA 一次做一个，哪怕几千个核心一起上，面对动辄上亿次乘加的深度学习层，也显得力不从心。于是硬件工程师做了一件事：**造一个专门做小矩阵乘的电路，一条指令吞下一整块乘加。**

## Tensor Core：一条指令，一整块矩阵乘加

**Tensor Core（张量核心）**是 Volta（2017）起每个 SM 里新增的专用单元。它不做标量 FMA，而是一条指令完成一次小规模的**矩阵乘累加（MMA，matrix multiply-accumulate）**：

$$D = A \times B + C$$

其中 `A`、`B`、`C`、`D` 都是小方块（例如 16×16）。一次 `16×16×16` 的 MMA 相当于 `16×16×16 = 4096` 次标量乘加——**一条指令干掉了几千次 FMA**。这就是 Tensor Core 相比 CUDA 核心在矩阵乘上能快一个数量级甚至更多的原因：它把"取指、译码、调度"的开销摊到了几千次运算上，而不是每次乘加都摊一遍。

### 混合精度：为什么输入用 FP16、累加用 FP32

Tensor Core 还有一个关键设计：**混合精度（mixed precision）**。它接受**低精度**的输入——FP16、BF16，新一代还有 FP8——但在内部**用 FP32 累加**，最后输出 FP32（或转回低精度）。

为什么这么设计？矩阵乘要把 `K` 个乘积加起来，`K` 一大，如果全程用 FP16 累加，小数值会被大数值"吃掉"（浮点吸收误差），精度崩坏。而输入用低精度有两个实打实的好处：**每个数占的比特少一半（或更少），同样带宽能喂进更多数据**（回忆第 3 章：大多数 kernel 卡在带宽）；**低精度乘法器电路更小更快，同样芯片面积能塞更多**。用低精度喂、用高精度攒——这是现代 AI 硬件的通用哲学，也是为什么训练里到处是 FP16/BF16。

<div class="warn">
<strong>你几乎从不直接写 Tensor Core 指令</strong>
Tensor Core 对数据在寄存器和 shared memory 里的<strong>排布（layout）</strong>有苛刻要求——哪个线程持有矩阵的哪几个元素，都有固定规矩。手写极易错且不可移植。所以现实中你通过<strong>层层封装</strong>去用它：最省心的是 <strong>cuBLAS</strong>（调一个 <code>gemm</code>）；要融合就用 <strong>CUTLASS</strong>（第 15 章 CuTe 就是它的底座）或 <strong>Triton 的 <code>tl.dot</code></strong>（第 13、14 章，一行就吃上 Tensor Core）；只有写库的人才碰底层的 <strong>WMMA API</strong> 或 PTX 的 <code>mma</code> 指令。本章末尾给你看一眼 WMMA，是为了让你<strong>认得它</strong>，不是让你天天写它。
</div>

## 异步拷贝：搬数据时别让计算干等

Tensor Core 把"算"加速了几十倍，瓶颈立刻回到"喂"——第 3 章那句老话又灵验了：**算得再快，数据搬不上来也白搭**。

回忆第 11 章的 tiling 矩阵乘：每个 block 反复地"从全局内存搬一小块（tile）到 shared memory → 算这一块 → 搬下一块"。传统写法里，"搬"和"算"是**串行**的：线程发起一次全局内存读，然后**卡住等它到货**，货到了才开始算。等的这几百个周期里，昂贵的 Tensor Core 在闲着。

现代硬件给了两件工具打破这个串行：

- **`cp.async`（异步拷贝，Ampere 起）**：发起一次"全局 → shared"的拷贝后**立即返回**，不等它完成。线程可以接着干别的（比如算上一块），等真要用数据时再用一条指令等它到齐。搬运和计算就能**重叠**。
- **TMA（Tensor Memory Accelerator，Hopper 起）**：更进一步，一条指令就能让专门的硬件单元把一整块多维 tile 从全局内存搬进 shared memory，连地址计算都交给硬件，线程完全不用逐元素操心。这大大简化了代码，也释放了寄存器和指令槽。

这和第 10 章的 **CUDA 流（stream）** 是同一种思想——用重叠掩盖延迟——只是尺度不同：流是在**整个 kernel / 拷贝**的粗粒度上重叠，而 `cp.async` / TMA 是在**一个 block 内、一块 tile** 的细粒度上重叠。

## 软件流水线：一边算这块，一边预取下一块

有了异步拷贝，就能搭出现代高性能 kernel 的核心套路——**软件流水线（software pipelining）**，也叫**多级缓冲（multi-stage buffering）**。

思想和 CPU 的指令流水线、或工厂流水线一模一样：**在计算当前 tile 的同时，异步预取（prefetch）下一个 tile**。这样当前 tile 算完，下一块数据已经躺在 shared memory 里了，Tensor Core 无缝接着算，几乎不用等内存。你在 shared memory 里开**两块（double buffering）或多块**缓冲轮流用：一块正在被计算，另一块正在被异步填充。

## 动手：流水线能省多少时间

下面这段程序**能在浏览器里直接跑**——点 **▶ 运行**。它用一个简单的时间模型对比：处理 `K` 个 tile，**非流水线**（每块先 load 再 compute，串行）要多久，**流水线**（预取重叠）又要多久。你会看到：tile 数越多，流水线越接近"只花计算时间"，也就是从访存受限逼近算力受限。

```python
# 对比流水线 vs 非流水线：处理 K 个 tile 的总时间
LOAD = 100      # 搬一个 tile 的耗时（时钟周期，示意值）
COMPUTE = 60    # 算一个 tile 的耗时

def no_pipeline(K):
    # 每个 tile：先 load 卡住等，再 compute。两者串行
    return K * (LOAD + COMPUTE)

def pipelined(K):
    # 稳态：load(k+1) 与 compute(k) 重叠，每步耗时 = max(load, compute)
    # 但第一个 tile 要先 load 进来（填流水线），最后一段计算没有 load 可重叠
    if K == 0:
        return 0
    return LOAD + K * max(LOAD, COMPUTE)

print(f"每 tile：load={LOAD}，compute={COMPUTE}（周期）\n")
print(f"{'K(tile数)':>10} | {'非流水线':>8} | {'流水线':>7} | {'加速比':>6}")
print("-" * 44)
for K in [1, 2, 4, 8, 16, 64]:
    a = no_pipeline(K)
    b = pipelined(K)
    print(f"{K:>10} | {a:>8} | {b:>7} | {a/b:>5.2f}x")

# 当 K 很大，流水线趋近于"只花每步 max(load,compute)"，非流水线是它们之和
big = 100000
ratio = no_pipeline(big) / pipelined(big)
print(f"\nK 很大时加速比趋近 (load+compute)/max(load,compute) = "
      f"{(LOAD+COMPUTE)/max(LOAD,COMPUTE):.2f}x")
print(f"（K={big} 实测 {ratio:.2f}x）")
```

```
K=100000 实测 1.60x
```

三个洞察：（1）`K=1` 时流水线反而**更慢**——只有一块，没得重叠，还白搭了填流水线的开销；流水线只有在 tile 多时才划算。（2）tile 越多，加速比越逼近上限 `(load+compute)/max(load,compute) = 160/100 = 1.6`——因为稳态下每步只花 `max(load, compute)`，较小的那个被完全藏住。（3）如果 `load` 远大于 `compute`（严重访存受限），流水线能把大部分 load 藏进 compute，但当 `load > compute` 时上限受 `load` 主导。请动手改：把 `COMPUTE` 从 60 调到 100（让 load 和 compute 相等），再运行——加速比上限会变成 `2.0x`，这是流水线的理想情形：搬和算完美对半重叠。

## 动手：Tensor Core 到底快多少（算一算）

再用一个小模型量化 Tensor Core 相对 CUDA 核心的吞吐优势。核心事实：一个 SM 一拍能做的标量 FMA 数有限，而一条 MMA 指令一次吞下一整块乘加。

```python
# 粗略对比：算一个 M×N×K 矩阵乘，CUDA 核心 vs Tensor Core 各需多少"指令发射"
M, N, K = 256, 256, 256
total_mac = M * N * K              # 总乘加次数
print(f"矩阵乘 {M}x{N}x{K}：共 {total_mac:,} 次乘加\n")

# CUDA 核心：一条 FMA 指令做 1 次乘加（一个线程一拍一次）
cuda_core_insts = total_mac
# Tensor Core：一条 MMA 指令做 16*16*16 次乘加
MMA = 16 * 16 * 16
tensor_core_insts = total_mac / MMA

print(f"CUDA 核心：需发射 {cuda_core_insts:,} 条 FMA 指令")
print(f"Tensor Core：一条 MMA = {MMA} 次乘加，需发射 {tensor_core_insts:,.0f} 条 MMA 指令")
print(f"指令数之比：{cuda_core_insts / tensor_core_insts:,.0f}x 更少的指令")
print("\n更少的指令 = 更少的取指/译码/调度开销，这是 Tensor Core 吞吐优势的根源之一")
print("（真实加速还叠加了低精度带来的带宽与面积收益，实测常达十几到几十倍）")
```

运行结果：

```
矩阵乘 256x256x256：共 16,777,216 次乘加

CUDA 核心：需发射 16,777,216 条 FMA 指令
Tensor Core：一条 MMA = 4096 次乘加，需发射 4,096 条 MMA 指令
指令数之比：4,096x 更少的指令
```

（运行时你还会看到最后两行说明文字。）这不是说 Tensor Core 就快 4096 倍——真实吞吐还受带宽、排布、精度等一堆因素约束——但它直观地说明了"一条指令吞一整块"为什么在指令开销上占尽便宜。请动手改：把 tile 从 `16×16×16` 换成新一代硬件更大的 MMA 形状，看指令数怎么进一步下降。

## 动手：认一眼 WMMA（不要求会写）

最后看一段 **WMMA（Warp Matrix Multiply-Accumulate）API** 的骨架。它是 CUDA C++ 里直接使用 Tensor Core 的相对高层的入口，一个 **warp 协作**算一块 `16×16` 的乘累加。下面是 `-norun` 片段，你**不必会写**，只要能认出这几个关键动作：声明 **fragment**（片段，warp 里的线程共同持有的一块矩阵）、`load_matrix_sync` 从 shared memory 装进 fragment、`mma_sync` 做矩阵乘累加、`store_matrix_sync` 写回。

```cuda-norun
#include <mma.h>
using namespace nvcuda;

// 一个 warp 协作，用 Tensor Core 算 C(16x16) += A(16x16) * B(16x16)
__global__ void wmma_tile(const half* A, const half* B, float* C) {
    // fragment：一块矩阵在 warp 内 32 个线程间的分布式表示
    wmma::fragment<wmma::matrix_a, 16, 16, 16, half, wmma::row_major> a_frag;
    wmma::fragment<wmma::matrix_b, 16, 16, 16, half, wmma::col_major> b_frag;
    wmma::fragment<wmma::accumulator, 16, 16, 16, float> acc_frag;

    wmma::fill_fragment(acc_frag, 0.0f);        // 累加器清零

    wmma::load_matrix_sync(a_frag, A, 16);      // 从内存装入 A 片段（FP16 输入）
    wmma::load_matrix_sync(b_frag, B, 16);      // 装入 B 片段

    wmma::mma_sync(acc_frag, a_frag, b_frag, acc_frag);  // ← 一条 MMA：FP32 累加

    wmma::store_matrix_sync(C, acc_frag, 16, wmma::mem_row_major);  // 写回 FP32 结果
}
```

看懂这段代码的三个要点就够了：（1）操作的单位是**整块矩阵（fragment）**，不是标量；（2）输入是 `half`（FP16），累加器是 `float`（FP32）——正是前面说的混合精度；（3）所有 `_sync` 结尾的函数都是**一个 warp 一起做**的，Tensor Core 的最小协作单位是 warp，不是单个线程。真实的高性能 GEMM 会在这之上叠加 tiling（第 11 章）、`cp.async` 预取、多级流水线——那就是 CUTLASS 干的活，也是为什么大多数人宁可调 cuBLAS 或写 Triton，而不手搓 WMMA。

<div class="note">
<strong>你需要记住的分寸</strong>
Tensor Core、<code>cp.async</code>、TMA、多级流水线——这些是<strong>库和编译器</strong>的战场。作为应用开发者，你的价值不在手写它们，而在：<strong>知道它们存在</strong>（于是能看懂性能分析工具里"Tensor Core 利用率""内存流水线停顿"这些指标），<strong>知道怎么触发它们</strong>（用对精度、对齐好数据、让矩阵维度是 tile 的整数倍、给 Triton 设合理的 <code>num_stages</code>），以及<strong>知道选哪个工具</strong>（简单矩阵乘用 cuBLAS，要融合用 Triton/CuTe）。这层"理解但不手写"的认知，恰恰是资深工程师和调库新手的分水岭。
</div>

## 小结与下一步

现代 GPU 之所以是 AI 引擎，靠的是三样在前面章节没出现过的硬件能力，而它们环环相扣：**Tensor Core** 把矩阵乘加速几十倍，于是瓶颈回到喂数据；**异步拷贝（cp.async / TMA）** 让搬数据不再阻塞计算；**软件流水线**用多级缓冲把访存延迟藏进计算，让昂贵的 Tensor Core 尽量不空转。这三者拼在一起，就是 FlashAttention、CUTLASS 快得离谱的秘密，也是 Triton、CuTe DSL 存在的意义——把这套复杂机制封装起来，让你不必手写 PTX。

你现在集齐了理解现代 GPU 高性能代码的全部拼图：从心智模型、执行层级、内存与同步，到四大技术、性能模型、访存优化、并行原语，再到这一章的现代硬件武器。下一章我们把它们**收束到一个具体问题上**：同一个 **softmax**，分别用 CUDA、SYCL、Triton、CuTe DSL 写一遍，让你在真实算子里看清四种技术的取舍与共性——你会发现，本章的归约（softmax 要求 max 和 sum）和上一章的原语，全都要用上。

---

**这一章的要点**

- **Tensor Core** 一条 **MMA** 指令完成一块小矩阵乘累加（如 16×16×16 = 4096 次乘加），把取指/调度开销摊薄，矩阵乘吞吐达 CUDA 核心的十几到几十倍。
- **混合精度**：输入用 FP16/BF16/FP8（省带宽、省面积），内部用 **FP32 累加**（保精度）——现代 AI 硬件的通用哲学。
- 你几乎从不手写 Tensor Core，而是通过 **cuBLAS / CUTLASS / Triton `tl.dot` / WMMA API** 分层使用；WMMA 的单位是 warp 协作的 **fragment**。
- **异步拷贝（cp.async，Ampere）与 TMA（Hopper）** 让"全局 → shared"的搬运不阻塞计算，是第 10 章 stream"用重叠掩盖延迟"思想在 tile 粒度上的延伸。
- **软件流水线 / 多级缓冲**：算当前 tile 时预取下一 tile，稳态下把访存藏进计算——正是 Triton `num_stages`（第 14 章）调的东西，也是 CuTe / CUTLASS（第 15 章）封装的模式。
- 作为应用开发者，价值在于**理解、触发、选型**这些能力，而非手写它们——这是高级 DSL 存在的理由，也是 FlashAttention 快的根源。
