# 第 8 章 协作与同步：barrier、shuffle 与原子操作

## 这一章想让你带走什么

<div class="goal-box">
<p>前面几章里，线程都在各干各的：一个线程算一个元素，互不打扰。但真实 kernel 里线程常常需要<strong>协作</strong>——一起算一个和、共享一块数据、往同一个地方累加。这一章讲清 GPU 上的三种协作机制，以及一个反直觉的事实：<strong>GPU 上几乎不用锁。</strong></p>
<p>你会看到 <strong>block 级 barrier</strong>（<code>__syncthreads()</code>）、<strong>warp 级 shuffle</strong>（寄存器直接交换）、以及 <strong>原子操作</strong>（<code>atomicAdd</code>）各自解决什么问题、代价在哪，还会亲手跑两个模拟器：一个让你看到"一个慢线程拖垮整个 block"，一个用 5 步把 32 个数归约成 1 个。</p>
</div>

## 为什么 GPU 几乎不用锁

在 CPU 上，多个线程要更新同一个共享变量时，你的第一反应是加锁：

```cpp-norun
std::mutex m;
long total = 0;
// 每个线程：
{
    std::lock_guard<std::mutex> lk(m);  // 抢锁
    total += local_sum;                 // 临界区
}                                       // 出作用域自动解锁
```

在 8 个线程上，这没问题：抢锁的冲突不多，偶尔等一下。但把这套原封不动搬上 GPU，会立刻崩溃——不是语法错，是**性能灾难**。想想第 1 章的世界观：GPU 上是**成千上万个线程**同时冲向同一把锁。锁的本质是"让临界区串行执行"，于是你几万个线程排成一队，一个接一个进临界区——你精心堆起来的并行度瞬间归零，还不如单核 CPU。

所以 GPU 的协作哲学是**尽量不串行化**。它给你三件工具，每一件都是为"让尽可能多的线程仍然并行"而设计的：

1. **block 级 barrier**：不锁数据，而是设一个"集合点"，让一个 block 内的所有线程**等齐**再一起往下走。
2. **warp 级 shuffle**：让一个 warp 内的 32 个线程**直接交换寄存器**，连内存都不碰，用来做超快的 warp 内归约。
3. **原子操作**：当实在躲不开"多个线程更新同一个地址"时，用硬件原语一步完成"读-改-写"，不必加锁。

一句话锚定：**CPU 靠锁保护数据，GPU 靠 barrier 协调节奏、靠 shuffle 交换数据、靠原子操作压缩冲突。** 下面逐一看。

## 机制一：block 级 barrier —— 全体到齐再走

barrier（屏障）是一句集合命令：**"这个 block 里的所有线程，都到这一行为止，谁先到谁等着，等全员到齐，再一起继续。"** CUDA 里它叫 `__syncthreads()`，SYCL 里叫 `item.barrier(...)`：

<div class="compare">
<div class="known">
<div class="compare-head">CUDA：__syncthreads()</div>

```cuda-norun
__shared__ float tile[256];
tile[threadIdx.x] = in[i];   // 各自往 shared 写
__syncthreads();             // 等全 block 写完
float left = tile[threadIdx.x - 1];  // 现在读邻居才安全
```

</div>
<div class="cpp-side">
<div class="compare-head">SYCL：item.barrier()</div>

```sycl-norun
auto tile = /* local_accessor */;
tile[it.get_local_id(0)] = in[i];
it.barrier(sycl::access::fence_space::local_space);
float left = tile[it.get_local_id(0) - 1];
```

</div>
</div>

barrier 最典型的用途，正是配合上一章讲的 **shared memory**：一批线程先各自把数据搬进 shared memory，然后必须**等所有线程都搬完**，才能安全地去读别的线程搬来的数据。没有这道 barrier，你就会读到别人还没写入的垃圾值——这是一种数据竞争。barrier 不保护某个变量，它保护的是**时序**：把"写阶段"和"读阶段"隔开。

barrier 有一个必须刻进肌肉记忆的陷阱：**`__syncthreads()` 要求 block 里每一个线程都能执行到它。** 如果你把它写进了一段只有部分线程会进入的分支里，那些没进分支的线程永远到不了这个集合点，而进了分支的线程会永远等它们——整个 block **死锁**：

```cuda-norun
// ❌ 致命错误：把 __syncthreads() 放进了分歧的分支里
__global__ void deadlock(float* x, int n) {
    int i = threadIdx.x;
    if (i < n) {              // 当 n < blockDim.x 时，只有部分线程进来
        x[i] *= 2.0f;
        __syncthreads();      // 进来的线程在此等待——但没进来的永远不会到达
    }                         // → 整个 block 永久卡死
}

// ✅ 正确：barrier 放在所有线程都会执行到的位置，用别的方式处理边界
__global__ void ok(float* x, int n) {
    int i = threadIdx.x;
    if (i < n) x[i] *= 2.0f;  // 边界检查只包住工作，不包住 barrier
    __syncthreads();          // 全 block 都会到达
}
```

<div class="warn">
<strong>barrier 的黄金法则：要么全体执行，要么全体不执行</strong>
<code>__syncthreads()</code> 必须在 block 内所有线程都会到达的控制流里。放进依赖 <code>threadIdx</code> 的 <code>if</code>、放进可能被部分线程提前 <code>return</code> 跳过的位置，都会导致<strong>死锁</strong>——而且它不报错，就是永远挂住。这是新手最常踩的 GPU 坑之一，回顾第 2 章的 warp 分歧：分歧本身只是变慢，但<strong>分歧 + barrier = 死锁</strong>。
</div>

图 8-1 还揭示了 barrier 的性能含义：整个 block 的进度由**最慢的那个线程**决定。这正是第 2 章讲 warp 分歧要紧的另一个原因——如果 block 里有个 warp 因为分歧跑得慢，barrier 会让其余所有 warp 陪它一起等。我们用一个能在浏览器里跑的模拟器把这件事量化出来。

```python
# 模拟 block 级 barrier：先到的线程干等，全员到齐才一起放行
# arrivals[t] = 第 t 号线程"到达 barrier"所需的步数（模拟各线程进度不同）
arrivals = [2, 5, 1, 8, 3, 2, 4, 1]

barrier_time = max(arrivals)          # barrier 放行时刻 = 最慢线程的到达时刻
slowest = arrivals.index(barrier_time)

print("各线程到达 barrier 的情况：")
for tid, t in enumerate(arrivals):
    wait = barrier_time - t           # 到得早的，就得干等这么多步
    bar = "已到达" if wait == 0 else f"到达后干等 {wait} 步"
    print(f"  线程 {tid}：第 {t} 步到达，{bar}")

total_wait = sum(barrier_time - t for t in arrivals)
print(f"\nbarrier 放行时刻 = max(到达时刻) = 第 {barrier_time} 步")
print(f"整个 block 的进度由最慢的线程 {slowest} 号（第 {barrier_time} 步）决定")
print(f"其余线程一共白等了 {total_wait} 步——这些就是被 barrier 浪费掉的算力")
```

  线程 2：第 1 步到达，到达后干等 7 步
  线程 3：第 8 步到达，已到达

  线程 7：第 1 步到达，到达后干等 7 步

改一行看看：把 `arrivals` 里那个 `8` 改成 `2`，让所有线程的到达时刻都接近——你会发现放行时刻骤降、白等的步数也大幅减少。这就是 GPU 优化里一条朴素但重要的原则：**让一个 block 里的线程负载尽量均衡**，别让个别线程拖后腿，否则大家都得在 barrier 前陪等。

## 机制二：warp 级 shuffle —— 连内存都不碰的交换

barrier 协调的是整个 block，但它有成本：要访问 shared memory、要全员等齐。如果需要协作的线程**恰好在同一个 warp 里**（第 6 章：32 个线程齐步走），有一条更快的路——**shuffle（洗牌）**。

回忆第 6 章：一个 warp 的 32 个线程本就在同一时刻齐步执行，硬件让它们**天然同步**。shuffle 指令利用这一点，让一个线程**直接读另一个线程寄存器里的值**——不经过 shared memory，不需要 `__syncthreads()`，一条指令完成。最常用的是 `__shfl_down_sync`（从"编号更大"的 lane 取值）：

```cuda-norun
// warp 内归约：把一个 warp 的 32 个 val 求和，结果落在 lane 0
// mask 指定参与的线程，0xffffffff 表示全 32 个都参与
float val = /* 本线程的值 */;
for (int offset = 16; offset > 0; offset >>= 1)
    val += __shfl_down_sync(0xffffffff, val, offset);
// 现在 lane 0 的 val 就是整个 warp 的和；其余 lane 的 val 是中间结果
```

这段循环只有 5 次迭代（`offset` 取 16、8、4、2、1），就能把 32 个值加成 1 个。它是一棵**归约树**：每一步，活跃的 lane 数减半，`lane[i]` 把 `lane[i+offset]` 的值加过来。`__shfl_xor_sync`（按 lane 编号异或取值）则能做"蝶形"交换，让**每个** lane 最后都拿到总和（广播式归约），常用于 warp 内的 all-reduce。

<div class="keypoint">
<strong>三种协作，选谁？看协作的范围</strong>
需要协作的线程在<strong>同一个 warp（32 线程）</strong>内 → 用 <strong>shuffle</strong>，最快，无需 barrier、不碰内存。跨 warp、但在<strong>同一个 block</strong> 内 → 用 <strong>shared memory + barrier</strong>。跨 block（整个 grid 层面）→ 只能用<strong>全局内存 + 原子操作</strong>，因为不同 block 可能在不同 SM 上、彼此无法 barrier。范围越大，协作越贵——这也是高性能归约的通用套路：先 warp 内 shuffle、再 block 内 shared memory、最后 block 间 atomic，一层比一层用得少。
</div>

## 机制三：原子操作 —— 躲不开的冲突，用硬件压缩

有时候多个线程就是得更新**同一个地址**：统计直方图（很多线程往同一个 bin 加一）、求全局的和、往一个共享计数器申请下标。这时既不能各写各的（会互相覆盖），又不该加锁（会串行化)。GPU 的答案是**原子操作（atomic operation）**：硬件保证"读-改-写"三步不被打断，一条指令原子地完成。

```cuda-norun
atomicAdd(&counter, 1);        // 原子地 counter += 1，多线程并发也不丢更新
atomicMax(&hi, val);           // 原子取最大
int slot = atomicAdd(&next, 1);// 常见技巧：原子地"领一个号"，返回旧值做下标
```

原子操作不加锁，但**不是免费的**。当很多线程同时对**同一个地址**做原子加，硬件必须把这些更新排队一个个应用——这叫**争用（contention）**。争用的线程越多，越接近串行。所以用原子操作的正确姿势不是"让每个线程都去 atomic 一下全局变量"，而是**先在便宜的层级把冲突压小**：让每个 block 用 shared memory 算出自己的局部和，最后**每个 block 只做一次** `atomicAdd` 到全局——把几十万次原子操作压缩成几百次。这正是本章动手环节要写的 kernel。

<div class="compare">
<div class="known">
<div class="compare-head">CPU：几个线程，靠锁 / std::atomic</div>

```cpp-norun
std::mutex m;
long total = 0;
// 每个线程抢锁进临界区（线程少，冲突可控）
{ std::lock_guard<std::mutex> lk(m);
  total += local_sum; }

std::atomic<long> a{0};

```

</div>
<div class="cpp-side">
<div class="compare-head">GPU：上万线程，靠 barrier + atomic</div>

```cuda-norun
// 绝不给几万个线程加锁——会彻底串行化
__shared__ float partial[256];
partial[tid] = val;
__syncthreads();          // 先 block 内归约（不锁）
// ... 树形归约到 partial[0] ...
if (tid == 0)
    atomicAdd(total, partial[0]);  // 每 block 只冲突一次
```

</div>
</div>

区别的根源还是第 1 章那句话：CPU 上线程少，锁的冲突可控；GPU 上线程数以万计，任何"让线程排队"的机制都是性能毒药。所以 GPU 把协作拆成**层级**——warp 内用 shuffle，block 内用 barrier + shared，实在要跨 block 才用原子，且尽量让原子操作的次数降到最低。

## 动手：用 5 步把 32 个数归约成 1 个

下面这个模拟器**能在浏览器里直接跑**——点 **▶ 运行**。它忠实地模拟图 8-2 的 warp shuffle 归约：每一步用 `offset` 让 `lane[i]` 加上 `lane[i+offset]` 的值，`offset` 从 16 一路减半到 1，打印每一步之后 lane 0 累加了多少。5 步之后，lane 0 就持有全部 32 个值的和。

```python
# 模拟 warp shuffle 归约：32 个值在 log2(32)=5 步内求和，结果落在 lane 0
WARP = 32
vals = list(range(1, WARP + 1))     # [1, 2, 3, ..., 32]
expected = sum(vals)
print(f"初始：{WARP} 个值 [1, 2, ..., 32]，直接求和应为 {expected}\n")

lane = vals[:]                      # 每个 lane 当前持有的值
offset = WARP // 2                  # 第一步 offset = 16
step = 1
while offset >= 1:
    # __shfl_down_sync：lane[i] 把 lane[i+offset] 的值加过来（同一时刻并行完成）
    new = lane[:]
    for i in range(WARP):
        if i + offset < WARP:
            new[i] = lane[i] + lane[i + offset]
    lane = new
    active = WARP // (2 ** step)     # 本步过后仍"有意义"的活跃 lane 数
    merged = 2 ** step               # lane 0 此刻已累加了多少个原始值
    print(f"第 {step} 步 offset={offset:2d}：{active:2d} 个 lane 活跃，"
          f"lane 0 已累加 {merged:2d} 个原始值，lane[0] = {lane[0]}")
    offset //= 2
    step += 1

print(f"\n归约完成：lane[0] = {lane[0]}，与直接求和 {expected} 一致：{lane[0] == expected}")
```

运行结果：

```
初始：32 个值 [1, 2, ..., 32]，直接求和应为 528

第 1 步 offset=16：16 个 lane 活跃，lane 0 已累加  2 个原始值，lane[0] = 18
第 2 步 offset= 8： 8 个 lane 活跃，lane 0 已累加  4 个原始值，lane[0] = 52
第 3 步 offset= 4： 4 个 lane 活跃，lane 0 已累加  8 个原始值，lane[0] = 120
第 4 步 offset= 2： 2 个 lane 活跃，lane 0 已累加 16 个原始值，lane[0] = 256
第 5 步 offset= 1： 1 个 lane 活跃，lane 0 已累加 32 个原始值，lane[0] = 528

归约完成：lane[0] = 528，与直接求和 528 一致：True
```

看那一列 lane 数：32 个值只用 **5 步**（而不是 31 次逐个相加）就归约完了——这就是"树形归约"的威力，也是为什么 GPU 求和、求最大值这类操作都写成这个形状。改一行试试：把 `WARP = 32` 改成 `WARP = 8`，你会看到只需 `log₂(8)=3` 步；这也解释了为什么 warp 大小取 2 的幂——归约树才能干净地对折。

## 把它们组合起来：一个真的求和 kernel

现在把三种机制拼成一个完整、可编译的程序：对一个大数组求和。策略正是上面反复强调的**分层压缩冲突**——每个 block 先用 shared memory + `__syncthreads()` 做树形归约算出自己的局部和，然后**每个 block 只做一次** `atomicAdd` 累加到全局计数器。几十万个线程，最后只产生几百次原子操作。

```cuda
#include <cstdio>
#include <cstdlib>
#include <cuda_runtime.h>

// 对数组求和：block 内 shared memory 树形归约 + 每 block 一次 atomicAdd
__global__ void sum_kernel(const float* in, float* out, int n) {
    extern __shared__ float sdata[];              // 动态 shared memory，大小在启动时给
    int tid = threadIdx.x;
    int i   = blockIdx.x * blockDim.x + threadIdx.x;

    sdata[tid] = (i < n) ? in[i] : 0.0f;          // 各线程读一个元素，越界补 0
    __syncthreads();                              // 等全 block 都写进 shared memory

    // block 内树形归约：步长每轮减半（对折），和 warp shuffle 是同一个形状
    for (int stride = blockDim.x / 2; stride > 0; stride >>= 1) {
        if (tid < stride) sdata[tid] += sdata[tid + stride];
        __syncthreads();                          // 每一轮之后都要再同步
    }

    // 每个 block 只由 0 号线程做一次原子加——把冲突压到最小
    if (tid == 0) atomicAdd(out, sdata[0]);
}

int main() {
    const int N       = 1 << 20;                  // 1,048,576 个元素
    const int threads = 256;                      // block 大小，取 32 的倍数、2 的幂
    const int blocks  = (N + threads - 1) / threads;  // 向上取整凑够线程

    float* h_in = (float*)malloc(N * sizeof(float));
    for (int i = 0; i < N; ++i) h_in[i] = 1.0f;   // 全填 1，总和应等于 N

    float *d_in, *d_out;
    cudaMalloc(&d_in,  N * sizeof(float));
    cudaMalloc(&d_out, sizeof(float));
    cudaMemcpy(d_in, h_in, N * sizeof(float), cudaMemcpyHostToDevice);
    cudaMemset(d_out, 0, sizeof(float));          // 全局计数器清零

    // 第三个 <<<>>> 参数：每个 block 的动态 shared memory 字节数
    sum_kernel<<<blocks, threads, threads * sizeof(float)>>>(d_in, d_out, N);

    float h_out = 0.0f;
    cudaMemcpy(&h_out, d_out, sizeof(float), cudaMemcpyDeviceToHost);
    printf("元素个数 N = %d\n", N);
    printf("GPU 求和结果 = %.1f（期望 %d）\n", h_out, N);

    cudaFree(d_in);
    cudaFree(d_out);
    free(h_in);
    return 0;
}
```

点开代码块的 **🔗 在 Compiler Explorer 打开**，可以看到 `nvcc` 把这个 kernel 编成的 PTX——留意 `atomicAdd` 对应的原子指令，以及 `__syncthreads()` 编成的 `bar.sync`。本地有 NVIDIA 卡的话，`nvcc sum.cu -o sum && ./sum` 会打印总和 `1048576.0`。这个 kernel 已经用上了本章全部三种机制里的两种（barrier + atomic），还留了一个练习：把 block 内归约的最后 6 步（`stride` 从 32 到 1）换成 warp shuffle，就能少掉几次 `__syncthreads()`——那是第 18 章优化归约时要做的事。

<div class="note">
<strong>浮点求和的顺序问题</strong>
上面全填 <code>1.0f</code>，结果精确。但如果数组是任意浮点数，<strong>归约的相加顺序会影响结果的最后几位</strong>——浮点加法不满足结合律。GPU 上线程调度不定，同一份数据两次运行甚至可能得到略微不同的和。这不是 bug，是浮点的固有性质；需要可复现结果时要用确定性归约或更高精度累加器。CPU 多线程归约其实有同样的问题，只是你可能没注意过。
</div>

---

**这一章的要点**

- GPU 上**几乎不用锁**：锁让上万线程排队，等于放弃并行。GPU 靠 barrier 协调节奏、靠 shuffle 交换数据、靠原子操作压缩冲突。
- **block 级 barrier**（`__syncthreads()` / `item.barrier()`）让一个 block 的线程**等齐再走**，常用来隔开 shared memory 的"写阶段"和"读阶段"。它由**最慢的线程**决定放行时刻。
- barrier 的黄金法则：**要么全体执行，要么全体不执行**。放进依赖 `threadIdx` 的分支会让 block **死锁**——分歧 + barrier = 死锁。
- **warp 级 shuffle**（`__shfl_down_sync` / `__shfl_xor_sync`）让一个 warp 的 32 个线程**直接交换寄存器**，不碰内存、无需 barrier，`log₂(32)=5` 步完成 warp 内归约。
- **原子操作**（`atomicAdd` 等）解决"多线程更新同一地址"，不加锁但有**争用**成本。正确姿势是**分层压缩冲突**：warp 内 shuffle → block 内 shared+barrier → block 间才用 atomic，且次数降到最低。
