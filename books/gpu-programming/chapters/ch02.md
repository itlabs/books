## 你可能已经见过它的哥哥：SIMD

如果你在 CPU 上做过性能优化，也许写过或读过 SIMD 代码。**SIMD（Single Instruction, Multiple Data，单指令多数据）**的意思是：一条 CPU 指令同时对**一排数据**做同样的运算。比如 AVX 指令一次能加 8 个 `float`：

```cpp
// CPU 的 SIMD：一条指令加 8 个 float（AVX）
#include <immintrin.h>
#include <cstdio>

int main() {
    float a[8] = {1, 2, 3, 4, 5, 6, 7, 8};
    float b[8] = {10, 20, 30, 40, 50, 60, 70, 80};
    float c[8];

    __m256 va = _mm256_loadu_ps(a);     // 把 8 个 float 装进一个 256 位寄存器
    __m256 vb = _mm256_loadu_ps(b);
    __m256 vc = _mm256_add_ps(va, vb);  // 一条指令，8 个加法同时做
    _mm256_storeu_ps(c, vc);

    for (int i = 0; i < 8; ++i) printf("%.0f ", c[i]);  // 11 22 33 ... 88
    printf("\n");
}
```

SIMD 很强，但写起来别扭：你得把数据手动打包进 `__m256` 这样的**向量寄存器**，用一套专门的 intrinsic 函数（`_mm256_add_ps`…），处理不够 8 个的"尾巴"，换个数据宽度还得重写。你写的是**向量**，不是标量——心智负担全在你身上。

<div class="note">
<strong>一句话记住 SIMD</strong>
一条指令，一个"宽"寄存器里的多个数据，被同一个运算处理。宽度是硬件定死的（AVX 是 8 个 float，AVX-512 是 16 个）。<strong>是你（或编译器的自动向量化）负责把标量代码改写成向量形式。</strong>
</div>

## GPU 的做法：SIMT——你写标量，硬件替你并起来

GPU 换了个思路。它让你**照常写标量代码**——就像上一章那个 `c[i] = a[i] + b[i]`，一个线程只管一个元素，读起来完全是普通的单线程代码。然后硬件在背后，把**一组线程**（NVIDIA 叫一个 **warp**，32 个线程；后面章节细讲）绑在一起，让它们**同一时刻执行同一条指令**，只是各自作用在自己的数据上。

这就是 **SIMT（Single Instruction, Multiple Threads，单指令多线程）**。

结果是：**SIMT 写起来像标量，跑起来像向量。** 你不用管向量宽度、不用打包数据、不用处理尾巴，只描述"一个线程做什么"，硬件负责把成百上千个这样的线程编成一个个 warp 齐步跑。这就是为什么 GPU 代码看着比 SIMD 干净得多——复杂度被移进了硬件和编译器。

## SIMT 的代价：warp 分歧

天下没有免费的抽象。SIMT 让你写标量的爽快，是建立在一个硬性约束上的：**一个 warp 里的 32 个线程，任一时刻只能执行同一条指令**。

那问题来了——如果代码里有 `if`，warp 里一半线程条件为真、一半为假，怎么办？它们要执行的可是**不同的指令**啊。

硬件的处理方式简单粗暴：**两条分支都执行，但每条分支只让"该走这条路"的线程生效，其余线程暂时"关灯"（掩码屏蔽）。** 也就是说，`if` 和 `else` 不再是二选一并行跑完，而是**先跑 if（else 那批线程干等），再跑 else（if 那批线程干等）**——串行化了！这就是 **warp 分歧（warp divergence）**。

```cuda-norun
// 一个会造成 warp 分歧的 kernel
__global__ void diverge(float* x) {
    int i = threadIdx.x;
    if (i % 2 == 0) {        // 偶数线程走这里
        x[i] = expensive_a(x[i]);   // 奇数线程被"关灯"，干等
    } else {                 // 奇数线程走这里
        x[i] = expensive_b(x[i]);   // 换偶数线程"关灯"，干等
    }
    // 结果：这个 warp 的耗时 ≈ 分支A + 分支B，而不是 max(A, B)
}
```

在 CPU 上，分支预测让 `if` 几乎免费，你从不为写分支而心虚。在 GPU 上，**warp 内的分歧是真金白银的性能损失**：分支越多、越"参差不齐"，串行化越严重。这是上一章说的"GPU 直觉和 CPU 相反"的头号例子。

<div class="warn">
<strong>关键限定：分歧只在 warp <em>内部</em> 才有代价</strong>
如果整个 warp 的 32 个线程都走同一条分支（比如按 warp 边界对齐地分工），那就<strong>没有分歧</strong>——大家还是齐步走，只是走的是同一条路。所以优化的方向不是"消灭所有 if"，而是"让走向不同的线程尽量落在<strong>不同的 warp</strong> 里"。这个"warp 是 32 个线程"的边界意识，第 6 章会专门讲。
</div>

## 动手：亲眼看分歧让代码变慢

下面这个模拟器**能在浏览器里直接跑**——点 **▶ 运行**。它不真用 GPU，而是用 Python 忠实地模拟 SIMT 的执行规则：一个 warp 一次只能执行一条指令；遇到分支时，先让走 `if` 的线程执行（其余"关灯"），再让走 `else` 的执行。我们数"这个 warp 一共花了几个执行步（cycle）"，对比"全员走同一条路"和"参差不齐地分歧"两种数据，你会看到时间差。

```python
# 模拟 SIMT：一个 warp 遇到 if/else 时如何串行化（warp 分歧）
WARP_SIZE = 8   # 真实 GPU 是 32，这里用 8 方便观察

def run_warp(values, if_cost=3, else_cost=3):
    """模拟一个 warp 执行 if(v>0){花 if_cost 步} else {花 else_cost 步}。
    返回这个 warp 实际消耗的执行步数。"""
    go_if   = [v for v in values if v > 0]     # 会走 if 分支的线程
    go_else = [v for v in values if v <= 0]    # 会走 else 分支的线程

    cost = 0
    if go_if:      # 只要 warp 里有任何线程走 if，整个 warp 就得为 if 花时间
        cost += if_cost
        print(f"  执行 if 分支：{len(go_if)} 个线程干活，{len(go_else)} 个被关灯干等")
    if go_else:    # 只要有任何线程走 else，整个 warp 再为 else 花一次时间
        cost += else_cost
        print(f"  执行 else 分支：{len(go_else)} 个线程干活，{len(go_if)} 个被关灯干等")
    return cost

print("情况一：全员走同一条分支（无分歧）")
uniform = [1, 2, 3, 4, 5, 6, 7, 8]          # 全部 > 0，都走 if
c1 = run_warp(uniform)
print(f"  → 这个 warp 花了 {c1} 步\n")

print("情况二：一半走 if、一半走 else（严重分歧）")
mixed = [1, -1, 2, -2, 3, -3, 4, -4]        # 正负交错
c2 = run_warp(mixed)
print(f"  → 这个 warp 花了 {c2} 步\n")

print(f"结论：同样 8 个线程、同样的活，分歧让耗时从 {c1} 步涨到 {c2} 步，慢了 {c2/c1:.0f} 倍。")
print("原因：if 和 else 被迫串行执行，而不是并行二选一。")
```

运行结果：

```
情况一：全员走同一条分支（无分歧）
  执行 if 分支：8 个线程干活，0 个被关灯干等
  → 这个 warp 花了 3 步

情况二：一半走 if、一半走 else（严重分歧）
  执行 if 分支：4 个线程干活，4 个被关灯干等
  执行 else 分支：4 个线程干活，4 个被关灯干等
  → 这个 warp 花了 6 步

结论：同样 8 个线程、同样的活，分歧让耗时从 3 步涨到 6 步，慢了 2 倍。
原因：if 和 else 被迫串行执行，而不是并行二选一。
```

看到了吗？同样的 8 个线程、同样的工作量，只因为它们**走向不同**，耗时就翻倍。请动手改一下：把 `mixed` 改成 `[1, 2, 3, 4, 5, 6, 7, -1]`（只有一个线程走 else），再运行——你会发现耗时**还是** 6 步！这揭示了分歧最反直觉的一点：**哪怕 warp 里只有一个线程"不合群"，整个 warp 都要为那条分支付出完整代价**。这也是为什么后面讲优化时，我们会那么在意"让一个 warp 里的线程尽量走同一条路"。

## SIMT ≠ SIMD：三个关键区别

它们思路相通（都是一条指令驱动多份数据），但别混为一谈：

| | SIMD（CPU 向量） | SIMT（GPU） |
|---|---|---|
| 你写的代码 | **向量**：手动打包、用 intrinsic | **标量**：像单线程，硬件替你并起来 |
| 分支怎么办 | 你得用掩码手动处理 | 硬件自动掩码，但会**分歧串行化** |
| 宽度 | 写死在指令里（8、16…） | 逻辑上任意多线程，硬件按 warp（32）打包调度 |
| 访存 | 通常要求连续 | 灵活，但**合并访问**才快（第 7 章） |

一句话：**SIMD 把并行的复杂度交给你，SIMT 把它藏进硬件——代价是你要理解 warp 这个"隐形的 32 人小队"，否则会在分歧和访存上莫名其妙地慢。** 理解这个隐形小队，正是第 6 章的任务。

## 小结与下一步

你现在懂了 GPU"线程齐步走"的机制，也懂了它的头号代价。但线程不是凭空齐步走的——它们跑在具体的硬件上：一颗 GPU 有几十个 **SM（流多处理器）**，每个 SM 有一堆计算核心、一块共享内存、还有那个负责调度 warp 的硬件。下一章我们就把 GPU 芯片"拆开"看一眼，建立硬件的心智模型——这样第 4 章开始讲 Grid/Block/Thread 时，你就知道这些软件概念到底落在哪块硅片上。

---

**这一章的要点**

- **SIMD**（CPU）：你手写向量代码，一条指令处理一"宽"寄存器里的多个数据，宽度写死。
- **SIMT**（GPU）：你照常写**标量**代码，硬件把一组线程（一个 **warp = 32 线程**）绑起来齐步执行同一条指令——写着像标量，跑着像向量。
- SIMT 的代价是 **warp 分歧**：warp 内线程走进不同 `if` 分支时，各分支被**串行执行**（该走的干活，其余关灯干等），哪怕只有一个线程不合群也要付全价。
- 分歧只在 **warp 内部**才有代价；优化方向是让走向不同的线程落在**不同的 warp**，而不是消灭所有分支。
- SIMT 把并行复杂度藏进硬件，代价是你必须理解 warp 这个"隐形的 32 人小队"（第 6 章）。
