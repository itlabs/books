## 从一个你天天写、却从没命名的东西说起

给你一个 `4×4` 的矩阵，行主序存在一维内存里。坐标 `(i, j)` 的元素在内存的第几个位置？你闭着眼都能写：

```cpp-norun
// 行主序：一行 4 个元素连续摆放
int offset = i * 4 + j;   // 第 i 行跳过 i*4 个元素，再往后数 j 个
```

那个 `4` 是什么？是**跨一行要跳过多少个元素**。如果换成列主序（一列连续摆放），就变成 `offset = i + j * 4`——那个 `4` 变成了**跨一列要跳过多少个元素**。这两个"跳过多少"的数，就是 **步长（stride）**。而矩阵的形状 `(4, 4)` 就是 **形状（shape）**。

你已经在用它了，只是从没给它起名字。CuTe 做的第一件事，就是把这对 `(Shape, Stride)` 提升成一个头等对象——**Layout**：

> **Layout = (Shape, Stride)**，它是一个函数，把逻辑坐标 `(i, j, ...)` 映射到线性内存偏移：
> `offset = i·stride₀ + j·stride₁ + ...`

<div class="keypoint">
<strong>一句话抓住 Layout</strong>
<strong>Shape</strong> 说"逻辑上有多大"（<code>(4,4)</code>），<strong>Stride</strong> 说"每个维度走一步，物理内存跳多远"。给定坐标，偏移就是<strong>坐标和 stride 的点积</strong>：<code>offset = Σ coordᵢ · strideᵢ</code>。同一个逻辑形状，配不同的 stride，就得到行主序、列主序、转置、分块……<strong>所有内存布局，都是同一套代数里换一组 stride 而已。</strong></div>

## 为什么值得抽象成一个对象

你可能想：不就是 `i*N+j` 吗，散在代码里也没什么。问题恰恰在"散"。真实的高性能 kernel 里，同一份数据要被**同时**用好几种坐标系看待：全局张量的坐标、切给某个 block 的分块坐标、块内某个线程负责的坐标、喂进 Tensor Core 前的重排坐标……每一层都是一次坐标到偏移的映射。如果全用裸下标算术手写，代码会变成一团谁也不敢碰的魔法数。

<div class="compare">
<div class="known">
<div class="compare-head">裸下标算术：散落各处的魔法数</div>

```cpp-norun
// 每处访问都手写一遍，换个布局要全局改
a[i * N + j]                      // 全局
tile[(bi*BM+ti)*N + (bj*BN+tj)]   // 分块+块内
smem[ti * BN + tj]                // shared memory 里又一套
// 转置？swizzle？每处都要重推下标
```

</div>
<div class="cpp-side">
<div class="compare-head">Layout 对象：把映射封装起来，可组合</div>

```python-norun
# 每种视角是一个 Layout，坐标 -> 偏移
A     = Layout((M, N), (N, 1))     # 全局，行主序
tile  = Layout((BM, BN), (N, 1))   # 一个分块的视角
# 布局怎么变，都是换 stride / 组合 Layout
# 访问统一写成 A(coord)，不再手推下标
off = A((i, j))
```

</div>
</div>

右边的好处不只是"干净"。因为 Layout 是对象，它能**组合（compose）**：一个"把矩阵切成分块"的 Layout，可以和一个"块内怎么排"的 Layout 复合，得到"全局第几个元素"的映射；一个"线程到它负责的数据"的映射（CuTe 里叫 **TV，thread-value 划分**）也是一个 Layout。**分块、转置、swizzle、线程分工——在 CuTe 里全都是 Layout 的代数运算，同一套规则。** 这就是 CUTLASS 能用一套通用代码逼近硬件峰值的原因：它不为每种情况手写下标，而是用 Layout 代数推导出来。

## 动手：亲手实现一个 Layout 类

理解 Layout 最快的路，是自己写一个。下面这段程序**能在浏览器里直接跑**——点 **▶ 运行**。它只用标准库，实现一个 `Layout(shape, stride)`：核心就是那个 `__call__`，把坐标（或一维序号）映射成偏移。我们用它构造同一个 `4×4` 的行主序和列主序，打印偏移表，再演示**分块作为组合**——用一个"块间 Layout"加一个"块内 Layout"复现出分块遍历的全局偏移。

```python
# 亲手实现 CuTe 的核心抽象：Layout(shape, stride) 把坐标映射到内存偏移
class Layout:
    def __init__(self, shape, stride):
        assert len(shape) == len(stride)
        self.shape = shape
        self.stride = stride

    def size(self):                       # 逻辑元素总数 = 各维之积
        n = 1
        for s in self.shape:
            n *= s
        return n

    def coord_of(self, idx):              # 一维逻辑序号 -> 多维坐标（行主序解读）
        coord = []
        for s in reversed(self.shape):
            coord.append(idx % s)
            idx //= s
        return tuple(reversed(coord))

    def __call__(self, coord):            # 核心：坐标 -> 线性内存偏移
        if isinstance(coord, int):        # 允许传一维序号，先转成坐标
            coord = self.coord_of(coord)
        off = 0
        for c, st in zip(coord, self.stride):
            off += c * st                 # offset = Σ coordᵢ · strideᵢ
        return off

def print_offsets(name, layout):
    print(f"{name}  shape={layout.shape} stride={layout.stride}")
    rows, cols = layout.shape
    for i in range(rows):
        line = "  ".join(f"{layout((i, j)):2d}" for j in range(cols))
        print("   " + line)
    print()

# 同一个逻辑 4×4，两种 stride
row_major = Layout((4, 4), (4, 1))    # 行主序：跨行跳 4，跨列跳 1
col_major = Layout((4, 4), (1, 4))    # 列主序：跨行跳 1，跨列跳 4

print("同一个逻辑 4×4，坐标 (i,j) 落到的线性偏移：\n")
print_offsets("行主序 row-major", row_major)
print_offsets("列主序 col-major", col_major)
print("坐标 (1,2)：", "行主序 ->", row_major((1, 2)), " 列主序 ->", col_major((1, 2)))

# 分块作为组合：把 4×4 看成 2×2 个 2×2 的块
# 块间 Layout：2×2 个块，块与块之间在行主序内存里的基址步长是 (8, 2)
# 块内 Layout：每块 2×2，在同一份行主序内存里步长是 (4, 1)
print("\n把 4×4 按 2×2 分块，用『块间 + 块内』两个 Layout 组合出全局偏移：")
tile_between = Layout((2, 2), (8, 2))   # 块(Ti,Tj) 的基址
tile_inner   = Layout((2, 2), (4, 1))   # 块内(i,j) 相对基址的偏移
for Ti in range(2):
    for Tj in range(2):
        base = tile_between((Ti, Tj))
        offs = [base + tile_inner((i, j)) for i in range(2) for j in range(2)]
        print(f"  块({Ti},{Tj}) 基址 {base:2d} -> 块内 4 个元素全局偏移 {offs}")
```

运行结果：

```

    4   5   6   7
    8   9  10  11

  块(0,1) 基址  2 -> 块内 4 个元素全局偏移 [2, 3, 6, 7]
```

停下来盯着最后四行看。四个块加起来正好覆盖了 `0..15` 全部 16 个偏移，一个不重不漏——但顺序被重排了：块 `(0,0)` 拿到的是内存里的 `0,1,4,5`（原矩阵左上角那个 `2×2`）。我们**没有搬动任何数据**，只是用"块间 Layout"定基址、"块内 Layout"定块内偏移，两者**组合**出了分块视角。这就是 CuTe 里 Layout 组合的精髓：`tiling`、`partition` 都是这种复合，同一套 `(shape, stride)` 代数。请动手改：把 `col_major` 的 stride 改成 `(1, 4)` 以外的值（比如 `(2, 8)` 表示每个逻辑元素间隔 2 个物理槽位），看偏移表怎么变——你会更直观地感到"stride 决定一切"。

<div class="note">
<strong>这和上一章的联系</strong>
还记得上一章 Triton 二维 kernel 里的 <code>row[:, None] * stride_m + col[None, :] * stride_n</code> 吗？那正是一次 <code>Layout((BLOCK_M, BLOCK_N), (stride_m, stride_n))</code> 的求值。区别在于：Triton 里 stride 是你临时手写的裸算术，块内布局由编译器决定；而 CuTe 把它变成头等对象，让你能<strong>组合</strong>它、<strong>推演</strong>它，从而精确掌控从全局内存到寄存器、再到 Tensor Core 的每一层数据布局。</div>

## cute.Tensor：指针 + Layout

有了 Layout，CuTe 的**张量（Tensor）**就顺理成章：

> **cute.Tensor = 一个数据指针（engine）+ 一个 Layout。**

Layout 负责"坐标 → 偏移"，指针负责"偏移 → 真实内存地址"，两者一合，`tensor(i, j)` 就能取到元素。这个拆分很关键：**逻辑形状/布局和物理存储解耦**了。同一块显存，套不同的 Layout，就是不同的张量视图（转置、切片、分块），全程零数据搬动——只是换了个"怎么解读这块内存"的规则。这正是 FlashAttention 那类 kernel 反复对同一块 shared memory 换视角、喂给 Tensor Core 的底层机制。

## 真实的 CuTe DSL 长什么样

上面的 Python 类是为了讲清原理。真实世界里，你用 NVIDIA 官方的 **CuTe DSL**（`pip install nvidia-cutlass-dsl`，模块名 `cutlass.cute`）在 Python 里直接写这套代数，它会编译成在 GPU 上跑的代码。API 的味道和你刚实现的那个类几乎一一对应——`make_layout` 造 Layout、`make_tensor` 把指针和 Layout 绑成张量、`print_layout` 打印出来看：

```cutedsl
# 需要 GPU 与 nvidia-cutlass-dsl 安装：pip install nvidia-cutlass-dsl
import cutlass
import cutlass.cute as cute

@cute.jit
def show_layouts():
    # make_layout(shape, stride=...)：和你手写的 Layout(shape, stride) 一个意思
    # 行主序 4×4：跨行跳 4，跨列跳 1
    row_major = cute.make_layout((4, 4), stride=(4, 1))
    # 列主序 4×4：跨行跳 1，跨列跳 4
    col_major = cute.make_layout((4, 4), stride=(1, 4))

    cute.printf("row-major (i,j)->offset:")
    cute.print_layout(row_major)          # 打印形状、步长与偏移表
    cute.printf("col-major (i,j)->offset:")
    cute.print_layout(col_major)

    # 坐标求值：等价于我们的 layout((1, 2))
    cute.printf("row_major(1,2) = {}", row_major((1, 2)))   # -> 6
    cute.printf("col_major(1,2) = {}", col_major((1, 2)))   # -> 9

    # 分块：用 tiler 把大 Layout 切成块视角——对应我们手写的『块间+块内』组合
    tiled = cute.tiled_divide(row_major, (2, 2))
    cute.print_layout(tiled)

# cute.Tensor = 指针(engine) + Layout。真实代码里通常这样绑：
#   ptr = cute.make_ptr(cutlass.Float32, gmem_address)
#   t   = cute.make_tensor(ptr, cute.make_layout((M, N), stride=(N, 1)))
#   x   = t[(i, j)]   # Layout 把 (i,j) 变偏移，engine 再变真实地址

show_layouts()
```

API 细节（函数名、切分算子）随 CUTLASS 版本可能微调，以官方文档为准；但**心智模型是稳定的**：`Layout = (Shape, Stride)` 是坐标到偏移的函数，`Tensor = 指针 + Layout`，一切分块/转置/线程分工都是 Layout 的代数。你手写的那个几十行的 Python 类，已经抓住了它的灵魂。

<div class="keypoint">
<strong>给 CuTe 定位：它在整个技术栈的哪一层</strong>
<strong>Triton</strong>（第 13、14 章）：你操作块和张量片，块内布局由编译器决定——高层、好写、放弃精细控制。<strong>CuTe DSL</strong>（本章）：你用 Layout 代数<strong>亲手</strong>描述每一层数据布局，能精确驾驭 shared memory 排布、避免 bank conflict、把数据以 Tensor Core 要的形状喂进去、用 TMA 异步搬运（第 19 章）——低层、更难、换来极致性能。CUTLASS、FlashAttention 这些逼近硬件峰值的库，正是建在 CuTe 之上。选 Triton 还是 CuTe，就是在<strong>开发效率</strong>和<strong>控制粒度</strong>之间做权衡。</div>

<div class="warn">
<strong>别一上来就用 CuTe</strong>
CuTe 的威力全在"精确控制"，而精确控制只有当你<strong>确知瓶颈在数据布局</strong>时才值得。大多数融合、逐元素、乃至不少矩阵乘，Triton 已经够快且快得多写出来。CuTe 是给你已经用第 16 章的性能模型定位到"卡在 shared memory 布局 / Tensor Core 喂数"这类问题、且愿意为最后百分之几的峰值付出复杂度时准备的。<strong>先会读它、理解 Layout 代数，比急着手写一个 CuTe kernel 更重要。</strong></div>

## 小结与下一步

你现在握住了 CuTe 的钥匙——**Layout**。它把你天天在写、却从没命名的下标算术 `i*stride₀ + j*stride₁`，提升成一个能组合、能推演的正式对象 `(Shape, Stride)`。同一个逻辑形状换一组 stride 就是行主序、列主序、转置；两个 Layout 组合就是分块、线程分工。`Tensor = 指针 + Layout`，让逻辑布局和物理存储彻底解耦。这套代数，就是 CUTLASS 和 FlashAttention 逼近 Tensor Core 峰值的地基，也是 CuTe DSL 让你在 Python 里能触碰的东西。

我们已经把四大技术（CUDA、SYCL、Triton、CuTe DSL）都过了一遍。它们不是四个互相竞争的"谁最好"，而是站在**抽象层级的不同位置**，各自解决不同的问题。把它们并成一张速查表，你以后选型时先看这张：

| 技术 | 抽象层级 | 主要解决什么 | 什么时候选它 |
|---|---|---|---|
| **CUDA** | 线程级，贴硬件 | **控制力**：对线程、内存、同步的完全掌控 + 最成熟的生态（cuBLAS/cuDNN/…） | 要极致性能、要用现成高性能库、只跑 NVIDIA |
| **SYCL** | 线程级，标准 C++ | **可移植**：一份源码跨 NVIDIA/AMD/Intel/CPU | 不想被单一厂商锁定、要跨硬件 |
| **Triton** | **块级**，Python | **易用性 × 性能**：用接近 Python 的成本写出高效 kernel，尤其擅长**算子融合**省掉访存受限算子的冗余显存往返 | 写 ML 算子、要融合、想快速迭代（这是现代深度学习框架底层越来越多用它的原因） |
| **CuTe DSL** | 数据布局级 | **极致控制**：用 Layout 代数精确驾驭 shared memory 排布、Tensor Core 喂数、TMA | 已定位到瓶颈在数据布局、要榨最后百分之几峰值（CUTLASS/FlashAttention 的地基） |

<div class="keypoint">
<strong>一句话记住四者的分工</strong>
它们大致沿"<strong>控制力 ←→ 易用性</strong>"这根轴排开：<strong>CuTe</strong> 和 <strong>CUDA</strong> 在"控制"这头（贴硬件、榨峰值，代价是复杂），<strong>Triton</strong> 在"易用"这头（块级、Python、融合友好，代价是放弃精细控制），<strong>SYCL</strong> 则多了一条正交的"<strong>跨厂商可移植</strong>"轴。没有谁取代谁——真实项目里常常混用：大部分算子用 Triton 快速写，少数性能关键路径下沉到 CuTe/CUDA，需要跨硬件时用 SYCL。
</div>

但工具再多，也得知道**往哪儿用力**。下一章进入本书的性能部分：怎么定量判断一个 kernel 到底卡在哪、还有多少余量——**occupancy、带宽与 Roofline 模型**。第 3 章那句"多数 kernel 卡在带宽上"，到那里会变成一张能读、能算、能指导优化的图。

---

**这一章的要点**

- **Layout = (Shape, Stride)**，是把逻辑坐标映射到内存偏移的函数：`offset = Σ coordᵢ · strideᵢ`。**Shape** 说逻辑多大，**Stride** 说每维走一步跳多远。
- 同一个逻辑形状，换一组 **stride** 就得到行主序 `(N,1)`、列主序 `(1,N)`、转置、分块……**所有内存布局都是同一套代数换 stride**。
- Layout 的威力在于**可组合**：分块（tiling）、转置、swizzle、线程到数据的 **TV 划分**，全是 Layout 的代数运算——这让 CUTLASS 能用一套通用代码逼近硬件峰值，而不必手写每种情况的下标。
- **cute.Tensor = 指针（engine）+ Layout**，让逻辑布局与物理存储解耦：同一块内存套不同 Layout 就是不同视图，零数据搬动。
- CuTe 比 Triton **低一层**：Triton 让编译器决定块内布局，CuTe 让你用 Layout 代数亲手控制到 shared memory、Tensor Core、TMA（第 19 章）——换来极致性能，代价是复杂度。
- 真实 CuTe DSL（`cutlass.cute`）用 `make_layout`、`make_tensor`、`print_layout` 把这套代数搬进 Python；API 细节随版本变，但 `(Shape, Stride) → offset` 的心智模型稳定。
